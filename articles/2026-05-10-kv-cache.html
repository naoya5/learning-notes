2026-05-10 · llm transformer inference kv-cache optimization

KVキャッシュの全体像：仕組み・ボトルネック・主要な最適化技術

LLM推論を支えるKVキャッシュの仕組みと、PagedAttention・量子化・エビクション・GQA/MLAなど主要な最適化技術を、図解多めで中級〜上級向けに整理する。

目次

    TL;DR

    • KVキャッシュは「過去トークンの Key/Value を保存して再計算を避ける」仕組みで、デコード時の計算量を O(n²) から O(n) に落とす代償としてメモリと帯域を消費する。
    • デコードは演算強度（arithmetic intensity）が低く、attention はメモリ帯域ボトルネックになる。GPU の FLOPs ではなく HBM 帯域がスループットの天井を決める。
    • 最適化は4軸ある — メモリ管理（PagedAttention）、ビット数削減（KIVI / NVFP4 などの量子化）、トークン数削減（StreamingLLM / H2O / SnapKV のエビクション）、アーキテクチャ変更（MQA / GQA / MLA）。
    • vLLM の PagedAttention は KVキャッシュ浪費を 60〜80% から 4% 未満に削減し、HF Transformers 比で最大 24× のスループットを実現した。
    • 長文・大バッチで詰まったらまず GQA + PagedAttention + 8〜4bit 量子化、が 2026 年現在の定石。

    俯瞰：KVキャッシュ最適化の全体マップ

    細部に入る前に、この記事で扱う 4 軸の最適化技術を 1 枚で見渡しておく。あとで個別を読むときに「自分はいま地図のどこにいるか」を意識しやすくなる。

    なぜ KV キャッシュが必要か

    自己回帰的な Transformer デコードは、新しいトークンを 1 個生成するたびに「これまで出力した全トークンに対する self-attention」を計算する必要がある。素朴に書けば t 番目のトークンを生成するために t 個の Key と Value を毎回作り直すので、シーケンス全体では O(n²) のオーダー。1000 トークン生成するなら 1000 回 attention をスクラッチから組み直すことになり、現実的な速度では動かない。

    ここで重要な観察は 「過去トークンの Key/Value テンソルは入力が変わらない限り変化しない」 こと。生成した瞬間にその位置の K, V は確定し、以降のステップで再利用できる。これらをまとめて GPU の HBM に置いておくのが KVキャッシュで、これにより:

    • プロンプト処理（prefill）: n 個のトークンに対して 1 回だけまとめて attention を計算し、K, V をキャッシュに格納。compute bound でハードウェアを飽和させやすい。
    • 生成（decode）: 1 トークンずつ、q は新しく作るが K, V はキャッシュから読み出すだけ。計算量は O(n) に落ちる。

    KVキャッシュのメモリ消費

    KVキャッシュ 1 トークン分のサイズはレイヤごと・ヘッドごとに K と V の 2 本ずつなので、ざっくり次の式で見積もれる: 

    cache_bytes_per_token = 2 (K, V)
                        × num_layers
                        × num_kv_heads
                        × head_dim
                        × bytes_per_element

    たとえば Llama-3 70B 相当（80 層、8 KV ヘッド、head_dim=128、fp16）だと、1 トークンで 2 × 80 × 8 × 128 × 2 ≈ 320 KB。8K トークンの会話なら 2.5 GB、128K トークンの長文なら 40 GB ものキャッシュが要る。モデル本体より KV のほうが GPU メモリを食うシナリオが普通に発生する。

    さらに厄介なのは、リクエストごとに最大長分を事前確保する素朴な実装だと、外部断片化と内部断片化（割当はしたが使ってない部分）で 60〜80% が無駄になることが報告されている。これが PagedAttention 登場の動機になった。

    メモリ帯域ボトルネックという真の問題

    decode フェーズで意識すべきは、容量だけでなく帯域。1 トークン生成するたびに K, V 全体を HBM から読み出して内積を取る必要があり、出てくる FLOPs はその割に少ない。Roofline モデルでいうと演算強度（FLOPs/byte）が極めて低く、GPU の演算ユニットがほぼ遊ぶ。

    Note: H100 SXM の HBM3 帯域は約 3.35 TB/s。Llama-3 70B を 8K コンテキストで decode すると、1 トークンあたり数 GB を読み込むことになり、1 ステップ時間は実質「KV を舐め切る時間」で決まる。FLOPs を倍にしても decode は速くならない。

    この観点から言うと、KVキャッシュ最適化の効果は 「いかに帯域消費を減らすか」 でほぼ決まる。容量削減と帯域削減は連動するが、量子化のように容量を削っても演算負荷を上げるだけのものは、decode 限界帯域より演算余力が大きい状況でしか効かない。設計選択を見るときは「これはどっちを削っているのか」を意識するとよい。

    軸1: メモリ管理 — PagedAttention

    PagedAttention (SOSP 2023) は OS のページング機構を attention に持ち込んだ。リクエストごとに連続メモリを確保するのではなく、固定サイズの ブロック（例: 16 トークン分）を仮想↔物理にマップし、必要に応じて非連続な物理ブロックに散らばらせる。

    • 外部断片化（隙間が使えない）が消える
    • 内部断片化はブロック粒度に閉じる（最後のブロックだけ無駄）
    • プロンプトを共有する複数リクエスト（system prompt を持つチャットなど）でブロックを共有でき、メモリ使用量がさらに減る
    • 結果として GPU 当たりの並列リクエスト数が増え、スループットが伸びる

    vLLM の実測では KVキャッシュの浪費が 4% 未満、Hugging Face Transformers 比で最大 24× のスループット改善が報告された。実装は CUDA カーネルとブロックテーブルが噛み合った非自明なエンジニアリングだが、概念としては「カーネルが論理ブロック番号 → 物理アドレスを引いて attention する」だけ。 

    # vLLM での KVキャッシュ管理イメージ（疑似コード）
# 各シーケンスは「論理ブロック → 物理ブロック」の表だけ持つ
seq_a.block_table = [P3, P7, P12]   # 連続じゃなくてOK
seq_b.block_table = [P3, P7, P9]    # 先頭2ブロックを seq_a と共有

# attention カーネルはこのテーブルを引いて非連続HBMから K,V を読む
out = paged_attention(q, block_table=seq.block_table,
                      block_size=16, num_blocks_used=k)

    軸2: ビット数削減 — KVキャッシュ量子化

    KVキャッシュの中身を fp16 から int8 / int4 / NF4 / 1bit に圧縮するアプローチ。容量も帯域も同じだけ減るので、decode が帯域ボトルネックの状況に直接効く。難点はキャッシュは「読まれる側」なので量子化誤差が attention 出力に直接乗ること。

    KIVI（2bit, plug-and-play）

    KIVI の鍵となる観察は Key と Value で外れ値の分布が違う こと。Key にはチャネル方向に強い外れ値があり、Value にはトークン方向に分散がある。そこで:

    • Key: per-channel に量子化（チャネルごとに scale/zero）
    • Value: per-token に量子化

    これだけで 2bit でも品質をほぼ保持し、ピークメモリ 2.6×、バッチサイズ 4× 増、スループット 2.35〜3.47× を報告。fine-tuning 不要で既存モデルにそのまま乗る点が普及を後押しした。

    Coupled Quantization（1bit）

    Coupled Quantization (NeurIPS 2024) は複数チャネルをまとめて量子化することで、チャネル間の相互依存を活用。1bit/channel まで落としても品質を保てると主張している。

    NVFP4 / FP8 ハードウェア量子化

    Hopper / Blackwell 世代の GPU は FP8、FP4 系のテンソル演算をハードウェアでサポートする。 NVFP4 KVキャッシュ は FP8 比で約 50% メモリ削減、長文・大バッチでのコンテキスト長を実用域に押し上げる。研究レベルの工夫より、まずこの「ハードウェアが速い形式に揃える」が現場の第一手。

    注意: 量子化は prefill よりも decode に効く。prefill は元から compute bound なので、量子化で帯域を減らしても演算側がボトルネックのまま。短プロンプト・長生成で効果が大きい。

    軸3: トークン数削減 — エビクション

    「全トークンを覚える必要があるのか？」という根本的な問い。長文ほどキャッシュは線形に増えるが、attention の重みは特定のトークンに偏ることが多い。重要トークンだけ残して他を捨てるのがエビクション系。

    StreamingLLM

    先頭のいくつか（attention sinks）+ 直近 N トークンのスライディングウィンドウだけ保持する。実装が単純で常に一定サイズのキャッシュで回せ、ほぼ無限長の生成を可能にする。重要トークンを捨てるリスクはあるが、対話のように直近重視のワークロードでは強い。

    H2O（Heavy Hitter Oracle）

    attention スコアの累積が大きいトークン（heavy hitters）が出力品質に支配的に効くという観察に基づき、累積スコアで上位 k 件と直近 m 件を残す。NeurIPS 2023 で提案された王道。

    SnapKV

    prefill 段階で長プロンプトを圧縮することに特化。プロンプト末尾の小さな観測ウィンドウから「今後どこに attention が向きそうか」を推定し、ヘッドごとに重要なクラスタを残す。長文プロンプト + 短応答で特に効く。

    注意: 2025 年に出た "The Pitfalls of KV Cache Compression" は、エビクション系がベンチマークでは強くても実運用では推論の連鎖を破壊する場面があると警鐘を鳴らしている。reasoning タスクで使う際は要評価。

    軸4: アーキテクチャ変更 — MQA / GQA / MLA

    学習時点で KVキャッシュが小さくなるようヘッド構造を変える系。後付けではなく事前学習段階の決定。

    • MHA (Multi-Head Attention): 各クエリヘッドに対応する KV ヘッドを別個に持つ素朴形。最大のキャッシュ。
    • MQA (Multi-Query Attention): 全クエリヘッドで KV を共有（1 セット）。キャッシュ最小だが品質劣化。
    • GQA (Grouped-Query Attention): クエリヘッドをグループ化し、グループごとに 1 セットの KV。MQA と MHA の中間で、Llama-2/3 など多数のモデルが採用。
    • MLA (Multi-head Latent Attention, DeepSeek): KV を低次元 latent に圧縮した上でキャッシュ。展開時に上射影。GQA より積極的にキャッシュを圧縮し、長文性能を保つ。

    運用上の意思決定マップ

    それぞれの軸はだいたい独立に併用できる。実運用での当て方をフローチャートにすると:

    • 大量の同時リクエスト × 共有プロンプト: PagedAttention（vLLM / SGLang）。これは入れない理由がない。
    • decode が帯域で詰まる: まず FP8/NVFP4 へ。さらに余力があれば KIVI 系の 4〜2bit。
    • 超長文プロンプト: SnapKV で prefill 段階を圧縮。効かないなら GQA/MLA のモデルへ切り替え。
    • 無限長対話・ストリーミング: StreamingLLM 形のスライディング。ただし長期記憶は別レイヤ（RAG / メモリモジュール）で持つ。
    • reasoning 系タスク: エビクションは慎重に。chain-of-thought を切り落とすと品質が落ちる。

    まとめ

    • KVキャッシュは Transformer 推論の前提インフラで、「容量」より「帯域」がボトルネックになる。
    • 最適化はメモリ管理 / ビット数 / トークン数 / アーキテクチャの 4 軸に整理して考えると見通しが良い。
    • 現場の標準装備は GQA + PagedAttention + 8/4bit 量子化。研究最前線は 1〜2bit 量子化、KVTC のような変換符号化、エビクションの reasoning ロバスト化。
