    1. CAP定理

    2000年に Eric Brewer が提唱。分散システムでは Consistency / Availability / Partition Tolerance の3つから 2つしか同時に満たせない。

    CAPの解釈

    要素意味
    C: Consistency全ノードで読み取った値が同じ（最新が見える）
    A: Availability一部ノードが落ちても応答が返る
    P: Partition Toleranceネットワーク分断（一部ノードが通信不可）でもシステム継続
    誤解されがち: 「3つから2つ選ぶ」と言われるが、現実の分散システムでは P（分断耐性）は必須。ネットワーク障害は不可避なので。実質「分断時にC を取るか A を取るか」の選択。

    PACELC — より実用的な拡張

    CAPは不完全。「正常時もLatency vs Consistencyのトレードオフがある」を加えたのが PACELC。

    • Partition時 → A or C
    • Else（正常時）→ Latency or Consistency
    分類例戦略
    PA/ELCassandra / DynamoDB常に低遅延・高可用性優先
    PA/ECMongoDB（デフォルト）分断時はA、正常時はC
    PC/EL—稀
    PC/ECHBase / Spanner常に強一貫性

    2. 一貫性モデル — 強→弱のスペクトル

    各モデルの特徴

    モデル説明例
    Strict全ノード即時一致（理論上のみ）単一DB
    Linearizable単一マシンに見える、強整合etcd, ZooKeeper, Spanner
    Sequential順序保証だが多少の遅延あり一部のRDB Replica
    Causal因果関係のみ保証多くの分散DB
    Eventual最終的に一致（中間で不一致あり）DNS, S3

    アプリ設計で覚える3つ

    • Strong consistency: 銀行残高、認証、決済
    • Read-your-writes: SNSの自分の投稿（自分には即見える）
    • Eventual consistency: いいね数、PV数、検索インデックス

    3. 主要分散パターン

    Saga — 分散トランザクション

    複数サービスにまたがる処理を 補償トランザクション で実現する。

    2種類のSaga

    方式特徴適用
    Choreography各サービスがイベントで連携、中央集権なしシンプルなフロー
    Orchestration中央の指揮者（Saga Manager）が制御複雑なフロー
     
    // Orchestration版 Saga（中央指揮）
class PlaceOrderSaga {
  constructor(
    private payment: PaymentService,
    private shipping: ShippingService,
    private inventory: InventoryService,
  ) {}

  async execute(order: Order) {
    const paymentId = await this.payment.charge(order);  // Step 1
    try {
      await this.inventory.reserve(order);                // Step 2
      try {
        await this.shipping.schedule(order);              // Step 3
      } catch (e) {
        await this.inventory.release(order);              // 補償2
        await this.payment.refund(paymentId);             // 補償1
        throw e;
      }
    } catch (e) {
      await this.payment.refund(paymentId);               // 補償1
      throw e;
    }
  }
}

    Outbox パターン

    「DB更新」と「イベント発行」を アトミック にする方法。

    -- DB更新とイベント記録を1トランザクションで
BEGIN;
UPDATE orders SET status='paid' WHERE id=1;
INSERT INTO outbox (event_type, payload, created_at)
  VALUES ('OrderPaid', '{"orderId":1}', NOW());
COMMIT;

-- 別プロセスが outbox をポーリングして Kafka へ
SELECT * FROM outbox WHERE published = false ORDER BY created_at LIMIT 100;
-- → 各レコードを Kafka に publish → published = true 更新

    Idempotency Key — 冪等性

    「同じリクエストを何回送っても結果が同じ」を保証する。

    // Express ミドルウェア
async function idempotencyMiddleware(req, res, next) {
  const key = req.headers["idempotency-key"];
  if (!key) return next();

  const cached = await cache.get(`idempotency:${key}`);
  if (cached) {
    return res.status(cached.status).json(cached.body);
  }

  const originalJson = res.json.bind(res);
  res.json = (body) => {
    cache.set(`idempotency:${key}`, { status: res.statusCode, body }, 24 * 3600);
    return originalJson(body);
  };
  next();
}

// クライアント側
fetch("/orders", {
  method: "POST",
  headers: { "Idempotency-Key": uuidv4() },
  body: JSON.stringify({ amount: 1000 }),
});

    Circuit Breaker — 障害遮断

    障害サービスへの呼び出しを 遮断 して連鎖障害を防ぐ。

    class CircuitBreaker {
  private state: "closed" | "open" | "half-open" = "closed";
  private failures = 0;
  private lastFailedAt: number = 0;

  constructor(
    private threshold = 5,
    private timeoutMs = 30000,
  ) {}

  async call<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === "open") {
      if (Date.now() - this.lastFailedAt > this.timeoutMs) {
        this.state = "half-open";
      } else {
        throw new Error("Circuit Breaker OPEN");
      }
    }

    try {
      const result = await fn();
      if (this.state === "half-open") {
        this.state = "closed";
        this.failures = 0;
      }
      return result;
    } catch (e) {
      this.failures++;
      this.lastFailedAt = Date.now();
      if (this.failures >= this.threshold) {
        this.state = "open";
      }
      throw e;
    }
  }
}

// 使う側
const breaker = new CircuitBreaker(5, 30000);
const data = await breaker.call(() => fetch("https://flaky-api.example.com"));

    Bulkhead — 区画化

    船の隔壁にちなむ命名。リソースを区画化して、1つの障害が全体に波及しないようにする。

    // 重要API用と一般API用でHTTP接続プールを分ける
const importantApiPool = new HttpAgent({ maxSockets: 50 });
const otherApiPool = new HttpAgent({ maxSockets: 20 });

// 一般APIが詰まっても、重要APIは無傷
fetch("/critical", { agent: importantApiPool });
fetch("/analytics", { agent: otherApiPool });

    Rate Limit / Throttling

    // Token Bucketアルゴリズム
class TokenBucket {
  private tokens: number;
  private lastRefill: number = Date.now();

  constructor(
    private capacity: number,
    private refillPerSec: number,
  ) {
    this.tokens = capacity;
  }

  tryAcquire(): boolean {
    this.refill();
    if (this.tokens >= 1) {
      this.tokens--;
      return true;
    }
    return false;
  }

  private refill() {
    const now = Date.now();
    const elapsed = (now - this.lastRefill) / 1000;
    this.tokens = Math.min(this.capacity, this.tokens + elapsed * this.refillPerSec);
    this.lastRefill = now;
  }
}

    4. Designing Data-Intensive Applications（DDIA）— 神本

    Martin Kleppmann の名著（2017）。「分散システムを学ぶ最良の本」と言われる。読破には30〜50時間。

    Part 1: 単一マシンのデータシステム

    • データモデル: リレーショナル / ドキュメント / グラフ
    • ストレージエンジン: B-tree（読み最適） vs LSM-tree（書き最適）
    • エンコーディング: JSON / XML / Protobuf / Avro / Thrift

    Part 2: 分散データ

    • Replication: Single-leader / Multi-leader / Leaderless
    • Partitioning: ハッシュ / レンジ / 動的リバランス
    • Transactions: ACID / 分離レベル
    • Distributed Consensus: Paxos / Raft

    Part 3: 派生データ

    • Batch処理: Hadoop / MapReduce
    • Stream処理: Kafka / Flink
    • Lambda Architecture: Batch + Stream の併用
    DDIAの読み方:
    • 一気に読まない（30〜50時間かかる）
    • 章単位で実装と紐づける
    • 「自分のシステムに当てはめると？」を毎章考える
    • 難解な章（Consensusなど）は2〜3周読み返す

    回答フレームワーク（7ステップ）

    各ステップで聞くべきこと

    Step確認事項
    1. 要件確認機能要件、非機能要件（QPS / レイテンシ / 可用性）
    2. 規模見積ユーザー数 × 1日のリクエスト数 × ピーク係数
    3. APIデザイン主要エンドポイントとパラメータ
    4. データモデルテーブル/ドキュメントスキーマ、関係
    5. アーキ図Web/API/DB/Cache/Worker/Queue の構成
    6. ボトルネックどこがネック？なぜ？
    7. スケール戦略水平分割、キャッシュ、CDN、レプリカ

    頻出の規模見積（数字暗記）

    • 1日 = 86,400秒 ≈ 10万秒
    • 1MB = 10^6 byte / 1GB = 10^9 / 1TB = 10^12
    • SSD読み 0.15ms / メモリ読み 0.1μs / ネットワークラウンドトリップ 0.5ms（同AZ）
    • 1QPS で1日 ≈ 10万リクエスト
    • HDD seek 10ms、SSD seek 0.1ms

    演習問題

    Q1. CAP の選択

    以下のシステム、CP系とAP系どちらを選ぶ？

    1. 銀行の口座残高DB
    2. SNSのいいね数カウンター
    3. ECサイトの商品在庫数
    4. Webサイトのアクセスログ
    解答を見る
    1. CP — 一貫性最重要、不一致でダブル引き出しは絶対NG
    2. AP — 多少カウントずれてもOK、表示できることが重要
    3. CP — 在庫切れ商品を「在庫あり」と見せたら大問題
    4. AP — ログは多少欠けても可、収集できることが重要

    Q2. Saga の補償順

    「課金 → 在庫引当 → 配送依頼」の順で実行し、配送依頼で失敗した場合の補償順は？

    解答を見る

    逆順で補償:

    1. 在庫引当を解除（reverse step 2）
    2. 課金を返金（reverse step 1）

    各ステップに対応する補償処理を必ず用意する（forward + compensate のペア）。

    Q3. 規模見積

    DAU 1000万、各ユーザーが1日10件の投稿を平均的に行うSNS。1秒あたりの書き込みQPSは？

    解答を見る

    1日の書き込み数 = 1000万 × 10 = 1億件
    1日 ≈ 10万秒
    平均QPS = 1億 / 10万 = 1000 QPS
    ピーク時はその3〜5倍 = 3000〜5000 QPS を想定
