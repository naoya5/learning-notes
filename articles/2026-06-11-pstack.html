pstack — Cursorを規律あるエンジニアリングチームに変えるプラグイン

React Core TeamのLauren Tan(poteto)が作ったCursor公式プラグインpstack。「速く行きたければ、まず深く行け」の哲学を、15のplaybookと20の原則という構造に落とし込んだスキル設計を読み解く。

目次

    TL;DR

    • pstackはcursor/pluginsで公開されているCursor公式プラグイン。作者はReact Compilerを開発するLauren Tan(poteto)で、「AIはslop（雑なコード）を書きすぎる」問題への回答として自身が日常で使うスキル群を公開したもの
    • 哲学は「if you want to go fast, go deep first」。LOC最大化の逆を行き、少なく・質の高いコードを書かせる。深掘りを信頼できるからこそ恐れずに並列化できる（fearless parallelism）
    • 中枢は/poteto-modeというルーティングスキル。タスクを15のplaybookにマッチさせ、手順をtodolistに逐語コピーしてから実行する
    • 20のエンジニアリング原則を「インラインの索引 + 1原則1スキルのleafファイル」という2層で持つ。索引はタスク開始時に必ず読み、適用時にleafを全文読む
    • モデルはロール別に使い分け（コード生成・判断・レビューパネル）。/setup-pstackで手持ちモデルに合わせて再構成でき、/automate-meで自分の作業履歴から「自分版mode」を生成できる

    pstackとは何か

    pstackはCursorのプラグインマーケットプレイスで配布されているスキル集で、/add-plugin pstack一発でインストールできる。作者のLauren Tan(poteto)はMeta・Netflix・Cursorで大規模コードベースに携わり、React Core TeamでReact Compilerを開発しているエンジニア。READMEの自己紹介がいきなり「i'm not a president or ceo」で始まるのが好い。

    リポジトリはMITライセンス。READMEには「fork it. improve it. make it yours.」とあり、PRも歓迎されている。

    出発点は「AIはコードを書きすぎる」という問題意識だ。READMEの言葉を借りれば、20人のslop職人チームのように出荷したいわけではない。スループットの最大化は目標ではなく、むしろその逆で、pstackは「少なく書くが、より質の高いコード」を書くための仕組みとして設計されている。

    面白いのは、品質への規律が並列化の前提になっている点。1つのエージェントに深い仕事を任せて「検証可能な良いコードを書く」と信頼できるなら、複数エージェントを安心して同時に走らせられる。これをREADMEはfearless parallelismと呼ぶ。品質は速度の敵ではなく、並列スケールの土台という整理だ。 

    # インストールはこれだけ
/add-plugin pstack

# 使い方も基本これだけ。タスクの先頭に付ける
/poteto-mode this pr has a subtle bug where the scroll drifts every 750ms
              even when idle. repro first, then fix and verify.

    全体アーキテクチャ

    プラグインの中身は驚くほどシンプルで、plugin.json（マニフェスト）、agents/poteto-agent.md（サブエージェント定義）、そしてskills/配下の34個のスキルだけ。スキルは役割で3種類に分かれる。

    • ルーティングスキル: poteto-mode。タスクの入口で、playbook選択と他スキルへの誘導を担う
    • ワークフロースキル: how / why / architect / arena / interrogate / tdd / unslop など13個。単体でも呼べる道具
    • 原則スキル: principle-*が20個。1ファイル1原則の短いリファレンス

    さらにagents/poteto-agent.mdというサブエージェント定義が同梱されていて、親エージェントからsubagent_type: "poteto-agent"で起動すると、作業前にpoteto-modeのSKILL.mdを原則インデックスごと全文読むことが保証される。汎用のgeneralPurposeで代用するとこの読み込みが飛んでスタイルがドリフトする、と定義ファイル自身が警告しているのが面白い。

    Note: pstackは意図的に全部入りではない。/deslopやcontrol-cli / control-uiは姉妹プラグインcursor-team-kit側にあり、併用が前提。また「planningスキルがないのはなぜ?」という問いにREADMEは「the best spec is code（最良の仕様書はコード）」と答えている。

    poteto-mode: ルーティングの中枢

    /poteto-modeを打つと、エージェントはまずtodolistを開き、最初の項目として原則インデックスを読む。これはSKILL.mdの冒頭で「Non-negotiables（交渉の余地なし）」と宣言されているルールで、さらに返信では「どの原則がどの判断を変えたか」を実際の選択に紐付けて引用することまで要求される。原則を読んだフリの引用（decisionの伴わないcitation）は禁止、という徹底ぶりだ。

    playbookの逐語コピー

    次にタスクを15のplaybookのどれかにマッチさせ、該当ファイルの手順をverbatim（逐語）でtodolistにコピーする。SKILL.mdはわざわざ失敗モードまで書いている。「playbookを読んだあと、名前付きステップを落とした自己流プランを書いてしまう」のが典型的な事故で、それを防ぐために逐語コピーを強制する。やらないステップも黙って消すのではなく、skip: 理由を一行添えてリストに残す。

    エージェントへの指示で「要約・言い換え」を許すと手順が静かに欠落していく問題への、構造による対策。スキル設計の参考になる。

    15のplaybook一覧
    playbook対象

    bug fix欠陥の再現・根本原因特定・ランタイム証拠つきの修正
    perf計測された遅さをトレースし、ベースライン比で改善

    このほか、全playbookの最後に呼ばれる「opening a pr」が別途ある。

    人間に聞く前に、実験で決める

    個人的に一番唸ったのがここ。「どっちのアプローチがいい?」と人間に質問しようとしたら、その前に質問を分類せよというルールがある。何かを動かせば観察できる事実（挙動・タイミング・レイアウト・性能）は人間に聞くものではない。prototype playbookで捨てるスケッチを作り、結果に決めさせる。人間への質問は、実験では決まらない純粋なプロダクト判断・好みの判断だけに留める。READMEいわく「質問は遅いパスであり、使い捨てのプローブの方がたいてい速く答えが出るうえ、人間には判断ではなく結果へのリアクションを渡せる」。

    20のエンジニアリング原則

    pstackの背骨は20個の原則スキル。各原則は「いつ適用するか」のトリガー条件つきでpoteto-mode内に索引化されていて、タスク開始時はこの索引だけを読む。実際にその原則が発火する場面になったら、leafスキル（principle-*）の全文を読む。索引で広く浅く、適用時に深くという2段階のprogressive disclosureになっている。

    カテゴリは5つ。core（9個）、architecture（5個）、verification（3個）、delegation（2個）、meta（1個）。代表としてcoreの筆頭「Laziness Protocol」を見ると、原則スキルの粒度がよく分かる。 

    # Laziness Protocol（抜粋・意訳）

コードを書くのは君にとって安価で、だからこそ過剰設計しやすい。
人間のメンテナの「疲労」を借りて、それに対抗せよ。

- 削除を優先せよ。リファクタを頼まれたら、追加より先に削除を探す
- 階層をフラットに保て。3ファイル/3層を超えるトレースが要るなら平らに
- 判断を1箇所に集約せよ。同じ選択を複数箇所で繰り返さない
- 差分を最小化せよ。「エレガントな」ボイラープレートより少ない行数
- シグナルの引き回しを疑え。型やスキーマを貫通する配線にはより直接の道がある

Prime directive: 人間の開発者が維持に疲弊するコードは悪い解。怠惰であれ。
    20原則の全リスト（カテゴリ別）
    • core: laziness-protocol（最小の変更と削除バイアス）/ foundational-thinking（ロジックの前に型とデータ構造）/ redesign-from-first-principles(新要件は初日からあったかのように再設計) / subtract-before-you-add（足す前に死荷重を除去）/ minimize-reader-load（読者の負荷を最小化）/ outcome-oriented-execution（互換の中間状態を温存せずターゲットへ収束）/ experience-first（実装都合よりユーザー体験）/ exhaust-the-design-space（前例なき判断は2〜3個の競合プロトタイプで比較）/ build-the-lever（手作業でなく、それを行う・証明するツールを作る）
    • architecture: boundary-discipline（境界でガード、内部は型を信頼）/ type-system-discipline（不正状態を表現不能に）/ make-operations-idempotent（クラッシュとリトライの中で同一終状態に収束）/ migrate-callers-then-delete-legacy-apis（呼び出し元を移行して旧APIを一括削除）/ separate-before-serializing-shared-state（共有を直列化する前に共有自体を排除）
    • verification: prove-it-works（プロキシでなく実物で検証）/ fix-root-causes（症状でなく根本原因）/ sequence-verifiable-units（検証可能な小単位に分割し、各単位をチェックで締める）
    • delegation: guard-the-context-window（バルクはサブエージェントへ、メインには要約だけ）/ never-block-on-the-human（可逆な作業は進めて結果を見せ、人間に軌道修正させる）
    • meta: encode-lessons-in-structure（同じ指示を2度書いたら、lint・チェック・スクリプトの形で構造に焼き込む）
    Tip: 原則スキルにはdisable-model-invocation: trueが付いている。モデルが勝手に呼ぶのではなく、poteto-modeの索引から明示的にナビゲートさせる設計。スキルの「自動発火させたくないが参照はさせたい」ケースで使えるテクニック。

    playbookの実例: bug fix

    playbookの実物がどれくらい具体的か、bug-fix.mdを見てみる。冒頭の宣言は「You own this task. Plan, review, verify.（このタスクの所有者は君だ。計画し、レビューし、検証せよ）」。調査と修正はサブエージェントに委譲してよいが、リードは渡さない。

    1. 自分で再現する。再現手順をユーザーに丸投げしない。直接再現しないなら、トリガーを合成し条件を締め上げ、発火するまで計装する
    2. 原因を二分探索する。仮説群を立て、毎回「残りの問題空間を最も削る分割」を選び、ランタイム証拠で消去する。推測は禁止。状態が不明ならログを足して動かして読む
    3. 修正を計画する。関数境界をまたぐなら先にarchitect。実装はモデル指定つきでサブエージェントに委譲し、自分でdiffをレビュー
    4. 同じ表面で検証する。元の再現手順が今は通ることを確認。「結果が不明瞭」はパスではない。ユニットテストは分岐の挙動を示すだけで、バグの不在は示さない
    5. コミットを物語る順に積む。失敗する再現を先に、修正をその上に。git履歴のdiffがそのままストーリーになる

    通底するのは「Be scientific. Every shipped line traces to runtime evidence.（科学的であれ。出荷される全行はランタイム証拠まで遡れる）」という規律。「効くかもしれない」保険的な変更は仮説であって修正ではないので出荷しない。証拠が仮説を棄却したら、その仮説が動機づけた変更はrevertする。

    サブエージェントとマルチモデル戦略

    pstackのもう1つの柱がモデルの使い分け。「すべてのフロンティアモデルには得手不得手がある」前提で、多くのスキルがマルチモデルワークフローになっている。

    • /arena は同じタスクにN個の並列候補を生成させ、ベースを選んで敗者の強い部分を移植する
    • /interrogate はdiffに対して4つの異なるモデルが壊そうと試みる敵対的レビュー。厳格なコード品質レンズも含む
    • /why は実行時に利用可能なMCPを発見し、ソース管理・課題トラッカー・ドキュメント・チャット・オブザーバビリティ・エラートラッキング・分析基盤という証拠カテゴリを並列に照会する

    ロールとモデルの対応は/setup-pstackで構成できる。手持ちのモデルを検出して、コード担当・判断担当・レビューパネルのそれぞれにモデルを割り当てる小さなルールを書き出し、各スキルはそれを読む（無ければ既定値にフォールバック）。既定はコード生成が高速系、散文と判断がthinking系という割り当てで、サブエージェント呼び出しはrun_in_background: true・ファイルポインタ渡し（コンテキストをインライン展開しない）が標準とされている。

    注意: 委譲しても責任は移らない。「You own every subagent's work」。サブエージェントの“done”要約を鵜呑みにせず自分でdiffをレビューし、要約も自分の言葉で書き直す。中断を挟んだresumeは指示を静かに落とすので、統合し直したスコープで新しいサブエージェントを起動し直せ、という運用ノウハウまで書かれている。

    文章まで設計対象: unslop

    pstackが他のスキル集と一線を画すのは、コードだけでなくエージェントの書く文章そのものを品質管理の対象にしている点。/unslopはAIっぽい文章の癖（AI tells）を除去するスキルで、poteto-modeの返信規範にも織り込まれている。

    • 短い宣言文。1文1思考
    • 長いダッシュ（em dash）は全面禁止。ファイル名と説明をダッシュで繋がず、文として書く
    • 文中接続のコロンも禁止（リスト導入のコロンは可）
    • 簡潔さを理由に内容を落とさない。詳細・トレードオフ・未決事項は全部残す
    • リンク・引用・トランスクリプト参照の捏造は絶対禁止。このセッションで自分が作ったか読んだものだけをリンクする

    根拠として面白いのが「書いた後のクリーンアップは失敗すると測定済み。だから悪い文を最初から生成するな」という一文。後処理パスに頼らず、生成時点の規範として効かせる。コメント規約も同様で、「// Phase 1: add cardsのようなナレーション型コメントを書くな、assertやログ文字列が唯一のドキュメントだ」と、エージェントが実際にやりがちな失敗を名指しで潰している。

    スキル設計として学べるパターン

    自分のClaude Codeスキル設計に持ち帰れるパターンを抜き出しておく。

    • 索引 + leafの2層構造。20原則をインライン索引で常時ロードし、全文は適用時にだけ読む。コンテキスト予算と網羅性の両立パターン
    • 手順の逐語コピー強制。playbookを「読んで参考にする」のではなくtodolistへverbatimコピーさせ、skipには理由を書かせる。エージェントの自己流ドリフトへの構造的対策
    • 失敗モードを名指しで書く。「読んだ後に自己流プランを書く」「ナレーションコメントを書く」など、実際に観測された失敗を具体的にスキルへ書き戻す。これ自体がmeta原則encode-lessons-in-structureの実践
    • 質問の前に分類。観察可能な事実は実験（プロトタイプ）で決め、人間には好みとプロダクト判断だけを聞く
    • カスタマイズの自動生成。/automate-meは最近のトランスクリプトをマイニングして「自分がどう働いてきたか」から<your-name>-modeを起草し、pstackを土台として残す。スキルをフォークさせるのではなく、ルーティング層だけ差し替えさせる設計
    • サブエージェントのスタイル保証。専用subagent定義（poteto-agent）でwrapperを固定し、汎用エージェント代用によるドリフトを防ぐ

    全体として、pstackは「良いエンジニアの暗黙知をどうやってエージェントの構造に焼き込むか」の実例集になっている。プロンプトに「気をつけて」と書くのではなく、todolist・逐語コピー・索引・wrapper・モデル割り当てルールといった構造で規律を作る。この姿勢が一番の学びだと思う。

    まとめ

    • pstackは「質を構造で担保すれば並列化を恐れなくてよい」というAIコーディング観の実装。量を出すためではなく、少なく正しく書くためのプラグイン
    • 中枢のpoteto-modeは、原則索引の強制読了・playbookの逐語コピー・原則引用の実判断トレースという3つの仕掛けでドリフトを抑える
    • 20原則・15playbook・13ワークフロースキルという分割は、コンテキスト予算を意識したprogressive disclosureの好例
    • 検証の規律（runtime evidence、同一表面での再現、失敗再現を先に積むgit履歴）はそのまま自分のワークフローに輸入できる
    • 次の一歩としては、cursor-team-kit側のdeslop / control-*を読むこと、そして自分のスキル群に「索引+leaf」「逐語コピー」「失敗モードの名指し」を移植してみること
