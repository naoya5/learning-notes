アーキテクチャ Lv1 — 基礎原則完全解説

SOLID 5原則・Clean Code・GoFパターンを Before/After コード + 図解で1から学べる詳細解説。シリーズ全5回の第1回目。

目次

    TL;DR

    • Lv1の到達点: 1関数・1クラスを「読みやすく・変更しやすく・テストしやすく」書ける
    • SOLID = 5つの原則の頭文字。守ると「変更が連鎖しないコード」になる
    • Clean Code = 命名・関数・コメントの3軸で読みやすさを担保
    • GoFパターン = 23ある中で実務で頻出は7つ（Strategy/Factory/Singleton/Adapter/Decorator/Observer/Facade）
    • このレベルが土台。Lv2以降のアーキは全部Lv1の上に乗る

    このシリーズの位置づけ

    「ソフトウェアアーキテクチャ完全学習ガイド Lv1〜Lv5」のシリーズ第1回。前回の総覧記事（Lv1〜5全体マップ）を、各レベル毎に「一から学べる」詳細レベルで深掘りする企画。

    Tip: Lv1は 個人開発でも必須。AIに「クラス分けて」と指示する時、ここが分かってないと AIの出力もぐちゃぐちゃになる。

    導入 — なぜ「コードレベルの設計」が大切か

    Lv2以降の派手なアーキテクチャ（Clean Architecture、DDD、マイクロサービス）は、すべて Lv1の原則の上に成り立つ。SOLIDが守られていない関数を100個集めてClean Architectureに並べても、結局メンテできないシステムになる。

    「変更コスト」がエンジニアの最大の敵

    SOLID — オブジェクト指向の5原則（完全版）

    1990年代後半に Robert C. Martin（Bobおじさん）が整理した、オブジェクト指向設計の中核5原則。順に詳しく見る。

    S — Single Responsibility Principle（単一責任原則）

    「1クラスを変更する理由は1つだけ」

    ❌ 違反例: User クラスに3つの責務

    // User がデータ + DB保存 + メール送信 + バリデーションを全部抱える
class User {
  constructor(public name: string, public email: string) {}

  save() {
    // PostgreSQL に保存
    db.query(`INSERT INTO users(name, email) VALUES ($1, $2)`,
             [this.name, this.email]);
  }

  sendWelcomeEmail() {
    // SMTP で送信
    smtp.send(this.email, "Welcome!", `Hi ${this.name}!`);
  }

  validate(): boolean {
    return this.name.length > 0 && /@/.test(this.email);
  }
}

    問題点:

    • DB変更（PostgreSQL → MySQL）で User を修正
    • メール送信先変更（SMTP → SendGrid）で User を修正
    • バリデーション追加（パスワード強度等）で User を修正
    • → 3つの異なる「変更理由」が1クラスに同居

    ✅ 修正後: 責務を分離

    // データだけ持つ
class User {
  constructor(public name: string, public email: string) {}
}

// DB保存だけ担当
class UserRepository {
  async save(user: User): Promise<void> {
    await db.query(`INSERT INTO users(name, email) VALUES ($1, $2)`,
                   [user.name, user.email]);
  }
}

// メール送信だけ担当
class WelcomeMailer {
  async send(user: User): Promise<void> {
    await smtp.send(user.email, "Welcome!", `Hi ${user.name}!`);
  }
}

// バリデーションだけ担当
class UserValidator {
  validate(user: User): boolean {
    return user.name.length > 0 && /@/.test(user.email);
  }
}
    注意: 「1クラス1メソッド」という意味ではない。「変更理由が1つ」が正しい解釈。同じ理由で変わるなら複数メソッドを持っていい。

    O — Open/Closed Principle（開放閉鎖原則）

    「拡張に対して開き、修正に対して閉じる」

    新機能追加時、既存コードを 修正せずに追加だけで済むようにする。

    ❌ 違反例: if 文増殖

    type PaymentMethod = "credit" | "paypal" | "crypto";

function pay(method: PaymentMethod, amount: number) {
  if (method === "credit") {
    chargeCreditCard(amount);
  } else if (method === "paypal") {
    chargePayPal(amount);
  } else if (method === "crypto") {           // ← 追加するたびに修正
    chargeCrypto(amount);
  }
}

    → 新しい支払い方法（Apple Pay等）を追加するたびに pay() を修正する必要がある。

    ✅ 修正後: Strategy パターン

    interface PaymentStrategy {
  execute(amount: number): void;
}

class CreditCardPayment implements PaymentStrategy {
  execute(amount: number) { chargeCreditCard(amount); }
}

class PayPalPayment implements PaymentStrategy {
  execute(amount: number) { chargePayPal(amount); }
}

class CryptoPayment implements PaymentStrategy {
  execute(amount: number) { chargeCrypto(amount); }
}

// pay() は変更不要、新しいクラスを追加するだけ
function pay(strategy: PaymentStrategy, amount: number) {
  strategy.execute(amount);
}

// 使う側
pay(new CreditCardPayment(), 1000);
pay(new ApplePayPayment(), 1000);  // ← 新規 class 追加だけで済む

    L — Liskov Substitution Principle（リスコフの置換原則）

    「親クラスを子クラスに差し替えても、プログラムは正しく動作する」

    ❌ 違反例: Penguinが飛ぼうとして例外

    class Bird {
  fly(): void {
    console.log("飛んでいます");
  }
}

class Penguin extends Bird {
  fly(): void {
    throw new Error("ペンギンは飛べません");  // ← 親の前提を壊す
  }
}

function makeBirdFly(bird: Bird) {
  bird.fly();  // Bird を期待してるのに Penguin が来ると死ぬ
}

makeBirdFly(new Penguin());  // ❌ 実行時エラー

    問題: Bird 型のつもりで使ったら例外で落ちる。「Bird = 飛べる」という暗黙の契約を子が破った。

    ✅ 修正後: 抽象を分ける

    interface Animal {
  move(): void;
}

class Sparrow implements Animal {
  move() { console.log("飛ぶ"); }
}

class Penguin implements Animal {
  move() { console.log("泳ぐ"); }
}

function makeAnimalMove(animal: Animal) {
  animal.move();  // どの実装でも安全
}
    判定法: 「親で書かれてた事前条件・事後条件・例外契約を、子が 同じか緩める方向にしか変更してない」ならOK。逆方向（厳しくする / 別の例外を追加）はNG。

    I — Interface Segregation Principle（インターフェース分離原則）

    「クライアントは、使わないメソッドへの依存を強制されるべきでない」

    ❌ 違反例: 太いインターフェース

    interface Worker {
  work(): void;
  eat(): void;
  sleep(): void;
}

// ロボットも実装することになると...
class Robot implements Worker {
  work() { /* OK */ }
  eat()  { throw new Error("ロボットは食べない"); }    // ❌
  sleep(){ throw new Error("ロボットは寝ない"); }      // ❌
}

    ✅ 修正後: 小さなインターフェースに分割

    interface Workable { work(): void }
interface Eatable   { eat(): void }
interface Sleepable { sleep(): void }

class Human implements Workable, Eatable, Sleepable {
  work()  { /* */ }
  eat()   { /* */ }
  sleep() { /* */ }
}

class Robot implements Workable {
  work() { /* OK、Eatable/Sleepable は実装不要 */ }
}

    D — Dependency Inversion Principle（依存性逆転原則）

    「上位モジュールは下位モジュールの具象に依存してはならない。両者は抽象に依存すべき」

    ❌ 違反例: ビジネスロジックがDB実装に直接依存

    class MySQLDatabase {
  save(order: Order) { /* MySQL固有実装 */ }
}

class OrderService {
  private db = new MySQLDatabase();  // ← 具象に直接依存

  placeOrder(order: Order) {
    this.db.save(order);
  }
}

    問題: DB を PostgreSQL に変えたい時、OrderService も書き換えが必要。テストもMySQL接続が必要になる。

    ✅ 修正後: interface 経由で依存性逆転

    // 抽象（インターフェース）
interface OrderRepository {
  save(order: Order): Promise<void>;
}

// 具体実装（差し替え可能）
class MySQLOrderRepository implements OrderRepository {
  async save(order: Order) { /* MySQL */ }
}
class PostgresOrderRepository implements OrderRepository {
  async save(order: Order) { /* PostgreSQL */ }
}

// 上位モジュール（抽象に依存、DI で具体を受け取る）
class OrderService {
  constructor(private readonly repo: OrderRepository) {}

  async placeOrder(order: Order) {
    await this.repo.save(order);
  }
}

// 使う側で具体を注入
const service = new OrderService(new MySQLOrderRepository());
    DIP と DI の違い:
    • DIP（原則）: 抽象に依存しろ
    • DI（実装パターン）: 依存を外から注入する技法（コンストラクタ注入など）
    DIP を実現する手段の1つが DI。原則とパターンを区別する。

    SOLID チェックリスト

    自分のコードを SOLID で点検する質問集。

    • ☐ S: このクラスを変更する理由は1つだけか？「○○の場合と××の場合に変える」と言いたくなったら違反
    • ☐ O: 機能追加で既存コードを変更してないか？ if-else が増殖してたら違反
    • ☐ L: サブクラスが親の前提を破ってないか？例外を追加してたら要警戒
    • ☐ I: 実装側で「使わないメソッド」を強制されてないか？throw new Error が並んでたら違反
    • ☐ D: ビジネスロジックが具体クラス（DB/API/ファイル）を直接 new してないか？してたら違反

    Clean Code — 読みやすさの原則（詳細版）

    SOLIDが「クラス間の構造」なら、Clean Codeは「1関数の中の表現」。

    await db.query("INSERT INTO order_items ...", [...]);

    ✅ After: 関数分解

    async function processOrder(orderData: OrderInput): Promise<Order> {
  const validated  = validate(orderData);
  const priced     = await calculatePrice(validated);
  const persisted  = await persist(priced);
  await sendConfirmationMail(persisted);
  await log("ORDER_CREATED", persisted);
  return persisted;
}

// 各関数は20行以内、1責務
function validate(input: OrderInput): ValidatedOrder { /* ... */ }
async function calculatePrice(o: ValidatedOrder): Promise<PricedOrder> { /* ... */ }
async function persist(o: PricedOrder): Promise<Order> { /* ... */ }
async function sendConfirmationMail(o: Order): Promise<void> { /* ... */ }

    関数の3原則:

    • 📏 20行以内（できれば10行）
    • 🎯 1つのことをする — 関数名以外のことをしない
    • 🚫 引数3つ以下 — 4つ以上ならオブジェクトにまとめる

    コメント — WHY を書く、WHAT を書かない

    ❌ NG: WHATを書くコメント

const userId = user.id;  // ← コードを読めば分かる、無駄

// ループで処理
for (const item of items) {  // ← 自明
  process(item);
}

    ✅ OK: WHY/制約を書くコメント

const numericId = parseInt(legacyResult.id, 10);

// 7回 = 顧客サポートチームと合意した最大リトライ数
// 経済的に許容できる失敗率を確保

    GoFパターン — 実務7選を完全解説

    23パターン全部覚える必要はない。実務で頻出する 7つを実装コード+UML図で見る。

    1. Strategy — アルゴリズムを差し替え可能にする

    使いどころ: 同じ目的の処理が複数あり、実行時に切り替えたい（支払い方法・ソート手法・圧縮形式）。

    interface SortStrategy { sort(arr: number[]): number[]; }

class QuickSort  implements SortStrategy { sort(a) { /* */ return a; } }
class MergeSort  implements SortStrategy { sort(a) { /* */ return a; } }
class BubbleSort implements SortStrategy { sort(a) { /* */ return a; } }

class Sorter {
  constructor(private strategy: SortStrategy) {}
  setStrategy(s: SortStrategy) { this.strategy = s; }
  execute(arr: number[]) { return this.strategy.sort(arr); }
}

const sorter = new Sorter(new QuickSort());
sorter.execute([3, 1, 4, 1, 5]);
sorter.setStrategy(new MergeSort());  // 実行時に差し替え

    2. Factory — 生成ロジックを隠蔽

    使いどころ: 環境やパラメータによって生成するクラスが変わる。

    interface Logger {
  log(msg: string): void;
}
class FileLogger    implements Logger { log(m) { /* file */ } }
class CloudLogger   implements Logger { log(m) { /* cloud */ } }
class ConsoleLogger implements Logger { log(m) { console.log(m); } }

class LoggerFactory {
  static create(env: "prod" | "staging" | "dev"): Logger {
    switch (env) {
      case "prod":    return new CloudLogger();
      case "staging": return new FileLogger();
      case "dev":     return new ConsoleLogger();
    }
  }
}

const logger = LoggerFactory.create(process.env.NODE_ENV as any);
logger.log("hello");

    3. Singleton — 1インスタンスだけ存在保証

    使いどころ: アプリ全体で共有する設定・接続プール・ロガー。

    class Config {
  private static instance: Config;
  private constructor(public readonly apiUrl: string) {}

  static getInstance(): Config {
    if (!Config.instance) {
      Config.instance = new Config(process.env.API_URL!);
    }
    return Config.instance;
  }
}

const c1 = Config.getInstance();
const c2 = Config.getInstance();
console.log(c1 === c2);  // true
    注意: Singletonはグローバル状態。テストが書きにくくなりがち。代替として「DIコンテナでスコープを Singleton にする」方が望ましい場面が多い。

    4. Adapter — 互換性のないAPIを橋渡し

    使いどころ: 外部ライブラリやレガシーAPIを、自分のアプリのインターフェースに合わせる。

    // 自分のアプリが期待するインターフェース
interface PaymentGateway {
  charge(amount: number, cardToken: string): Promise<{ id: string }>;
}

// 外部の Stripe SDK
class StripeSDK {
  async createCharge(opts: { amount_cents: number; source: string }) {
    return { stripe_charge_id: "ch_xyz" };
  }
}

// 互換性をつなぐ Adapter
class StripeAdapter implements PaymentGateway {
  constructor(private stripe: StripeSDK) {}
  async charge(amount: number, cardToken: string) {
    const result = await this.stripe.createCharge({
      amount_cents: amount * 100,
      source: cardToken,
    });
    return { id: result.stripe_charge_id };
  }
}

    5. Decorator — 機能を動的に追加

    使いどころ: ミドルウェア、ロギング、認証、キャッシュなど横断的関心事を後付けする。

    interface Handler {
  handle(req: Request): Promise<Response>;
}

class BaseHandler implements Handler {
  async handle(req: Request) { return new Response("ok"); }
}

// ロギングを追加するDecorator
class LoggingHandler implements Handler {
  constructor(private inner: Handler) {}
  async handle(req: Request) {
    console.log(`-> ${req.url}`);
    const res = await this.inner.handle(req);
    console.log(`<- ${res.status}`);
    return res;
  }
}

// 認証を追加するDecorator
class AuthHandler implements Handler {
  constructor(private inner: Handler) {}
  async handle(req: Request) {
    if (!req.headers.get("Authorization")) return new Response("401", { status: 401 });
    return this.inner.handle(req);
  }
}

// 重ね合わせ
const handler = new LoggingHandler(new AuthHandler(new BaseHandler()));

    6. Observer — 状態変化を通知

    使いどころ: イベント駆動UI、Redux、リアクティブシステム。

    type Listener<T> = (value: T) => void;

class Subject<T> {
  private listeners: Listener<T>[] = [];
  subscribe(fn: Listener<T>) {
    this.listeners.push(fn);
    return () => { this.listeners = this.listeners.filter(l => l !== fn); };
  }
  notify(value: T) {
    this.listeners.forEach(fn => fn(value));
  }
}

const orderUpdated = new Subject<{ id: string; status: string }>();

const unsub1 = orderUpdated.subscribe(o => console.log(`Logger: ${o.id}`));
const unsub2 = orderUpdated.subscribe(o => sendNotification(o.id));

orderUpdated.notify({ id: "ord_1", status: "paid" });

    7. Facade — 複雑なサブシステムを単純化

    使いどころ: 複数のクラス操作を1つのインターフェースで隠蔽（API Gateway、ライブラリのエントリポイント）。

    // 内部の複雑なサブシステム
class InventoryService { reserve(itemId: string, qty: number) { /* */ } }
class PaymentService   { charge(userId: string, amount: number) { /* */ } }
class ShippingService  { schedule(orderId: string, addr: string) { /* */ } }
class MailService      { send(to: string, body: string) { /* */ } }

// クライアントが使う Facade
class OrderFacade {
  constructor(
    private inventory: InventoryService,
    private payment:   PaymentService,
    private shipping:  ShippingService,
    private mail:      MailService,
  ) {}

  async placeOrder(userId: string, items: Item[], address: string) {
    items.forEach(i => this.inventory.reserve(i.id, i.qty));
    const total = items.reduce((s, i) => s + i.price * i.qty, 0);
    this.payment.charge(userId, total);
    const orderId = generateId();
    this.shipping.schedule(orderId, address);
    this.mail.send(userId, `Order ${orderId} placed`);
    return { orderId };
  }
}

// クライアントは Facade だけ知ればOK
facade.placeOrder("user1", items, "Tokyo");
    パターン乱用の罠: 「学んだから使う」じゃなく「問題があるから使う」。Singletonをグローバル変数代わりにする、全部Factoryでくるむ等は 過剰設計でコードを複雑にするだけ。

    演習問題

    理解度チェック。各問の「考えるポイント」を見る前にまず自分で考える。

    Q1. 以下のクラスは何が問題？

    class Report {
  generate() { /* PDF生成 */ }
  email() { /* メール送信 */ }
  saveToS3() { /* S3保存 */ }
  parseJSON(s: string) { /* JSON parse */ }
}
    解答を見る

    S違反。生成・メール・保存・パースの4責務が同居。

    修正: ReportGenerator / ReportMailer / ReportS3Storage / JsonParser に分離。

    Q2. 以下のコードを OCP に従う形に書き換えて

    function calculateDiscount(type: string, price: number) {
  if (type === "regular") return price * 0.95;
  if (type === "premium") return price * 0.85;
  if (type === "vip")     return price * 0.75;
  return price;
}
    解答を見る 
    interface DiscountStrategy {
  apply(price: number): number;
}
class RegularDiscount implements DiscountStrategy { apply(p) { return p * 0.95; } }
class PremiumDiscount implements DiscountStrategy { apply(p) { return p * 0.85; } }
class VipDiscount     implements DiscountStrategy { apply(p) { return p * 0.75; } }

function calculateDiscount(strategy: DiscountStrategy, price: number) {
  return strategy.apply(price);
}

    新割引追加時 calculateDiscount は変更不要。

    Q3. 以下の Bird は LSP違反？

    class Bird {
  fly() { /* 飛ぶ */ }
}
class Eagle extends Bird {}
class Ostrich extends Bird {
  fly() { throw new Error("ダチョウは飛べない"); }
}
    解答を見る

    LSP違反。Bird型として使った時、Ostrich は例外で死ぬ。
    解決: FlightlessBird を別クラスに分けるか、Animal { move() } という抽象に変える。

    まとめ — Lv1の要点

    1. SOLIDは変更コストを下げる5原則 — 1つでも崩れると保守性が落ちる
    2. Single Responsibility — 1クラス1責務（変更理由が1つ）
    3. Open/Closed — 拡張に開く、修正に閉じる（Strategy が定石）
    4. Liskov Substitution — 親の契約を子が破らない
    5. Interface Segregation — interface は小さく分割
    6. Dependency Inversion — 抽象に依存、具象に依存しない
    7. Clean Code — 命名・関数長・コメントの3軸で読みやすく
    8. GoF 7パターン — Strategy/Factory/Singleton/Adapter/Decorator/Observer/Facade
    9. パターンは問題に対する解 — 問題なければ使わない
