Agent Skills を読み解く — OMC と OpenCode、50本のSKILL.mdからプロンプト設計を学ぶ

oh-my-claudecode（Claude Code向け）と oh-my-opencode / oh-my-openagent（OpenCode向け）の3プロジェクトに同梱されたスキル50本のSKILL.md全文を読み込み、自動起動の仕組み・プロンプトの定型句・サブエージェント委譲・出力契約のパターンを抽出。最終章は自作SKILL.mdの雛形と設計チェックリスト。

目次

    TL;DR

    • 3プロジェクト: oh-my-claudecode (38スキル) / oh-my-opencode (4 builtin + 4 commands) / oh-my-openagent (4 skills + 4 commands)。Claude Code または OpenCode に「マルチエージェント・オーケストレーション層」を被せる。
    • frontmatter は name + description が標準。 OMC は level argument-hint pipeline handoff 等で frontmatter を小さなワークフロー DSL に拡張している。
    • description は自動起動の唯一の指標。 Use when … + Trigger phrases include "…" + 必要なら MUST USE for ANY … でドメインを占有する。
    • 本文は XML タグで意図を分割 (<Purpose> <Use_When> <Do_Not_Use_When> <Why_This_Exists>)。長いスキルは STEP 0: REGISTER TODO LIST で自己拘束する。
    • 頻出キーワード: MUST=67、NEVER=29、CRITICAL=39、evidence=89、subagent=70、permalink=33。ガード文・証拠主義・並列化が3本柱。
    • 自作の判断は Skillify Quality Gate の3問: 「ググって5分で出るか」「自リポ固有か」「発見に本物の労力がかかったか」。3問とも YES のときだけ作る。

    はじめに — なぜ50本のスキルを読み解くのか

    Claude Code（Anthropic 公式 CLI）と OpenCode（OSS エージェント CLI）には、エージェントの能力を「スキル（SKILL.md）」というファイルで拡張する仕組みがある。Anthropic が Agent Skills という共通フォーマットを公開していて、CLI ごとに方言はあるが基本は揃っている。

    このフォーマットの上に、より重武装な「オーケストレーション層」を被せたコミュニティ製のメタディストリビューションが2系統ある。

    • oh-my-claudecode (OMC) — Claude Code 向け。38本のスキル + tmux 並列実行 + 自己進化 + マルチモデル協調まで詰め込んだ重武装系。
    • oh-my-opencode / oh-my-openagent — OpenCode 向け。前者は後者のフォーク。日常運転に効く builtin skill を厳選 + 敵対多エージェント計画のような派手なスキルも持つ。

    この記事はこの3つのリポジトリの 50本のSKILL.mdを全文ダウンロードして読み解いた 学習ノート。目標は「自分用にスキルを書くための材料集」。

    Note: 本文はクリーンに整理してあるが、調査の生データ（一覧表）は Obsidian 側の リサーチ/oh-my-claudecode-and-oh-my-opencode-skills.md にもある。

    SKILL.md の解剖図

    ここからは 50 本全部の SKILL.md から抽出した 共通の作りのパターン。「自分用にスキルを書く」ための材料集。

    frontmatter のキー集計

    50 ファイル全部からキーを集計（出現数）。

    キー出現役割
    description50 / 50必須。スキル自動マッチに使う本文。Anthropic Agent Skills の標準キー
    name46 / 50スキル ID。kebab-case 推奨
    level30 / 50OMC 独自。1=軽量、4=自律実行系
    argument-hint18 / 50OMC 独自。スラッシュコマンドの引数テンプレ
    aliases4 / 50OMC 独自。スキルの別名
    pipeline3 / 50OMC 独自。次に呼ぶスキルの連鎖
    handoff3 / 50OMC 独自。次のスキルに渡すファイルパス
    handoff-policy2 / 50OMC 独自。approval-required で人間承認をゲート
    triggers3 / 50学習スキルが使う配列
    agent1 / 50OMC 独自。委譲先 built-in agent
    user-invocable1 / 50false で /skill list から隠す（裏方スキル用）
    教訓: Anthropic 公式 Agent Skills だけ意識するなら name + description でいい。OMC はオーケストレーション機能を frontmatter に押し込んでいて、スキル自体が小さなワークフロー DSL になっている。自作するなら標準2キーで始め、必要が出てから足す。

    description の書き方（自動起動のキモ）

    description は LLM がスキル選択を判断する 唯一の指標。長さ・密度・トリガー語の入れ方で精度が変わる。実例3種：

    (a) 主語＋動詞のみ — 最小

    description: Strategic planning with optional interview workflow

    短いと自動起動に乗りにくい。ユーザーが明示的に呼ぶ前提のスキル向け。

    (b) "Use when …" 句で文脈を埋め込む — 標準

    description: Browser automation with persistent page state. Use when users ask to
  navigate websites, fill forms, take screenshots, extract web data, test web apps,
  or automate browser workflows. Trigger phrases include "go to [url]", "click on",
  "fill out the form", "take a screenshot", "scrape", "automate", "test the website",
  "log into", or any browser interaction request.

    Use when … で意図、Trigger phrases include "…" で具体的な発話例を列挙。自動起動の精度を上げる王道。

    (c) MUST USE / STRONGLY RECOMMENDED で押し込む — 強制

    description: "MUST USE for ANY git operations. Atomic commits, rebase/squash,
  history search (blame, bisect, log -S). STRONGLY RECOMMENDED: Use with
  delegate_task(category='quick', load_skills=['git-master'], ...) to save context.
  Triggers: 'commit', 'rebase', 'squash', 'who wrote', 'when was X added',
  'find the commit that'."

    MUST USE を頭に書くと、競合する他スキルを押しのけて起動しやすくなる。git-master のようにドメインを「占有」したいスキル向け。

    本文の典型構造

    50本ぜんぶの最初の見出しを集計したら、上位パターンは：

    見出し出現
    Usage8
    Overview4
    Goal4
    Routing3
    When to use / When not to use各2
    Setup / Commands / Parameters / Workflow / Phase 1各2
    STEP 0: REGISTER TODO LIST2

    派生して、deep-interview のような重量級スキルでは XML 風タグ で意図を分割しているのが特徴的： 

    <Purpose>
Deep Interview implements Ouroboros-inspired Socratic questioning ...
</Purpose>

<Use_When>
- User has a vague idea and wants thorough requirements gathering
- User says "deep interview", "interview me", "ask me everything"
...
</Use_When>

<Do_Not_Use_When>
- User has a detailed, specific request with file paths -- execute directly
- User wants to explore options or brainstorm -- use `omc-plan` skill instead
...
</Do_Not_Use_When>

<Why_This_Exists>
AI can build anything. The hard part is knowing what to build. ...
</Why_This_Exists>

    XML タグは Anthropic の prompt engineering ガイドの推奨そのもの。「いつ使うか」「いつ使ってはいけないか」「なぜ存在するか」を本文先頭で明示 するのは、自動起動と誤起動防止の両方に効く。

    STEP 0: REGISTER TODO LIST パターン

    opencode-cmd 系（publish / remove-deadcode）は本文の最初に必ずこれが来る： 

    ## STEP 0: REGISTER TODO LIST (MANDATORY FIRST ACTION)

**Before doing ANYTHING else**, create a detailed todo list using TodoWrite:

TodoWrite([
  {"id": "scan", "content": "PHASE 1: ..."},
  {"id": "verify", "content": "PHASE 2: ..."},
  ...
])

    意図：スキルが LLM に「忘れずに最後まで完走する」自己拘束をかける。長いワークフローを実行させたいときの定石。その後は PHASE 1: PHASE 2: と連番で続く。

    プロンプト技法カタログ

    ガード文（MUST / NEVER / CRITICAL）

    50 ファイル中に MUST=67、NEVER=29、CRITICAL=39 の出現。頻出度が高いほど、その仕事の失敗コストが大きい ことを暗示している。

    実例: false-positive を一切許さないガード（remove-deadcode）

    ## CRITICAL RULES

1. **LSP is law.** Never guess. Always verify with `LspFindReferences` before
   removing ANYTHING.
2. **One removal = one commit.** Every dead code removal gets its own atomic commit.
3. **Test after every removal.** Run `bun test` after each. If it fails, REVERT and skip.
4. **Leaf-first order.** Remove deepest unused symbols first.
5. **Never remove entry points.** `src/index.ts`, test files, config files,
   and files in `packages/` are off-limits unless explicitly targeted.

    実例: 偽陽性ガード（False Positive Guards）

    **NEVER mark as dead code if:**
- Symbol is in `src/index.ts` (package entry point)
- Symbol is referenced in test files (tests are valid consumers)
- Symbol has `@public` or `@api` JSDoc tags
- Symbol is in a file listed in `package.json` exports
...
    教訓: ガードは「やってはいけないこと」を 箇条書きで具体的に列挙 する。LLM は曖昧な禁止より具体例の禁止のほうが守る。

    サブエージェント委譲（delegate_task / task）

    50 ファイル中に subagent=70、delegate=22。並列化はスキルの主要装備。

    OpenAgent / OpenCode のサブタスク呼び出し

    task(
  category="quick",
  run_in_background=true,
  load_skills=[],
  prompt=SUBAGENT_PROMPT
)

    Claude Code 互換の delegate_task

    delegate_task(subagent_type="explore", run_in_background=true,
  prompt="Find ALL exported functions, classes, types ...")

    サブエージェントの絶対ルール（github-triage より）

    **ABSOLUTE RULES for Subagents:**
- **ONLY ANALYZE** - Never take action on GitHub (no comments, merges, closes)
- **READ-ONLY** - Use tools only for reading code/GitHub data
- **WRITE REPORT ONLY** - Output goes to `{REPORT_DIR}/{issue|pr}-{number}.md` via Write tool
- **EVIDENCE REQUIRED** - Every claim must have GitHub permalink as proof
    教訓: サブエージェントは「権限の薄いコピー」として動かす。load_skills=[] で文脈を切り、run_in_background=true で並列化。親エージェントは集約だけする。

    出力契約（artifact path）

    evidence=89、permalink=33、/tmp/=6 の出現。スキルが どこにファイルを書くか を最初から決め打ちしているのが特徴。

    用途パス例出典
    実行ログ・トレース/tmp/{datetime}/{issue|pr}-{number}.mdgithub-triage
    プラン成果物.omc/plans/ralplan-*.mdomc-plan
    インタビュー仕様.omc/specs/deep-interview-{slug}.mddeep-interview
    リリースルール.omc/RELEASE_RULE.mdrelease

    evidence rule（証拠主義）

    特に github-triage / pre-publish-review が徹底している：

    ## Evidence Rule (MANDATORY)

### Rules
- **No permalink = no claim.** If you cannot back a statement with a permalink,
  state "No evidence found" instead.
- Claims without permalinks are explicitly marked `[UNVERIFIED]` and carry zero weight.
- Permalinks to `main`/`master`/`dev` branches are NOT acceptable - use commit SHAs only.

### How to generate permalinks
1. Find the relevant file and line(s) via Grep/Read.
2. Get the current commit SHA: `git rev-parse HEAD`
3. Construct: `https://github.com/{REPO}/blob/{SHA}/{filepath}#L{line}`
    教訓: スキルに「証拠を伴わない主張は無効」と書くだけで、LLM のハルシネーションが 目に見えて減る。Branch ではなく SHA を要求するのも巧い（時間が経っても URL が壊れない）。

    progressive disclosure（段階的開示）

    サブディレクトリで本文を軽くするテク。dev-browser が代表例：

    src/features/builtin-skills/dev-browser/
├── SKILL.md                       ← 短い概要 + 使い方の入り口
└── references/
    ├── installation.md            ← 必要時に Read で読み込む
    └── scraping.md                ← 同上

    OMC の self-improve はもっと派手：

    skills/self-improve/
├── SKILL.md
├── data_contracts.md
├── si-benchmark-builder.md
├── si-goal-clarifier.md
├── si-researcher.md
├── scripts/
└── templates/
    教訓: SKILL.md は「LLM が最初に読む 1 ページ」として軽く保ち、詳細手順や I/O 契約は references/ templates/ scripts/ に逃がす。

    数学的曖昧性スコアリング（deep-interview 流）

    deep-interview は「いつ質問を止めるか」を 数値で決める。普通のスキルにも応用できる発想：

    Score RangeMeaningAction
    0.0 - 0.1Crystal clearProceed immediately
    ≤ resolved thresholdClear enoughProceed
    > threshold + minor gapsSome gapsContinue interviewing
    ModerateSignificant gapsFocus on weakest dimensions
    HighVery unclearMay need reframing
    ExtremeAlmost nothing knownEarly stages, keep going

    5次元（Goal Clarity / Constraint Clarity / Success Criteria / Component / Risk）を 0.0-1.0 で評価し、平均が閾値を切ったら次フェーズへ。

    教訓: ループ系スキル（ralph / autoresearch / deep-interview）は 「終わる条件」を数値化 している。「LLM の判断にお任せ」ではなく、明示的なメトリクスとしきい値を持たせる。

    mode composition（スキルの入れ子）

    OMC は重量スキルを 他のスキルから呼ぶ 設計。pipeline: frontmatter が連鎖の hint：

    # omc-plan
pipeline: [deep-interview]

# deep-interview
pipeline: [deep-interview, plan]
    教訓: 大きいスキルを1個書くより、小さいスキル群と "それを順番に呼ぶ" 司令塔スキル に分けたほうが再利用しやすい。

    代表スキル深掘り

    verify — 最小例

    ---
name: verify
description: Verify that a change really works before you claim completion
---

# Verify

Use this skill when the user wants help diagnosing ...

## Rules
- Do not say a change is complete without evidence.
- If a check fails, include the failure clearly.
- If no realistic verification path exists, say that explicitly instead of bluffing.
- Prefer concise evidence summaries over noisy logs.

    たった 38 行。Rules だけで動作を縛る。「短くても効くスキル」のお手本。bluffing 禁止 という1行が効く。

    remember — 記憶の整理

    ## Rules
- Do not dump everything into one store.
- Prefer project memory for durable team knowledge.
- Prefer notepad for short-lived working context.
- Keep entries concise and actionable.
- If something is uncertain, mark it as uncertain rather than storing it as fact.

    42 行。「メモリストアの振り分け先」だけを教えるスキル。判断ヒューリスティクスを5行に圧縮した好例。

    autopilot / ralph / ultrawork — 自律実行三段重ね

    autopilot → ralph → ultrawork の入れ子。autopilot は Phase 0: 要件展開 から始まり、内部で deep-interview を呼ぶ。

    ralph の状態機械は教科書的：

    1. iteration 1 開始
2. team pipeline 実行: team-plan → team-prd → team-exec → team-verify
3. team-verify pass → architect 検証 (STANDARD tier)
4. architect approve → 完了 (run /cancel)
5. team-verify fail OR architect reject → team-fix → team-exec へ戻る
6. fix loop > max_fix_loops → ralph iteration を増やしてリトライ
7. ralph iteration > max_iterations → terminal failed
    学び: 「失敗したら何をして、何回までリトライするか」を 全部明文化 してある。自分のスキルでも、無限ループ防止の上限値を必ず書く。

    deep-interview — 曖昧性スコアによる対話制御

    frontmatter で pipeline と handoff を宣言：

    pipeline: [deep-interview, plan]
handoff-policy: approval-required
handoff: .omc/specs/deep-interview-{slug}.md

    本文構造：

    1. <Purpose> 何のためのスキルか
    2. <Use_When> 起動条件
    3. <Do_Not_Use_When> 起動禁止条件
    4. <Why_This_Exists> モチベーション
    5. Phase 1: Topology Lock — 質問対象のコンポーネント確定
    6. Phase 2: Interview Loop — 質問→回答→曖昧性スコア更新
    7. Phase 3: Spec Generation — 仕様書を .omc/specs/ に書き出す
    8. Ambiguity Score Interpretation — しきい値テーブル

    git-master — ドメイン占有スキル

    ### 3.0 Calculate Minimum Commit Count FIRST

FORMULA: min_commits = ceil(file_count / 3)

 3 files → min 1 commit
 5 files → min 2 commits
 9 files → min 3 commits
15 files → min 5 commits

**If your planned commit count < min_commits → WRONG. SPLIT MORE.**

    定量的な強制力。LLM が「まとめてコミット」しがちな性質を、式と表で抑え込む。

    ## Quality Gate

Before extracting a skill, all three should be true:
- "Could someone Google this in 5 minutes?" → No.
- "Is this specific to this codebase, project, or workflow?" → Yes.
- "Did this take real debugging, design, or operational effort to discover?" → Yes.

Prefer skills that encode decision-making heuristics, constraints,
pitfalls, and verification steps. Avoid generic snippets, boilerplate,
or library usage examples that belong in normal documentation.

    A. 自律実行・並列実行（ループ）系

    OMC が最も力を入れている領域。「自分で考えてループする」エージェントをどう作るか。

    入れ子構造（OMC 内部）

    autopilot (autonomous execution / full lifecycle)
 └─ ralph (persistence wrapper / loop + verification reviewer)
     └─ ultrawork (parallelism only)
    スキル役割抜けるもの
    ultrawork並列実行だけ。タスクを N 個に分けて同時に走らせる検証なし。終了しない
    ralphultrawork に「ループ + 検証レビューア」を被せる要件展開なし
    autopilotralph に「Phase 0: 要件展開 (deep-interview 経由)」を頭に追加起動時のすり合わせは依然必要
    設計の含意: 同じ「ループする実行」でも、層を分けて単機能のスキルを重ねている。1スキル = 1責務 を徹底すると、組み合わせが効く。

    ultraqa — QA に特化したループ

    ループだが対象が QA。最大 5 サイクル：

    1. RUN QA — --tests / --build / --lint / --typecheck / --custom / --interactive のいずれかで検証
    2. CHECK RESULT — pass なら exit、fail なら次へ
    3. ARCHITECT DIAGNOSIS — architect subagent (opus) に根本原因と修正案を出させる
    4. FIX ISSUES — executor subagent (sonnet) に修正させる
    5. REPEAT — Step 1 へ
    学び: 「役割分担した3体（QA tester / architect / executor）を直列に呼ぶ」がループ系の鉄板パターン。診断と修正を別エージェントに分ける ことで責務が明確になり、各役の system prompt をチューニングできる。

    self-improve — 進化アルゴリズム

    トーナメント選抜による自律進化的コード改善：

    1. 複数候補を executor 並列生成
    2. 各候補を benchmark_score で評価（Tournament Selection）
    3. status: "success" のみ filter
    4. benchmark_direction を尊重した ranked-candidate loop
    5. 勝者を merge、敗者ブランチを git-master 経由で archive
    6. 勝者なし → 何も merge せず、improvement branch は前回値のまま
    7. ラウンド毎に Merge Report JSON を <self-improve-root>/state/merge_reports/round_{n}.json に保存
    注意: 「LLM が自分でコードを書き、自分で評価する」を 進化アルゴリズム的に並列化 するなら、評価関数（benchmark）を最初に厳格に書く必要がある。評価が定量化できない領域には適用できない。

    「実行する前に何を実行すべきか決める」スキル群。ループ系の前段 として機能する。

    plan / ralplan / deep-interview / deep-dive の関係

    deep-dive (causal investigation + requirements crystallization)
  ├─ Phase 3: trace (3並列laneの競合仮説)
  └─ Phase 4: deep-interview (3-point injection)
       └─ Ambiguity score < threshold → spec を .omc/specs/deep-dive-{slug}.md へ

omc-plan (strategic planning)
  └─ オプションで deep-interview 起動 (--interactive)
  └─ 出力: .omc/plans/ralplan-*.md
  └─ handoff-policy: approval-required

ralplan = omc-plan の consensus mode alias

    deep-dive の 3-point injection

    deep-dive の Phase 4 では、deep-interview に 3 つの情報を同時注入 して質問を始める：

    1. initial_idea — ユーザの当初発話
    2. codebase_context — 該当リポの構造・規約
    3. question_queue — Phase 3 の trace で出た per-lane の "critical unknowns"

    trace 由来のテキストは <trace-context> 区切り文字で囲む（untrusted data guard）— LLM がユーザ発話と取り違えないため。

    学び: 複数の情報源を1つの LLM プロンプトに混ぜるとき、ソースごとに区切り文字を分ける のは prompt injection 対策として有効。

    trace — 因果調査の並列化

    agent: tracer を frontmatter で宣言（OMC 独自）。built-in tracer エージェントを 3 lane 並列 で走らせ、競合仮説を立てる。

    • Lane 1: 観察の機械的な原因
    • Lane 2: 設計上の原因
    • Lane 3: MOVE 候補 (リファクタリングで解消できる場所)。ownership_scope を必ず含め、cross-boundary なら警告
    学び: 「同じ問題を異なる視点で3つ独立に解析」は、特に "わからない" を最速で潰す手段。Lane 3 の MOVE 候補に ownership_scope を持たせるのは、コードオーナーシップ境界を越える変更を default 推奨にしない安全装置。

    autoresearch — 評価器契約付きのループ

    deep-interview --autoresearch で起動できる「研究ミッション専用」ループ。重要な制約：

    • mission clarity と evaluator clarity を hard readiness gate にする（通常の ambiguity threshold とは別）
    • evaluator command を取得できないと起動しない
    • 起動後は omc-plan / autopilot / ralph / team のいずれにもブリッジしない（自前ループ）
    • mission slug / evaluator command / max-runtime / artifact location を最初に announce
    学び: ループを開始する前に 「終わる条件」と「評価方法」を強制的に取りに行く のは reliability の肝。

    external-context — 並列 doc-specialist

    # 同時発火、独立 facet（最大5体）
Task(subagent_type="oh-my-claudecode:document-specialist", model="sonnet",
     prompt="Search for: <facet 1>. Use WebSearch and WebFetch. Cite all sources.")
Task(subagent_type="oh-my-claudecode:document-specialist", model="sonnet",
     prompt="Search for: <facet 2>. ...")
    学び: external-context は facet 分解 が肝。「React Server Components」を5体に同じプロンプトで投げても収穫は減衰する。「公式」「実装例」「パフォーマンス」「移行手順」「批判」のように切り分ける。

    ccg — Claude + Codex + Gemini の役割分担

    Execution Protocol：

    1. Decompose Request:
      • Codex prompt: architecture, correctness, backend, risks, test strategy
      • Gemini prompt: UX/content clarity, alternatives, edge-case usability, docs polish
      • Synthesis plan: how to reconcile conflicts
    2. それぞれの結果を Claude が synthesize
    3. 片方が落ちている場合は available provider + Claude のみで継続、欠落視点をリスクとして明記
    学び: 単に「全モデルに同じ質問」ではなく、モデルごとに役割を分けて聞く。Codex はバックエンド寄り、Gemini はユーザ体験寄りの返答を返す傾向があるから、prompt を分けるのが効く。

    OMComc-plan --consensus複数視点のレビューアが計画を批評

    「セッションを跨いで何を残すか」のスキル群。種類が多いのは「何を残すか」の選択肢が複数あるから。

    wiki — 永続知識ベース（Karpathy型）

    「全部覚える」より「4種のうちどれに該当するか判定する」を本体にしたほうが、後で何度も読み返すコストが下がる。

    • Characters (캐릭터): 감정궤도 / 태도 / 대사톤 / 화법

    D. 検証・QA・分析系

    「コードや出力が信頼できるか」を測るスキル群。evidence 主義 が共通。

    verify — 最小例（38行）

    本体は4行：

    ## Rules
- Do not say a change is complete without evidence.
- If a check fails, include the failure clearly.
- If no realistic verification path exists, say that explicitly instead of bluffing.
- Prefer concise evidence summaries over noisy logs.

    bluffing 禁止 の1行が効く。短くても LLM の挙動を縛れる。

    visual-verdict — スクショ vs リファレンス比較

    UI のリグレッションテストで「人間に何を見せれば判断できるか」を構造化する Visual QA verdict。

    ai-slop-cleaner — 削除ファーストのレビュー

    /oh-my-claudecode:ai-slop-cleaner <target>
/oh-my-claudecode:ai-slop-cleaner <target> --review     # レビューだけ
/oh-my-claudecode:ai-slop-cleaner <file-a> <file-b>

    Ralph から呼ばれた場合は Ralph のセッションで変更されたファイルだけを対象にして、Ralph に戻る（post-cleanup regression verification）。

    学び: 「AI が書いたコードを掃除する」は、削除を default にすると過剰実装が消える。リライトより削除が安全。

    github-triage — 読み取り専用の徹底ぶり

    Anti-Patterns（重大度付き）：

    ViolationSeverity
    ANY GitHub mutation (comment/close/merge/review/label/edit)CRITICAL
    Claim without permalinkCRITICAL
    Using category other than quickCRITICAL
    Batching multiple items into one taskCRITICAL
    run_in_background=falseCRITICAL
    git checkout on PR branchCRITICAL
    Guessing without codebase evidenceHIGH
    Not writing report to {REPORT_DIR}HIGH
    Using branch name instead of commit SHA in permalinkHIGH
    学び: アンチパターン表で 重大度を明示 するのは強い。「これをやると重大」を本文に書くと、LLM は実行前に該当しないか自己点検する。

    pre-publish-review — 16 エージェントゲート

    仕組み：

    1. /get-unpublished-changes で前回リリース以降の変更を全列挙
    2. 最大10体の ultrabrain エージェントが per-change 深堀り分析
    3. /review-work（5体）で全体的レビュー
    4. 1 oracle が総合判定

    トリガー語の網羅性が特徴的：pre-publish review, review before publish, release review, ready to publish?, can I publish?, safe to publish など、命令形だけでなく 疑問形 や 形容詞 + 名詞 も入れている。

    E. ブラウザ自動化・UI/UX系（OpenCode 系のみ）

    OMC にはないカテゴリ。OpenCode 系の builtin として 3 本：

    agent-browserCLI 型のブラウザ自動化（コマンド毎回起動・状態を持たない）
    dev-browser永続セッション付きブラウザ自動化（ログイン状態など保持）
    frontend-ui-uxデザインモックなしで UI/UX を作る「デザイナー出身デベロッパー」

    開発中のテスト一発 → agent-browser。ログインしてから複数操作 → dev-browser。UI コードを書きたい → frontend-ui-ux。

    git-master の最小コミット数公式

    他のスキルにも応用できる：

    FORMULA: min_commits = ceil(file_count / 3)

 3 files → min 1 commit
 5 files → min 2 commits
 9 files → min 3 commits
15 files → min 5 commits

If your planned commit count < min_commits → WRONG. SPLIT MORE.

    LLM は「まとめてコミット」しがち。式と表で抑え込む のは、抽象的な指示より強い。

    H. 設計思想のサマリ

    軸OMCOpenAgentOpenCode (fork)
    哲学汎用基盤の合成特定ユースケース専用の大砲日常運転の最小セット
    自律実行autopilot / ralph / ultrawork の3層work-with-pr 専用ループなし
    計画系同意形成型（plan / deep-interview / ccg）敵対批判型（hyperplan）なし
    QAverify / visual-verdict / ai-slop-cleanergithub-triage / pre-publish-review (16体)/remove-deadcode
    知識管理wiki / remember / writer-memoryなしなし
    ブラウザなしなしagent-browser / dev-browser
    Gitreleaseなしgit-master / /publish / etc.
    セットアップ多数なしなし
    示唆: OMC は「インフラ的に重武装」、OpenAgent は「攻撃的・批判的」、OpenCode は「足元固め」。自分でスキルを作るなら、目的に合わせて3つの思想のうちどれを採るか先に決める と設計がブレない。
    • 日常運転を楽にしたい → OpenCode 寄り（短く具体的）
    • レビューや判断の質を上げたい → OpenAgent 寄り（敵対多体、ガード強め）
    • 長時間ループを安全に回したい → OMC 寄り（層分け、評価器契約、cancel）

    まとめ

    • SKILL.md は「LLM が自動的に呼ぶプロンプトファイル」。description が全て。
    • 長いスキルほど <Use_When> / <Do_Not_Use_When> / STEP 0: REGISTER TODO LIST / 数値しきい値 / 状態機械 を入れる。
    • ガードは具体的に。MUST NEVER CRITICAL + 偽陽性ガードのリストで LLM の判断を抑え込む。
    • サブエージェント委譲は READ-ONLY + WRITE REPORT ONLY + EVIDENCE REQUIRED の3点セットで運用。
    • 出力先（artifact path）は最初に決める。/tmp/{datetime}/ や .omc/specs/ のように パスが宣言される ことでスキル間連携が成立する。
    • 自作の判断は Skillify Quality Gate の3問だけで十分。3問とも YES なら作る、それ以外はドキュメントに書く。
    • universal に書くなら frontmatter は name + description だけにし、本文は素の Markdown + 標準ツール名で書く。

    次の一歩は、「これから自分用に書きたいスキルのアイデア」から1本（たぶん llm-wiki-page が一番 Skillify 3問通る）を最小版（30〜50行）で起こす。
