アーキテクチャ Lv2 — アプリ構造完全解説

レイヤード/ヘキサゴナル/Clean Architecture/DDD（戦略+戦術）を Before/After のフォルダ構成+TypeScript実装+SVG図解で1から学べる詳細解説。シリーズ全5回の第2回。

目次

    TL;DR

    • Lv2の到達点: 1アプリケーションを「変更しやすい構造」に組める。ビジネスロジックとインフラの境界を引ける
    • 4つのアーキは思想の発展形: レイヤード → ヘキサゴナル → Clean Architecture → DDD
    • 共通する黄金ルール: 依存方向は外→内のみ。Domain（業務ルール）が一番内側、フレームワークやDBに依存しない
    • DDDは戦略+戦術の2層: 戦略=境界の引き方（Bounded Context）、戦術=Context内の構成要素（Entity/VO/Aggregate）
    • 過剰設計に注意: 個人プロト/PoCではClean Archは過剰。チーム3人以上+長期保守+テスト重視で初めて費用対効果が出る

    このシリーズの位置づけ

    シリーズ第2回。Lv1（基礎原則）で身につけたSOLIDを土台に、複数クラス・複数モジュールをどう組み立てるか（=アプリケーション構造）を扱う。

    導入 — なぜアプリ構造を設計するのか

    SOLIDで個別のクラス品質を上げても、「フォルダの並べ方」「依存の向き」がぐちゃぐちゃだと結局メンテ困難になる。アプリ構造の設計は 「規模が大きくなった時に崩壊しないための地図」。

    1. レイヤードアーキテクチャ — 最も基本

    「Presentation → Application → Domain → Infrastructure」の4層に分け、上から下へ依存する。最古典かつ最も学習コストが低い。

    各レイヤーの責務とコード例

    // Domain層 — 業務ルール（フレームワーク非依存）
class Order {
  constructor(public id: string, public items: OrderItem[]) {}
  total(): number {
    return this.items.reduce((s, i) => s + i.price * i.qty, 0);
  }
  canCancel(): boolean {
    return this.status === "pending";
  }
}

// Application層 — ユースケース調整
class PlaceOrderUseCase {
  constructor(private repo: OrderRepository, private mailer: Mailer) {}

  async execute(input: PlaceOrderInput): Promise<Order> {
    const order = new Order(generateId(), input.items);
    if (order.total() < 100) throw new Error("最低金額未満");
    await this.repo.save(order);
    await this.mailer.sendConfirmation(order);
    return order;
  }
}

// Presentation層 — HTTP受け口
app.post("/orders", async (req, res) => {
  const useCase = new PlaceOrderUseCase(orderRepo, mailer);
  const order = await useCase.execute(req.body);
  res.json(order);
});

// Infrastructure層 — DB実装
class PostgresOrderRepository implements OrderRepository {
  async save(order: Order) {
    await db.query("INSERT INTO orders ...", [...]);
  }
}

    「貧血ドメインモデル」の罠

    注意: レイヤードでよくある失敗が「貧血ドメインモデル」。

    Entityがgetter/setterだけになり、ビジネスロジックがすべてApplication/Serviceに流れ込む現象。結果として Service が肥大化し、ドメインの意味が消える。

    対策: ロジックは可能な限り Entity 内に置く（order.canCancel()、user.changeEmail() 等）。

    2. ヘキサゴナル / ポート＆アダプター

    別名「Ports & Adapters」。アリスター・コックバーン提唱。ビジネスロジックを中心に置き、外部接続はすべてアダプターで隔離する思想。

    ポートとアダプターの実装

    // Port（インターフェース） — Domain側で定義
interface OrderRepository {
  save(order: Order): Promise<void>;
  findById(id: string): Promise<Order | null>;
}

interface NotificationService {
  notify(userId: string, message: string): Promise<void>;
}

// Domain Core（DBもメールも知らない、Portだけ知る）
class OrderService {
  constructor(
    private repo: OrderRepository,
    private notifier: NotificationService,
  ) {}

  async place(order: Order) {
    order.validate();
    await this.repo.save(order);
    await this.notifier.notify(order.userId, `Order ${order.id} placed`);
  }
}

// Outbound Adapter（具体実装）
class PostgresOrderRepository implements OrderRepository {
  async save(order: Order) { /* SQL */ }
  async findById(id: string) { /* SQL */ }
}

class SendGridNotifier implements NotificationService {
  async notify(userId: string, msg: string) { /* SendGrid API */ }
}

// Inbound Adapter（HTTP / CLI / イベントなどを Domain に橋渡し）
app.post("/orders", async (req, res) => {
  const service = new OrderService(
    new PostgresOrderRepository(),
    new SendGridNotifier(),
  );
  await service.place(new Order(req.body));
  res.send("OK");
});

    利点

    • ✅ ビジネスロジックがフレームワーク非依存 — Express、NestJS、何でも差し替え可
    • ✅ テストが容易 — Adapter をモック化して Core だけテスト
    • ✅ 入出力を後から差し替え可能 — REST → GraphQL、PostgreSQL → MongoDB

    3. Clean Architecture（Bobおじさん）

    ヘキサゴナルを発展させた 同心円モデル。Robert C. Martin が2012年に提唱。

    4層の意味

    層役割例
    Entity（最内）業務ルール（最も普遍的）Userの年齢制限、注文の最低金額
    Use Case「何ができるか」注文する、退会する、検索する
    Interface Adapter入出力の橋渡しController / Presenter / Gateway
    Frameworks & Drivers（最外）具体技術Express / React / MongoDB

    フォルダ構成例

    src/
├── entities/                # Lv4: 内側
│   ├── Order.ts
│   ├── User.ts
│   └── Money.ts             # Value Object
│
├── usecases/                # Lv3
│   ├── PlaceOrder.ts
│   ├── CancelOrder.ts
│   └── ports/               # interfaces (Repository, Notifier)
│       ├── OrderRepository.ts
│       └── NotificationService.ts
│
├── adapters/                # Lv2
│   ├── controllers/
│   │   └── OrderController.ts
│   ├── presenters/
│   │   └── OrderPresenter.ts
│   └── repositories/
│       └── PostgresOrderRepository.ts
│
└── infrastructure/          # Lv1: 外側
    ├── express/
    │   └── server.ts
    └── postgres/
        └── client.ts

    黄金ルール

    依存ルール: 外側の層は内側の層を知ってよい。内側は外側を知らない。

    例: UseCase は Entity を import するが、Entity は UseCase を import しない。

    Clean Architecture を採用すべきか？

    状況適用
    個人プロト / PoC / 小規模アプリ❌ 過剰、レイヤードで十分
    チーム3人以上 / 長期保守 / テスト重視✅ 真価を発揮
    業務ロジックが複雑なシステム✅ 推奨
    シンプルなCRUDアプリ❌ 過剰

    4. DDD（Domain-Driven Design）

    Eric Evans が2003年に著書 Domain-Driven Design で提唱。ドメインモデル（業務知識）を中心に据えた設計手法。戦略設計（マクロ）と戦術設計（ミクロ）の2層で構成される。

    戦略設計（マクロ） — 大きな境界の引き方

    Bounded Context（境界づけられたコンテキスト）

    「同じ言葉が違う意味を持つ境界」を引く。

    Ubiquitous Language（ユビキタス言語）

    チーム全員（開発者・PM・ドメインエキスパート）が 同じ言葉を同じ意味で使う。コードと会話の用語を一致させる。

    Subdomain（サブドメイン）

    種別説明戦略
    Core Subdomainビジネスの差別化要素自社開発で力を注ぐ
    Supporting Subdomain必要だが差別化要素ではないシンプルに自社開発 or アウトソース
    Generic Subdomain業界共通（認証、決済等）SaaS / OSS で済ます

    戦術設計（ミクロ） — Context内の構成要素

    Entity vs Value Object

    // Entity — 同一性は「ID」で決まる
class User {
  constructor(
    public readonly id: string,  // ← これが同一性の基準
    public name: string,
    public email: Email,         // Value Object
  ) {}

  // 名前を変えても「同じユーザー」
  changeName(newName: string) {
    this.name = newName;
  }
}

// Value Object — 同一性は「値」で決まる、不変
class Email {
  constructor(public readonly value: string) {
    if (!/@/.test(value)) throw new Error("Invalid email");
  }

  equals(other: Email): boolean {
    return this.value === other.value;  // 値が同じなら同じVO
  }
}

class Money {
  constructor(public readonly amount: number, public readonly currency: string) {}

  add(other: Money): Money {
    if (this.currency !== other.currency) throw new Error("通貨違い");
    return new Money(this.amount + other.amount, this.currency);  // 新しいインスタンス
  }
}

    Aggregate（集約）

    // Aggregate Root: Order
// Order と OrderItem は強く結びついた整合性の単位
class Order {
  private items: OrderItem[] = [];

  constructor(public readonly id: string, public readonly userId: string) {}

  addItem(productId: string, qty: number, price: Money) {
    if (this.items.length >= 100) throw new Error("100品まで");
    this.items.push(new OrderItem(productId, qty, price));
  }

  total(): Money {
    return this.items.reduce(
      (sum, i) => sum.add(i.subtotal()),
      new Money(0, "JPY"),
    );
  }
}

class OrderItem {
  constructor(
    public readonly productId: string,
    public readonly qty: number,
    public readonly price: Money,
  ) {
    if (qty <= 0) throw new Error("数量>0");
  }
  subtotal(): Money {
    return new Money(this.price.amount * this.qty, this.price.currency);
  }
}

    Repository

    // Aggregate単位で永続化する
interface OrderRepository {
  findById(id: string): Promise<Order | null>;
  save(order: Order): Promise<void>;
  findByUser(userId: string): Promise<Order[]>;
}

// 使う側はAggregateを丸ごと取得・保存
const order = await orderRepo.findById("ord_123");
order.addItem("prod_1", 2, new Money(1000, "JPY"));
await orderRepo.save(order);

    Domain Service

    Entity / Value Object に収まらないドメインロジックを置く場所。「複数のAggregateにまたがる処理」などに使う。

    // 「振込」は送信元アカウントと送信先アカウント両方に関係する
class TransferService {
  constructor(private accountRepo: AccountRepository) {}

  async transfer(fromId: string, toId: string, amount: Money) {
    const from = await this.accountRepo.findById(fromId);
    const to   = await this.accountRepo.findById(toId);
    from!.withdraw(amount);
    to!.deposit(amount);
    await this.accountRepo.save(from!);
    await this.accountRepo.save(to!);
  }
}

    Domain Event

    「業務的に意味のある出来事」を表すイベント。Aggregateが発行し、別のContext / Aggregateが反応する。

    interface DomainEvent {
  readonly occurredAt: Date;
  readonly type: string;
}

class OrderPlaced implements DomainEvent {
  readonly type = "OrderPlaced";
  readonly occurredAt = new Date();
  constructor(public readonly orderId: string, public readonly userId: string) {}
}

class Order {
  private events: DomainEvent[] = [];
  place() {
    this.events.push(new OrderPlaced(this.id, this.userId));
  }
  pullEvents(): DomainEvent[] {
    const out = this.events;
    this.events = [];
    return out;
  }
}

// 使う側
const order = new Order(...);
order.place();
const events = order.pullEvents();
events.forEach(e => eventBus.publish(e));

    イベントストーミング — 業務フローを可視化する手法

    付箋の色を分けて業務フローを時系列に並べる手法。Bounded Context の発見にも使える。

    実プロダクト分析: ariadne (takt-runner)

    演習問題

    Q1. 以下のクラスのどこが問題？

    class Order {
  id: string = "";
  items: OrderItem[] = [];
  total: number = 0;
  status: string = "";
  // getterとsetterだけ
  getId() { return this.id; }
  setId(id: string) { this.id = id; }
  getItems() { return this.items; }
  setItems(items: OrderItem[]) { this.items = items; }
}
    解答を見る

    貧血ドメインモデル。データを持つだけで業務ルールが不在。getter/setterだけのクラスはほぼ struct と同じ。

    修正: 業務ルールを Entity 内に置く。例: order.canCancel()、order.calculateTotal()、order.addItem()（数量・上限チェック付き）など。

    Q2. 以下の Service コードを Clean Architecture に変えて

    class OrderService {
  async place(input: any) {
    const result = await mysqlClient.query("INSERT ...", input);
    await sendgridClient.send(input.email, "Welcome");
    return result;
  }
}
    解答を見る 
    // usecase（最内寄り）
interface OrderRepository { save(order: Order): Promise<void>; }
interface Mailer { send(to: string, body: string): Promise<void>; }

class PlaceOrderUseCase {
  constructor(
    private repo: OrderRepository,
    private mailer: Mailer,
  ) {}
  async execute(input: PlaceOrderInput) {
    const order = new Order(input);
    await this.repo.save(order);
    await this.mailer.send(input.email, "Welcome");
    return order;
  }
}

// adapters（外側）
class MySQLOrderRepository implements OrderRepository {
  async save(o: Order) { await mysqlClient.query("INSERT ...", o); }
}
class SendGridMailer implements Mailer {
  async send(to: string, body: string) { await sendgridClient.send(to, body); }
}

    UseCase は具体クラスを知らない（DI で受け取る）。テストでは MockRepository / MockMailer を渡せる。

    Q3. 以下は同じ Bounded Context にすべき？

    • EC機能（カタログ・注文・在庫）
    • 会員機能（登録・退会・プロフィール）
    • 分析機能（売上ダッシュボード・コホート分析）
    解答を見る

    分けるべき。各々のドメイン用語が違う：

    • EC: 「商品」「注文」「在庫」が中心
    • 会員: 「ユーザー」「セッション」「プロファイル」が中心
    • 分析: 「KPI」「コホート」「ファネル」が中心

    同じ「ユーザー」も意味が違う（EC=購入者 / 会員=登録者 / 分析=セグメント）→ 別Bounded Context。

    まとめ — Lv2の要点

    1. 4つのアーキは思想の発展形: レイヤード → ヘキサゴナル → Clean → DDD
    2. 共通の黄金ルール: 依存方向は 外 → 内。Domain が最内
    3. レイヤード: 4層（Presentation/Application/Domain/Infrastructure）。学習コスト低、過剰設計リスクも低
    4. ヘキサゴナル: Port&Adapterで外部接続を隔離。テスト容易性◎
    5. Clean Architecture: 同心円。チーム3人以上+長期保守で真価
    6. DDD戦略設計: Bounded Context, Ubiquitous Language, Subdomain
    7. DDD戦術設計: Entity / VO / Aggregate / Repository / Domain Service / Domain Event
    8. 貧血ドメインモデルに注意: ロジックはEntityに置く
    9. 過剰設計に注意: 規模に合わせて選ぶ
