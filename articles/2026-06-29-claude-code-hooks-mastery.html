Claude Code Hooks 完全マスターガイド

全 13 Hook イベント・Sub-Agents・Team Validation・Status Lines・Output Styles を体系的に学ぶ。Hook はプロンプトで「お願い」する代わりに、確定的な制御をコードとして書く仕組みだ。

目次

    TL;DR

    • Hook は Claude Code のライフサイクル 13 地点で自動実行されるシェルコマンド。LLM の判断に依存せず確定的に動く。
    • exit code 2 でツール実行をブロック、stdout に JSON を出せば承認・拒否・コンテキスト追加が細かく制御できる。
    • UV シングルファイルスクリプトを使うと Hook ごとに依存を宣言でき、venv 管理が不要になる。
    • Sub-Agent ファイル（.claude/agents/*.md）はシステムプロンプト。Builder + Validator のペアでコード品質を自動担保できる。
    • Status Lines v1〜v9 でコスト・トークン・コンテキスト使用量をリアルタイム表示、Output Styles でレスポンス形式を切り替えられる。

    Claude Code Hooks とは

    Claude Code Hooks はセッション中の特定タイミングで自動実行されるユーザー定義のシェルコマンドだ。

    公式ドキュメント: Claude Code Hooks
    LLM に「毎回フォーマットしてください」と頼む代わりに Hook に書けば、必ず実行される。これが最大の利点だ。

    何ができるか

    • セキュリティ: rm -rf や .env アクセスをブロック
    • 品質管理: .py 編集後に ruff / ty を自動実行し lint エラーでブロック
    • コンテキスト注入: プロンプト送信前に git status・issue を自動ロード
    • 音声通知: 完了時に TTS で「Work complete!」を読み上げ
    • ログ収集: 全ツール実行を JSON でログ蓄積
    注意: Hook はシステム上で任意のコマンドを自動実行する。設定前にコードを必ず確認すること。悪意のある Hook はデータ漏洩の原因になりうる。

    Hook ライフサイクル — 13 イベント

    セッション中、Hook は以下の順で発火する。同一イベントにマッチする Hook はすべて並列実行される。

    イベント別チートシート

    イベント 発火タイミング ブロック 主な用途
    UserPromptSubmitプロンプト送信直後✅ 可バリデーション、コンテキスト注入
    PreToolUseツール実行前✅ 可危険コマンド防止、自動承認
    PermissionRequest権限ダイアログ表示時✅ 可読み取り専用の自動許可
    PostToolUseツール成功後❌（既実行）lint・フォーマット・ログ
    PostToolUseFailureツール失敗後❌エラーログ記録
    Notification通知送信時（非同期）❌デスクトップ通知・TTS
    Stop応答完了時✅ 可TTS 通知・タスク完了チェック
    SubagentStartSub-Agent スポーン時❌起動ログ
    SubagentStopSub-Agent 完了時✅ 可完了 TTS
    PreCompactコンテキスト圧縮前❌トランスクリプトバックアップ
    SessionStartセッション開始・再開時❌git status ロード・env 設定
    SessionEndセッション終了時❌クリーンアップ・ログ
    Setup--init/--maintenance 時のみ❌依存インストール・マイグレーション
    Tip: SessionStart と Setup の違いは頻度。SessionStart は毎回実行されるので軽量に保つ。Setup は --init / --maintenance フラグ指定時のみ動き、重い初期化処理に向く。

    settings.json で Hook を設定する

    Hook は .claude/settings.json（プロジェクト）または ~/.claude/settings.json（ユーザー全体）に記述する。

    {
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command", "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/pre_tool_use.py" }]
      },
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command", "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/validators/ruff_validator.py", "timeout": 30 }]
      }
    ],
    "UserPromptSubmit": [
      {
        "hooks": [{ "type": "command", "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/user_prompt_submit.py --log-only" }]
      }
    ],
    "Stop": [
      {
        "hooks": [{ "type": "command", "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/stop.py --chat" }]
      }
    ]
  }
}

    matcher の書き方

    • "Bash" — 完全一致（Bash ツールのみ）
    • "Edit|Write" — 正規表現 OR
    • "Notebook.*" — 前方一致
    • "*" または "" — 全ツール
    • "mcp__memory__.*" — MCP ツールのパターン
    matcher が使えるのは PreToolUse・PermissionRequest・PostToolUse・Notification・PreCompact・SessionStart のみ。UserPromptSubmit・Stop 等は matcher 不要。
    Note: Hook パスには $CLAUDE_PROJECT_DIR を使うこと。Claude がどのサブディレクトリにいても正しく解決される。

    フロー制御 — exit code と JSON 出力

    exit code による基本制御

    exit code意味stdoutstderr
    0成功UserPromptSubmit/SessionStart はコンテキストに追加。他は verbose モード(Ctrl+O)無視
    2ブロッキングエラー無視（JSON も処理されない）Claude またはユーザーにフィード
    その他非ブロッキングエラー無視verbose モードでユーザーに表示

    exit 2 の実装例 — rm -rf をブロック

    #!/usr/bin/env -S uv run --script
# /// script
# requires-python = ">=3.8"
# ///
import json, sys, re

data = json.load(sys.stdin)
tool_name = data.get("tool_name", "")
tool_input = data.get("tool_input", {})

# .env ファイルアクセスをブロック
if tool_name in ["Read", "Edit", "Write"]:
    path = tool_input.get("file_path", "")
    if ".env" in path and not path.endswith(".env.sample"):
        print("BLOCKED: .env ファイルへのアクセスを禁止", file=sys.stderr)
        sys.exit(2)  # exit 2 → Claude にエラーとしてフィード

# rm -rf をブロック
if tool_name == "Bash":
    cmd = tool_input.get("command", "")
    if re.search(r"\brm\s+.*-[a-z]*r[a-z]*f", cmd.lower()):
        print("BLOCKED: 危険な rm -rf コマンドを防止", file=sys.stderr)
        sys.exit(2)

sys.exit(0)

    JSON 出力による高度な制御

    exit 0 のときに stdout に JSON を出力すると、より細かい制御が可能になる。

    PreToolUse — ツール実行の承認・拒否・入力書き換え 
    {
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow",
    "permissionDecisionReason": "自動承認: ドキュメントファイル",
    "updatedInput": { "command": "npm run lint" },
    "additionalContext": "本番環境です。慎重に操作してください。"
  }
}

    permissionDecision は "allow"（権限スキップ） / "deny"（ブロック） / "ask"（ユーザーに確認）の3択。

    UserPromptSubmit — プロンプトのブロックとコンテキスト追加 
    import json, sys, re, datetime

data = json.load(sys.stdin)
prompt = data.get("prompt", "")

# シークレットが含まれていたらブロック
if re.search(r"(?i)\b(password|secret|key|token)\s*[:=]", prompt):
    output = {
        "decision": "block",
        "reason": "セキュリティ違反: シークレットが含まれています。"
    }
    print(json.dumps(output))
    sys.exit(0)  # exit 0 で JSON を処理させる（exit 2 では JSON は無視される）

# コンテキストを追加（stdout に書くだけでいい）
print(f"現在時刻: {datetime.datetime.now()}")
sys.exit(0)
    Stop — テスト失敗時に停止をブロック 
    {
  "decision": "block",
  "reason": "テストが失敗しています。失敗しているテストを修正してから完了してください。"
}
    注意: Stop Hook では stop_hook_active フラグを必ず確認すること。このフラグが true の時はすでに Stop Hook が実行中なので、もう一度ブロックすると無限ループになる。

    フロー制御の優先順位

    1. "continue": false — 全制御より最優先
    2. "decision": "block" — Hook 固有のブロック
    3. exit code 2 — stderr を Claude にフィード
    4. その他の exit code — 非ブロッキングエラー

    プロンプト型 Hook（type: "prompt"）

    bash コマンドの代わりに LLM（Haiku）に判断させる方式も選べる。コンテキストを理解した動的な判断が必要な場合に有効。

    {
  "hooks": {
    "Stop": [{
      "hooks": [{
        "type": "prompt",
        "prompt": "Claude が停止すべきか評価してください。Context: $ARGUMENTS\n全タスクが完了していますか？\nJSON で回答: {\"ok\": true} or {\"ok\": false, \"reason\": \"説明\"}",
        "timeout": 30
      }]
    }]
  }
}
    Note: プロンプト型 Hook は API 呼び出しが発生するため bash Hook より遅い。確定的なルールは bash、コンテキスト理解が必要な判断はプロンプト型、と使い分けるとよい。

    UV シングルファイルスクリプトアーキテクチャ

    このリポジトリのすべての Hook は UV シングルファイルスクリプトで書かれている。Hook ファイル自体に依存関係を宣言でき、venv 管理が一切不要になる。

    #!/usr/bin/env -S uv run --script
# /// script
# requires-python = ">=3.11"
# dependencies = [
#     "python-dotenv",
#     "anthropic",
#     "openai",
# ]
# ///

# ↑ これだけで UV が依存を自動インストール・実行する
import anthropic
from dotenv import load_dotenv

    Hook から呼び出す際は uv run $CLAUDE_PROJECT_DIR/.claude/hooks/xxx.py とするだけでよい。キャッシュが効くため 2 回目以降は瞬時に起動する。

    Sub-Agents — 専門化された AI アシスタント

    Sub-Agent は特定タスクに特化した AI で、プライマリ Claude がタスクを委譲する。独立したコンテキストウィンドウと制限されたツール権限を持つ。

    最重要: .claude/agents/*.md の内容はシステムプロンプトであり、ユーザープロンプトではない。Sub-Agent はあなたに直接応答するのではなく、プライマリ Agent の指示に応答する。

    情報フロー

    Agent ファイルの書き方

    ---
name: crypto-market-agent
description: "仮想通貨市場を分析する。「PROACTIVE」と言われたら自動使用する"
tools: WebSearch, Bash, Read    # 省略すると全ツール継承
color: Cyan                      # ターミナルでの視覚識別
model: haiku                     # haiku | sonnet | opus
---

# あなたは仮想通貨市場アナリストです

## 指示
1. 市場データを収集する
2. トレンドを分析する
3. 結果をプライマリ Agent に報告する

## 報告フォーマット
JSON 形式で: {market_trend, top_movers, recommendation}
    description フィールドがプライマリ Agent が「いつ・なぜ使うか」を判断する唯一の手がかり。「PROACTIVE に使う」「〇〇と言われたら」などトリガーワードを入れると自動委譲されやすい。

    Meta-Agent — エージェントを作るエージェント

    .claude/agents/meta-agent.md は新しい Sub-Agent ファイルを自動生成する特殊な Agent だ。「テストを実行して失敗を修正する Sub-Agent を作って」と伝えるだけで、ベストプラクティス準拠の Agent ファイルが生成される。「Build the thing that builds the thing」の原則で開発速度を指数関数的に向上させる。

    Team-Based Validation System

    /plan_w_team コマンドは Builder + Validator のペアでコード品質を保証する。

    Agentファイルツール自動検証役割
    Builderteam/builder.md全ツールRuff + Ty（.py）実装
    Validatorteam/validator.md読み取り専用なし検証

    /plan_w_team の 3 つの特徴

    1. 自己バリデーション: Stop Hook でプランファイルの必須セクションを自動確認。不足があれば Agent が継続して修正する。
    2. Agent オーケストレーション: Task システムで Builder/Validator を並列調整。bash sleep ループ不要。
    3. テンプレート生成: 確定フォーマットのプランを出力する「テンプレートメタプロンプト」。成果物が予測可能になる。
    PostToolUse バリデーターの仕組み

    Builder が .py ファイルを書くと PostToolUse Hook が自動発火し、ruff と ty を実行する。lint エラーや型エラーがあれば decision: "block" で Claude にフィードバックし、Claude 自身が修正するループが回る。

    {
  "PostToolUse": [{
    "matcher": "Write|Edit",
    "hooks": [
      { "type": "command", "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/validators/ruff_validator.py" },
      { "type": "command", "command": "uv run $CLAUDE_PROJECT_DIR/.claude/hooks/validators/ty_validator.py" }
    ]
  }]
}

    Status Lines と Output Styles

    Custom Status Lines（v1〜v9）

    ターミナル下部にリアルタイム情報を表示する。.claude/status_lines/ に配置し settings.json の "statusLine" で指定する。

    バージョン主な追加機能
    v1Git ブランチ・ディレクトリ・モデル（基本 MVP）
    v2最新プロンプト（250文字）・タスクタイプ別カラーコーディング
    v3Agent 自動命名（Phoenix, Sage 等）・直近 3 プロンプト
    v4カスタムメタデータ（/update_status_line で追加）
    v5コストトラッキング（$）・行変更数（+/-）
    v6コンテキストウィンドウ使用量バーと残りトークン
    v7セッション継続時間タイマー
    v8入出力トークン数・キャッシュ作成/読み取り統計
    v9Powerline ミニマルスタイル（セグメント区切り付き）

    Output Styles

    .claude/output-styles/*.md に置いたスタイルは /output-style [名前] で切り替えられる。Claude のレスポンス形式が変わる。

    • genui ⭐ — スタイル込みの美しい HTML を生成。ブラウザで即プレビュー可能
    • table-based — 全情報をマークダウンテーブルで整理
    • ultra-concise — 最小限の言葉で最大の情報
    • tts-summary — タスク完了を TTS でアナウンス
    • 他: yaml-structured / bullet-points / html-structured / markdown-focused

    まとめ

    • Hook は「プロンプトでお願い」から「コードとして確定的に実行」への転換。これが Agentic Engineering の核心。
    • exit code 2 でシンプルにブロック、JSON 出力で承認・拒否・コンテキスト追加を細かく制御できる。
    • UV シングルファイルスクリプトで Hook を書くと、依存管理が自動化されポータビリティが高まる。
    • Sub-Agent の description フィールドは「いつ使うか」を左右する最重要フィールド。トリガーワードを入れる。
    • Builder + Validator パターンと PostToolUse の lint Hook を組み合わせると、コード品質チェックが自動化される。
    • Status Lines v1〜v9 はコスト・トークン・コンテキスト使用量を可視化し、長時間セッションの管理が楽になる。
