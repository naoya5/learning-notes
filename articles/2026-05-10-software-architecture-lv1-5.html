ソフトウェアアーキテクチャ完全学習ガイド — Lv1〜Lv5

SOLIDからAIエージェント設計まで、5レベルで体系化したアーキテクチャ学習ガイド。AIに正しく指示を出すための「設計の地図」を頭に作る。

目次

    TL;DR

    • AI時代こそ「設計の地図」 — AIは実装は得意、設計判断は苦手。指示者の頭の中の解像度がそのまま出力品質になる
    • Lv1〜Lv5の段階構造 — コード(Lv1)→アプリ構造(Lv2)→システム全体(Lv3)→分散・大規模(Lv4)→AI時代(Lv5)
    • 個人開発はLv1+2+5 必修。チーム開発はLv3まで。エンタープライズはLv4以降も
    • Lv5「Harness Engineering」 がAIプロダクト運用の核心。観測・評価・安全装置・コスト制御の7軸
    • 実例分析つき — ariadne(takt-runner) と SkillHub の実コードで「どう適用するか」を見せる

    はじめに — なぜ今アーキテクチャか

    AIが実装を肩代わりする時代、エンジニアの価値は「何をどう作るかを設計する力」へシフトしている。設計が曖昧 = AIの出力も曖昧。設計の解像度がそのままプロダクトの品質に直結する。

    Tip: このガイドのゴールは「プロダクトを作る人」ではなく「プロダクトを設計してAIに任せる人」になること。

    全体マップ — 5レベルのピラミッド

    下から積み上げる5階層。各レベルは独立ではなく、下位レベルが上位レベルの土台になる。

    例: Single Responsibility 違反 → 修正

    // ❌ 1クラスに3つの責務
class User {
  save() { /* DB */ }
  sendEmail() { /* メール */ }
  validate() { /* バリデーション */ }
}

// ✅ 責務を分離
class User { /* データだけ */ }
class UserRepository { save(user: User) {} }
class EmailService { send(user: User) {} }
class UserValidator { validate(user: User) {} }

    ADR（Architecture Decision Records）

    「なぜその設計を選んだか」を残す軽量ドキュメント。1決定 = 1ファイル、連番管理。

    # ADR-001: SQLite を採用する

## Status
Accepted (2026-05-10)

## Decision
SQLite を採用する。

## Consequences
- ✅ Tauri から native アクセス可能
- ✅ FTS5 / sqlite-vec が使える
- ❌ ブラウザ単体では動かない

    まとめ — アーキテクトの10原則

    1. 問題が先、パターンが後
    2. トレードオフを言語化する
    3. 可逆性を最大化する
    4. YAGNI — 今いらない機能は作らない
    5. 抽象化はコスト
    6. 境界を引く — Bounded Context、レイヤー、サービス境界
    7. 依存方向を制御する — 外→内のみ
    8. 観測性を最初から組み込む
    9. 失敗を前提に設計する
    10. コンウェイの法則を意識する
    一周して気付くこと: AIに正しく指示出すには、自分の頭の中の「設計の地図」の解像度が全て。Lv1〜Lv5 = AIへの指示書を書ける言語の獲得。
