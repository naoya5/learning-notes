モデルからハーネスへ：13週間の学習ログ（W12–W24）

2026年春・13週間で学んだことの総集編。通底するのは「競争の主戦場はモデルからハーネスへ移った」という一本の軸。

目次

    TL;DR

    • 2026年 W12〜W24 の学びは、バラバラの点ではなく「モデルの賢さより、それを動かすハーネス（ツール・ループ・メモリ・スキル）が成果の天井を決める」という一本の線に収束した。
    • エージェントの自己進化は L1（スキル）→ L2（スキル+重み）→ L3（改善プロセス自体） で整理でき、メモリ設計は「何を覚えるか」より「何を保存しないか」が難所だった。
    • ソフトウェアアーキテクチャは規模を問わず「依存を外→内に向け、滅多に変わらないビジネスルールを守る」が共通原理。CQS・parse・マイクロサービスの「4つの柱」まで一本で繋がった。
    • LLM 内部も KV キャッシュ最適化（Paged Attention / TurboQuant）からエンコーダーレス（Gemma4）まで、推論を「使う分だけ・無駄なく」に寄せる方向で進化していた。
    • 休息週（W16）が教えてくれた最大の学び：受動的インプットは自動で回るが、能動的アウトプット（自分の言葉化）は意識しないと止まる。

    13週間を貫いた一本の軸

    この記事は、2026年 W12〜W24（3/16〜6/14）の週次レポートを再構成した学習ログの総集編だ。

    元データは毎週の Weekly レポート13本（W21・W25は欠番）。日次の点を週でまとめ、さらに本記事で13週ぶんをテーマ別に束ね直した。
    振り返ると、最初は「AIエージェント基盤を片っ端から調査する」ところから始まり、ハーネス・メモリ・アーキテクチャへと掘り下げ、最後は業界そのものの地殻変動を肌で感じる、という流れになっていた。

    個別のトピックは多岐にわたるのに、読み返すと驚くほど一つの主張に収束する —— 「賢いモデルを手に入れることがゴールではない。モデルを動かす"周辺"（ハーネス）の設計こそが、これからの成果を決める」。 以下、時系列の羅列ではなく、この軸に沿ってテーマ別に再構成する。

    Tip: 各章は独立して読める。気になる章だけつまんでも、全体を通して「モデル→ハーネス」という地殻変動の一枚絵が見えるように並べている。

    ハーネスエンジニアリング — 天井を決めるのはモデルではない

    13週間でいちばん太い幹がこれだった。きっかけは Anthropic の 「Harness design for long-running application development」 の精読（W14）と、論文 Scaling the Harness in Agentic AI / Harness-Bench（W22）。 ここで得た中心命題は明快だ —— agentic AI の次のボトルネックは「強いモデル」ではなく harness 設計（memory・context constructor・skill routing・governance）にある。

    さらに刺さったのが HyperAgents の 「制約は改善の天井になる」という原則（W13）。 初期ハーネスを意図的に最小にしておくほど、進化の余地が最大化される。 実際 MiniMax M2.7 は、エージェント自身がハーネスを自律的に書き換え、 100回ループ・人間介入ゼロで 30% の性能向上を、再トレーニングなしで達成していた（W14）。

    W24 では視点がさらに進み、「プロンプトを書く時代から、ループを書く時代へ」という言い方で腑に落ちた。 ゴール保持とループ操舵をシステム側に作り込む「ループエンジニアリング」、 回すほど壊れにくくなる self-repairing harness —— 関心のど真ん中が、単発の生成からループの設計へと移っていった。

    Note: 「強いモデル＝信頼できる実行」ではない。能力の高いエージェントがなぜ失敗するのか、を言語化できたのが講義 Learn Harness Engineering（W22）の収穫だった。

    エージェントの自己進化とスキル — L1 / L2 / L3 で世界を整理する

    自己進化を語る論文は多いが、「何が変わるのか」の射程で3段階に整理すると一気に見通しがよくなった（W13）。

    スキルは「3段階ロード」で読み込まれる

    Agent Skills（SKILL.md）の仕組みも整理できた（W14）。 Tier 1（起動時 YAML のみ・〜100 token）→ Tier 2（マッチ時にフル SKILL.md・〜5000 token）→ Tier 3（動的ロード）という段階構成で、 必要になるまで重い本体を読み込まない。Anthropic が 2025年12月に標準化し、30以上のエージェント製品が採用済みだ。

    ただしスキルが増えると、どうセマンティッククラスタして検索するかが次の課題になる。 関連して InsForge の MCPMark V2 の反直観的な結果が効いた（W19）—— 賢い Claude ほどトークン使用量が 54% 増える。 賢いモデルはバックエンド状態を確認したがるので、スキルを「狭いスコープ」に切ると認知負荷が下がる。

    スキル化のボトルネックは「手順の言語化（SOP化）」。作業を録画 → Gemini 等で解析 → Claude Code で SKILL.md 化、という流れがこの壁と相性が良かった（W18）。

    エージェントのメモリ設計 — 「何を保存しないか」が難所

    メモリは13週を通じて何度も戻ってきたテーマだ。実装は MemGPT / Mem0 / LangMem / Letta / Zep / Graphiti / Cognee、 論文系は A-MEM / HippoRAG / LongMemEval / AgeMem などを横断整理した（W20〜W24）。 比較軸を持てたことで「自分の運用がどの設計に近いか」を言葉にできるようになった。

    いちばんの転換は PROJECTMEM 論文の精読（W24）。 「何を覚えるかより、何を保存しないか」「失敗 attempt を first-class event として扱う」という Memory-as-Governance の設計思想だ。メモリを「成功の記録」ではなく「ガバナンスの装置」として捉え直す視点が新しかった。

    手前の低レイヤでは、VectorDB 不要・3並列エージェントが 異なる観点（事実・文脈・時系列）で検索する ASMR（W13）や、 長時間調査がコンテキスト制限でクラッシュする問題に Compaction（アクティブメモリ圧縮）+ Memory（再利用可能ハビット保存）で対処する OpenAI の論文（W19）が実装の手触りを与えてくれた。

    Tip: コミュニティの収束パターンは「ローカル SQLite + 常駐 markdown」（Hermes）。派手な VectorDB より、素朴で監査しやすい構成に落ち着きつつある。

    LLM の内部と推論最適化 — 「使う分だけ・無駄なく」

    モデルの中身にも解像度を上げた13週間だった。通底するのは 「事前に大きく確保するのをやめ、使う分だけ無駄なく割り当てる」という発想だ。

    同じ方向の進化が、推論まわりに次々現れた：

    • TurboQuant（W13）— KV キャッシュを再学習なしで6倍圧縮、精度ロスほぼゼロ、H100 で最大8倍速。
    • Attention Residuals（AttnRes）（W13）— 残差接続の重要度希釈問題を各層の softmax attention で解決し、GPQA で最大 +7.5pt。
    • SubQ（W19）— サブクワドラティック Sparse Attention で 12M トークンを線形スケール処理。FlashAttention の52倍速・コンピュート63%減。

    極めつけが Gemma4 12B のエンコーダーレス設計（W23）。 従来は音声・画像用の専用エンコーダーがレイテンシ要因だった。Gemma4 はこれを取り払い、 画像はパッチに分割 → 1層の Embedding 層でトークン化 → Positional Encoding で位置情報を保持することで、 アテンションなしでも画像の位置を覚えたまま LLM へ入力できる。

    補足: なぜ「事前確保」は無駄なのか（KVキャッシュの基礎）

    通常の KV キャッシュは、生成しうる最大トークン数ぶんのメモリをあらかじめ確保する。 だが実際の出力長は予測できず、短い応答では確保した領域の大半が遊んでしまう。 Paged Attention はこれを OS の仮想メモリページングになぞらえ、トークンを固定サイズのブロックで管理して 「使う分だけ・必要になったら足す」に切り替える。アナロジーが効いて理解が一気に繋がった。

    ソフトウェアアーキテクチャ — 依存の向きを内側に守る

    W18 以降、O'Reilly『ソフトウェアアーキテクチャの基礎』を軸に設計を体系的に積み上げた。 最初の心構えがよかった —— アーキテクチャ決定で重要なのは知識の「深さ」より「幅」。正解はないので常にトレードオフで考える（W20）。 循環的複雑度は5以下が望ましい、巨大な泥団子は避ける、といった具体の物差しも手に入れた。

    依存性の逆転がすべての背骨

    レイヤード／DDD の責務分割（Repository=DBアクセス、Entity=ビジネスロジック、Service=処理の主体、Controller=外との受付）から、 Hexagonal / Onion / Clean まで比較したが、結局すべては一つの原理に還元された —— 依存は外側（変化が速い DB・フレームワーク）から内側（滅多に変わらないビジネスルール）へ。コアロジックを変化から守る（W23）。

    堅牢なコード: CQS と parse

    コードレベルでは コマンド・クエリ分離（CQS）が効いた（W23）。 状態を変更する処理（コマンド）と内容を取得する処理（クエリ）を別関数に分ける。 「取得のつもりが状態を変えていた」副作用がいちばん危ない。 検証も validate で終わらせず、データを型にはめる parse まで行う。 

    // NG: 取得のつもりが状態を変える（隠れた副作用）
function getNextId(): number {
  return ++this.counter; // 呼ぶたびに counter が増えてしまう
}

// OK: コマンド（変更）とクエリ（取得）を分離する
function currentId(): number {
  return this.counter;   // 読むだけ。副作用なし
}
function advance(): void {
  this.counter++;        // 変更だけ。値は返さない
}

    マイクロサービスの「4つの柱」

    W24 で teach スキルを使い、マイクロサービスを1週間で完走した。 サービス境界 → サービス間通信・Saga・Context Mapping → レジリエンス・可観測性 → システム設計（CAP定理・Saga パターン）という 「4つの柱」を段階的に踏破。教材化しながら学ぶと、定義の暗記ではなく「人に説明できる形」まで落ちた。 同時に得た区別が 「コード設計のアーキテクチャ ≠ 物理設計のアーキテクチャ」（W22）だ。

    注意: マイクロサービスは分散トランザクションやログのトレースが難しく、運用コストが大きい（W23）。「分割すれば良くなる」わけではなく、ここでもトレードオフがすべて。

    開発ワークフローと運用 — レビューループ・小さなコミット・定点観測

    学びは設計論だけでなく、日々の回し方にも及んだ。効いた実践を挙げる：

    • Claude × Codex レビューループ（W18）— Claude にプランを書かせ → Codex で適合性レビュー → Claude が書き直す、を Codex の指摘がゼロになるまで回す。プランが曖昧なフェーズで効く。プラン段階で HTML モックや ASCII ワイヤーフレームを生成させると UI 精度が上がる。
    • CLAUDE.md にリポジトリ種別を明記（W19）— 本番／個人／プロトタイプを書くと、Claude が無駄に厳格にならず作業効率が上がる。前提共有は最重要メタ情報。
    • AM/PM バッチ化と集約アーカイブ（W14）— 情報収集を朝夕2回に束ね、日次ファイルの積み上げより集約ファイルにすると参照・再利用しやすい。second-brain の23:00更新で点を線にする運用が定着した。

    git pre-commit に Claude を仕込むハック（W19）も実用的だった。 .git/hooks/pre-commit に置いて chmod +x するだけで、コミット前に自動レビューが走る： 

    #!/bin/bash
# .git/hooks/pre-commit に置いて chmod +x するだけ
git diff --cached | claude -p "Spot any console.logs, hardcoded secrets, or unfinished TODOs in this diff?"

    設計パターンとしては ツールディスパッチ・パターンが引き出しに加わった（W17）。 ツールをオブジェクトとして登録し、AI が呼んだ名前で実処理へ振り分ける。 if/elif が増えそうな場面をテーブル駆動に置き換える発想で、新ツール追加もエントリを足すだけになる： 

    type Tool = (input: string) => string;

// キー＝ツール名、バリュー＝処理関数として登録する
const tools: Record<string, Tool> = {
  read:   (path)  => readFile(path),
  write:  (path)  => writeFile(path),
  search: (query) => grep(query),
};

// AI が選んだ名前でディスパッチ。分岐の if/elif が消える
function dispatch(name: string, input: string): string {
  const tool = tools[name];
  if (!tool) throw new Error(`unknown tool: ${name}`);
  return tool(input);
}

    業界の地殻変動 — 行レビュー放棄と、非エンジニアの台頭

    13週間は、開発文化そのものが動いた期間でもあった。

    • AI 生成コードの行単位レビュー放棄が本格化（W19）。Simon Willison（Django 共同創設者）が本番コードでも行レビューを諦めたと公言し、Coinbase は非技術者が本番コードをリリースする体制に転換。 「品質の証明は本番での稼働時間だけ」という時代観へ。
    • 非エンジニアの台頭（W24）。Anthropic ハッカソンで上位が非エンジニアだった件から、「現代モデルの性能があれば、コードを書ける人よりドメイン知識を持つ人の方が強い」可能性を実感した。
    • モデルのコモディティ化（W24）。Claude Fable 5 の公開、DeepSeek V4 Pro の恒久75%値下げなど価格競争が加速。競争の主戦場がランタイム／ハーネスへ移りつつある。
    この潮流が自分のスタンスに与えた影響

    モデルが安く強くなるほど、差がつくのは「何を作らせるか」「どう回すか」になる。 だからこの13週間、アプリそのものよりハーネス（スキル・メモリ・ワークフロー）を磨く方に時間を寄せた。 「アプリを作る」より「エージェントにアプリを作らせる足場を作る」—— W14 で言語化したこの転換が、最後まで一貫した行動原理になった。

    メタ学習 — 休息週が教えてくれたこと

    最後は、学び方そのものについての学び。W16 は日記の「学んだこと」が7日とも空白の完全な休息週だった。 この週が逆に教えてくれたのは、「受動的インプット（RSS 自動収集）は回るが、能動的アウトプット（自分の言葉化）は意識しないと止まる」という非対称性だ。 強いインプット週（W15）の直後に消化・実装ターンが来ないと、学びは揮発しかねない。

    対策として効いたのが2つ。ひとつは teach スキルでの教材化（W24）—— 人に説明できる形まで落とすと定着する。もうひとつが、まさにこの learning-notes への HTML 化だ。 「学習結果は Markdown より HTML の方が人間に優しい」という気づき（W19）から、 気になったテーマを HTML 記事にして一箇所に集約する仕組みを作った。この記事自体がその実践でもある。

    Note: インプット量に対して蒸留（自分の主張への言語化）は常に手薄になりがちだった。「集めて終わり」を避け、蒸留のリズムを持つことが、学びを資産に変える唯一の鍵だと13週間で痛感した。

    まとめ — 13週間で残ったもの

    • 軸: 競争の主戦場は「モデルの賢さ」から「ハーネスの設計」へ移った。天井を決めるのは周辺の作り込みだ。
    • エージェント: 自己進化は L1→L2→L3、スキルは3段階ロードと狭いスコープ、メモリは「何を保存しないか」。
    • LLM内部: KVキャッシュ最適化からエンコーダーレスまで、「使う分だけ・無駄なく」へ収束。
    • 設計: 規模を問わず「依存を内側に守る」。CQS・parse でコードを堅牢に、マイクロサービスは4つの柱とトレードオフで捉える。
    • 学び方: 受動インプットは放っておくと揮発する。教材化と HTML 蓄積で「説明できる・読み返せる」資産にする。

    次の一歩: 積み上がったインプットを evergreen ノート（1ノート=1主張）へ蒸留し続けること。 そして、ここで学んだハーネス設計を、自分の手で動くものに落とすこと —— 「理解止まり」を「動くもの」に変える番だ。
