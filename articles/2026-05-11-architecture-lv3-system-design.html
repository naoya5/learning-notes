アーキテクチャ Lv3 — システム全体完全解説

C4モデル / マイクロサービス vs モノリス / イベント駆動アーキテクチャ / CQRS+Event Sourcing を実装コード+SVG図解で1から学べる詳細解説。シリーズ全5回の第3回。

目次

    TL;DR

    • Lv3の到達点: 複数サービス・複数コンポーネントが連携するシステム全体を「図と言葉」で表現できる
    • C4モデル = 解像度を変えて4枚の図を描く（Context / Container / Component / Code）
    • マイクロサービスは銀の弾ではない: "Monolith First"。30人未満のチームで導入は過剰
    • イベント駆動 = サービス間を Pub/Sub で疎結合化。障害が分離され可用性UP
    • CQRS+ES = 監査要件・複雑な読み書き分離が必要な場合のみ採用。学習コスト高

    このシリーズの位置づけ

    シリーズ第3回。Lv2（アプリ構造）で1アプリ内の構造を学んだ次は、複数サービスをまたぐシステム全体の設計。チーム開発に入る前に必修。

    1. C4モデル — アーキテクチャ表現の標準

    Simon Brown が提唱した、解像度を変えて4枚の図でシステムを表現する標準手法。UMLよりずっと軽量で実用的。

    各レベルの使い分け

    レベル対象読者含む情報
    Lv1 Context経営層・新規参加者システム1個 + 外部の人/システムとの関係
    Lv2 Containerアーキテクト・運用Web/API/DB/Cache/Workerなどデプロイ単位
    Lv3 Component開発リーダーContainer内の主要モジュール
    Lv4 Code実装者クラス・関数（自動生成推奨）

    使いどころ

    Tip:

    • 設計レビュー → Lv2/Lv3で議論（具体的な構造を検討）
    • コード変更時 → Lv4はコードから自動生成（手書きしない）

    2. マイクロサービス vs モノリス

    2010年代にバズったマイクロサービス。"銀の弾ではない"。組織規模・業務複雑性によって正解が変わる。

    詳細比較表

    項目モノリスマイクロサービス
    初期開発速度速い遅い（基盤整備）
    デプロイ一括サービス単位（独立）
    スケール全体（コスト悪）サービス単位（コスト◎）
    チーム構成全員1コードサービスごと小チーム
    言語選択1言語固定サービス毎に自由
    運用負荷低い高い（K8s / 監視 / Tracing）
    トランザクションACID簡単分散TX困難（Saga必要）
    適性立ち上げ期 / 〜10人30人以上 / 業務複雑

    "Monolith First" — Martin Fowler の教え

    注意: 最初からマイクロサービスにする失敗パターンが頻発。
    • 境界の引き方は 実装してわかる（机上では決まらない）
    • 小規模では運用コストが利益を上回る
    • サービス分割の失敗コストが高い（後から統合困難）
    正解: モノリスで始めて、ボトルネックが明確になったら段階的に切り出す。

    移行判断フローチャート

    サービス間通信の選択肢

    方式特徴適用
    RESTシンプル、HTTP標準外部公開API
    gRPC高速、型安全内部サービス間
    GraphQL柔軟なクエリBFF / 集約API
    イベント（Kafka等）非同期、疎結合長時間処理 / 通知

    3. イベント駆動アーキテクチャ

    サービス間通信を イベント で疎結合化する手法。Pub/Sub パターンの応用。

    同期通信 vs 非同期通信

    軸同期（API）非同期（イベント）
    応答性即座に結果結果は別経路
    結合度強（依存先必須）弱（疎結合）
    障害伝播連鎖（Cascading Failure）分離
    整合性強整合性結果整合性
    適用認証 / 決済通知 / ログ / 分析

    イベント発行 → 購読のコード例

    // イベント定義
interface DomainEvent { type: string; occurredAt: Date; }

class OrderPlaced implements DomainEvent {
  type = "OrderPlaced";
  occurredAt = new Date();
  constructor(
    public orderId: string,
    public userId: string,
    public total: number,
  ) {}
}

// イベント発行
class OrderService {
  constructor(private bus: EventBus) {}

  async place(input: PlaceOrderInput) {
    const order = new Order(input);
    await orderRepo.save(order);
    await this.bus.publish(new OrderPlaced(order.id, order.userId, order.total));
    return order;
  }
}

// 購読側 — Inventory
bus.subscribe("OrderPlaced", async (e: OrderPlaced) => {
  await inventoryService.reserveItems(e.orderId);
});

// 購読側 — Mail
bus.subscribe("OrderPlaced", async (e: OrderPlaced) => {
  await mailService.sendConfirmation(e.userId, e.orderId);
});

// 購読側 — Stats
bus.subscribe("OrderPlaced", async (e: OrderPlaced) => {
  await statsRepo.increment("orders.placed", 1, { userId: e.userId });
});

    4. CQRS（Command Query Responsibility Segregation）

    「書き込み（Command）と読み取り（Query）のモデルを分離する」設計パターン。

    利点と欠点

    利点欠点
    書き込み・読み取りを別々に最適化システム複雑性UP
    スケール戦略を分離可能整合性管理が必要（結果整合性）
    複雑な読み取り処理が簡単に同期遅延の考慮
    セキュリティ境界を分けやすい運用工数増

    コード例

    // Command Side
class CreateOrderCommand {
  constructor(public userId: string, public items: Item[]) {}
}

class CreateOrderHandler {
  async execute(cmd: CreateOrderCommand) {
    const order = new Order(cmd.userId, cmd.items);
    await orderRepo.save(order);
    await eventBus.publish(new OrderCreated(order));
  }
}

// Query Side（非正規化された読み取り専用モデル）
interface OrderListItem {
  orderId: string;
  userName: string;        // User テーブルとJOIN済
  total: number;
  itemCount: number;
  status: string;
  createdAt: Date;
}

class OrderQueryService {
  // SQLは ElasticSearch等で最適化済み
  async listForUser(userId: string): Promise<OrderListItem[]> {
    return await readDb.query(`
      SELECT * FROM order_list_view WHERE user_id = $1 ORDER BY created_at DESC
    `, [userId]);
  }
}

// Read Modelの更新（非同期）
eventBus.subscribe("OrderCreated", async (e) => {
  await readDb.query(`
    INSERT INTO order_list_view (order_id, user_name, total, ...) VALUES (...)
  `);
});

    5. Event Sourcing（ES）

    「状態を保存しない。状態変化のイベント全部を保存」する手法。CQRS と組み合わせることが多い。

    コード例

    // イベント
type AccountEvent =
  | { type: "Deposited"; amount: number; at: Date }
  | { type: "Withdrawn"; amount: number; at: Date };

// Aggregate（イベントから状態を再生）
class Account {
  private balance = 0;
  private uncommitted: AccountEvent[] = [];

  // 過去のイベントから状態復元
  static replay(events: AccountEvent[]): Account {
    const a = new Account();
    for (const e of events) a.apply(e);
    return a;
  }

  deposit(amount: number) {
    this.apply({ type: "Deposited", amount, at: new Date() });
  }
  withdraw(amount: number) {
    if (this.balance < amount) throw new Error("残高不足");
    this.apply({ type: "Withdrawn", amount, at: new Date() });
  }

  private apply(e: AccountEvent) {
    if (e.type === "Deposited") this.balance += e.amount;
    if (e.type === "Withdrawn") this.balance -= e.amount;
    this.uncommitted.push(e);
  }

  getBalance() { return this.balance; }
  pullEvents() {
    const evs = this.uncommitted;
    this.uncommitted = [];
    return evs;
  }
}

// Event Store
class EventStore {
  async append(streamId: string, events: AccountEvent[]) {
    for (const e of events) {
      await db.query("INSERT INTO events (stream_id, type, payload, occurred_at) VALUES ($1, $2, $3, $4)",
        [streamId, e.type, JSON.stringify(e), e.at]);
    }
  }
  async load(streamId: string): Promise<AccountEvent[]> {
    const rows = await db.query("SELECT * FROM events WHERE stream_id = $1 ORDER BY occurred_at", [streamId]);
    return rows.map(r => ({ ...JSON.parse(r.payload), at: r.occurred_at }));
  }
}

// 使う側
const events = await store.load("acc_1");
const account = Account.replay(events);
account.withdraw(500);
await store.append("acc_1", account.pullEvents());

    利点と欠点

    ✅ 利点❌ 欠点
    完全な監査ログ（全変更履歴）クエリが複雑（再生が必要）
    過去の任意時点の状態復元可能学習コスト高
    バグ修正後の再計算可能スキーマ進化が困難（バージョニング必要）
    強力なデバッグ可能性イベント数膨張対策（スナップショット）必要
    導入判断:
    • 監査要件が厳しい（金融、医療） → ✅ Yes
    • 過去の任意時点の状態が必要（投資商品の口座履歴） → ✅ Yes
    • シンプルなCRUDで十分 → ❌ No（過剰）

    演習問題

    Q1. C4のどのレベルで描く？

    以下のシチュエーションで、C4のどのレベルの図を見せる？

    1. 新人エンジニアの初日オンボーディング
    2. 運用チームへのデプロイ構成説明
    3. 同僚開発者へのコード変更レビュー
    4. 経営層への新システム提案
    解答を見る
    1. Lv1 Context — 全体像を最初に見せる
    2. Lv2 Container — Web/API/DBの構成が分かる
    3. Lv3 Component or Lv4 Code — 該当モジュール
    4. Lv1 Context — 経営層は技術詳細不要

    Q2. マイクロサービス導入すべき？

    10人のスタートアップで、Web/iOS/Androidの3クライアント、月間DAU 5万人。どう判断する？

    解答を見る

    モノリスで開始すべき。理由:

    • 10人 = マイクロサービスの運用負荷を吸収できない
    • 5万DAUはモノリスで余裕
    • 初期は仕様変動が激しく、サービス境界を間違えやすい

    将来チーム30人 + ボトルネック明確になったら、Order/Payment等から切り出す。

    Q3. 同期 vs 非同期？

    以下のフローを設計するとき、各ステップは同期/非同期どっちが適切？

    1. ログイン認証
    2. 注文確定後のメール送信
    3. クレカ決済
    4. 商品購入後のレコメンド学習データ収集
    5. 在庫引当
    解答を見る
    1. 同期 — 即座に結果が必要
    2. 非同期 — 1秒遅れてもOK、メール失敗で注文自体を失敗させたくない
    3. 同期 — 決済結果を待ってから注文確定
    4. 非同期 — ユーザー体験に影響しない、後でバッチでもOK
    5. 同期 — 注文成立に必要、即座にロックすべき

    まとめ — Lv3の要点

    1. C4モデル — Context/Container/Component/Code の4階層で図を描き分ける
    2. Monolith First — マイクロサービス導入は 30人以上 + ボトルネック明確 が条件
    3. イベント駆動 — 通知/ログ/分析など非同期で良い処理に最適
    4. 同期 vs 非同期 — リアルタイム必要なら同期、結果待たないなら非同期
    5. CQRS — 書き込み/読み取りを分離。複雑な読み取りに有効
    6. Event Sourcing — 全変更を保存。監査要件が厳しい時のみ
    7. 過剰設計に注意 — シンプルなCRUDで足りるなら CQRS/ES は要らない
