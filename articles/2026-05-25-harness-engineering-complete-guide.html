ハーネスエンジニアリング完全ガイド

2026年初頭に OpenAI・Anthropic・Martin Fowler・LangChain・Mitchell Hashimoto らが相次いで発信した「Harness Engineering」を、英日33本の一次情報を実取得して読み解いた統合ガイド。命名・歴史・3層モデル・構成要素・7つのアーキテクチャパターン・ケーススタディの定量証拠・実装チェックリストまでを 1 ページに収めた。

目次

    TL;DR

    • Harness Engineering = エージェント周辺の環境設計工学。Agent = Model + Harness (LangChain・Böckeler)。Harness とは制約・ツール・ドキュメント・フィードバックループの永続的ソフトウェア資産 (Charlie Guo)。
    • 命名は Mitchell Hashimoto (2026-02-05)。「同じミスを二度させない仕組みを作る」が原点。制度化は OpenAI Ryan Lopopolo (2026-02-11):5ヶ月・100万行・1,500 PR・人間手書きコード 0 行の社内製品出荷レポート。
    • 業界標準階層: Prompt ⊂ Context ⊂ Harness。比喩は Model=CPU / Context=RAM / Harness=OS / Agent=Application (Schmid → Hightower)。
    • 定量証拠: LangChain が モデル固定 で Terminal Bench 2.0 を 52.8 → 66.5 (+13.7pt)、Top30→Top5 に押し上げ。Stripe Minions は週 1,300+ PR。
    • 反例: 2026-03 Amazon AWS の 13 時間停止・630 万件オーダー喪失 ── ハーネス不在の AI 生成コードが本番障害を起こす "AI Coding Hangover" の最大事例。
    • 標語: "Humans steer. Agents execute." / "Agents aren't hard; the Harness is hard." / "Give Codex a map, not a 1,000-page instruction manual."

    ハーネスとは何か ── メタファーと定義

    Harness の原義は 馬具・引き具。強い動物の力を意図方向に向けるための装具を指す。 エージェント文脈では3つのニュアンスが重ねられている。

    「ハーネス=馬具、手綱」は梶川琢馬 (MOSHTech 2026-03-26) で日本語圏に定着。NxCode は horse = model, harness = infrastructure, rider = engineer の三項関係まで明示する。

    • 馬具・手綱 (reins) ── 強力な LLM を意図方向へ導く
    • ロッククライミング安全帯 ── 暴走・墜落 (誤動作・本番障害) を防ぐ命綱
    • Test harness ── 既存ソフトウェア工学の "周辺コード" 概念の拡張

    10名の論者による一行定義

    業界が同じ概念に収束する過程で、各論者が独自のフレーズで定義を打ち出した。最頻出引用は Hashimoto の「二度と同じミスをさせない仕組み」と Schmid の CPU/OS 比喩。

    10名の定義 (原文・全文)
    • Ryan Lopopolo (OpenAI): "A software engineering team's primary job is no longer to write code, but to design environments, specify intent, and build feedback loops that allow Codex agents to do reliable work."
    • Mitchell Hashimoto (命名者): "Anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent never makes that mistake again."
    • Birgitta Böckeler / Martin Fowler: "Agent = Model + Harness." ── Guides (feedforward) と Sensors (feedback) の総体
    • Anthropic: "Imagine a software project staffed by engineers working in shifts" ── シフト勤務エンジニア群を回す環境
    • Charlie Guo: "The set of constraints, tools, documentation, and feedback loops that keep an agent productive and on track."
    • LangChain (Vivek Trivedy): "The goal of a harness is to mold the inherently spiky intelligence of a model for tasks we care about."
    • Philipp Schmid (DeepMind): "An Agent Harness is the infrastructure that wraps around an AI model to manage long-running tasks. It is not the agent itself."
    • Rick Hightower: "The model is the CPU. The harness is the OS."
    • Cobus Greyling: "The missing architectural layer that determines whether AI Agents actually work in production."
    • 梶川琢馬 (TechBowl): 「自律的な AI Agent の実現のために、LLM モデルの性能を最大限引き出すための環境を設計すること」
    本ガイドのまとめ: Harness = (1) エージェントランタイム + (2) 仕様・知識ベース (AGENTS.md / docs/ / Skills) + (3) ツール群 (MCP / linter / test) + (4) 評価・観測装置 + (5) ループ制御 (Hooks / Permissions / human-on-the-loop) の5レイヤー統合システム。

    3層モデル ── Prompt ⊂ Context ⊂ Harness

    HumanLayer・Hightower・Charlie Guo・kazu・estブログ ── ほぼ全ての記事で同じ階層が採用される。Schmid の CPU/OS 比喩が伝播の中心になった。

    重要なのは 否定ではなく包含 の関係であること。「ハーネスはコンテキストエンジニアリングを 置き換えるのではなく、包含する 上位概念」(nogataka)。Greyling は SDK → Framework → Scaffolding → Harness の4世代モデルを提示し、Harness を新世代として位置付ける。

    Hightower の補足: 「context engineering を完璧にしてもエージェントは 20-40% 失敗する」── これが harness 層が必須な数値的根拠。失敗の主因は単発推論の質ではなく 連続実行の制御 にある。

    ハーネスの構成要素 ── Böckelerの4象限フレーム

    martinfowler.com (2026-04-02 更新版) が提示した 業界で最も洗練された分類軸。Guides (事前) / Sensors (事後) × Computational (決定論的) / Inferential (LLM) の交差で 4 象限に整理する。

    Keep Quality Left 原則: 「ルールで表現できる品質基準はすべて Computational Sensors に落とし込み、『意味的に正しいか』だけを Inferential Sensors に任せる」(nogataka, Martin Fowler 読解より)。コストと信頼性が同時に最適化される。

    5レイヤーで構成要素を整理

    1. 仕様・知識ベース ── AGENTS.md / CLAUDE.md は100行前後の「目次 (map)」。本体は docs/design-docs/ docs/exec-plans/{active,completed}/ ARCHITECTURE.md QUALITY_SCORE.md RELIABILITY.md SECURITY.md に分割 (OpenAI)。Progressive Disclosure。
    2. 構造的制約 ── 依存方向の自動強制 (BE: deptrac, FE: dependency-cruiser), 型 × 静的解析 × テストの3層 (梶川)。
    3. ツール群 ── MCP 集約 (Stripe Toolshed 400+), Chrome DevTools MCP でエージェントに「目」を与える, custom linter のエラー文に修正手順を埋め込む (OpenAI: Charlie Guo が「最も賢いアイデア」と評価)。
    4. 評価・観測 ── OpenTelemetry + LogQL/PromQL/TraceQL, LangSmith Trace Analyzer, Generator+Evaluator (GAN), LLM-as-judge + few-shot rubric。
    5. ループ制御 ── Initializer + Coding agent (Anthropic), Hooks (PreToolUse/PostToolUse/Stop/UserPromptSubmit), Garbage Collection エージェント, Ralph Wiggum loop (完了直前の再検証強制)。

    主要アーキテクチャパターン 7選

    1. Initializer + Coding Agent (Anthropic, 2025-11)

    「シフト勤務エンジニア」メタファ。各セッションは記憶ゼロで来る前提で、状態を claude-progress.txt + git history に集約する。 

    ┌─────────────────────┐
│ Initializer Agent   │  起動時に1回だけ
│  - init.sh 生成      │  - 開発サーバ起動方法を確立
│  - claude-progress  │  - 機能リストをJSON化
│  - 初回 git commit   │
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Coding Agent (loop) │  毎セッション
│  ・進捗を読む         │  - 1機能ずつ漸進
│  ・1機能実装          │  - passes フィールド更新のみ
│  ・progress更新       │  - git commit
│  ・commit             │
└─────────────────────┘

    JSON feature list は 200+ 機能を JSON で管理し、passes: false を true にしか変更不可な制約をかける。「テストを消すのは unacceptable」と強い文言で改ざんを抑止。Markdown より JSON のほうがエージェントによる不適切な書き換えが少ないという経験則。

    2. Generator + Evaluator + Planner (Anthropic, 2026-03)

    GAN (敵対的生成ネットワーク) からの着想。「作る役」と「採点する役」を分離して自己評価バイアスを回避する。Evaluator は few-shot examples + 採点基準 (Design quality / Originality / Implementation quality / Usability) で "taste" を再現する。

    この記事は同時に "Context anxiety" という重要発見を提示している ── context limit が近いと感じたモデルが作業を早めに畳む現象 (Sonnet 4.5 で顕著)。

    対処は compaction ではなく context reset + 構造化 handoff。圧縮は context anxiety を解消しない。

    3. Deep Agent (LangChain, 2026-02)

    モデルを gpt-5.2-codex に固定したまま harness のみ反復改善 ── これで Terminal Bench 2.0 のスコアを 52.8% → 66.5% (+13.7pt) 押し上げ、ランキング Top30 → Top5 に躍進した実証実験。3つのノブ (System Prompt / Tools / Middleware) を boosting 的に更新する。

    実装上の決定打は LocalContextMiddleware (起動時にcwd/dir/toolsを注入) と PreCompletionChecklistMiddleware (完了直前の再検証を強制 = Ralph Wiggum 安全実装)。さらに「Planning → Build → Verify → Fix」を system prompt で強制し、build-verify ループを自然に組み込む。

    モデルごとに harness を tailor すべし: 同じ harness で Claude Opus 4.6 は 59.6% に留まった。「Codex 向け harness」と「Claude 向け harness」は別物として扱う必要がある。

    4. Codex App Server (OpenAI, 2026-02)

    CLI / VS Code / JetBrains / Xcode / macOS app / Web app という複数 surface を支える Codex harness を、共通の双方向 JSON-RPC API として OSS 化した経緯のレポート。MCP では表現しきれない rich session semantics (workspace exploration / streaming progress / diff emission) をカスタムプロトコルで実装。

    3つのプリミティブ:

    • Items ── atomic I/O with lifecycle (started/delta/completed)
    • Turns ── user-initiated unit
    • Threads ── durable container

    Simon Willison が引用する Gabriel Chua (OpenAI) の三分割整理 Codex = Model + Harness + Surfaces はこの App Server 構造を背景にしている。決定的に重要な一文:

    "Codex models are trained in the presence of the harness. Tool use, execution loops, compaction, and iterative verification aren't bolted on behaviors — they're part of how the model learns to operate." (Gabriel Chua, via Simon Willison)
    ── モデルが harness の存在を前提に 訓練されている。HumanLayer の「Opus 4.6 は Claude Code 外で順位が上がる (over-fitting to harness)」観察と整合する。

    5. Sub-agents as Context Firewall (HumanLayer)

    Sub-agent を context firewall として運用する。親 (Opus, full context) → 子 (Sonnet/Haiku, isolated) と分離することで:

    • 中間ノイズが親の文脈を汚さない
    • 重い思考だけ Opus、定型作業は Sonnet/Haiku でコスト制御
    • 並列実行で時短

    背景には Chroma の "context rot" 研究がある。長文脈になるとモデルが "Dumb Zone" に入り性能低下する。HumanLayer はこれを "Instruction Budget" (トークンではなく 処理すべき指示の量) の概念で再定義している。

    6. ローカル・オーケストレーター (国内事例)

    国内エクスプラザ社の Tsucchi 氏が GitHub Actions の max-turns 限界を超えるために実装した、手元の Mac/Linux で 24/7 常駐する TypeScript オーケストレーター。 

    Asana → Issue 検出 → orchestrate.ts (ローカル常駐)
                           │
                ┌──────────┼──────────┐
                ▼          ▼          ▼
            難易度判定  Explore  Implement  Review
            (E/M/H)    フェーズ  フェーズ   フェーズ
                           │
                  ループ検出 (stderr正規化, LOOP_THRESHOLD=3)
                           │
                       Draft PR
                           │
              launchd で週次/月次品質スキャン

    ポイントは --allowedTools での最小権限、Git worktree での隔離、Claude Max 定額 ($100/月) でのコスト固定、そして stderr 正規化によるループ検出 (同じエラーが3回続いたら強制中断)。

    7. Ralph Loop (ghuntley系)

    1セッションをループで放置し、エージェントに自己修正を任せるパターン。Böckeler は「正しく理解しないと誤用される」と注意喚起しており、LangChain の PreCompletionChecklistMiddleware は Ralph Loop の安全実装と位置付けられる。

    ケーススタディ ── 定量証拠が物語る

    OpenAI: 5ヶ月100万行・人間手書きコード 0 行

    分野の制度化記事となった Ryan Lopopolo (2026-02-11) のレポートにある具体的な数字。 

    チーム規模          3 → 7 人 (拡大後も生産性低下せず)
期間                5 ヶ月 (2025年8月下旬〜)
生成コード          約 100 万行
PR                  1,500 件
1人/日あたり PR     3.5 件
工数                従来比 約 1/10
手書きコード        0 行 (チームポリシー)
自律実行時間        最大 6 時間

    ハーネス構成: AGENTS.md = 100 行の 目次、本体は docs/ 配下。階層型ドメインアーキテクチャ + Providers パターン (boring tech 主義)。Custom linter にエラー文として 修正手順を埋め込む。一時的ローカル observability stack (LogQL / PromQL / TraceQL)。Chrome DevTools MCP で UI 検証。Garbage Collection エージェントが定期的に drift 検出 → 修正 PR を開く。

    LangChain: モデル固定で +13.7pt

    既述の通り、gpt-5.2-codex 固定で harness のみ反復改善 → Terminal Bench 2.0 のスコアを 52.8% → 66.5% (+13.7pt)、Top30 → Top5 に押し上げ。

    教訓: 「モデル世代差」よりも「harness 設計差」のほうがベンチマークスコアを大きく動かす。次世代モデルを待つ前に、今ある harness を磨くほうが投資対効果が高い。

    Stripe Minions: 週 1,300+ PR

    Charlie Guo / Epsilla 経由で報告されている Stripe 社の事例。内製ツール群 "Toolshed" (400+ ツール) を MCP で統一提供。各エージェントは devbox で隔離。

    2-strike CI rule: CI 失敗 2 回で human escalation。"Blueprint" = deterministic nodes (確定処理) + agentic nodes (LLM 処理) の混在ワークフロー。

    反例: Amazon AWS Outage 2026-03

    AI Coding Hangover の最大事例: AWS コスト管理機能が 13 時間停止、3月5日に 630 万件のオーダーが喪失。"AI assisted code" 起因と報じられる。Amazon が 335 重要システムに 90日安全リセット を発動し、Kiro (agentic IDE) 主導の変更プロセスを見直した。

    教訓: ハーネス不在の AI コード生成は本番障害に直結する。 Spillwave / Hightower (2026-03) が示した「ガードレール6種 (standards / reference architectures / dependency controls / performance budgets / deployment policies / data contracts)」を最初から強制する必要がある。

    Mitchell Hashimoto: AI懐疑派 → 命名者

    HashiCorp 創業者の個人ジャーニー。6 ステップで導入し、Step 5 で「harness engineering」と命名した。Ghostty (彼のターミナル OSS) の src/inspector/AGENTS.md を 「過去のミスを 1 行ずつ積み上げた error log」として運用している。

    Hashimoto の 6 ステップ adoption
    1. Drop the Chatbot ── チャット UI は brownfield では非効率、agent ループへ移れ
    2. Reproduce Your Own Work ── 手動コミットを agent で再現する苦行で「3原則」を発見: タスク細分化 / planning と execution を分離 / 検証手段を渡せ
    3. End-of-Day Agents ── 1日の最後の30分をエージェント着火に充てる
    4. Outsource the Slam Dunks ── 自信のあるタスクだけ並走で投げる
    5. ★ Engineer the Harness ── 同じミスを二度させない仕組みを作る (命名の原点)
    6. Always Have an Agent Running ── GPT-5.2-Codex deep mode 等を常時稼働

    実装プラクティス・チェックリスト

    主要記事を統合した、新しくハーネスを組む際の作業項目。必須 → 中・上級 → 組織レベル の3段階で並べた。

    必須 (どの記事でも一致)

    • AGENTS.md / CLAUDE.md を 60〜100行の「目次」として配置。本体は docs/ 配下に分割
    • docs/ 階層を System of Record として整備 (design-docs / exec-plans / ARCHITECTURE / QUALITY_SCORE / RELIABILITY / SECURITY)
    • Progressive Disclosure: エントリポイント → 必要時に深部
    • エージェントが読む前提でドキュメント再整備 (agent legibility)
    • カスタムリンター + テスト + 型 の3層。lint エラー文に 修正手順を埋め込む
    • 依存方向の自動強制 (deptrac / dependency-cruiser / ArchUnit)
    • Permissions deny で機密ファイルを「存在しない」状態に
    • Sub-agents を context firewall として使う: Opus 親 + Sonnet/Haiku 子
    • Hooks で決定論的ゲート: PreToolUse / PostToolUse / Stop / UserPromptSubmit
    • Stop hook で typecheck + lint を再実行、エラー時 exit 2 で agent を再エンゲージ

    HumanLayer 由来の Stop hook 例

    最小実装。~/.claude/settings.json または プロジェクトの .claude/settings.json に。 

    {
  "hooks": {
    "Stop": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "biome check . & tsc --noEmit; wait; if [ $? -ne 0 ]; then echo 'Fix lint/type errors before stopping' >&2; exit 2; fi"
          }
        ]
      }
    ]
  }
}

    exit 2 は Claude Code に back-pressure を返す合図で、エージェントが自動的に再エンゲージして修正する。typecheck と lint を並列実行することでフィードバックループを短縮できる。

    中・上級

    • Generator + Evaluator を分離。自己評価が困難な領域に evaluator agent を配置
    • Initializer Agent で起動時の環境整備 (init.sh, progress.txt, JSON feature list) を自動化
    • JSON feature list: passes: false を true にしか変更不可な制約
    • Context reset > compaction。「context anxiety」を観測したらリセット
    • Garbage Collection エージェントを定期 cron で稼働
    • 2-strike CI rule: CI 失敗 2 回で human escalation
    • Trace Analyzer Skill: trace → 並列誤り解析 → harness 自動更新
    • モデル別に harness を tailor
    • ハーネス自体に CI をかける (リグレッション検知)
    • AGENTS.md を「失敗の error log」として運用

    組織レベル

    • Toolshed パターン: 内部ツール群を MCP で統一カタログ化
    • Blueprint パターン: deterministic nodes + agentic nodes の混在ワークフロー
    • Agents Captain 役割を任命 (Brockman / Charlie Guo)
    • brownfield retrofit ロードマップを策定
    • "AI Manager's schedule" への移行 ── エンジニアの仕事を「環境構築」と「仕事の管理」の二分化として再定義
    • "build to delete" 原則 ── harness はモデル更新で陳腐化するので捨てやすく作る (Schmid / Bitter Lesson)

    関連概念との比較

    Humans Out / In / On the Loop (Böckeler, 2026-03-04)

    • Out of the loop (Vibe Coding) ── 内容を読まずデプロイ。リスク: 内部品質崩壊・本番障害 (Amazon 事例)
    • In the loop (毎行レビュー) ── 人間がボトルネックに。スループット低下
    • ★ On the loop (= harness engineering) ── artefact ではなく harness を直す。初期投資が大きいが永続的
    「The 'in the loop' way is to fix the artefact. The 'on the loop' way is to change the harness that produced the artefact so it produces the results we want.」── Böckeler の最重要フレーズ。問題が再発したらコードではなく harness を直す。

    The Agentic Flywheel

    究極形は agents directing agents to improve the harness ── エージェントが harness 改善案を生成 → スコアリング → 自動承認する自己進化ループ。

    経営視点 (note allegrubato)

    • AI が生産性を 2 倍にすれば人月モデル上は売上半減 ── 自己矛盾
    • 希少資源が「人の作業量」→「人の確認時間と注意力」に移行
    • SIer の生存戦略: ハーネスのテンプレート化 / 暗黙知のAI翻訳 / 成果ベース契約
    「AIの事業価値は、AIの出力の総量ではなく、確認コストをどこまで落とせるかで決まります。」── ため (note) の経営フレーミング。

    まだ揺れている論点

    • Brownfield retrofit 問題 ── 10年物コードベースに harness を後付けできるかは明確な解答なし (Charlie Guo)
    • Behaviour harness の困難さ ── 機能・振る舞いの検証は依然 LLM-as-judge + 人間に依存 (Böckeler)
    • モデルと harness の co-evolution ── ベンダー側がモデル訓練時に harness を前提化 → 設計が戦略と結合 (Willison / HumanLayer)
    • Harness debt ── harness 自体が次の保守対象になる (Bouchard)

    まとめ

    • 2026年初頭の半年で確立した新分野。命名 (Hashimoto) → 制度化 (OpenAI Lopopolo) → フレーム化 (Böckeler) → 理論的下地 (Schmid) の分業構造で進化した。
    • Harness = エージェント周辺の永続的ソフトウェア資産。プロンプトの揮発性を超え、5つのレイヤー (ランタイム / 知識ベース / ツール / 評価 / ループ制御) の統合システムとして設計する。
    • 4象限フレーム (Guides × Sensors × Computational × Inferential) を用いて自プロジェクトを棚卸せよ。3 regulation categories のうち Behaviour harness が最も未解決領域。
    • モデルより harness のほうがスコアを動かす ── LangChain の +13.7pt が示す通り。次世代モデルを待たず、今ある harness を磨くほうが投資対効果が高い。
    • 反面教師は Amazon AWS Outage。ハーネス不在の AI 生成コードは本番障害に直結する。"AI Coding Hangover" を起こす前に品質ゲートを設計せよ。
    • Stop hook + exit 2 で back-pressure を実装するのが最小の入口。CLAUDE.md は 60〜100 行の目次に絞り、本体は docs/ に分割する。
    • 人間の役割は "on the loop" ── artefact ではなく harness を直す。問題が再発したら必ず harness を更新する反復ループを文化として確立する。
