Gemma 4いちばんやさしい仕組みガイド

Google DeepMind の小型〜中型LLM「Gemma 4」。その中身を、専門用語を噛み砕いて図とたとえ話で解説します。長い注意機構の話から MoE、スマホで動く秘密、爆速化の仕掛けまで。

01 / 全体像

Gemma 4 は「4兄弟」サイズ違いの4モデルがあり、自分のPCやスマホに載るものを選べます。どれも画像を理解できる「マルチモーダル」です。

E2B
Dense · 実質2B
超軽量。スマホ向け。文字+画像に加えて音声も扱える。
E4B
Dense · 実質4B
軽量。E2Bと同じくオンデバイス&音声対応。
26B A4B
MoE · 動くのは4B
合計26Bだが、推論時は4B分しか動かない「専門家チーム」型。大きいのに速い。
31B
Dense · 31B
いちばん素直で大きい「王道」モデル。性能の基準。

この4つは大きく2タイプに分かれます。全パラメータを毎回使う Dense(密) 型と、必要な専門家だけを呼び出す Mixture of Experts(MoE) 型です。まずは全モデルに共通する「土台」から見ていきましょう。

02 / 共通の土台

注意の仕組み:近くだけ見る層と、全部見る層LLMは「どの単語に注目するか(Attention)」で文章を理解します。Gemma 4 は2種類の注目方法を交互に使って、賢さと速さを両立させています。

局所スライディング・ウィンドウ注意

ローカル注意(局所注意)は、直前の決まった数のトークンだけに注目します。全部を見ないぶん計算が軽い。Gemma 4 では小型モデルが直前 512トークン、大型モデルが直前 1024トークンだけを見ます。

たとえ話

走る電車の窓のようなもの。窓に映る景色(=直近の数トークン)だけを見て進みます。過去の景色は窓から消えますが、頭の中のメモ(隠れ状態)として記憶は引き継がれるので、完全に忘れるわけではありません。

なぜ全部を局所にしないの?

局所注意を重ねれば情報は伝わりますが、伝言ゲームのように層を通るたびに少しずつ薄まります。だから時々「全体を見渡す層」が必要になります。

窓の外に出たトークンも、完全に消えるわけではありません。各層が計算した 隠れ状態(hidden states) を通じて、過去の情報は上の層へ少しずつ受け渡されていきます。ただし下の図のように、層を経るほど元の情報は薄まる——だからグローバル層で「本物の全体ビュー」を補う必要があるのです。

大域グローバル注意とインターリーブ

グローバル注意(全域注意)は、これまでの全トークンに注目する通常のやり方。正確ですが重い。そこで Gemma 4 は、軽い局所層を何枚か挟んでから時々グローバル層を入れるインターリーブ(交互配置)を採用します。

パターンは 5:1(局所5枚→グローバル1枚)が基本(E2Bだけ4:1)。そして Gemma 3 からの改良点として、最後の層は必ずグローバル注意になるよう揃えられました。

03 / 効率化のワザ

重いグローバル層を軽くする3つの工夫全部を見る層はメモリを大量に食います。Gemma 4 はここに3つのテクニックを効かせて「速くて省メモリ」を実現します。

GQAグループ化クエリ注意

注意機構は Query(質問)・Key(鍵)・Value(値) の3役で動きます。Key と Value はキャッシュに溜まりメモリを圧迫しがち。そこで複数のQueryで1組のKVを共有させて節約します。グローバル層では 8つのQueryが1組のKVを共有(局所層は2:1)。共有で性能が落ちないよう、Keyのサイズは倍にして補っています。

局所層とグローバル層で比率が違う

局所層は見る範囲が狭いので 2 : 1(Q2つでKV1組)で十分。一方グローバル層は全文を保持するためキャッシュ負担が桁違いに大きいので、より強く節約する 8 : 1 を採用しています。「重い所ほど強く絞る」という考え方です。

K=V鍵と値を同じにする

さらに大胆な節約。グローバル層に限ってKey と Value を同じ値にしてしまう。すると保存するのは実質1種類で済み、KVキャッシュ(実質Kキャッシュ)がもう一段軽くなります。性能への悪影響はほとんどないとのこと。

p-RoPE位置情報を一部だけに入れる

LLMは RoPE という仕組みで「単語の順番」を覚えます。ベクトルを2個ずつのペアに分け、ペアごとに少しずつ回転させて位置を表す方法です。回転が大きい(高周波)ペアは位置に敏感、回転が小さい(低周波)ペアは意味を担当します。

問題は、低周波ペアに無理に位置情報を入れるとノイズになり、長文では誤差が積み重なって遠くの単語の関係が崩れること。そこで p-RoPE は位置情報を最初の25%のペアだけに入れ、残りは意味専用に温存します。長文を扱うグローバル層でこれが効きます。

グローバル層に効いている工夫まとめ

  • 最終層は必ずグローバル注意にする
  • GQA:Query 8 : KV 1 の比率で共有
  • Keyのサイズを倍にして共有の悪影響を補正
  • K = V:鍵と値を同一化してキャッシュを削減
  • p-RoPE (p=0.25):位置情報を25%のペアだけに限定
04 / 画像を理解する

ビジョンエンコーダ:絵を「言葉」に変換全モデルが画像を扱えます。仕組みは Vision Transformer(ViT)。画像を小さなタイルに切って、それぞれを埋め込みベクトルに変えます。

テキストを単語に分けるように、画像は 16×16ピクセルのパッチ(タイル)に分割されます。各パッチがTransformerを通って、1つずつ埋め込みに変換される——という流れです。

工夫1いろんな縦横比に対応(2D RoPE)

従来のViTは画像を正方形に潰すため、横長の絵は不自然に歪んでしまいます。Gemma 4 はパッチの縦・横の位置を別々に位置情報化する 2D RoPEと、足りない分をパディング(余白埋め)する仕組みで、元の縦横比を保ったまま処理できます。

なぜ縦横比が問題かというと——「4番目のパッチ」の意味が画像の形で変わってしまうからです。3×3なら4番目は真ん中ですが、2×4なら左下。位置の意味が崩れるので、縦と横の座標を別々に持たせる必要があるのです。

工夫2解像度を選べる「トークン予算」

同じ縦横比でも解像度はさまざま。そこでソフトトークン予算を導入。70 / 140 / 280 / 560 / 1120から選べて、予算が大きいほど高解像度=多くのパッチで細かく見られます。物体検出には大きい予算、動画には小さい予算、と使い分けられます。

工夫3本体に合わせる「線形射影」

画像から作った埋め込みは、そのままでは Gemma 4 に渡せません。テキストの単語埋め込みとは住んでいる座標空間が違うからです。そこで小さなニューラルネットでサイズと値の分布を本体の期待に合わせて変換(射影)し、最後に RMSNorm でスケールを整えます。これでようやく画像が「言葉と同じ土俵」に乗ります。

補足

最終的に隣り合う3×3パッチを平均(プーリング)して1つにまとめるため、見た目のパッチ数より少ないトークン数になります。小型モデルのビジョンエンコーダは150M、それ以外は550Mパラメータです。

05 / モデル別の個性

3タイプの中身を深掘り共通の土台の上に、各モデルが「秘密のひと工夫」を載せています。

本題に入る前に、2大アーキテクチャ Dense と MoE の違いを押さえましょう。Denseは全パラメータを毎回フル稼働、MoEは一部の「専門家」だけを起動します。なお、ひとつのモデルが層ごとにDenseとMoEを混ぜることはなく、全層どちらか一方です。

31B素直な王道 Dense モデル

いちばんクセのない大型モデル。Gemma 3 の27B版に近い構成で、局所×グローバルのインターリーブや前後のRMSNormも踏襲。違いは前述の K=V と p-RoPE など。層は少なめ(60層)だけど横に広いのが特徴で、他モデルを理解する「基準」になります。

26B A4B専門家チーム型(MoE)

名前の A は Active(実際に動く)。合計26Bをメモリに載せつつ、推論で動くのは4B分だけ。だから大きいのに4Bモデル並みに速い、という美味しいモデルです。

たとえ話

大きな病院の総合受付(ルーター)を想像してください。患者(トークン)が来ると、受付が症状を見て 128人の専門医から最適な 8人だけを呼び出します。さらに何でも診る総合医(共有エキスパート)は常に対応。全員を毎回動かさないから速いのです。

ここで大事な2つの言葉。スパースパラメータはモデルの全パラメータで、起動時にメモリへ載せる必要があります。一方アクティブパラメータは実際の計算で使う分。26B A4B は 26B載せるけれど、動くのは8人+総合医ぶんの約4Bだけ。残り119人の専門家は出番が来るまで待機。だから容量は大きくても、速度は4Bモデルに近いのです。

E2B / E4Bスマホで動く Dense モデル

名前の E は Effective(実効)。秘密は Per-Layer Embeddings(層ごとの埋め込み, PLE) です。

通常は性能を上げたいなら層やパラメータを増やします。PLEは発想が逆で、各層ごとに小さな「メモ」を用意します。例えば「cat」というトークンに対し——

  • 第2層のメモ:「私は名詞。たいてい後ろに動詞が来る」
  • 第18層のメモ:「私は小動物で『ペット』『ネコ科』に近い」

このメモ表(巨大なルックアップテーブル)はフラッシュメモリ(ストレージ)に置いておくのがポイント。計算に使う貴重な (V)RAM を消費しません。推論開始時に必要なメモだけサッと取り出すので軽い。だからスマホのような限られたメモリでも動くのです。Eは「PLE以外の実際に計算で使うパラメータ」を指します。

音声E2B/E4B だけのオーディオ対応

小型モデルは音声も扱えます。流れは:
① 生の音声を メルスペクトログラム(時間×周波数の2D画像)に変換 → ② チャンクに区切る → ③ 畳み込みで圧縮して埋め込み列に。エンコーダには Conformer(畳み込みを足したTransformer)を使い、最後はビジョンと同じく本体のサイズに合わせて射影します。音声認識や翻訳が得意です。

06 / 爆速化の仕掛け

下書き役で速くする:投機的デコード本体とは別に、小さな「下書き(drafter)」モデルが同時公開されました。これで推論が大きく速くなります。

LLMは1トークンずつ生成し、簡単な単語でも難しい単語でも同じ計算量がかかります。これが遅さの原因。そこで小さな下書きモデルが数トークン先回りで予測し、本体(target)がまとめて一気に検証します。これが 投機的デコード(Speculative Decoding)、下書き役を Multi-Token Prediction(MTP)ヘッドと呼びます。

たとえ話

英語のことわざ「Actions speak …」と来れば、続きは「louder than words」だと誰でも予想できますよね。こんな簡単な続きを本体がいちいち1語ずつ書くのは時間の無駄。下書き役が一気に4語先回りして、本体は「OK!」とまとめて承認するだけ。これで何倍も速くなります。

本体が下書きに同意すれば全部採用。途中で間違いがあれば、そこまでを採用し間違った語を本体が自分の答えに差し替えます。本体は1語生成する時間で複数語を検証できるので、結果は正確なまま大幅に高速化します。

中身下書きモデルを賢く速くする3工夫

  • ターゲット活性化の流用:本体の最終層の計算結果を下書きに渡し、より良い予測に。「本体が頑張った成果を捨てるのはもったいない」発想。
  • KVキャッシュの共有:下書きが自前でKVを作らず、本体が計算済みのKVキャッシュを借りる。
  • 効率的エンベッダ:26万トークン全部の確率を計算せず、似た意味のクラスタに絞ってから候補を出す(E2B/E4Bのみ)。

3つ目の「効率的エンベッダ」をもう少し。通常、次トークンを決める最終工程(LM Head)では26万語ぜんぶの確率を計算します。小さな下書きモデルにはこれが重い。そこで似た意味の語をクラスタ(かたまり)にまとめ、まず「どのクラスタが当たりか」をざっくり選んでから、その中の語だけ詳しく計算します。図書館で棚を先に選んでから本を探すイメージです。

どれくらい小さい?

E2B用の下書きモデルは約76Mパラメータ・4層だけ。本体(target)とほぼ同じ品質を保ちつつ、低遅延・オンデバイス用途にぴったりです。

07 / 全体像

すべてを1枚に組み立てるここまでのパーツを積み上げると、Gemma 4 の1モデルが完成します。入力(文字・画像・音声)から出力までの流れを俯瞰しましょう。

どのモデルもこの骨格を共有し、FFNNをDenseにするかMoEにするか、PLEを足すか、音声エンコーダを付けるかといった「載せ替え」でキャラクターが決まります。共通の土台があるからこそ、サイズ違いの4兄弟を一貫した設計で作れるわけです。

08 / まとめ

Gemma 4 を一言でいうと

30秒で振り返り

  • 4兄弟:E2B / E4B(スマホ向け・音声対応)、26B A4B(MoE)、31B(王道Dense)
  • 注意の二刀流:軽い局所注意と全域グローバル注意を5:1で交互配置、最終層は必ずグローバル
  • 省メモリ三種の神器:GQA(8:1共有)・K=V・p-RoPE(25%)でグローバル層を軽量化
  • マルチモーダル:ViT+2D RoPE+トークン予算で画像、Conformerで音声を処理
  • MoEの妙:128人中8人+総合医だけ動かし「26B載せて4Bで走る」
  • PLEの妙:層ごとのメモをストレージに逃がしてRAMを節約、スマホで動く
  • 投機的デコード:小さな下書き役が先回り予測→本体が一括検証で爆速化

どの工夫も根っこは同じ哲学——「全部を毎回フルパワーでやらない。必要な所に必要なだけ計算を割く」。これが Gemma 4 が小さくても賢く、そして速い理由です。
