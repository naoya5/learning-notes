ソフトウェアアーキテクチャの基礎 — アーキテクチャ特性と6つのスタイル

アーキテクチャ特性(イリティ)と適応度関数を軸に、レイヤード〜マイクロサービスまで6つのスタイルをMermaid図で整理した学習ノート

目次

    TL;DR

    • アーキテクチャ特性＝システムが満たすべき「〜性（-ility / イリティ）」。要件には明記されない暗黙知から特定する力が設計の肝。
    • 特性は互いにトレードオフする（セキュリティを上げると性能が落ちる等）。だから"全部入り"は不可能で、絞るほど良い設計になる。
    • 適応度関数で特性を数値化し CI/CD で継続計測すると、「アーキテクチャが壊れていないか」を自動で守れる。
    • アーキテクチャ決定は「どう作るか」のルール＝ガードレール。特性（何を満たすか）とセットで考える。
    • スタイルは目的別の道具箱。レイヤード→単純さ / マイクロカーネル→拡張性 / サービスベース→実用的な分散 / EDA→非同期スケール / スペースベース→高並行 / マイクロサービス→独立デプロイ。

    アーキテクチャ特性（イリティ）

    ソフトウェアアーキテクチャは大きく 構造・アーキテクチャ特性・アーキテクチャ決定・設計指針 の4つで語れる（Richards & Ford『ソフトウェアアーキテクチャの基礎』）。このうち最初に向き合うのが「このシステムは何が"良い状態"なのか」を決めるアーキテクチャ特性だ。

    特性は要件定義書に「可用性99.9%」と書かれることもあるが、多くは「うちのドメインなら当然これが要る」という暗黙知として埋もれている。それを掘り起こすのが設計者の腕。

    適応度関数 — 良さを数値化する

    適応度関数は「解の良さ」を数値化する関数で、局所最適ではなく大域最適を目指すための物差しだ（遺伝的アルゴリズム由来の比喩）。アーキテクチャの文脈では、CI/CD で継続的に計測する品質ゲートとして実装する。結合度・レイテンシ・セキュリティスキャンなど「望ましい状態」を関数に落とし込み、壊れたら気づける状態を作る。これが進化的アーキテクチャの中核になる。 

    # 適応度関数の一例：p95 レイテンシが予算を破っていないか CI で検証する
def test_p95_latency_within_budget():
    samples = load_latency_samples("checkout")   # 直近の本番計測
    p95 = percentile(samples, 95)
    assert p95 < 100, f"p95={p95}ms が予算 100ms を超過"
    Tip: 適応度関数は性能だけでなく「層を飛び越えた依存がないか（結合度）」「禁止された import がないか」のような構造ルールにも使える。ArchUnit などがこの用途の定番。

    システムが満たす「〜性」を3つに分類する

    特性は数えると無数にあるが、おおむね次の3カテゴリで整理できる。

    分類意味代表例
    運用特性稼働・運用時の振る舞い可用性 / 性能 / スケーラビリティ / 弾力性 / 信頼性 / 回復性
    構造特性コード構造・変更しやすさ保守性 / 拡張性 / 設定容易性 / 移植性 / 再利用性
    横断特性分野をまたいで効く品質セキュリティ / アクセシビリティ / プライバシー / ユーザビリティ / 認証・認可

    トレードオフ — 特性は少ないほどいい

    アーキテクチャ特性はほとんどの場合、相互に影響しあう。セキュリティ向上のために処理を追加すればパフォーマンスは落ちるし、可用性を上げれば構成が複雑になる。すべてを同時に最大化することはできない。だから特性は取捨選択して、できるだけ少なく絞るほど、設計は明確になり守りやすくなる。

    Note: 目標は「最高のアーキテクチャ」ではなく「最も無理のないアーキテクチャ」。ドメインの関心事・要件・暗黙知から効く特性を特定する能力そのものが、アーキテクトの中心スキルになる。

    アーキテクチャ決定

    アーキテクチャ決定とは、システムをどう構築すべきかのルールを定めるものだ。たとえばレイヤードアーキテクチャで「ビジネス層とサービス層のみがデータベースに直接アクセスできる」と決める、といったもの。決定はチームの選択肢を意図的に狭めるガードレールとして働く。

    Tip: 重要な決定は ADR（Architecture Decision Record）として「背景・決定・理由・結果」を1ファイルに残すと、後から「なぜこうなっているのか」を辿れる。

    なお本記事に頻出するハッピーパスとは、エラーや例外が一切発生しない理想的な正常系の処理経路のこと。後述するマイクロカーネルのコアは、この「ハッピーパスだけ」を担当する。

    アーキテクチャスタイルの地図

    ここからは代表的なアーキテクチャスタイルを見ていく。スタイルは大きく、1プロセスに収めるモノリシック系（レイヤード・マイクロカーネル）と、ネットワーク越しに分割する分散系（サービスベース・イベント駆動・スペースベース・マイクロサービス）に分かれる。どれを選ぶかは「どの特性を最大化したいか」で決まる。

    レイヤードアーキテクチャ

    層ごとに役割を分離するアーキテクチャ。シンプルで、アーキテクチャを本格的に検討する前の出発点としてよく使われる。

    原則は閉鎖的で、上の層は必ずすぐ下の層を通してデータを渡す。ただし必要に応じて層を開放し、層を飛ばして取りにいけるようにする（パフォーマンス改善などのため）。

    特性とトレードオフ

    強い特性弱い特性
    シンプルさ / 低コスト / テスト容易性（層単位でモックしやすい）スケーラビリティ / 弾力性 / 性能 / デプロイ容易性（全体を再デプロイ）

    トレードオフ： 単純さと低コストの代わりに、変更容易性とスケールを諦める。各層をただ素通りするだけのリクエストが増えるとアーキテクチャシンクホール（層分割がオーバーヘッドにしかならないアンチパターン）に陥る。素通りが8割を超えるなら層分割を見直す合図。

    採用される場面

    • 小規模アプリ・社内ツール・PoC / MVP など「まず動かす」段階
    • 予算・納期・チーム規模が限られ、分散の複雑さを背負えないとき
    • アーキテクチャを本格検討する前のデフォルトの出発点

    マイクロカーネルアーキテクチャ

    プラグインアーキテクチャとも呼ばれる。製品ベース（ダウンロード／インストール型）のアプリに最適で、Eclipse・Chrome・Jira・VS Code などプラグインで拡張する製品で採用されている。

    項目内容
    目的最低限の機能を形式的に定義。「ハッピーパス」のみ担当
    設計原則循環的複雑度を最小化。カスタム処理はプラグインへ委譲
    知らなくてよいことプラグインの内部処理・ビジネスロジック

    肝はコアとプラグインの明確な分離、そして両者の境界を定めるコントラクトの設計だ。

    特性とトレードオフ

    強い特性弱い特性
    進化性 / 拡張性（プラグイン追加で機能拡張）/ テスト容易性 / デプロイ容易性（プラグイン単位）スケーラビリティ / 弾力性 / 耐障害性（コアがモノリシックなため）

    トレードオフ： 機能の追加・差し替えは抜群にしやすい一方、全体としてはモノリシックなのでスケールは苦手。コアとプラグインのコントラクトをどうバージョン管理するかが運用上の肝になる。

    採用される場面

    • IDE・ブラウザ・SaaS などプラグインで拡張する製品（VS Code / Eclipse / Jira）
    • ビジネスルールのバリエーションが多い領域（税計算・保険商品・申請フローなど、州や条件ごとに分岐）
    • サードパーティに拡張を開放したいエコシステム型プロダクト

    サービスベースアーキテクチャ

    マイクロサービスのハイブリッド。柔軟性が高く、最も実用的な分散スタイルの一つ。マイクロサービスやイベント駆動ほどの複雑さ・コストを払わずに、分散の利点を享受できる。

    項目内容
    粒度粗い（アプリケーションの一部）
    デプロイ個別デプロイ（EAR/WAR/アセンブリ）
    コンテナ化必須ではない
    DB通常は単一のモノリシックDBを共有
    通信通常 REST。メッセージング・RPC・SOAP も可

    マイクロサービスとレイヤードの中間。DB やUIが分かれることもある。ACID トランザクションが重要で「どこかで失敗したらロールバックしたい」処理に向く。

    特性とトレードオフ

    強い特性弱い特性
    デプロイ容易性 / 進化性 / テスト容易性 / 耐障害性（サービス単位で隔離）/ 分散の中では高いシンプルさ弾力性 / スケーラビリティ（共有DBが天井になる）

    トレードオフ： マイクロサービスの運用コストを払わずに分散の利点を得られるが、共有DBがスケールの上限になる。裏を返せば、DBを共有するおかげでACIDトランザクションを保ちやすく、サーガが要らない場面が多いのが実用上の強み。

    採用される場面

    • モノリスから段階的に分割していきたいとき（いきなりマイクロサービスにしない）
    • ACIDトランザクションが重要で、整合性を崩したくない業務系
    • マイクロサービスがオーバーキルな中規模アプリ、DDDで境界が引けているチーム

    イベント駆動アーキテクチャ（EDA）

    高度にスケーラブルで高パフォーマンスな分散・非同期型スタイル。「何かが起きた（イベント）」という事実を中心に、各コンポーネントが非同期に連携する。従来のリクエスト/レスポンス型が「A が B を呼び、結果を待って次へ進む」のに対し、EDA では 「A はイベントを発行するだけ。誰が受け取るかは A の関心事ではない」。

    向いている場面は「リアルタイムに反応すべき多消費システム」「監査が重要な領域」「マイクロサービス間の疎結合」。一つの出来事に複数の関心事がぶら下がるサービスは、EDA の恩恵を受けやすい。

    実世界の採用例
    • 配車・デリバリー：Uber/Lyft はドライバー位置を毎秒イベント化 → マッチング・料金計算・ETA が独立に subscribe。DoorDash は注文ステータス変化を全関係者に fan-out。
    • 配信：Netflix は視聴開始/一時停止/離脱をイベント化 → レコメンド学習・品質監視・課金集計を別チームが消費。
    • 決済：Stripe の Webhook はまさに EDA。銀行の不正検知は取引イベントをリアルタイムに ML 推論へストリーミング。
    • EC：Amazon は「注文」1つから在庫引当・倉庫指示・配送追跡・レコメンド更新・領収書メールが独立に走る。
    2つのトポロジ： イベントを各消費者が自由に受け取るブローカー型（シンプル・最も疎結合）と、中央のメディエーターが処理手順を制御するメディエーター型（エラー処理・調整がしやすい）がある。疎結合を取るか、制御しやすさを取るかの選択。

    特性とトレードオフ

    強い特性弱い特性
    性能 / スケーラビリティ / 弾力性 / 耐障害性 / 進化性（消費側を自由に足せる）シンプルさ / テスト容易性（非同期は追跡しづらい）/ データ整合性（結果整合性）

    トレードオフ： 高スループットと疎結合の代わりに、処理フローの追跡・デバッグが難しくなり、エラーハンドリング・重複・順序保証の設計が複雑になる。「いま全体がどうなっているか」を一望しにくいのが最大の難点。

    採用される場面

    • リアルタイムに多数が反応する系（配車のマッチング、配信の視聴イベント、決済Webhook）
    • 1つの出来事に複数の関心事がぶら下がるfan-out（EC注文 → 在庫・配送・通知・レコメンド）
    • 監査が重要で全イベントを記録したい領域、マイクロサービス間の疎結合な連携

    スペースベースアーキテクチャ

    WebサーバーをいくらスケールしてもDBが1つだとそこに処理が集中してボトルネックになる。スペースベースは、DBとは別にインメモリのデータグリッドを処理ユニットに持たせて処理を捌き、そこから非同期でDBへ書き込むことでボトルネックを解消する。 

    通常のWebアプリ:  Web → App → DB（中央）  ← ここが詰まる
スペースベース:    Web → 処理ユニット（インメモリ）  ← DBを直接読み書きしない
                              ↓（非同期）
                          データポンプ → DB（結果整合性）

    中核は仮想ミドルウェアの4コンポーネント。中でもデータグリッドがレプリケーションキャッシュを担う最重要要素だ。

    特性とトレードオフ

    強い特性弱い特性
    弾力性 / スケーラビリティ / 性能（いずれも極めて高い）シンプルさ / テスト容易性（本番規模の負荷再現が困難）/ 全体コスト / データ整合性

    トレードオフ： 桁外れの同時実行とスパイク耐性を得る代わりに、キャッシュ整合性とインメモリ運用の難しさ、高い構築・運用コストを払う。DBへは非同期反映なので結果整合性を許容できることが前提になる。

    採用される場面

    • 負荷が予測不能に急増する系（コンサートチケット、フラッシュセール、オークション、座席予約）
    • 同時接続数が読めず、DBが真っ先にボトルネックになる高並行アプリ
    • スパイク時でも一貫した低レイテンシを維持したいとき

    マイクロサービスアーキテクチャ（MSA）

    単一の責務を持つ小さなサービスの集合体で、各サービスが独立してデプロイ・スケール・開発できるアーキテクチャ。モノリシックでは規模拡大に伴い「一部を変えても全体を再デプロイ」「特定機能だけスケールしにくい」といった問題が出る。それを分割で解く。

    哲学: DDD の境界づけられたコンテキストを物理化したもの。そして「再利用よりも重複」——共通化は結合を生み独立デプロイを壊すので、多少コピーしても分離を優先する。

    特徴：

    • データは完全分離。共有DB・共有テーブル・他サービスDBへの直接アクセスは禁止。欲しいデータは「信頼できる唯一の情報源」へ問い合わせる。
    • 各サービスは1つのビジネス能力だけに責任を持つ。「ユーザー管理が注文データを直接触る」のは境界違反。
    • トランザクション境界とコレオグラフィを常に意識する。特にトランザクションが鬼門（サーガが頻発する設計は避ける）。
    • スケーラビリティ・弾力性・進化性・デプロイ容易性・テスト容易性が強い一方、分散の複雑さとネットワーク呼び出しの多さで性能は弱め。DevOps（CI/CD・k8s）なしには成立しない。

    サービス間通信は同期（REST/gRPC）と非同期（イベント駆動）を使い分ける。

    サーガとは: 各サービスのローカルトランザクションを順に実行し、途中で失敗したら成功分を逆順に補償トランザクションで打ち消す仕組み。例）注文→決済→在庫引当→配送手配で在庫引当が失敗したら、決済を返金・注文をキャンセルと逆順に巻き戻す。実行用とは別に取り消し用メソッドが要るので、使いすぎ注意。中央指揮役のいるオーケストレーション型と、イベント連鎖で進むコレオグラフィ型がある。

    特性とトレードオフ

    強い特性弱い特性
    スケーラビリティ / 弾力性 / 進化性 / デプロイ容易性 / テスト容易性 / 耐障害性 / モジュール性（いずれも最高クラス）性能（ネットワークホップが増える）/ シンプルさ / 全体コスト

    トレードオフ： サービスごとの独立性と細かいスケールを手に入れる代わりに、分散の複雑さ・性能低下・運用コストを背負う。とりわけ複数サービスをまたぐトランザクションはサーガが必要になり一気に難しくなる。CI/CD・k8s・監視といったDevOps基盤の成熟が前提。

    採用される場面

    • 大規模・多チームで独立して頻繁にデプロイしたいプロダクト
    • 機能ごとにスケール要件が大きく異なる（決済は堅く、レコメンドは大量に、など）
    • ドメインを明確に境界づけられたコンテキストへ分割でき、組織もそれに沿っている（コンウェイの法則）

    まとめ

    • アーキテクチャ特性（イリティ）は要件の裏にある暗黙知。効く特性を特定し、少数に絞るのが設計の出発点。
    • 特性はトレードオフする。全部入りは不可能。だから「何を諦めるか」を決めるのが本質。
    • 適応度関数で特性を数値化し CI/CD で継続計測すれば、アーキテクチャの劣化を自動で防げる。
    • スタイルは特性を実現する道具。単純さならレイヤード、拡張性ならマイクロカーネル、独立デプロイならマイクロサービス——目的から逆引きする。
    • 分散系は強力だがトランザクション・整合性・運用コストという代償を伴う。サーガを多用する設計は黄信号。
