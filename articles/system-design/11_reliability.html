信頼性・可用性・耐障害性の定義

信頼性 (Reliability)

システムが期待通りに動作する確率。MTBF（平均故障間隔）で測る。

定義
Reliability = MTBF / (MTBF + MTTR)

例：MTBF=1000時間、MTTR=1時間 → 99.9%信頼性

可用性 (Availability)

稼働時間の割合。ユーザーがサービスを使用できる時間の比率。

計算式
A = MTBF / (MTBF + MTTR)

SLO・SLAで表現され「99.9%」「99.99%」のように表記する。

耐障害性 (Fault Tolerance)

コンポーネントが故障しても、システム全体として正常動作を続ける性質。

冗長化・フェイルオーバー・グレースフルデグラデーションが主な実現手段。

高可用性（HA）ダウンタイム換算表

直列 vs 並列の可用性合成：直列接続では可用性が掛け合わされ低下する（A×B）。2つの99.9%コンポーネントを直列にすると99.8%になる。並列冗長化では 1-(1-A)^n で向上する。2台並列なら 1-(0.001)² = 99.9999%。

障害の種類

ハードウェア障害

ディスク故障・メモリ不良・NIC不良・電源障害・データセンター停電。大規模クラウド環境では常態として起きることを前提に設計する。

ソフトウェア障害

バグ・メモリリーク・デッドロック・設定ミス・依存ライブラリの不具合。人的ミスによるデプロイ失敗も含む。

ネットワーク分断

パーティション障害・高遅延・パケットロス・DNS障害。CAP定理のP（分断耐性）に関わる根本的な分散システムの課題。

人的ミス

Google/Amazon の調査では障害の70〜80%が人的ミス。誤ったconfig変更・誤削除・未検証デプロイなど。自動化とレビューフローで軽減する。

サーキットブレーカー (Circuit Breaker)

なぜ必要か — カスケード障害の防止

依存先サービスが遅延・障害を起こしたとき、呼び出し元がタイムアウトを待ち続けることでスレッドが枯渇し、障害が上流サービスへ連鎖する（カスケード障害）。サーキットブレーカーは依存先への呼び出しを素早く失敗させることで、障害の連鎖を断ち切る。

サーキットブレーカー 状態遷移ダイアグラム

3つの状態と遷移条件

状態動作遷移条件
CLOSEDリクエストを通常通り通過させる。失敗カウンターを管理するエラー率がしきい値を超えるか、連続失敗N回でOPENへ
OPENすべてのリクエストを即座に拒否（フォールバック実行）。依存先への呼び出しをスキップ設定したタイムアウト時間経過後にHALF-OPENへ
HALF-OPEN限定的なリクエストを通過させ回復を試験する成功→CLOSED / 失敗→再度OPEN

実装コード例（Python 疑似コード）

import time, threading
from enum import Enum

class State(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"

class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=30, half_open_max=3):
        self.failure_threshold = failure_threshold   # 連続失敗しきい値
        self.recovery_timeout = recovery_timeout     # OPEN後の待機秒数
        self.half_open_max = half_open_max           # HALF-OPENで通すリクエスト数
        self._state = State.CLOSED
        self._failure_count = 0
        self._last_failure_time = None
        self._lock = threading.Lock()

    def call(self, func, *args, **kwargs):
        with self._lock:
            if self._state == State.OPEN:
                elapsed = time.time() - self._last_failure_time
                if elapsed >= self.recovery_timeout:
                    self._state = State.HALF_OPEN
                    self._failure_count = 0
                else:
                    raise Exception("CircuitBreaker OPEN: fast fail")

        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

    def _on_success(self):
        with self._lock:
            self._failure_count = 0
            self._state = State.CLOSED

    def _on_failure(self):
        with self._lock:
            self._failure_count += 1
            self._last_failure_time = time.time()
            if self._failure_count >= self.failure_threshold:
                self._state = State.OPEN

パラメーター選定の指針
  • failureRateThreshold：SLOのエラー率上限を参考に設定（通常 50〜70%）
  • waitDurationInOpenState：依存サービスの平均復旧時間（MTTR）を参考に設定
  • minimumNumberOfCalls：低トラフィック環境では誤発動しやすいため高め設定
  • slidingWindowSize：短すぎると誤発動、長すぎると検知遅延。20〜100件が目安
  • 本番では必ずメトリクスをモニタリングし段階的チューニング

リトライとバックオフ (Retry & Backoff)

なぜリトライが必要か

ネットワークの一時的な障害・503エラー・レートリミット超過などは、再試行することで回復できる。単なる「再実行」ではなく、適切な戦略で行うことが重要。

単純リトライの危険性

大量クライアントが一斉にリトライすると、復旧しかけたサーバーに負荷が集中し再び障害を引き起こす（Thundering Herd 問題）。バックオフとジッターで分散させる必要がある。

Exponential Backoff の仕組み

待機時間の計算式
wait_time = min(base × 2^attempt, max_delay)

例：base=1s, max=60s の場合：1秒 → 2秒 → 4秒 → 8秒 → 16秒 → 32秒 → 60秒（以降上限）

Jitter（ランダム要素）の3種類

種類計算式特徴推奨度
Full Jitterrandom(0, min(cap, base×2^n))最も分散効果が高い。平均待機時間が短い推奨
Equal Jitterhalf + random(0, half)（halfは指数バックオフの半分）最低限の待機を保証。適度な分散適宜
Decorrelated Jittermin(cap, random(base, prev×3))前回の待機時間を使って次を計算。よりランダム適宜
No Jittermin(cap, base×2^n)Thundering Herd が発生しうる非推奨

実装コード例（Python — Exponential Backoff + Full Jitter）

import time, random, requests

def retry_with_backoff(
    func,
    max_retries: int = 5,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
    retryable_status: set = {429, 500, 502, 503, 504},
    deadline: float = None,          # デッドライン伝播用
):
    """Exponential Backoff + Full Jitter リトライ"""
    start = time.time()
    last_exception = None

    for attempt in range(max_retries + 1):
        # デッドライン超過チェック
        if deadline and (time.time() - start) >= deadline:
            raise TimeoutError(f"Deadline exceeded after {attempt} attempts")

        try:
            return func()
        except requests.HTTPError as e:
            status = e.response.status_code
            if status not in retryable_status or attempt == max_retries:
                raise
            last_exception = e
        except (requests.ConnectionError, requests.Timeout) as e:
            if attempt == max_retries:
                raise
            last_exception = e

        # Full Jitter: 0 ~ min(cap, base * 2^attempt)
        exp_backoff = base_delay * (2 ** attempt)
        wait = random.uniform(0, min(max_delay, exp_backoff))
        print(f"Attempt {attempt+1} failed. Retrying in {wait:.2f}s...")
        time.sleep(wait)

    raise last_exception

デッドライン伝播（Deadline Propagation）

上限時間を超えたリトライを止める

gRPC や HTTP ヘッダー（X-Request-Deadline）で呼び出し元から残り時間を伝播させ、タイムアウト済みのリクエストに対してリトライを行わないようにする。

例：フロントエンドのタイムアウトが3秒の場合、バックエンドに「残り2.5秒しかない」という情報を渡すことで、無意味なリトライによるシステム負荷を防ぐ。

冪等性との関係

リトライ可能な操作の条件：リトライは冪等な操作（同じリクエストを複数回送っても同じ結果になる操作）にのみ安全に適用できる。
冪等：GET、PUT、DELETE（RFC 7231）、冪等キー付きPOST
非冪等：冪等キーなしPOST（決済・注文など二重実行リスクあり）

タイムアウト設計

Connection Timeout

TCP接続確立までの最大待機時間。サーバーが応答しない状態を検知する。一般的に短め（1〜5秒）に設定。

Read Timeout

接続後、レスポンスデータ受信完了までの最大待機時間。処理時間の見積もりに基づいて設定（最も重要）。

Write Timeout

リクエスト送信完了までの最大待機時間。大きなペイロードを送信する際の上限として機能する。

適切な値の決め方

アプローチ計算方法例
SLO ベースSLO × 0.9 を上限とする（SLO内で完了させる余裕）SLO=1秒 → タイムアウト=900ms
P99 レイテンシ本番の P99 レイテンシ × 1.5〜2.0 倍P99=300ms → タイムアウト=500ms
エンドtoエンド呼び出しチェーン全体のSLOから各サービスに予算配分全体3秒 → A:1s, B:1s, C:0.5s, buffer:0.5s
段階的調整まず高めに設定し、実測レイテンシから段階的に引き下げる初期30秒→実績200ms→設定500ms

Timeout + Retry 組み合わせ戦略

# gRPC デッドライン設定例 (Python)
import grpc
from datetime import datetime, timedelta

def call_with_deadline(stub, request, total_budget_seconds=3.0):
    """全体バジェット内でリトライしながら呼び出す"""
    deadline = datetime.utcnow() + timedelta(seconds=total_budget_seconds)

    for attempt in range(3):
        remaining = (deadline - datetime.utcnow()).total_seconds()
        if remaining <= 0:
            raise grpc.RpcError("Deadline exceeded")

        # 個別タイムアウト = 残り時間の半分（上限1秒）
        per_call_timeout = min(remaining * 0.5, 1.0)
        try:
            return stub.DoWork(request, timeout=per_call_timeout)
        except grpc.RpcError as e:
            if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
                raise  # 全体デッドライン超過は再試行しない
            # その他のエラーはバックオフ後にリトライ
            time.sleep(0.1 * (2 ** attempt))

タイムアウトなし設定は禁止：タイムアウトを設定しないと、依存サービスの遅延がスレッドを永遠に保持し、最終的にスレッドプールが枯渇してサービス全体が停止する。すべての外部呼び出しにはタイムアウトを設定すること。

バルクヘッドパターン (Bulkhead Pattern)

船の隔壁（Bulkhead）のアナロジー

タイタニック号のような大型船には複数の防水隔壁（バルクヘッド）があり、一区画に浸水しても他の区画は保持される。ソフトウェアでも同様に、あるサービスへの障害が他のリソースを使い尽くさないよう隔離する。

バルクヘッドあり vs なし — スレッドプール分離の比較

分離の種類と比較

分離方法仕組みオーバーヘッドユースケース
スレッドプール分離サービスごとに独立したスレッドプールを割り当て中（スレッドコンテキストスイッチ）レイテンシ要件が緩いバッチ処理・非同期処理
セマフォ分離同時実行数の上限（セマフォ）のみ制御、同一スレッドで実行低低レイテンシ要件・高スループット
プロセス分離サービスをプロセスやコンテナで分離高重大なサービス間の完全分離
接続プール分離DBコネクションプールをサービスごとに分離低〜中DB依存の多いバックエンド

Saga パターンと補償トランザクション

分散環境でのトランザクション管理の課題

マイクロサービス環境では各サービスが独自のDBを持つため、複数サービスをまたぐ2PCトランザクションは適用できない（可用性・スケーラビリティが大きく損なわれる）。Sagaパターンは一連のローカルトランザクションを連鎖させ、失敗時には補償トランザクション（Undo操作）を実行することで整合性を保つ。

Choreography Saga — イベント駆動型（注文フロー）
Orchestration Saga — 中央コーディネーター型

Choreography vs Orchestration

観点Choreography（コレオグラフィ）Orchestration（オーケストレーション）
結合度疎結合（イベントで連携）中結合（中央コーディネーター依存）
可観測性全体フローの追跡が困難フローが一箇所で管理・追跡しやすい
複雑性イベント連鎖の把握が難しいビジネスロジックが集中し理解しやすい
単一障害点なしオーケストレーターが SPOFになりうる
適するケースシンプルなフロー・イベント駆動アーキテクチャ複雑なフロー・ビジネスプロセス管理

補償トランザクション設計

冪等性 (Idempotency) の設計

冪等性とは：同じ操作を1回実行しても100回実行しても、システムの状態と結果が同じであるという性質。ネットワーク障害でのリトライ・二重実行・メッセージキューのat-least-once配信に対して安全なシステムを構築する基盤となる。

冪等性キー（Idempotency Key）の実装

# 決済API 冪等性実装例（FastAPI + PostgreSQL）
from fastapi import FastAPI, Header, HTTPException
from sqlalchemy import text
import uuid

app = FastAPI()

@app.post("/v1/payments")
async def create_payment(
    payment: PaymentRequest,
    idempotency_key: str = Header(..., alias="Idempotency-Key")
):
    """
    クライアントは一意のIdempotency-Keyを生成してヘッダーに含める。
    同じキーで再送した場合は前回の結果を返す（二重決済を防止）。
    """
    async with db.transaction():
        # 1. キーが既に存在するか確認
        existing = await db.fetchrow(
            "SELECT response, status FROM idempotency_keys WHERE key = $1",
            idempotency_key
        )

        if existing:
            if existing["status"] == "processing":
                raise HTTPException(409, "Request is still being processed")
            # 既存のレスポンスをそのまま返す（キャッシュ）
            return existing["response"]

        # 2. キーを「処理中」として登録（UNIQUE制約で競合防止）
        try:
            await db.execute(
                """INSERT INTO idempotency_keys
                   (key, status, created_at) VALUES ($1, 'processing', now())""",
                idempotency_key
            )
        except UniqueViolationError:
            raise HTTPException(409, "Concurrent request with same key")

        # 3. 実際の決済処理
        result = await payment_processor.charge(payment)

        # 4. 結果を保存（TTL付き）
        await db.execute(
            """UPDATE idempotency_keys
               SET status='completed', response=$1, expires_at=now()+interval'24h'
               WHERE key=$2""",
            result.dict(), idempotency_key
        )
        return result

DBのUNIQUEキーを使った重複防止

-- 冪等性キーテーブルの設計
CREATE TABLE idempotency_keys (
    key         UUID PRIMARY KEY,              -- クライアント生成UUID
    status      VARCHAR(20) NOT NULL,          -- processing / completed / failed
    request_hash VARCHAR(64),                  -- リクエストボディのハッシュ（改ざん検知）
    response    JSONB,                         -- キャッシュするレスポンス
    created_at  TIMESTAMPTZ DEFAULT now(),
    expires_at  TIMESTAMPTZ                    -- TTL（24時間程度）
);

-- TTL切れを自動削除するインデックス
CREATE INDEX idx_idempotency_expires ON idempotency_keys (expires_at);

-- 決済テーブルにも冪等性キー参照を持たせる
ALTER TABLE payments ADD COLUMN idempotency_key UUID REFERENCES idempotency_keys(key);

Graceful Degradation と Fallback

Graceful Degradation（優雅な縮退）

一部の機能が使用不能になっても、コアとなる機能を継続して提供する設計思想。「全か無か」ではなく「できる範囲でサービスを継続」することを優先する。

例：レコメンドシステムが落ちても、デフォルトの人気商品リストを表示する。

Fallback 戦略
  • デフォルト値：空のリスト・ゼロ・"N/A"など静的な値を返す
  • キャッシュ値：直前にキャッシュされた古いデータを返す（stale-while-revalidate）
  • 静的コンテンツ：あらかじめ生成済みのHTMLを返す（静的サイトフォールバック）
  • 劣化版機能：軽量な代替処理を実行

Feature Flag による機能切り替え

# Feature Flag による障害時の機能制御（Python + LaunchDarkly/ローカル実装）
class FeatureFlags:
    """実行時に機能を ON/OFF する"""
    def __init__(self):
        self._flags = {
            "enable_recommendations": True,
            "enable_inventory_check": True,
            "enable_fraud_detection": True,
        }

    def is_enabled(self, flag: str) -> bool:
        return self._flags.get(flag, False)

flags = FeatureFlags()

async def get_product_page(product_id: str) -> dict:
    product = await product_service.get(product_id)

    # レコメンデーションが無効化されている場合はフォールバック
    if flags.is_enabled("enable_recommendations"):
        try:
            recs = await recommendation_service.get(product_id)
        except Exception:
            recs = await get_popular_products()  # フォールバック
    else:
        recs = await get_popular_products()      # Feature Flag で無効化

    # 在庫確認が無効化されている場合は「在庫あり」を仮定
    if flags.is_enabled("enable_inventory_check"):
        in_stock = await inventory_service.check(product_id)
    else:
        in_stock = True  # 楽観的フォールバック（後で在庫切れ対処）

    return {"product": product, "recommendations": recs, "in_stock": in_stock}

カオスエンジニアリング (Chaos Engineering)

定義（Netflix原則）："Chaos Engineering is the discipline of experimenting on a system in order to build confidence in the system's capability to withstand turbulent conditions in production." — 本番のような乱れた条件でもシステムが耐えられる確信を得るために、システムへの実験を行う規律。

カオス実験の5ステップ

1. 定常状態の定義

「正常」の定量的な定義を決める。例：成功率99.9%・P99レイテンシ200ms以下・1秒あたり10,000リクエスト処理可能。これが崩れたら「障害」と判断するしきい値。

2. 仮説の立案

「DBの1ノードが落ちても定常状態は維持される」のような形で仮説を立てる。仮説が明確でないと実験の意味がない。

3. 実験の実行

実際に障害を注入する（CPU負荷・ネットワーク遅延・プロセス停止・ディスク満杯など）。最初は小さいブラストラディウス（影響範囲）から始める。

4. 観察と検証

メトリクス・ログ・トレースを観察し、定常状態が維持されているか検証する。仮説が成立すれば自信、崩れれば改善のチャンス。

5. 改善とループ

発見した弱点を修正し、より複雑な実験へと移行する。定期的な実施（Game Day）で継続的に検証する。

ブラストラディウス最小化

実験の影響範囲を最小化する原則。最初はステージング環境・次に本番の数%のトラフィック・監視を強化した状態で実施し、問題を即座にロールバックできる準備をする。

DR (Disaster Recovery) と RTO/RPO

RTO（Recovery Time Objective）

目標復旧時間：障害発生から業務を再開できるまでの最大許容時間。「何時間までなら止まっていてもいいか」を定義する。

例
RTO = 4時間（金融）/ 1時間（EC）/ 5分（ミッションクリティカル）
RPO（Recovery Point Objective）

目標復旧時点：障害発生時にどの時点までのデータ損失を許容できるか。「最悪どこまで巻き戻っていいか」を定義する。

例
RPO = 24時間（日次バックアップ）/ 1時間 / 0秒（ゼロデータ損失）

バックアップ戦略

フルバックアップ

全データを毎回バックアップ。復元が最もシンプル。ストレージ・時間コストが高い。週次が一般的。

増分バックアップ

前回のバックアップ以降に変更されたデータのみ保存。日次で実施。フル+増分の組み合わせが標準的。

スライディングウィンドウ: Count-based vs Time-based

Resilience4j はCLOSED状態でのエラー率評価に2種類のスライディングウィンドウをサポートします。

import time, threading
from collections import deque
from enum import Enum

class WindowType(Enum):
    COUNT_BASED = "count"
    TIME_BASED = "time"

class SlidingWindow:
    """
    Count-based: 直近N回の結果を評価
    Time-based:  直近N秒間の結果を評価
    """
    def __init__(self, window_type: WindowType, size: int):
        self.window_type = window_type
        self.size = size          # COUNT_BASED: 件数、TIME_BASED: 秒数
        self._results = deque()   # (timestamp, is_success) のタプル
        self._lock = threading.Lock()

    def record(self, is_success: bool):
        with self._lock:
            now = time.time()
            self._results.append((now, is_success))
            self._trim()

    def _trim(self):
        """ウィンドウ外のエントリを除去"""
        if self.window_type == WindowType.COUNT_BASED:
            while len(self._results) > self.size:
                self._results.popleft()
        else:  # TIME_BASED
            cutoff = time.time() - self.size
            while self._results and self._results[0][0] < cutoff:
                self._results.popleft()

    def get_failure_rate(self) -> float:
        """エラー率（0.0〜1.0）を返す。件数が最小値に満たなければ-1"""
        with self._lock:
            self._trim()
            if len(self._results) == 0:
                return 0.0
            failures = sum(1 for _, ok in self._results if not ok)
            return failures / len(self._results)

    def get_call_count(self) -> int:
        with self._lock:
            self._trim()
            return len(self._results)


class AdvancedCircuitBreaker:
    """
    スライディングウィンドウ + HALF-OPEN試験リクエスト管理
    + メトリクス収集
    """
    def __init__(
        self,
        name: str,
        failure_rate_threshold: float = 0.5,    # 50%超でOPEN
        minimum_calls: int = 10,                  # 最小評価呼び出し数
        wait_duration_open: float = 30.0,         # OPEN継続秒数
        half_open_permitted: int = 5,             # HALF-OPENで通す試験数
        window_type: WindowType = WindowType.COUNT_BASED,
        window_size: int = 20,
    ):
        self.name = name
        self.failure_rate_threshold = failure_rate_threshold
        self.minimum_calls = minimum_calls
        self.wait_duration_open = wait_duration_open
        self.half_open_permitted = half_open_permitted

        self._state = "CLOSED"
        self._window = SlidingWindow(window_type, window_size)
        self._open_time = None
        self._half_open_calls = 0
        self._lock = threading.Lock()

        # メトリクス
        self._metrics = {
            "total_calls": 0,
            "successful_calls": 0,
            "failed_calls": 0,
            "not_permitted_calls": 0,  # OPEN時の拒否数
            "state_transitions": [],
        }

    def _transition_to(self, new_state: str):
        old_state = self._state
        self._state = new_state
        self._metrics["state_transitions"].append({
            "from": old_state,
            "to": new_state,
            "at": time.time(),
        })
        print(f"[CircuitBreaker:{self.name}] {old_state} → {new_state}")

    def call(self, func, *args, **kwargs):
        with self._lock:
            state = self._check_state()

        if state == "OPEN":
            self._metrics["not_permitted_calls"] += 1
            raise Exception(f"CircuitBreaker '{self.name}' is OPEN")

        if state == "HALF_OPEN":
            with self._lock:
                if self._half_open_calls >= self.half_open_permitted:
                    self._metrics["not_permitted_calls"] += 1
                    raise Exception(f"CircuitBreaker '{self.name}' HALF_OPEN limit reached")
                self._half_open_calls += 1

        self._metrics["total_calls"] += 1
        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

    def _check_state(self) -> str:
        if self._state == "OPEN":
            if time.time() - self._open_time >= self.wait_duration_open:
                self._half_open_calls = 0
                self._transition_to("HALF_OPEN")
        return self._state

    def _on_success(self):
        with self._lock:
            self._metrics["successful_calls"] += 1
            self._window.record(True)
            if self._state == "HALF_OPEN":
                # HALF_OPEN中に成功: CLOSEDに戻る
                self._window = SlidingWindow(
                    WindowType.COUNT_BASED, 20)  # ウィンドウリセット
                self._transition_to("CLOSED")

    def _on_failure(self):
        with self._lock:
            self._metrics["failed_calls"] += 1
            self._window.record(False)

            if self._state == "HALF_OPEN":
                # HALF_OPEN中に失敗: 再度OPEN
                self._open_time = time.time()
                self._transition_to("OPEN")
                return

            # CLOSED: エラー率チェック
            if (self._window.get_call_count() >= self.minimum_calls and
                    self._window.get_failure_rate() >= self.failure_rate_threshold):
                self._open_time = time.time()
                self._transition_to("OPEN")

    def get_metrics(self) -> dict:
        return {
            **self._metrics,
            "current_state": self._state,
            "current_failure_rate": self._window.get_failure_rate(),
        }
Bulkheadとの組み合わせ（Semaphore方式）
import threading

class BulkheadCircuitBreaker:
    """
    セマフォBulkhead + CircuitBreaker の組み合わせ
    同時実行数を制限しつつ障害時は遮断
    """
    def __init__(self, cb: AdvancedCircuitBreaker,
                 max_concurrent: int = 10):
        self.cb = cb
        self._semaphore = threading.BoundedSemaphore(max_concurrent)

    def call(self, func, *args, **kwargs):
        acquired = self._semaphore.acquire(blocking=False)
        if not acquired:
            raise Exception("Bulkhead full: too many concurrent calls")
        try:
            return self.cb.call(func, *args, **kwargs)
        finally:
            self._semaphore.release()

Error Budget Policy

SLO（例: 99.9%）を満たせなかった場合、残りのError Budgetに応じて開発速度を調整するポリシーです。

Error Budget残量アクション
>100%（余裕あり）機能追加・リスクある変更を許可
50〜100%通常のリリースペース
10〜50%リリース頻度を下げる。信頼性改善を優先
<10%リリース凍結。信頼性作業のみ許可
0%（使い切り）次の期間まで新機能リリース禁止

Error Budgetの計算：月間ダウンタイム許容量 = (1 - SLO) × 月間分数。SLO=99.9%なら月43.8分がError Budget。

Postmortem文化: 非難しない根本原因分析

Googleのpostmortemの基本姿勢は Blameless（非難なし）。障害は個人の失敗ではなくシステムの失敗と捉え、再発防止に集中します。

Chaos Monkey: 実際の実行コードと判断基準

NetflixはAWS上でランダムにEC2インスタンスをシャットダウンする「Chaos Monkey」を本番常時稼働させています。その思想は「コントロールされた障害の方が、予期しない障害より良い」です。

import random, boto3, time, logging
from datetime import datetime, time as dt_time

logger = logging.getLogger("ChaosMonkey")

class ChaosMonkey:
    """
    Netflixの Chaos Monkey を模した本番インスタンス自動停止
    ビジネス時間内のみ実行（意図的に対応力を鍛える）
    """
    def __init__(self,
                 aws_region: str = "ap-northeast-1",
                 kill_probability: float = 0.02,   # 2%: 1日1台程度
                 safe_hours: tuple = (9, 17),       # 9〜17時のみ実行
                 excluded_groups: list = None):
        self.ec2 = boto3.client("ec2", region_name=aws_region)
        self.kill_probability = kill_probability
        self.safe_hours = safe_hours
        self.excluded_groups = excluded_groups or ["critical-db", "kafka"]

    def _is_safe_hour(self) -> bool:
        """ビジネスアワー内かどうか"""
        now = datetime.now().time()
        return dt_time(self.safe_hours[0]) <= now <= dt_time(self.safe_hours[1])

    def _get_eligible_instances(self) -> list:
        """停止候補のインスタンスを取得（除外グループ以外）"""
        response = self.ec2.describe_instances(
            Filters=[{"Name": "instance-state-name", "Values": ["running"]}]
        )
        instances = []
        for reservation in response["Reservations"]:
            for inst in reservation["Instances"]:
                tags = {t["Key"]: t["Value"] for t in inst.get("Tags", [])}
                group = tags.get("chaos-monkey-group", "default")
                if group not in self.excluded_groups:
                    instances.append(inst)
        return instances

    def run(self, dry_run: bool = True):
        """メインループ: 確率的にインスタンスを停止"""
        if not self._is_safe_hour():
            logger.info("Outside safe hours, skipping chaos injection")
            return

        instances = self._get_eligible_instances()
        if not instances:
            logger.info("No eligible instances found")
            return

        for instance in instances:
            if random.random() < self.kill_probability:
                instance_id = instance["InstanceId"]
                logger.warning(f"CHAOS: Terminating {instance_id}")
                if not dry_run:
                    self.ec2.terminate_instances(InstanceIds=[instance_id])
                # Blast Radius制御: 1回の実行で停止するのは最大1台
                break

Serializable Snapshot Isolation (SSI)

PostgreSQL 9.1以降が実装するSSIは、MVCCのSnapshotをベースにしつつ直列化可能性（Serializability）を保証する手法です。

  • 読み取りロックなしでSerializableを実現（2PLより高スループット）
  • 「Anti-dependency（読み書きサイクル）」を検知しアボートで解決

Fencing Token: Split Brain防止

分散ロックを使ってリソースを保護する際、ロック保持者がGCポーズやネットワーク遅延でロックの期限切れに気付かずに処理を続けると Split Brain（2つのノードが同時にリソースを操作）が発生します。Fencing Tokenはこれを防ぐ手法です。

"""
Fencing Tokenの仕組み:

1. クライアントAがロック取得: token=100 を受け取る
2. クライアントAがGCポーズ（数十秒）でロックが期限切れ
3. クライアントBがロック取得: token=101 を受け取る
4. クライアントBがtoken=101でDBに書き込み（成功）
5. クライアントAが復帰し古いtoken=100でDBに書き込もうとする
   → DBがtoken=100 < current_token(101) を検知して拒否！
"""

class FencedStorage:
    """Fencing Tokenを検証するストレージ"""

    def __init__(self):
        self._data = {}
        self._fencing_tokens = {}  # key → 最後に書き込んだtoken

    def write(self, key: str, value, fencing_token: int) -> bool:
        current_token = self._fencing_tokens.get(key, -1)
        if fencing_token <= current_token:
            raise Exception(
                f"Stale write rejected: token={fencing_token}, "
                f"current={current_token}"
            )
        self._data[key] = value
        self._fencing_tokens[key] = fencing_token
        return True


# ZooKeeperでのFencing Token取得例
import kazoo.client as kz

def acquire_lock_with_fencing(zk: kz.KazooClient, lock_path: str):
    # ZooKeeperのzxid（単調増加するトランザクションID）をTokenとして使用
    lock = zk.Lock(lock_path)
    if lock.acquire(timeout=5):
        stat = zk.exists(lock_path)
        fencing_token = stat.mzxid  # ZooKeeper Transaction ID（常に増加）
        return lock, fencing_token
    raise Exception("Failed to acquire lock")
Compare-and-Swap (CAS)

「現在の値が期待値と一致する場合のみ新しい値に更新する」原子操作です。分散ロックや楽観的ロックの基盤となります。

# Redis での CAS（WATCH + MULTI/EXEC）
import redis
r = redis.Redis()

def cas_update(key: str, expected: str, new_val: str) -> bool:
    with r.pipeline() as pipe:
        try:
            pipe.watch(key)
            current = pipe.get(key)
            if current and current.decode() != expected:
                return False  # 値が変わっていた
            pipe.multi()
            pipe.set(key, new_val)
            pipe.execute()
            return True
        except redis.WatchError:
            return False  # 競合: 別クライアントが変更

# DynamoDB条件付き書き込み（同等）
dynamodb.update_item(
    Key={"id": "user:42"},
    UpdateExpression="SET balance = :new",
    ConditionExpression="balance = :expected",
    ExpressionAttributeValues={
        ":new": 900, ":expected": 1000
    }
)
Increment-and-Fetch vs Compare-and-Swap
操作使うべき場面注意点
Increment-and-Fetch（INCR）カウンター・シーケンスID・いいね数単調増加のみ。「残高を1000に設定」はできない
Compare-and-Swap（CAS）楽観的ロック・在庫の原子的減算・設定の条件更新競合時はアボート＆リトライが必要。競合多発時はスループット低下
Fetch-and-Addバッチ処理のオフセット確保加算量が決まっている場合のみ

if current == nil then return -1 end
if current < tonumber(ARGV[1]) then return -2 end
return redis.call('DECRBY', KEYS[1], ARGV[1])
"""

演習1: 本番インシデントのPostmortemを書く

シナリオ

ECサービスのキャッシュTTLを商品データ・カテゴリデータ・レコメンドデータすべてで同じ値（3600秒）に設定していたため、1時間ごとに大量のキャッシュが同時に期限切れを起こし、DBへの大量クエリが発生。3時間にわたりAPIのレスポンスタイムが10倍に悪化した。

Postmortem 模範解答

根本原因

  1. 直接原因：全キャッシュキーのTTLが同一値（3600秒）で設定されており、Redisリスタート後に全キーが同時にCold Startを起こした（Thundering Herd / Cache Stampede）。
  2. 設計上の問題：Redisリスタートによるキャッシュ全消去のリスクをメンテナンス計画に含めていなかった。
  3. 監視の問題：キャッシュヒット率のアラートが設定されておらず、問題の根本原因特定に30分かかった。

対策

即時対応（当日実施）

  • DB接続プールの一時的拡大
  • キャッシュウォームアップスクリプトの手動実行（人気商品のプリロード）

短期対応（1週間以内）

  • TTLにランダムジッター追加（±20%）: TTL = base_ttl + random(0, base_ttl * 0.2)
  • Redisメンテナンス時の自動キャッシュウォームアップ手順を追加
  • キャッシュヒット率アラートを追加（<80%でWarning、<50%でCritical）

長期対応（1ヶ月以内）

  • Stale-While-Revalidate パターンの導入（期限切れキャッシュをバックグラウンドで更新、古い値はしばらく返す）
  • Cache-Aside → Read-Through への移行検討（ミスオープンをライブラリ側でシリアライズ）
  • Probability-based early expiration: prob = exp(-delta * beta * log(remaining_ttl))で期限前に確率的に更新
  • Redisメンテナンス手順書の更新とリハーサル実施

将来検討（四半期以内）

  • マルチ層キャッシュ（ローカルメモリキャッシュ → Redis → DB）の導入でRedis単点依存を解消
  • GameDayの実施（意図的なRedis停止訓練）

演習3: RTO/RPO設計問題（3シナリオ）

各シナリオのRTO/RPO要件に最適なDR（Disaster Recovery）戦略を設計します。

RPO（Recovery Point Objective）= データ損失の許容上限時間
RTO（Recovery Time Objective）= サービス復旧までの許容時間

シナリオA: 金融決済システム（RTO=15分, RPO=0秒）

DR戦略: Active-Active マルチリージョン

  • 2リージョン（例: ap-northeast-1, ap-northeast-3）でActive-Active構成
  • DBは同期レプリケーション（Amazon Aurora Global Database / Google Spanner）
  • RPO=0秒を達成するには、書き込みが両リージョンに同期コミットされるまで成功を返さない
  • Route53 ヘルスチェック + フェイルオーバールーティングで自動切り替え（RTO: 数分）

バックアップ戦略・インフラ

  • RDS Aurora: 同期レプリケーション + 継続的バックアップ（PITR: Point-in-Time Recovery）
  • S3 Cross-Region Replication（CRR）+ Versioning
  • コスト: 通常の約2倍（2リージョン全稼働）
  • レイテンシトレードオフ: 同期書き込みで数ms増加

シナリオB: SNSのタイムライン（RTO=1時間, RPO=1時間）

DR戦略: Warm Standby（温スタンバイ）

  • DR リージョンに最小構成（1/4スケール）でスタンバイ環境を常時起動
  • DBは非同期レプリケーション（MySQL Binlog Replication / DynamoDB Global Table）
  • フェイルオーバー時に DR リージョンのスケールアウトを実行（Auto Scaling）
  • タイムライン・いいね数は Eventual Consistency でOK（SNSは厳密整合性不要）

バックアップ頻度・コスト

  • DB スナップショット: 1時間ごと（RPO=1時間を満たす）
  • S3: 同期レプリケーション（メディアファイルはデータロスが大きい）
  • コスト: 本番の約30〜40%追加（スタンバイの最小構成）
  • RTO=1時間: スタンバイ起動（既に起動済み）+ スケールアウト（15〜30分） + ヘルスチェック（10分）

シナリオC: ログ分析システム（RTO=24時間, RPO=24時間）

DR戦略: Backup & Restore（コールドスタンバイ）

  • DR環境は常時起動不要（コールド）。障害時に初めてインスタンスを起動
  • DBは日次フルバックアップ + WAL/Binlogを S3に保存
  • インフラは Terraform/CloudFormation として IaC 化し、緊急時にコマンド一発で再構築

バックアップ頻度・コスト

  • Redshift/BigQuery スナップショット: 日次（RPO=24時間を満たす）
  • S3 Glacier（低コストアーカイブ）: 長期保存（30日〜7年）
  • コスト: S3ストレージのみ（本番の5〜10%追加）
  • RTO=24時間: インスタンス起動（30分）+ DBリストア（数時間〜24時間）+ 検証
