01

スケーラビリティとは

定義

スケーラビリティとは、負荷（リクエスト数・データ量・ユーザー数）が増大した際に、パフォーマンスを維持しながらシステムを拡張できる能力のことです。

単に「大きくする」ことではなく、増加した負荷に対してコストと複雑度を抑えながら線形に近い性能を保つことが目標です。

Throughput（スループット）

単位時間あたりに処理できるリクエスト数（RPS / QPS）。スケールアウトで線形向上が理想。

Latency（レイテンシ）

1リクエストの処理時間。p50/p95/p99で測定。負荷増大でキューイングによる遅延が増加する。

Concurrency（同時接続数）

同時に処理できるリクエスト数。スレッド数・コネクションプール・イベントループ方式で決まる。

Amdahlの法則（並列化の限界）

プログラムの並列化で得られる高速化の理論上限を示す法則。直列部分がボトルネックになる。

Amdahlの法則
Speedup(N) = 1 / (S + (1-S)/N)

S = 直列実行が必要な割合（並列化不可能部分） N = プロセッサ（コア）数

例: S=0.1（10%が直列）の場合、N→∞でも最大10倍の高速化が上限
Gustafsonの法則（問題規模の拡大）

Amdahlの法則は「問題サイズ固定」が前提。Gustafsonの法則は問題サイズをコア数に合わせてスケールさせることで、より楽観的なスケーリングを示す。

Gustafsonの法則
Speedup(N) = N - S×(N-1)

= α + N×(1-α)　where α = 直列部分の割合

実際のクラウドシステムでは問題規模も増大するため、Gustafsonがより現実的

Amdahlの法則 — 並列化の上限曲線

02

垂直スケーリング (Scale Up) vs 水平スケーリング (Scale Out)

垂直スケーリング (Scale Up)

単一サーバーのCPU・メモリ・ディスクを強化する方法。アプリケーションの変更が不要で導入が容易。

限界：物理的な上限（例：最大数TB RAM）があり、コストが指数的に増加する。単一障害点のリスクも残る。

  • ✓ 実装変更不要
  • ✓ レイテンシへの影響が少ない
  • ✗ 物理的上限がある
  • ✗ ダウンタイムが必要な場合あり
  • ✗ コストが非線形増加
水平スケーリング (Scale Out)

サーバーの台数を増やして負荷を分散する方法。ステートレスなアプリケーション設計が前提条件。

必要条件：セッション共有（Redis）・共有ストレージ・ロードバランサーが必要。

  • ✓ 理論上無制限にスケール可能
  • ✓ 冗長性・高可用性が向上
  • ✗ 分散システムの複雑性増加
  • ✗ ステートレス化の設計変更が必要
  • ✗ データ一貫性の考慮が必要

コスト比較：垂直スケール vs 水平スケール

クラウドでのAuto Scaling

AWS Auto Scaling / GCP Autoscaler / Azure VMSS などは、CPU使用率・リクエスト数などのメトリクスに基づいてインスタンスを自動追加/削除する。

スケールアウトのトリガー例（AWS）

# Target Tracking Policy（CPU 70%を目標）
{
  "TargetValue": 70.0,
  "PredefinedMetricSpec": {
    "PredefinedMetricType": "ASGAverageCPUUtilization"
  },
  "ScaleOutCooldown": 60,
  "ScaleInCooldown": 300
}

注意点

  • • ウォームアップ時間（新インスタンス起動に数十秒〜数分）
  • • スケールイン時のGraceful Shutdown（処理中リクエストの保護）
  • • クールダウン期間（過剰なスケール防止）
  • • 最小・最大インスタンス数の設定
  • • Predictive Scalingで予測的事前スケール
項目 垂直スケーリング（Scale Up） 水平スケーリング（Scale Out）
コスト曲線指数関数的増加ほぼ線形
上限物理的上限あり（数TBなど）理論上無限
実装の複雑さ低（アプリ変更不要）高（ステートレス化必要）
可用性単一障害点リスク冗長性あり
ダウンタイムスケール時に必要な場合ありゼロダウンタイム可能
データ一貫性問題なし分散トランザクションが必要な場合
適用例DB（初期）、モノリシックアプリWeb/APIサーバー、マイクロサービス
03

コンシステントハッシング (Consistent Hashing)

通常のモジュロハッシュの問題

ノード数Nでハッシュする場合：node = hash(key) % N

ノードを1台追加してN+1になると、ほぼすべてのキーのマッピングが変わるため、大量のキャッシュミスや大規模なデータ移動が発生する。

再マッピング率 ≈ N/(N+1) ≈ 100%（Nが大きい場合もほぼ全キーに影響）
コンシステントハッシングの仕組み

0〜2³²-1（または0〜360°）のハッシュ空間をリング（環）状に配置し、各ノードをリング上の点にマッピング。キーは「時計回りに最初に見つかるノード」に割り当てられる。

メリット：ノード追加・削除時に影響を受けるキーは 1/N（平均） のみ。

ノード追加時の再マッピング = 全キーの約 1/(N+1) ノード削除時の再マッピング = 削除ノードのキー数のみ（隣接ノードへ移行）

コンシステントハッシュリング（バーチャルノードあり）

バーチャルノード（Virtual Nodes / vnodes）

各物理ノードをリング上の複数の仮想ノード（通常100〜200個）にマッピングすることで、偏りのない均等な負荷分散を実現する。

新ノード追加時は全既存ノードからキーを少量ずつ移行できるため、リバランシングが滑らか。スペックの異なるサーバーには仮想ノード数を変えることで重み付けが可能。

Cassandraのvnode設定例

# cassandra.yaml
num_tokens: 256   # デフォルトvnode数
# 高スペックノード
num_tokens: 512   # 2倍の負荷を受け持つ

利用システム例

  • Cassandra vnodeで均等分散
  • DynamoDB 内部でコンシステントハッシング
  • Memcached クライアント側ライブラリで実装
  • CDN エッジノード選択に利用
  • Redis Cluster 16384スロット方式（変形版）

面接ポイント：ホットスポット対策

  • 特定のキー（有名人のUserIDなど）が超集中する場合、そのキーを key + random_suffix(0..99) に分割してシャードに分散させる
  • 読み取りホットスポットはレプリカの増加で対応
  • 書き込みホットスポットはバッファリング＋バッチ処理で緩和
04

データシャーディング (Sharding)

水平シャーディングの原理

大きなデータセットをシャード（Shard）と呼ぶ小さな単位に分割し、複数のデータベースノードに分散して格納する技術。各シャードは全体の部分集合を持ち、独立して動作する。

例：1億ユーザーを4シャードに分割 Shard 0: user_id % 4 == 0 → 2500万ユーザー Shard 1: user_id % 4 == 1 → 2500万ユーザー ...

シャーディング方式の比較

シャードキーの設計原則

良いシャードキーの条件

  • ✓ 高カーディナリティ（値の種類が多い）
  • ✓ 均等なデータ分散を促す
  • ✓ クエリパターンに合っている
  • ✓ 変更頻度が低い
  • ✓ 関連データを同じシャードに集約できる

悪いシャードキーの例

  • ✗ タイムスタンプ（書き込みが最新シャードに集中）
  • ✗ ステータス（active/inactiveで偏り）
  • ✗ 国コード（人口の多い国に偏り）
  • ✗ 頻繁に更新される値
オンラインシャードリバランシング

データが偏ってきたシャードを再分割する作業。ダウンタイムなしで行うには以下のステップが必要：

  1. 新しいシャードを追加し、データを二重書き込み開始
  2. 既存データをバックグラウンドで移行（マイグレーション）
  3. 移行完了後、新シャードへの読み取りに切り替え
  4. 古いシャードの不要データを削除

Vitess Citus MongoDB などがオンラインリバランスをサポート

05

データベース読み書き分離 (Read/Write Splitting)

Primary-Replica構成

書き込みはPrimaryに集中し、読み取りはReplicaへ分散する構成。読み取りが多いアプリ（一般的に70-90%が読み取り）に効果的。

読み書き分離アーキテクチャ

レプリケーションラグ問題と対処

問題：Read-Your-Writes違反

ユーザーが書き込んだデータを即座に読もうとするとレプリカがまだ同期しておらず、古いデータが返ることがある。

対処法

  • • 自分が書いたデータはPrimaryから読む
  • • タイムスタンプでレプリカの同期状態チェック
  • • セッション単位でPrimaryに固定
  • • Causal Consistency（因果整合性）の利用
# ProxySQL設定例（読み書き分離）
mysql_query_rules:
  - rule_id: 1
    match_pattern: "^SELECT"
    destination_hostgroup: 10    # Replicaグループ
    apply: 1
  - rule_id: 2
    match_pattern: ".*"
    destination_hostgroup: 0     # Primaryグループ
    apply: 1

非同期処理とキューイング

同期処理を非同期化してスループットを向上

重い処理（メール送信・画像変換・決済処理など）をメッセージキューを通じて非同期化することで、APIのレスポンスタイムを大幅に短縮できる。

リクエストを受け取った瞬間にACKを返し、実際の処理は別のワーカーが担当する。

ワーカープール（Worker Pool）パターン

バッチ処理 vs ストリーム処理
方式特徴適用例
バッチ一定間隔でまとめて処理日次集計・レポート
ストリームイベント発生ごとにリアルタイム処理不正検知・通知
マイクロバッチ数秒単位のバッチ（中間）Spark Streaming
Backpressure（背圧）

消費者（Worker）の処理能力を超えた速度で生産者（Producer）がメッセージを送り続けることを防ぐ仕組み。

// Kafka Consumer の max.poll.records
// でバッチサイズを制限
props.put("max.poll.records", "50");
props.put("fetch.max.bytes", "1048576");
07

データの非正規化とマテリアライズドビュー

読み込み最適化のための非正規化

正規化（3NF）はデータ整合性のために重要だが、JOIN多発・クロスシャードクエリのパフォーマンスが悪化することがある。読み取り性能を優先する場合、意図的に非正規化（冗長データを持つ）設計を行う。

正規化（3NF）

-- 注文時にJOINが必要
SELECT o.*, u.name, u.email,
       p.name AS product_name
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id

非正規化（読み取り最適化）

-- ordersテーブルに冗長データを保持
orders {
  id, user_id,
  user_name,    -- 冗長
  user_email,   -- 冗長
  product_name, -- 冗長
  ...
}
-- JOINなしで取得可能
Materialized View（マテリアライズドビュー）

クエリ結果を事前に計算してテーブルとして保存する仕組み。複雑な集計クエリを高速化できる。

-- PostgreSQL のマテビュー
CREATE MATERIALIZED VIEW
  daily_sales AS
SELECT
  DATE(created_at) AS day,
  SUM(amount) AS total,
  COUNT(*) AS count
FROM orders
GROUP BY DATE(created_at);

-- インデックス追加
CREATE INDEX ON daily_sales(day);

-- 定期的にリフレッシュ
REFRESH MATERIALIZED VIEW
  CONCURRENTLY daily_sales;

トレードオフ

  • ✓ 読み取り速度が大幅向上
  • ✓ 複雑な集計をシンプルに
  • ✗ ストレージ増加
  • ✗ データが古くなる可能性
  • ✗ リフレッシュに時間がかかる
  • ✗ 書き込み時のオーバーヘッド
Twitterホームタイムライン設計：Push vs Pull

Pull方式（Fan-out on Read）

タイムライン表示時に全フォロイーのツイートをリアルタイム取得。読み取り負荷が高い。大量フォロイーで遅延。

SELECT tweets.*
FROM tweets
WHERE user_id IN (
  SELECT followee_id
  FROM follows
  WHERE follower_id = :me
)
ORDER BY created_at DESC
LIMIT 100

Push方式（Fan-out on Write）

ツイート投稿時にフォロワー全員のタイムラインキャッシュへ書き込む。読み取りは高速。フォロワー数百万人のセレブには不向き（ハイブリッドで対応）。

// ツイート投稿時
for follower_id in get_followers(user_id):
    timeline_cache[follower_id]
      .prepend(tweet)

// 読み取り時
return timeline_cache[me][:100]

Twitterの実際の実装：一般ユーザーはPush、セレブはPull、それ以外はハイブリッド。フォロワー数がしきい値（例：300万）を超えるとPull方式に切り替え。

CDN とエッジコンピューティング

CDN（Content Delivery Network）による静的アセット配信

世界各地にエッジサーバーを配置し、ユーザーに最も近いノードからコンテンツを配信。オリジンサーバーの負荷を削減し、レイテンシを大幅に低減する。

適するコンテンツ

  • • 画像・動画・音声ファイル
  • • CSS・JavaScript・HTML
  • • PDF・静的ドキュメント
  • • フォントファイル

主要プロバイダー

  • Cloudflare 世界最大規模
  • AWS CloudFront AWS統合
  • GCP Cloud CDN GCP統合
  • Fastly リアルタイムパージ

Cache-Control設定例

Cache-Control:
  public,
  max-age=31536000,
  immutable
# 変更時はURLにhash付与
# /app.a1b2c3d4.js
エッジコンピューティング（動的コンテンツのエッジ処理）

従来のCDNはキャッシュのみ。エッジコンピューティングではエッジノード上でコードを実行できるため、オリジンへの往復なしに動的処理が可能。

Cloudflare Workers の例

// エッジでA/Bテストを実行
addEventListener('fetch', event => {
  event.respondWith(
    handleRequest(event.request)
  )
})
async function handleRequest(req) {
  const variant = Math.random() < 0.5
    ? 'A' : 'B';
  const url = new URL(req.url);
  url.searchParams.set('v', variant);
  return fetch(url.toString());
}

エッジで処理できる例

  • • A/Bテスト・フィーチャーフラグ
  • • 認証トークン検証
  • • リクエストのルーティング・リダイレクト
  • • レスポンスの動的変換（HTML書き換え）
  • • レート制限・DDoS対策
  • • 地理的コンテンツパーソナライゼーション
  • • SSR（サーバーサイドレンダリング）の一部

分散キャッシュのスケーリング

Redis Cluster（自動シャーディング）

16384スロットにキー空間を分割し、各マスターノードが担当スロットを持つ。コンシステントハッシングの変形版。

slot = CRC16(key) % 16384 例: 3ノード構成 Node A: slot 0 - 5460 Node B: slot 5461 - 10922 Node C: slot 10923 - 16383 
# Redis Cluster起動
redis-cli --cluster create \
  127.0.0.1:7001 127.0.0.1:7002 \
  127.0.0.1:7003 \
  --cluster-replicas 1
Redis Sentinel（高可用性）

Masterの障害をSentinel（番兵）プロセス群が監視し、自動フェイルオーバーを実行する。

# sentinel.conf
sentinel monitor mymaster \
  127.0.0.1 6379 2
# quorum=2（多数決でフェイルオーバー）
sentinel down-after-milliseconds \
  mymaster 30000
sentinel failover-timeout \
  mymaster 180000

Sentinelは最低3台（奇数）で構成し、過半数の合意でフェイルオーバーを実行する。

Memcached のマルチスレッドスケール

Redisとは異なりシングルスレッドではなく、マルチスレッドでCPUコアを有効活用できる。シンプルなKey-Valueキャッシュに特化しており、データ構造はStringのみ。

  • ✓ マルチスレッドで高スループット
  • ✓ シンプルなユースケースに最適
  • ✗ 永続化なし
  • ✗ クラスタリング非対応（クライアント側でシャーディング）
キャッシュウォーミング（Cold Start問題）

キャッシュが空の状態（コールドスタート）では、全リクエストがDBに到達して過負荷になる可能性がある。

対策：

  • • デプロイ前にキャッシュをプリフィル（事前ロード）
  • • 新ノード追加時に隣接ノードからコピー
  • • リクエストを段階的にルーティング（カナリアリリース）
  • • DBからの読み取りレートを制限（Circuit Breaker）
Redis Cluster vs Sentinel vs スタンドアロン
構成スケーラビリティ可用性複雑さ適用場面
スタンドアロン低単一障害点低開発・小規模
Sentinel中（垂直のみ）自動フェイルオーバー中中規模・高可用性重視
Cluster高（水平シャーディング）レプリカ付き高大規模・高トラフィック

スケーラビリティのアンチパターン

⚠ Chatty I/O（大量の小さいリクエスト）

1つの操作を完了するために多数の小さいI/Oリクエストを繰り返すパターン。ネットワーク往復コストが積み重なる。

悪例：ユーザーのプロフィール表示のために、name取得→email取得→avatar取得→設定取得と4回DBクエリ。

改善：1回のクエリで必要なフィールドを全て取得。バッチAPIやGraphQLのデータローダーで複数リクエストを結合。

⚠ Monolithic Persistence（単一DBへの集中）

全てのサービスが1つのデータベースに接続するパターン。DBがボトルネックになり、スケーリングが困難。

改善：サービスごとに独自のDBを持つ（Database per Service）。読み取り専用のReplica・Read Modelを別途作成。

⚠ Synchronous Chain（同期呼び出しチェーン）

サービスA→B→C→Dと同期で連鎖するパターン。1つのサービスの遅延が全体に波及（レイテンシの加算）。障害時にカスケード障害が起きやすい。

改善：独立した処理を非同期キューで分離。必要な依存のみを同期で呼び出し、その他は非同期化。Circuit Breakerで連鎖障害を防止。

同期チェーンの遅延 = L_A + L_B + L_C + L_D 各サービスが50ms → 総レイテンシ200ms 非同期化後 → 最長パスのみ（並列化可能）

⚠ N+1 クエリ問題

1件のリスト取得後、各アイテムに対して個別クエリを実行するパターン。N件のリストに対しN+1回のDBアクセスが発生。

// 悪例（N+1問題）
const posts = await db.query('SELECT * FROM posts'); // 1クエリ
for (const post of posts) {
  post.author = await db.query(         // N クエリ
    'SELECT * FROM users WHERE id = ?', [post.user_id]
  );
}

// 改善（Eager Loading / JOINまたはIN句）
const posts = await db.query(`
  SELECT posts.*, users.name AS author_name
  FROM posts
  JOIN users ON posts.user_id = users.id
`); // 1クエリで解決

ORM での対処：Rails の includes、Hibernate の JOIN FETCH、Prisma の include などを使用。DataLoader（GraphQL）でバッチ化。

Monolithic PersistenceDBボトルネック・スケール不可DB分離・CQRS・Read Replica

SortedListを使ったリング実装

コンシステントハッシングを実際に動くPythonコードで実装します。sortedcontainersライブラリのSortedListを使うことで、ノード追加・削除・検索をO(log N)で実現します。

# pip install sortedcontainers
import hashlib
from sortedcontainers import SortedList
from typing import Optional

class ConsistentHashRing:
    """
    コンシステントハッシュリングの完全実装
    - SHA-256ハッシュ関数（32bit空間に収める）
    - バーチャルノードで均等な負荷分散
    """

    def __init__(self, num_virtual_nodes: int = 150):
        self.num_virtual_nodes = num_virtual_nodes
        self._ring: SortedList = SortedList()           # ソート済みハッシュ位置リスト
        self._hash_to_node: dict = {}                   # ハッシュ位置 → ノード名
        self._nodes: set = set()                        # 追加済みノード名

    def _hash(self, key: str) -> int:
        """SHA-256で文字列をハッシュし32bit整数に変換"""
        h = hashlib.sha256(key.encode()).hexdigest()
        return int(h, 16) % (2 ** 32)

    def add_node(self, node: str) -> dict:
        """
        ノードを追加し、そのノードが受け持つことになるキー範囲を返す
        Returns: {"moved_from": str, "keys_affected": range(start, end)}
        """
        if node in self._nodes:
            return {}
        self._nodes.add(node)

        # num_virtual_nodes 個の仮想ノードをリングに配置
        for i in range(self.num_virtual_nodes):
            vnode_key = f"{node}#vnode#{i}"
            position = self._hash(vnode_key)
            self._ring.add(position)
            self._hash_to_node[position] = node

        return {"status": "added", "node": node,
                "virtual_nodes": self.num_virtual_nodes}

    def remove_node(self, node: str) -> bool:
        """ノードを削除。そのノードのキーは次のノードに引き継がれる"""
        if node not in self._nodes:
            return False
        self._nodes.discard(node)

        for i in range(self.num_virtual_nodes):
            vnode_key = f"{node}#vnode#{i}"
            position = self._hash(vnode_key)
            self._ring.remove(position)
            del self._hash_to_node[position]

        return True

    def get_node(self, key: str) -> Optional[str]:
        """
        キーを担当するノードを返す
        リングを時計回りに検索し、最初に見つかったノードを返す
        """
        if not self._ring:
            return None

        key_hash = self._hash(key)

        # key_hash 以上の最小値を検索（二分探索 O(log N)）
        idx = self._ring.bisect_left(key_hash)

        # リングを一周（末尾を超えたら先頭に戻る）
        if idx == len(self._ring):
            idx = 0

        position = self._ring[idx]
        return self._hash_to_node[position]

    def get_nodes_for_replication(self, key: str, n: int = 3) -> list:
        """
        レプリケーション用: キーを担当する連続したn個の物理ノードを返す
        同一物理ノードの仮想ノードはスキップ
        """
        if len(self._nodes) < n:
            return list(self._nodes)

        key_hash = self._hash(key)
        idx = self._ring.bisect_left(key_hash) % len(self._ring)

        result = []
        seen_nodes = set()
        steps = 0

        while len(result) < n and steps < len(self._ring):
            position = self._ring[(idx + steps) % len(self._ring)]
            node = self._hash_to_node[position]
            if node not in seen_nodes:
                result.append(node)
                seen_nodes.add(node)
            steps += 1

        return result

    def load_distribution(self) -> dict:
        """各物理ノードが担当するバーチャルノード数（負荷分散の均等性確認）"""
        distribution = {node: 0 for node in self._nodes}
        for node in self._hash_to_node.values():
            distribution[node] += 1
        return distribution

シャードリバランシング: Traffic Shadowingを使った移行

本番ダウンタイムなしでシャードを再分割する際の安全な手順です。

  1. 準備フェーズ：新しいシャード（例: shard-4）を空の状態で追加。ルーティングテーブルにはまだ含めない。
  2. 二重書き込み開始：shard-3 の担当データに対する書き込みを shard-3 と shard-4 両方に送信（Traffic Mirroring）。shard-4 への書き込みエラーは無視（まだ本番外）。
  3. 初期データコピー：バックグラウンドで shard-3 の既存データを shard-4 にバルクコピー。書き込みの二重化で差分は随時同期。
  4. 読み取り検証：shard-4 への読み取りを影で実行し、shard-3 の結果と比較。一致率が100%に達したら次のフェーズへ。
  5. カットオーバー：ルーティングテーブルを更新し、shard-4 担当のキー範囲を shard-3 から shard-4 に切り替え（ダウンタイムなし）。
  6. クリーンアップ：数日後 shard-3 の不要データを削除。

Traffic Shadowingの実装ポイント

  • Envoyプロキシの mirror_policy や NGINX の mirror ディレクティブで透過的に実装可能
  • 影のリクエストへのエラーは本番レスポンスに影響させない（fire-and-forget）
  • 読み取り検証は別プロセスで非同期に行う
Cross-Shardクエリの最適化: スキャッタ-ギャザーパターンとその限界

スキャッタ-ギャザー（Scatter-Gather）とは、複数シャードにクエリを並列投げ（Scatter）、結果を集約する（Gather）パターンです。

利点

  • • 並列実行でレイテンシを抑制可能
  • • シャード数に関係なく実装できる
import asyncio

async def scatter_gather_query(shards, sql):
    tasks = [shard.query(sql) for shard in shards]
    results = await asyncio.gather(*tasks)
    # 全シャードの結果を結合
    return [row for shard_rows in results
            for row in shard_rows]

限界

  • • ORDER BY + LIMIT は全シャードで過剰取得が必要（例: LIMIT 10 → 各シャードで LIMIT 10取得→マージで最終10件選択）
  • • GROUP BY は2段階集計が必要（shard内集計→マスター集計）
  • • 全シャードのうち1つが遅いとレスポンス全体が遅くなる（Tail Latency問題）
  • • シャード数が増えるほどDBへの接続数が増大

ホームタイムライン設計の変遷

初期: Pure Push（Fan-out on Write）

ツイート投稿時に全フォロワーのタイムラインキャッシュに書き込む。Ladyガガ（1億フォロワー）がツイートすると1億回の書き込みが発生。

読み取り時にフォロイーのツイートをSELECT。ユーザーがTL表示のたびにN人分のクエリ。アクティブユーザーが多いと読み取りが爆発。

通常ユーザー → Push方式（TLキャッシュに事前書き込み）。有名人（高フォロワー数）→ 読み取り時にのみ取得してマージ。

Snowflake ID生成器の完全実装

Twitterが開発した分散ID生成アルゴリズム。DBの自動インクリメントに依存せず、複数ノードで毎秒400万以上のユニークIDを生成できます。

64-bit ID構造
[1bit: 符号] [41bit: タイムスタンプ(ms)] [10bit: マシンID] [12bit: シーケンス番号]

最大生成数/秒 = 2^12 × 2^10 × 1000ms = 4,194,304 / ノード タイムスタンプ有効期間 = 2^41 ms = 69.7年（Twitter Epoch 2010-11-04から） 
import time
import threading

class SnowflakeIDGenerator:
    """
    Twitter Snowflake 互換ID生成器
    64-bit = 1符号 + 41タイムスタンプ + 10マシンID + 12シーケンス
    """
    # Twitter Epoch: 2010-11-04T01:42:54.657Z（ミリ秒）
    EPOCH = 1288834974657

    TIMESTAMP_BITS = 41
    MACHINE_ID_BITS = 10
    SEQUENCE_BITS = 12

    MAX_MACHINE_ID = (1 << MACHINE_ID_BITS) - 1   # 1023
    MAX_SEQUENCE = (1 << SEQUENCE_BITS) - 1        # 4095

    # ビットシフト量
    MACHINE_ID_SHIFT = SEQUENCE_BITS               # 12
    TIMESTAMP_SHIFT = SEQUENCE_BITS + MACHINE_ID_BITS  # 22

    def __init__(self, machine_id: int):
        if not 0 <= machine_id <= self.MAX_MACHINE_ID:
            raise ValueError(f"machine_id must be 0-{self.MAX_MACHINE_ID}")
        self.machine_id = machine_id
        self._sequence = 0
        self._last_timestamp = -1
        self._lock = threading.Lock()

    def _current_time_ms(self) -> int:
        return int(time.time() * 1000)

    def _wait_next_ms(self, last_ts: int) -> int:
        """同じmsで4096以上のIDが必要なら次のmsまで待機"""
        ts = self._current_time_ms()
        while ts <= last_ts:
            ts = self._current_time_ms()
        return ts

    def generate_id(self) -> int:
        with self._lock:
            ts = self._current_time_ms()

            if ts == self._last_timestamp:
                # 同じミリ秒内: シーケンスインクリメント
                self._sequence = (self._sequence + 1) & self.MAX_SEQUENCE
                if self._sequence == 0:
                    # シーケンスが溢れた: 次のmsまで待機
                    ts = self._wait_next_ms(self._last_timestamp)
            else:
                # 新しいms: シーケンスリセット
                self._sequence = 0

            self._last_timestamp = ts

            # 64-bit ID の組み立て
            snowflake_id = (
                ((ts - self.EPOCH) << self.TIMESTAMP_SHIFT) |
                (self.machine_id << self.MACHINE_ID_SHIFT) |
                self._sequence
            )
            return snowflake_id

    @staticmethod
    def parse_id(snowflake_id: int) -> dict:
        """IDを逆パースして各フィールドを抽出"""
        EPOCH = 1288834974657
        sequence = snowflake_id & 4095           # 下12bit
        machine_id = (snowflake_id >> 12) & 1023 # 次の10bit
        timestamp_ms = (snowflake_id >> 22) + EPOCH

        from datetime import datetime, timezone
        created_at = datetime.fromtimestamp(
            timestamp_ms / 1000, tz=timezone.utc
        )

        return {
            "id": snowflake_id,
            "timestamp_ms": timestamp_ms,
            "created_at": created_at.isoformat(),
            "machine_id": machine_id,
            "sequence": sequence,
        }


# 使用例
gen = SnowflakeIDGenerator(machine_id=1)
ids = [gen.generate_id() for _ in range(5)]
for id_ in ids:
    info = SnowflakeIDGenerator.parse_id(id_)
    print(f"ID={id_} seq={info['sequence']} time={info['created_at']}")

import random

def get_cache_key_for_hot_user(user_id: int, num_shards: int = 10) -> str:
    """
    ホットなUserIDをnum_shards個のキャッシュキーに分散させる
    読み取り時はランダムに選択, 書き込み時は全shardに書く
    """
    suffix = random.randint(0, num_shards - 1)
    return f"user:{user_id}:shard:{suffix}"

# 書き込み（全shardに書く）
def write_hot_user_cache(user_id: int, data: dict, cache, num_shards=10):
    for i in range(num_shards):
        key = f"user:{user_id}:shard:{i}"
        cache.set(key, data, ttl=300)

# 読み取り（ランダムに1shardから読む）
def read_hot_user_cache(user_id: int, cache, num_shards=10):
    shard = random.randint(0, num_shards - 1)
    return cache.get(f"user:{user_id}:shard:{shard}")

解決策2: バーチャルノード数の調整効果（シミュレーション）

from collections import Counter

def simulate_distribution(ring: ConsistentHashRing, num_keys: int = 100000):
    """仮想ノード数別の負荷分散均等性をシミュレート"""
    results = Counter()
    for i in range(num_keys):
        node = ring.get_node(f"user:{i}")
        results[node] += 1

    counts = list(results.values())
    avg = sum(counts) / len(counts)
    std_dev = (sum((c - avg)**2 for c in counts) / len(counts)) ** 0.5
    return {
        "distribution": dict(results),
        "std_dev_ratio": std_dev / avg * 100,  # 変動係数（%）
    }

# 仮想ノード数別の比較
for num_vnodes in [10, 50, 150, 300]:
    ring = ConsistentHashRing(num_virtual_nodes=num_vnodes)
    for server in ["A", "B", "C", "D", "E"]:
        ring.add_node(f"server-{server}")
    stats = simulate_distribution(ring)
    print(f"vnodes={num_vnodes:3d}: 変動係数={stats['std_dev_ratio']:.1f}%")

演習3: Snowflake IDジェネレーターの完全実装と生成能力の計算

先ほどの実装コードを参照しつつ、生成能力の計算を詳しく行います。

1ノードの最大生成能力
1ms あたりの最大ID数 = 2^12 = 4,096 1s あたりの最大ID数 = 4,096 × 1,000ms = 4,096,000 ≈ 410万 ID/s/ノード

マシンIDのビット数 = 10bit → 最大 2^10 = 1,024 ノード 全ノード合計 = 4,096,000 × 1,024 = 4,194,304,000 ID/s（約42億/s）
タイムスタンプの有効期間
41bitで表現可能な最大値 = 2^41 - 1 = 2,199,023,255,551 ms 年換算 = 2,199,023,255,551 / (1000 × 60 × 60 × 24 × 365) ≈ 69.7年

Twitter Epoch（2010-11-04）から69.7年 = 約2080年まで使用可能 
# Snowflakeの亜種: Instagram ID（SequentialID）
# Instagram方式: Postgresのストアドファンクションで生成（DBレス）
# 41bit timestamp + 13bit shard ID + 10bit sequence

CREATE OR REPLACE FUNCTION next_id(OUT result bigint) AS $$
DECLARE
    our_epoch bigint := 1314220021721;   -- Instagram Epoch (2011-08-25)
    seq_id bigint;
    now_ms bigint;
    shard_id int := 5;                  -- このDBシャードのID（0〜8191）
BEGIN
    -- シーケンスから次の値を取得（1〜1024でラップ）
    SELECT nextval('global_id_sequence') % 1024 INTO seq_id;
    -- 現在時刻（ミリ秒）
    SELECT FLOOR(EXTRACT(EPOCH FROM clock_timestamp()) * 1000) INTO now_ms;
    -- ビット結合: 41bit ms + 13bit shard + 10bit seq
    result := (now_ms - our_epoch) << 23;
    result := result | (shard_id << 10);
    result := result | seq_id;
END;
$$ LANGUAGE plpgsql;

-- 使用例
SELECT next_id();

Snowflake vs UUIDv4 vs ULID の比較

  • Snowflake：単調増加→BTREEインデックス効率最高。ノードIDが必要（機器管理コスト）。時刻情報内包でデバッグ容易。
  • UUIDv4：完全ランダム→インデックスフラグメンテーション。ノード管理不要でシンプル。128bitで大きい。
  • ULID（Universally Unique Lexicographically Sortable Identifier）：タイムスタンプ + ランダム性。辞書順ソート可能。128bit。ノード管理不要でSnowflakeより実装が容易。
