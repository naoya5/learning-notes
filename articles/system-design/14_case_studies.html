Case Study 01
URL短縮サービス (TinyURL)
シンプルに見えて奥が深い。ID生成・リダイレクト・キャッシュ戦略の本質を学ぶ
要件定義

機能要件

  • 長いURLを短縮URLに変換する
  • 短縮URLにアクセスすると元URLへリダイレクト
  • カスタムエイリアス（任意の文字列）の指定
  • URLの有効期限設定（デフォルト: 無期限）
  • アクセス統計（クリック数、地域情報）

非機能要件

  • 100M DAU（デイリーアクティブユーザー）
  • 読み取り / 書き込み比率 = 100 : 1
  • 可用性 99.9%（年間ダウンタイム8.7時間以下）
  • リダイレクトレイテンシ < 10ms（キャッシュ）
  • 短縮URLは推測困難であること（セキュリティ）
  • スケーラブル：将来10倍のトラフィックに対応

ポイント: ステートレス設計
APIサーバーはセッション情報を持たないステートレス設計にする。これにより水平スケールが容易になり、任意のサーバーにリクエストをルーティング可能。
ID生成戦略

Base62エンコード

使用文字: a-z A-Z 0-9（62文字）。7文字の場合: 62⁷ ≈ 3,521億の一意キー

-- 短縮URLの例
https://tiny.url/ab3Kp9x  -- 7文字のBase62 = 十分な一意性

-- Base62エンコード (Python風の擬似コード)
def encode_base62(num: int) -> str:
    chars = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789"
    result = []
    while num > 0:
        result.append(chars[num % 62])
        num //= 62
    return ''.join(reversed(result)).zfill(7)
方式メリットデメリット推奨場面
UUID v4 実装が簡単、分散生成可能 128ビット、URLには長い、衝突確率は低いが0ではない 小規模
連番ID + Base62 衝突なし、短い、予測可能な増加 単一SPOF（採番サーバー）、推測されやすい 中規模
ZooKeeper採番 分散対応、衝突なし保証 ZooKeeper依存、レイテンシ増加 大規模
MD5/SHA-1の先頭7文字 実装簡単 ハッシュ衝突の可能性 非推奨
リダイレクト戦略: 301 vs 302
HTTPステータスキャッシュ動作サーバー負荷アクセスカウント推奨
301 Moved Permanently ブラウザにキャッシュ（次回はサーバー不要） 低 計測不可 パフォーマンス優先
302 Found 毎回サーバーにリクエスト 高 正確に計測可能 分析・トラッキング優先
⚡
設計判断
アクセス統計が要件にある場合は302を選ぶ。ただし高トラフィックではRedisでのカウント集計を非同期で行い、リダイレクト自体のレイテンシには影響させない。

Push vs Pull — Fan-outモデルの詳細
モデル仕組みメリットデメリット向いているケース
Push on Write
(Fan-out on Write)		 ツイート時に全フォロワーのタイムラインキャッシュに書き込む 読み取り超高速（キャッシュから返すだけ） フォロワー数百万のユーザーは書き込みが膨大（有名人問題） フォロワー少ない一般ユーザー
Pull on Read
(Fan-out on Read)		 タイムライン表示時にフォロー中の全ツイートを動的集約 書き込み負荷が低い、キャッシュが不要 読み取りが遅い（フォロー数が多いと複数DBクエリ） 有名人アカウント
Hybrid
推奨		 フォロワー < 1万 → Push / フォロワー ≥ 1万 → Pull 両方のメリットを享受 実装が複雑、閾値の調整が必要 本番Twitterと同等
💡
ポイント: Celebrity Problem（有名人問題）
Elon Muskのような1.5億フォロワーのユーザーがツイートすると、Push方式では1.5億件のキャッシュ書き込みが発生。これはキュー詰まりと書き込みストームを引き起こす。Hybridアプローチでは有名人のツイートはキャッシュに書き込まず、読み取り時にリアルタイムで取得・マージする。
Redisによるタイムラインキャッシュ設計
-- Redisデータ構造: Sorted Set (ZSet)
-- Key: timeline:{user_id}
-- Score: タイムスタンプ (UNIX epoch, 降順)
-- Member: tweet_id

-- タイムラインへのツイート追加 (Fan-out時)
ZADD timeline:12345 1703952000 tweet:98765

-- タイムライン取得 (最新20件)
ZREVRANGE timeline:12345 0 19 WITHSCORES

-- キャッシュサイズ制御: 最大800件を保持、古いものを削除
ZREMRANGEBYRANK timeline:12345 0 -801

-- パイプラインで複数ユーザーに一括配信 (Lua Script)
local followers = redis.call('SMEMBERS', 'followers:' .. tweet_author_id)
for _, follower_id in ipairs(followers) do
    redis.call('ZADD', 'timeline:' .. follower_id, timestamp, tweet_id)
    redis.call('ZREMRANGEBYRANK', 'timeline:' .. follower_id, 0, -801)
end

トレードオフまとめ
採用した設計とその理由

Hybrid Fan-out で通常ユーザーのタイムラインを高速化。Cassandraでツイートデータの高スループット書き込みに対応。RedisのSorted Setでタイムラインをメモリ管理。

残る課題と解決アプローチ

タイムラインの結果整合性（数秒の遅延は許容）。有名人フォロワーへの配信タイムラグをMergeで補完。フォロー解除後の古いキャッシュ無効化（Lazy Delete）。

YouTube / 動画配信サービス設計
大容量動画のアップロードパイプライン、トランスコーディング、アダプティブストリーミングの全貌
要件定義

機能要件

  • 動画のアップロード（最大4K、10GB）
  • 動画のストリーミング再生
  • 複数解像度での自動変換（360p〜4K）

  • 動画再生開始まで < 2秒（バッファリング）
  • グローバルに低レイテンシ（CDN必須）
  • 可用性 99.9%
  • ストレージコスト最適化

項目計算結果

動画アップロードパイプライン
⚡
DAG（有向非循環グラフ）によるジョブ管理
トランスコードは複数のステップ（動画分割 → 並列エンコード → サムネイル生成 → マージ → メタデータ更新）をDAGで管理する。各ノードはコンポーネント（タスク）、エッジは依存関係を表す。AirflowやAWS Step Functionsに相当する。
アダプティブビットレートストリーミング (ABR)

DASH / HLS の仕組み

  • 動画を2〜10秒のセグメントに分割
  • 各セグメントを複数のビットレートで用意
  • マニフェストファイル（.m3u8 / .mpd）で解像度一覧を提供
  • クライアントがネット速度に応じてセグメントを動的選択
  • 回線が遅ければ低解像度に、速ければ高解像度に自動切替

CDN設計のポイント

  • 人気動画はエッジノードにプリロード（Pre-warming）
  • GeoIPで最寄りのエッジにルーティング
  • Long-tail動画はOriginからオンデマンドフェッチ
  • Cold Start軽減：再生開始直後は低解像度セグメント先読み
  • Range Requestによる部分取得（シーク対応）

トレードオフまとめ
重要設計判断

動画と非動画データを分離（オブジェクトストレージ vs RDB）。非同期パイプラインでアップロード後のUXをブロックしない。CDNにより世界規模でレイテンシを均一化。

コスト最適化の課題

ストレージコスト削減: 90日アクセスなし動画をS3 Glacier(コールドストレージ)に移行。人気上位1%の動画がトラフィックの95%を占めるため、エッジキャッシュ集中が有効。

Case Study 04
分散KVストア（DynamoDB風）
コンシステントハッシング・LSMツリー・Gossipプロトコルで99.99%可用性を実現する内部設計
要件定義

機能要件

  • Get(key) — 値の取得
  • Put(key, value) — 値の設定・更新
  • Delete(key) — 値の削除
  • キーとバリューのサイズは10KB以下
  • データの永続化保証

非機能要件

  • 可用性 99.99%（年間ダウンタイム52分以下）
  • 読み取りレイテンシ < 1ms（P99）
  • 水平スケールアウト可能
  • CAP定理: APを優先（Availability + Partition Tolerance）
  • 自動障害検知と復旧
コンシステントハッシング
💡
バーチャルノードの重要性
物理ノード1台をリング上に複数の仮想ノードとして配置することで、データ分散が均等になる。ノード追加/削除時の再配置量も最小化できる。例えば物理ノード3台に対し各々100個の仮想ノードを配置すると、300点のハッシュ値でリングを分割する。
データレプリケーションとQuorum

Quorum パラメータ (N, W, R)

パラメータ意味推奨値効果
N（レプリカ数）データを保持するノード数N = 31台障害でもデータ保持
W（書き込みQuorum）書き込み成功に必要なノード数W = 2過半数確認で強い一貫性
R（読み取りQuorum）読み取り成功に必要なノード数R = 2W + R > N で強一貫性
設定特性向くケース
W=1, R=1最速だが一貫性弱いセッションカウンター等
W=1, R=N書き込み高速、読み取り低速だが強一貫書き込み多、読み取り少
W=N, R=1書き込み低速だが読み取り強一貫設定値、参照データ
W=2, R=2 (N=3)バランス重視（推奨）汎用KVストア
書き込みパス — LSMツリー構造
💡
Bloom Filterの役割
「キーが存在しないこと」を高速に判定する確率的データ構造。偽陽性（ないのにあると言う）はあるが偽陰性はない。存在しないキーの検索でSSTableのディスクI/Oを回避し、読み取り性能を大幅改善する。
障害検知 — Gossipプロトコル

Gossipプロトコルの仕組み

  • 各ノードがランダムなピアにハートビートを定期送信
  • 受信ノードはその情報を別のランダムなピアに転送
  • 感染モデルで全ノードに状態が伝播（O(log N)）
  • 特定のコーディネーターが不要（分散型）
  • 一定時間ハートビートがなければそのノードを障害と判断

ベクタークロック（競合解決）

  • 各更新に[ノードID: バージョン番号]のベクターを付与
  • 競合発生時: 複数バージョンをクライアントに返す
  • クライアント側で最終的な値を決定（Application-level）
  • DynamoDBの「Last Write Wins」または「Vector Clock」
  • 簡略化: タイムスタンプによるLWW（Last Write Wins）
コアコンポーネントまとめ

データ分散

  • コンシステントハッシング
  • バーチャルノード（均等分散）
  • N=3レプリカ（耐障害性）

書き込みパス

  • WAL（クラッシュ復旧）
  • MemTable（高速インメモリ）
  • SSTable + Compaction（ディスク）

読み取り最適化

  • MemTable優先検索
  • Bloom Filter（不存在高速判定）
  • SSTableのインデックスキャッシュ
トレードオフまとめ
APシステムの設計哲学

DynamoDB/Cassandraは可用性を最優先。ネットワーク分断時も書き込みを受け付け、結果整合性で最終的に収束。"Always writable"を実現するためにベクタークロックで競合を追跡。

LSMツリーのデメリット

Read Amplification（複数レベルを検索）とWrite Amplification（Compaction時の多重書き込み）が発生。頻繁なCompactionはI/O負荷を上げるため、タイミングとレベル構成の調整が必要。

WebSocketによるリアルタイム接続設計
💡
ポイント: なぜWebSocketか
HTTPはリクエスト/レスポンスモデルのため、リアルタイム配信にはLong Pollingかサーバーサイドプッシュが必要。WebSocketは初回ハンドシェイク後に双方向の永続接続を確立するため、チャットのような低レイテンシ配信に最適。ただしステートフルなのでLBは「スティッキーセッション」または「IPハッシュ」を使う必要がある。

データモデル — Cassandraの選択理由
⚡
なぜCassandraか
チャットデータは「特定チャットルームの最新N件を取得」という時系列クエリが主。Cassandraのパーティションキー(chat_id)でデータをノード分散し、クラスタリングキー(message_id降順)で時系列ソートを実現。高書き込みスループット(100万msg/s)かつ線形スケールが可能。MySQLでは書き込みボトルネックになる。

グループサイズ配信戦略実装
小グループ (〜20人) Push: 全メンバーのWebSocket接続を探して直接配信 Redis HashでuserID→serverIDマッピング

大グループ (200〜500人) Fan-out Service: 専用ワーカーが非同期で順次配信 最終的にはPull（次回接続時に未読取得）も組み合わせ
オンライン状態管理 — HeartbeatとRedis

プレゼンス管理の仕組み

  • クライアントが5秒ごとにハートビートを送信
  • サーバーがRedisに presence:{user_id} を更新（TTL=30秒）
  • TTL切れ = オフラインと判定（自動）
  • フォロー中ユーザーのステータス変化をWebSocketでPush通知
  • 大規模では結果整合性（多少の遅延は許容）

既読確認の実装

  • 1チェック（✓）: サーバーがメッセージを受信
  • 2チェック（✓✓）: 受信者のデバイスに配信完了
  • 青チェック（✓✓）: 受信者がメッセージを開封
  • 配信確認: ACKをサーバー経由で送信者に逆送信
  • オフライン中は再接続時にまとめてACK処理
トレードオフまとめ
設計の核心判断

WebSocket+KafkaでリアルタイムかつスケーラブルなアーキテクチャをWebSocket接続の多数のサーバーで管理。Cassandraの時系列パーティショニングで数十億メッセージの書き込みに対応。Redisでプレゼンス情報をサブ秒で管理。

複雑さのトレードオフ

WebSocketはステートフルなのでスケールアウト時の接続移行が難しい。メッセージ順序保証（Snowflake ID）とAt-least-once配信の組み合わせで重複除去が必要。E2E暗号化はサーバー側でのメッセージ内容検索を不可能にする。

面接でシステムデザインを語る際の必須パターン一覧

5つのケーススタディで登場した設計パターンを体系化。面接では「なぜその選択をしたか」のトレードオフを必ず説明できるようにしておく。

Base62 URL エンコード

a-z/A-Z/0-9の62文字で7桁→35億の一意空間。UUID代替として短くて予測困難なIDを生成。URL短縮・短縮コード生成に必須。

301 vs 302 リダイレクト

301はブラウザキャッシュで高速だがアクセス計測不可。302は毎回サーバーヒットで計測可能。要件次第で選択する。

Hybrid Fan-out（Push+Pull）

Twitterの有名人問題解決策。フォロワー閾値（1万人）でPush（Write-time fan-out）とPull（Read-time fan-out）を切り替える。

Redis Sorted Set タイムライン

ZADD + ZREVRANGE でタイムスタンプスコアのツイートリストを管理。O(log N)の追加、O(log N + M)の範囲取得。

動画トランスコードパイプライン

アップロード→オブジェクトストレージ→Kafkaキュー→DAG管理のWorker群→複数解像度→CDN。非同期・並列処理が鍵。

アダプティブビットレート（ABR）

DASH/HLSでセグメントを複数品質で用意し、クライアントが回線速度に応じて動的選択。Cold Startは低品質から始める。

コンシステントハッシング

ハッシュリング上にノードとキーを配置。ノード追加/削除時のデータ移動を最小化。バーチャルノードで均等分散を実現。

Quorum (N, W, R)

N=レプリカ数、W=書き込み確認数、R=読み取り確認数。W+R>Nで強一貫性。W=1,R=1で高可用性低一貫性。

LSMツリー（MemTable + SSTable）

書き込みを順次ディスクに追記（高速）。MemTable→L0→L1→L2のCompactionで読み取り最適化。Cassandra/RocksDBが採用。

Bloom Filter

確率的データ構造で「キーが存在しないこと」を高速判定。偽陰性なし（ないと言ったら確実にない）。LSMツリーのRead最適化に不可欠。

Gossipプロトコル

P2P方式でノード状態をO(log N)で全体伝播。中央コーディネーター不要の分散障害検知。Cassandraが採用。

WebSocket + Kafka

WebSocketで永続双方向接続（リアルタイム）、Kafkaでサーバー間メッセージルーティング。チャット/通知系の標準構成。

Snowflake ID

41bit時刻+10bit機械ID+12bitシーケンスの64bit ID。時刻順ソート可能、分散生成、毎秒4096IDまでSPOFなし。

プレゼンス管理（Redis TTL）

HeartbeatでRedisキーをUPDATE（TTL=30秒）。TTL切れ=オフライン。大規模では結果整合性（数秒の遅延を許容）。

CDN プリウォーミング

公開前に人気コンテンツをエッジノードへ事前配布。Cold Start時のOriginへの集中アクセスを防ぐ。動画・画像配信の必須戦略。

CAP定理の実践的選択

CP（一貫性優先）: 銀行・決済。AP（可用性優先）: SNS・チャット。ネットワーク分断は避けられないのでCまたはAの選択が核心。

Sliding Window Log の Redis 実装（Lua Script）
-- sliding_window.lua: Redis EVAL で原子的に実行
local key = KEYS[1]          -- "ratelimit:{user_id}:{endpoint}"
local now = tonumber(ARGV[1]) -- 現在時刻（ミリ秒）
local window = tonumber(ARGV[2]) -- ウィンドウサイズ（ミリ秒）: 60000
local limit = tonumber(ARGV[3])  -- 上限: 100

-- 1. ウィンドウ外の古いエントリを削除
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- 2. 現在のカウントを取得
local count = redis.call('ZCARD', key)

-- 3. 上限チェック
if count >= limit then
  return {0, count, limit}  -- 拒否: {allowed, current, limit}
end

-- 4. 現在のリクエストを記録（スコア=タイムスタンプ、メンバー=ユニークID）
redis.call('ZADD', key, now, now .. '-' .. math.random(100000))
redis.call('EXPIRE', key, math.ceil(window / 1000) + 1)

return {1, count + 1, limit}  -- 許可
# Python クライアントからの呼び出し
import redis, time

r = redis.Redis(host='localhost')

with open('sliding_window.lua') as f:
    lua_script = r.register_script(f.read())

def check_rate_limit(user_id: str, endpoint: str, limit: int = 100) -> dict:
    key = f"ratelimit:{user_id}:{endpoint}"
    now_ms = int(time.time() * 1000)
    window_ms = 60000  # 1分

    result = lua_script(keys=[key], args=[now_ms, window_ms, limit])
    allowed, current, lim = result

    return {
        "allowed": bool(allowed),
        "remaining": max(0, lim - current),
        "retry_after": 60 if not allowed else None
    }
Token Bucket の分散 Redis 実装
-- token_bucket.lua: バースト対応レートリミット
local key = KEYS[1]
local capacity = tonumber(ARGV[1])  -- バケット容量: 100
local fill_rate = tonumber(ARGV[2]) -- 補充レート: 10トークン/秒
local now = tonumber(ARGV[3])       -- 現在時刻（秒）
local tokens_requested = tonumber(ARGV[4]) -- 消費トークン数: 1

local last = redis.call('HGETALL', key)
local tokens = capacity
local last_time = now

if #last > 0 then
  tokens = tonumber(last[2])
  last_time = tonumber(last[4])
  -- 経過時間に応じてトークンを補充
  local elapsed = now - last_time
  tokens = math.min(capacity, tokens + elapsed * fill_rate)
end

if tokens >= tokens_requested then
  tokens = tokens - tokens_requested
  redis.call('HMSET', key, 'tokens', tokens, 'last_time', now)
  redis.call('EXPIRE', key, math.ceil(capacity / fill_rate) + 1)
  return {1, math.floor(tokens)}  -- 許可
else
  return {0, math.floor(tokens)}  -- 拒否
end
フェイルオープン vs フェイルクローズの判断基準
🔓
フェイルオープン（Redis障害時に許可）
Redisに接続できない場合、すべてのリクエストを通す。
適用場面: 収益に直結するAPI（決済、認証）。ダウンタイムのコストがDDoSより高い場合。
リスク: Redis障害中は無制限アクセスが可能。
🔒
フェイルクローズ（Redis障害時に拒否）
Redisに接続できない場合、すべてのリクエストを拒否（HTTP 503）。
適用場面: セキュリティクリティカルなAPI。スクレイピング防止など。
リスク: Redis障害がサービス全体のダウンに直結。
💡
推奨: ローカルキャッシュによるハイブリッド
Redis障害時はプロセス内のin-memoryカウンター（ローカルリミット）にフォールバック。精度は落ちるが完全な障害伝播を防ぐ。複数インスタンスが独立してカウントするため、合計は若干多めになるがサービスは継続できる。
マルチリージョン: 階層設計

ローカルカウンター + グローバル集約の2層構造

# グローバルリミット: 1000 req/min
# リージョン割り当て（東京: 60%, 大阪: 40%）
#   東京ローカルリミット: 600 req/min
#   大阪ローカルリミット: 400 req/min
#
# メリット: リージョン間通信なしでlow latency判定
# デメリット: リージョンをまたいだリクエストは若干超過する可能性

class MultiRegionRateLimiter:
    def __init__(self, region: str, global_limit: int, region_weight: float):
        self.local_limit = int(global_limit * region_weight)
        self.local_redis = RedisClient(region=region)
        # グローバルRedis（定期的に同期）
        self.global_redis = RedisClient(region="global")

    def is_allowed(self, key: str) -> bool:
        # まずローカルで高速チェック
        local_count = self.local_redis.incr(f"local:{key}")
        if local_count > self.local_limit:
            return False
        # 非同期でグローバルカウンターを更新（精度より速度優先）
        asyncio.create_task(self.sync_global(key))
        return True
