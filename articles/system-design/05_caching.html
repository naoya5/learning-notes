01キャッシュとは & なぜ重要か

キャッシュは高速なストレージに頻繁にアクセスされるデータを一時保存することで、レイテンシを削減しスループットを向上させる技術です。

レイテンシ改善の原理

コンピュータシステムにおける記憶階層では、速度と容量はトレードオフの関係にあります。CPUレジスタは超高速ですが容量は数バイト、HDDは大容量ですがアクセスに数十ミリ秒かかります。キャッシュはこのギャップを埋める役割を果たします。

各記憶層のアクセスレイテンシ（対数スケール的に可視化）

キャッシュヒット率の重要性

ヒット率 95% の場合

100リクエストのうち95件がキャッシュから返る。残り5件はDBアクセス（10ms）が発生。

平均レイテンシ =
  0.95 x 1ms  (キャッシュ)
+ 0.05 x 10ms (DB)
= 1.45ms

ヒット率 99% の場合

100リクエストのうち99件がキャッシュから返る。DBアクセスは1件のみ。

平均レイテンシ =
  0.99 x 1ms  (キャッシュ)
+ 0.01 x 10ms (DB)
= 1.09ms
重要: ヒット率を95%から99%に改善するだけで平均レイテンシが約25%改善。さらに重要なのはDBへの負荷が1/5に削減されること。高トラフィック環境ではこの差がシステムの生死を分けます。

キャッシュに適するデータ・不適なデータ

✓ キャッシュに向くデータ

  • 読み取り頻度が高く書き込みが少ないデータ（商品カタログ、ユーザープロフィール）
  • 計算コストが高い結果（集計クエリ結果、レコメンデーション）
  • 複数リクエストで共有されるデータ（設定情報、静的コンテンツ）
  • 一時的なセッション情報
  • レート制限カウンター

✗ キャッシュに向かないデータ

  • リアルタイム性が求められるデータ（残高、在庫数）
  • ユーザー固有で共有されないデータ（個人の下書き）
  • 頻繁に変更されるデータ（株価、スポーツスコア）
  • 機密性が高くキャッシュから漏洩リスクがあるデータ
  • 書き込みが読み取りより多いデータ

02キャッシュの階層

現代のシステムはCPUからブラウザまで複数の層でキャッシュを活用しています。各層の特性を理解することが設計の基本です。

キャッシュ階層 — ユーザーに近いほど高速・小容量、データソースに近いほど低速・大容量

各層の特性比較

キャッシュ層レイテンシ容量スコープ管理
CPU L1/L2/L30.5〜40nsKB〜MBCPU単体ハードウェア自動
アプリ内キャッシュ<1ms数百MBプロセス内アプリ実装
分散キャッシュ0.5〜5msGB〜TB全サーバー共有Redis/Memcached
CDNキャッシュ5〜50ms無制限エッジノードCDNプロバイダー
ブラウザキャッシュ<1ms数GBブラウザHTTPヘッダー
設計のポイント: 可能な限り「ユーザーに近い層」でキャッシュすることが重要です。ブラウザキャッシュがヒットすれば、ネットワーク通信自体が不要になります。一方、分散キャッシュは複数サーバー間でデータを共有できるため、水平スケーリング環境では必須です。

03書き込み戦略 (Write Strategies)

データを書き込む際、キャッシュとデータベースの間にどのような整合性ポリシーを取るかが重要な設計上の決定です。

Write-Through (同時書き込み)

書き込みリクエストが来たとき、キャッシュとDBに同時に書き込む戦略。データの一貫性が高いが、書き込みレイテンシはDBのレイテンシに依存する。

Write-Behind / Write-Back (遅延書き込み)

書き込みをまずキャッシュに行い、DBへの書き込みは非同期で後から行う戦略。書き込みスループットが高いが、DBへの反映前にシステム障害が起きるとデータを失う可能性がある。

リスク: キャッシュサーバーがクラッシュした場合、未反映データが失われる。金融取引などには不向き。ゲームのスコアや分析ログなど、多少の欠損が許容される用途に適する。

Write-Around (キャッシュスキップ)

書き込みはDBに直接書き込み、キャッシュはスキップする戦略。書き込まれたデータへの読み取りは、最初にキャッシュミスが発生してDBから読み込まれる。一度しか書き込まれないような大量データの取り込み時に有効。

書き込み戦略比較表

戦略一貫性書き込み速度データ損失リスク適用場面
Write-Through高中低ECサイト商品情報、ユーザーデータ
Write-Behind中高高SNSいいね数、ゲームスコア、ログ
Write-Around高高低バッチ取り込み、一度限りの大量書き込み

04キャッシュ立ち退きポリシー (Eviction Policies)

キャッシュの容量に上限があるとき、何のデータを削除するかを決定するアルゴリズムです。選択を誤るとヒット率が大きく低下します。

LRU (Least Recently Used)

最後にアクセスされた時刻が最も古いエントリを削除。最もよく使われるポリシー。時間的局所性（最近使ったものはまた使われる）の前提に基づく。

適用場面: ほとんどのユースケース。Webアプリのセッション、APIレスポンスキャッシュ。
# Redis での LRU 設定
maxmemory-policy allkeys-lru

LFU (Least Frequently Used)

アクセス頻度が最も低いエントリを削除。頻度カウンターを維持するため実装コストが高い。頻度の低いがたまに使われるアイテムも保持できる。

適用場面: コンテンツ配信、人気楽曲/動画のキャッシュ。アクセスパターンが安定している場合に有効。
# Redis での LFU 設定
maxmemory-policy allkeys-lfu

FIFO (First In, First Out)

最初にキャッシュに追加されたエントリを削除。シンプルな実装だが、古くても頻繁に使われるデータも削除してしまう問題がある。

適用場面: アクセスパターンが均一で予測可能な場合。バッファキュー的な用途。

Random (ランダム削除)

ランダムにエントリを選択して削除。実装が最もシンプル。特定のパターンに対してはLRUより良い場合もあるが、一般的には最悪ケースの保証がない。

適用場面: アクセスパターンが完全にランダムな場合。LRUのオーバーヘッドを避けたい組み込みシステム。

Redisで使用できるポリシー一覧

ポリシー対象キー説明
noeviction全キーメモリ上限到達時にエラーを返す（デフォルト）
allkeys-lru全キー全キーからLRUで削除（最も推奨）
volatile-lruTTL付きキーTTL設定済みキーからLRUで削除
allkeys-lfu全キー全キーからLFUで削除
volatile-lfuTTL付きキーTTL設定済みキーからLFUで削除
allkeys-random全キー全キーからランダムで削除
volatile-ttlTTL付きキーTTLが最も近いキーを優先削除

05キャッシュパターン

読み書きの観点からキャッシュをどのように扱うかの設計パターンです。それぞれトレードオフがあり、用途に応じて選択します。

Cache-Aside (Lazy Loading)

最もシンプルで広く使われるパターン。アプリケーションがキャッシュを直接操作する。キャッシュミス時のみDBにアクセスし結果をキャッシュに格納する。

async function getUser(userId) {
  // 1. キャッシュを確認
  const cached = await redis.get(`user:${userId}`);
  if (cached) {
    return JSON.parse(cached);  // キャッシュヒット
  }
  // 2. DBから取得
  const user = await db.findUser(userId);
  // 3. キャッシュに保存（TTL: 1時間）
  await redis.setex(`user:${userId}`, 3600, JSON.stringify(user));
  return user;
}
メリットデメリット
シンプルな実装、疎結合最初のリクエストでキャッシュミス（Cold Start）
不要なデータをキャッシュしないTTL期限切れ直後に負荷集中の可能性

Read-Through

アプリはキャッシュのみと通信し、キャッシュがDBの前段に位置してデータ取得を担当する。Cache-Asideとの違いは、キャッシュミス時のDB取得をキャッシュ層が自動的に行う点。

適用場面: AWS ElastiCache (DAX for DynamoDB) など、DBとキャッシュが密結合したマネージドサービス。アプリロジックを単純化できる。

Write-Behind (書き込みパターン)

大量の書き込みがある場合、キャッシュに書いてバッチでDBに反映することでDBへの書き込み負荷を大幅に削減できる。

# Redis リストを使った Write-Behind キュー
# 書き込み時: キューに追加
LPUSH write_queue '{"id":1,"action":"update","data":...}'

# バッチワーカー: 定期的にキューを処理
LRANGE write_queue 0 99    # 100件取得
# DB バルク INSERT/UPDATE 後
LTRIM write_queue 100 -1   # 処理済みを削除

06Redis の特徴と使い方

Redis (Remote Dictionary Server) は高速なインメモリデータストアで、多様なデータ型を提供します。単純なキャッシュを超えた幅広いユースケースに対応します。

String — 基本の文字列型

バイナリセーフな文字列。JSON、カウンター、フラグなどに最適。

# セット / 取得
SET user:123 '{"name":"田中"}' EX 3600
GET user:123

# アトミックカウンター（ページビュー）
INCR pageview:article:456
INCRBY pageview:article:456 10

# NX: 分散ロックの実装
SET lock:resource 1 NX EX 30

Hash — フィールドと値のマップ

オブジェクトの部分更新に最適。全体をJSON文字列で保存するより効率的。

# ユーザープロフィールをHashで保存
HSET user:123 name "田中太郎"
HSET user:123 email "t@example.com"
HSET user:123 age 30

# 特定フィールドのみ更新
HSET user:123 age 31

# 全フィールド取得
HGETALL user:123

List — 順序付きリスト

最新N件の取得、メッセージキュー、アクティビティフィードに活用。

# アクティビティフィード
LPUSH feed:user:123 "アクション1"
LPUSH feed:user:123 "アクション2"

# 最新10件取得
LRANGE feed:user:123 0 9

# リストを最新100件に保つ
LTRIM feed:user:123 0 99

# ブロッキングデキュー（キュー用）
BLPOP queue:jobs 0

Set — 重複なし集合

ユニークユーザー追跡、タグ管理、集合演算（和・積・差）に活用。

# 記事のタグ管理
SADD article:456:tags "tech" "redis" "cache"
SMEMBERS article:456:tags

# アクティブユーザー数（重複なし）
SADD active:2024-01-15 user:123
SCARD active:2024-01-15

# 共通フォロワーを集合積で取得
SINTER followers:user1 followers:user2

Sorted Set — スコア付き順序集合

ランキング・リーダーボード実装に最適。O(log N)で挿入・取得が可能。

# ゲームランキング
ZADD leaderboard 9850 "player:tanaka"
ZADD leaderboard 12400 "player:suzuki"
ZADD leaderboard 8200 "player:sato"

# TOP3取得（降順）
ZREVRANGE leaderboard 0 2 WITHSCORES

# 自分の順位確認
ZREVRANK leaderboard "player:tanaka"

# スコア加算
ZINCRBY leaderboard 150 "player:tanaka"

Pub/Sub — メッセージングパターン

リアルタイム通知、チャット、イベント駆動アーキテクチャに活用。

# パブリッシャー
redis.publish('notifications:user:123',
  JSON.stringify({
    type: 'new_message',
    from: 'user:456'
  }))

# サブスクライバー
const sub = redis.subscribe('notifications:user:123');
sub.on('message', (channel, msg) => {
  ws.send(msg);  // WebSocketで通知
});

セッション管理の実装例

# セッション作成
HSET session:abc123 user_id 123 role "admin" created_at "2024-01-15T10:00:00Z"
EXPIRE session:abc123 1800  # 30分TTL

# セッション確認とスライディングウィンドウ延長（アクティブ中は延長）
HGETALL session:abc123
EXPIRE session:abc123 1800  # リクエストのたびにリセット

# セッション削除（ログアウト）
DEL session:abc123
RedisとMemcachedの選択基準: RedisはList/Set/SortedSet/Hashなどのデータ型、永続化（AOF/RDB）、Pub/Sub、Lua スクリプト、クラスタ構成をサポート。単純なキーバリューのみ必要でシンプルさを優先するならMemcached、それ以外は基本的にRedisを選択。

07キャッシュの問題と解決策

キャッシュは強力ですが、適切に設計しないと深刻な問題を引き起こします。面接でも頻出の4大問題を理解しましょう。

⚡ Cache Stampede (キャッシュスタンピード)

人気のキャッシュエントリのTTLが切れた瞬間、大量のリクエストが同時にDBへ殺到する問題。DBに数百倍の負荷がかかり、最悪の場合クラッシュする。

解決策 1: Mutex Lock (ミューテックスロック)

async function getWithMutex(key) {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  // ロック取得を試みる（NX: 存在しない時のみ、EX: 10秒TTL）
  const locked = await redis.set(`lock:${key}`, 1, 'NX', 'EX', 10);

  if (locked) {
    // ロック取得成功 → DBから取得してキャッシュ更新
    const data = await db.fetch(key);
    await redis.setex(key, 3600, JSON.stringify(data));
    await redis.del(`lock:${key}`);
    return data;
  } else {
    // ロック取得失敗 → 少し待って再試行
    await sleep(50);
    return getWithMutex(key);
  }
}

解決策 2: Probabilistic Early Expiration (確率的早期更新)

// XFetch アルゴリズム: 期限が近づくほど更新確率が上がる
function shouldRecompute(expireAt, beta = 1.0) {
  const delta = -beta * Math.log(Math.random());
  return (Date.now() / 1000 + delta) >= expireAt;
}
// TTL残り時間が少なくなるにつれて確率的に早期更新を行い
// 全リクエストが同時に期限切れを検出することを防ぐ

解決策 3: Background Refresh (バックグラウンド更新)

// 古いデータを返しながらバックグラウンドで更新
async function getWithStaleWhileRevalidate(key) {
  const entry = await redis.hgetall(key);  // {data, expire_at}

  const isExpired = Date.now() > entry.expire_at;
  const isSoftExpired = Date.now() > (entry.expire_at - 60000); // 1分前

  if (isSoftExpired && !isExpired) {
    // まだ有効だが古い: バックグラウンドで更新しつつ古いデータを返す
    refreshInBackground(key);  // 非同期、awaitしない
  }
  return entry.data;  // 古くても即返却
}

❄️ Cache Avalanche (キャッシュ雪崩)

多数のキャッシュエントリが同時刻に一斉に期限切れになり、大量のDBリクエストが同時に発生する問題。同じTTLを設定したバッチ処理後などに発生しやすい。

具体的シナリオ: 毎朝5時に全商品情報をキャッシュに1時間TTLで格納 → 毎朝6時に全キャッシュが同時期限切れ → DBに数万件のクエリが同時に殺到

解決策: TTL Jitter (ジッター追加)

// 悪い例: 全エントリが同時に期限切れになる
products.forEach(p => redis.setex(`product:${p.id}`, 3600, p));

// 良い例: +/-10分のランダムジッターを追加
products.forEach(p => {
  const jitter = Math.floor(Math.random() * 600);  // 0〜600秒
  const ttl = 3600 + jitter;  // 3600〜4200秒
  redis.setex(`product:${p.id}`, ttl, p);
});
// 期限切れが600秒間に分散し、DBへの負荷が平準化される

その他の解決策

  • Circuit Breaker: DBへのリクエストが閾値を超えたら一時的にサーキットをオープンにしてエラーを返す
  • 多層キャッシュ: Redisが落ちてもアプリ内ローカルキャッシュが一次防衛線になる
  • 永続キャッシュ + バックグラウンド更新: TTLで削除する代わりに古いデータを返しつつバックグラウンドで更新

🔍 Cache Penetration (キャッシュ貫通)

存在しないキーへのリクエストを大量に送り、キャッシュを完全にスキップしてDBへ負荷をかける攻撃または設計上の問題。

攻撃例: GET /user/999999999, GET /user/888888888 ... 存在しないIDを次々にリクエスト → 全てキャッシュミス → 全てDBアクセス

解決策 1: Null Caching (null値のキャッシュ)

async function getUser(userId) {
  const cacheKey = `user:${userId}`;
  const cached = await redis.get(cacheKey);

  if (cached !== null) {
    if (cached === '__NULL__') return null;  // 存在しないことをキャッシュ
    return JSON.parse(cached);
  }

  const user = await db.findUser(userId);

  if (!user) {
    // 存在しないユーザーも短いTTLでキャッシュ（5分）
    await redis.setex(cacheKey, 300, '__NULL__');
    return null;
  }
  await redis.setex(cacheKey, 3600, JSON.stringify(user));
  return user;
}

解決策 2: Bloom Filter (ブルームフィルター)

確率的データ構造で、「そのキーがDBに存在するか」を非常に小さいメモリで判定（偽陰性なし、偽陽性あり）。

# Redis Bloom Filter (RedisBloom モジュール)
# 初期化（エラー率0.1%, 想定要素数100万）
BF.RESERVE users:bloom 0.001 1000000

# DBの全ユーザーIDをBloom Filterに追加
BF.MADD users:bloom 1 2 3 ... 999999

# リクエスト処理時
BF.EXISTS users:bloom 99999999  # 0: 存在しない確定 → DBアクセス不要
BF.EXISTS users:bloom 123       # 1: 存在する可能性あり → DBで確認

🔥 Hot Key Problem (ホットキー問題)

特定のキーに極端にリクエストが集中し、そのキーを保持するRedisノードが過負荷になる問題。

具体例: 有名人のプロフィールページがバイラルし、特定ユーザーIDのキャッシュキーへのアクセスが毎秒10万件を超える

解決策 1: Local Cache (アプリ内ローカルキャッシュ)

// アプリサーバー内のメモリキャッシュ（5秒TTL）
const localCache = new LRUCache({ max: 1000, ttl: 5000 });

async function getHotData(key) {
  // 1. ローカルキャッシュを確認（ネットワーク不要）
  const local = localCache.get(key);
  if (local) return local;

  // 2. Redisを確認
  const remote = await redis.get(key);
  if (remote) {
    localCache.set(key, remote);  // ローカルにも保存
    return remote;
  }
  // 3. DBから取得...
}
// 同一サーバーへの繰り返しアクセスはRedisにすら届かない

解決策 2: Key Sharding (キーのシャーディング)

// ホットキーを複数のキーに分散させる
// user:viral_id → user:viral_id:0 〜 user:viral_id:9

function getShardKey(key, shards = 10) {
  const shard = Math.floor(Math.random() * shards);
  return `${key}:${shard}`;
}

// 書き込み: 全シャードに書き込む
for (let i = 0; i < 10; i++) {
  redis.setex(`user:viral_id:${i}`, 3600, data);
}
// 読み取り: ランダムシャードから取得（10ノードに分散）
const val = await redis.get(getShardKey('user:viral_id'));

08CDNキャッシュ

CDN (Content Delivery Network) は地理的に分散したエッジサーバーでコンテンツをキャッシュし、ユーザーに最も近いサーバーから高速配信する仕組みです。

CDNエッジノードが世界各地にコンテンツをキャッシュし、ユーザーに近い場所から配信

静的コンテンツ vs 動的コンテンツ

✓ CDNキャッシュ向き（静的）

  • HTML/CSS/JS ファイル
  • 画像・動画・フォント
  • 全ユーザー共通のAPIレスポンス（商品一覧など）
  • ダウンロードファイル、PDF

△ 設定次第（動的）

  • パーソナライズされたAPIレスポンス（Varyヘッダーで分岐）
  • 認証が必要なページ（Surrogate-Keyでのパージが必要）
  • A/Bテストコンテンツ

Cache-Control ヘッダー設定

# 静的アセット（ハッシュ付きファイル名）: 永続キャッシュ
Cache-Control: public, max-age=31536000, immutable
# main.abc123.js は内容が変わらないので1年キャッシュ

# HTMLページ: ブラウザキャッシュさせないがCDNは1時間
Cache-Control: public, max-age=0, s-maxage=3600
# max-age=0: ブラウザは毎回確認
# s-maxage=3600: CDN（共有キャッシュ）は1時間保持

# ユーザー固有データ: キャッシュ禁止
Cache-Control: private, no-cache

# APIレスポンス（全ユーザー共通）: stale-while-revalidate
Cache-Control: public, max-age=300, stale-while-revalidate=60
# 300秒間キャッシュ、期限切れ後60秒間は古いデータを返しつつバックグラウンドで更新

Vary ヘッダー

同じURLでもリクエストの条件によって異なるレスポンスを返す場合、VaryヘッダーでCDNがキャッシュを分岐させます。

# 言語によってキャッシュを分岐
Vary: Accept-Language
# /api/products は Accept-Language: ja と en で別々にキャッシュ

# エンコーディングで分岐（推奨）
Vary: Accept-Encoding

# 注意: User-Agentは無限パターンでヒット率激減
# Vary: User-Agent  ← 使わない！レスポンシブデザインで対応する
注意: Vary: User-Agent はUser-Agentの組み合わせが無限にあるためキャッシュヒット率が激減します。モバイル対応はレスポンシブデザインや別URLで実現することを推奨。

💡面接対策 — よく聞かれる質問

システムデザイン面接でキャッシュについて聞かれる定番質問と、評価される回答のポイントです。

01

「キャッシュ戦略をどう選ぶか？」

読み取り頻度 vs 書き込み頻度、データの一貫性要件、TTLの許容範囲を軸に説明。Cache-Asideが汎用的で最初の選択肢と述べた上で、要件に応じてWrite-Throughなどを組み合わせる方針を示す。

02

「StampedeとAvalancheの違いは？」

Stampede = 1つの人気キーのTTL切れで多数のリクエストがDB直撃。Avalanche = 大量のキーが同時にTTL切れ。どちらも解決策（Lock, Jitter）とトレードオフを説明できると高評価。

03

「RedisとMemcachedの違いは？」

Redisはデータ型が豊富（List/Set/SortedSet/Hash）、永続化対応、Pub/Sub機能あり、クラスタ構成が容易。Memcachedはシンプルなキーバリューのみ。通常はRedisを推奨。

04

「どのデータをキャッシュすべきか？」

読み取り頻度が高く変更が少ないデータ。計算コストが高いクエリ結果。共有できるデータ（ユーザー固有でないもの）。80/20ルール: 20%のデータが80%のアクセスを受ける。

05

「キャッシュの整合性をどう保つか？」

DBとキャッシュの同期方法: (1) TTLによる自然な期限切れ、(2) DB更新時に即座にキャッシュを無効化（Cache Invalidation）、(3) Write-ThroughでDB更新と同時にキャッシュ更新。"There are only two hard things in CS: cache invalidation and naming things"

06

「適切なTTLをどう決める？」

データの更新頻度（更新頻度 < 1/TTL が目安）、整合性の許容範囲、メモリコストのバランスで決定。長すぎると古いデータ提供リスク、短すぎるとヒット率低下。Jitterも忘れずに言及。

面接での加点ポイント: 「キャッシュを使うとメモリコスト、整合性リスク、複雑性が増す。これらのトレードオフを理解した上でキャッシュが必要かどうかから検討する」というスタンスを最初に示すと、経験豊富なエンジニアとして評価されます。

設計チェックリスト

  • ✅ キャッシュすべきデータを特定した（読み取り多/書き込み少/共有可能）
  • ✅ 適切なキャッシュ層を選択した（アプリ内 / Redis / CDN）
  • ✅ 書き込み戦略を決定した（Write-Through / Write-Behind / Write-Around）
  • ✅ TTLとEviction Policyを設定した（Jitter追加も忘れずに）
  • ✅ Cache Stampede対策を実装した（Mutex Lock か Early Expiration）
  • ✅ Cache Penetration対策を実装した（Null Caching か Bloom Filter）
  • ✅ Hot Key対策を検討した（Local Cache か Key Sharding）
  • ✅ 整合性更新戦略を決定した（TTL自然期限切れ か 明示的Invalidation）
  • ✅ モニタリング指標を設定した（Hit Rate, Miss Rate, Eviction Count, Memory Usage）

10深掘り: Redisの内部実装

Redisが高速かつ多機能な理由は内部データ構造の最適化にある。面接で差をつける実装レベルの知識を押さえる。

ziplist / listpack vs skiplist — ZSetの内部表現

Redis の Sorted Set (ZSet) は要素数とサイズに応じて内部表現を自動的に切り替える。小規模データはlistpack（旧ziplist）、大規模データはskiplist + hashが使われる。

切り替え閾値のデフォルト:
  • 要素数が 128 以下 かつ 各要素が 64 バイト以下 → listpack（連続メモリ、キャッシュ効率◎）
  • いずれかを超えると → skiplist + hash（O(log N)検索）
# 閾値の確認・変更
CONFIG GET zset-max-listpack-entries  # デフォルト: 128
CONFIG GET zset-max-listpack-value    # デフォルト: 64 bytes

# メモリ効率を優先する場合（listpack維持）
CONFIG SET zset-max-listpack-entries 256
CONFIG SET zset-max-listpack-value 128

# 現在の内部表現を確認
OBJECT ENCODING leaderboard   # "ziplist" / "skiplist"

# skiplist の特徴
# - レベル0: 全要素へのリンク（線形走査）
# - レベルN: 確率的にスキップ（平均O(log N)）
# - 探索・挿入・削除: 平均 O(log N)
内部表現メモリ検索速度適用条件
listpack少ないO(N)線形≤128要素 かつ ≤64B/要素
skiplist+hash多いO(log N)閾値超過後

RDB vs AOF — 永続化の仕組み

RDB (Redis Database Snapshot)

指定時刻または変更件数に達したとき、メモリ全体をスナップショットとしてディスクに書き出す。fork()でコピーオンライトを使うため、書き込み中も本プロセスは稼働続ける。

# redis.conf の RDB 設定
# 「N秒以内にM回変更があった場合に保存」
save 3600 1      # 1時間に1回以上変更
save 300  100    # 5分間に100回以上変更
save 60   10000  # 1分間に1万回以上変更

dbfilename dump.rdb
dir /var/lib/redis/
利点: コンパクト・リストア高速・I/O負荷小 / 欠点: 最後のスナップショット以降のデータを失う可能性

AOF (Append Only File)

全ての書き込みコマンドをログに追記する。appendfsync設定でfsyncのタイミングを制御する。

# redis.conf の AOF 設定
appendonly yes
appendfilename "appendonly.aof"

# fsync タイミング（重要）
appendfsync always    # 毎コマンド後: 最安全・最低速
appendfsync everysec  # 毎秒: 推奨・最大1秒分のデータ損失
appendfsync no        # OS任せ: 最速・データ損失リスク大

# AOF Rewrite: ファイルを圧縮（不要コマンド除去）
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
everysec の意味: 最大1秒分のデータを失う可能性があるが、パフォーマンスと耐久性のバランスが最も良い。本番環境での推奨設定。

Redis Cluster のスロット設計

Redis Cluster はキーを16,384スロットに分散する。スロット番号は CRC16(key) mod 16384 で決定される。

# スロット計算の仕組み
slot = CRC16("user:123") % 16384  # → 例: 5649

# ハッシュタグ: {} 内の文字列でスロットを固定
# 同じスロットに置きたいキーに使う（MULTI/EXEC用途）
"user:{123}:profile"   # CRC16("123") でスロット決定
"user:{123}:orders"    # 同じスロット → 同じノード
"user:{456}:profile"   # 別スロット → 別ノード

# クラスター構成例（3マスター + 3レプリカ）
# Node A: スロット 0〜5460
# Node B: スロット 5461〜10922
# Node C: スロット 10923〜16383
MOVED vs ASK リダイレクト:
  • MOVED: キーが恒久的に別ノードに移った → クライアントはノード情報を更新し再送
  • ASK: リシャーディング中で一時的に別ノードにある → 今回だけ別ノードへ送る（更新しない）

Lua スクリプトによるアトミック操作

RedisはEVALコマンドでLuaスクリプトを実行する。スクリプト全体はアトミックに実行され、他のコマンドが割り込めない。チェック&セット（CAS）操作に最適。

# CAS: 現在値を確認してから更新（競合を防ぐ）
# EVAL script numkeys key [key ...] arg [arg ...]

local script = [[
  local current = redis.call('GET', KEYS[1])
  if current == ARGV[1] then
    redis.call('SET', KEYS[1], ARGV[2])
    return 1  -- 成功
  else
    return 0  -- 失敗（値が変わっていた）
  end
]]

const result = await redis.eval(
  script,
  1,                    // numkeys
  'stock:product:456', // KEYS[1]
  '100',               // ARGV[1]: 期待する現在値
  '99'                 // ARGV[2]: 新しい値
);
// result=1: 更新成功 / result=0: 他者が先に変更

11深掘り: キャッシュ一貫性の保証

データ更新時にキャッシュとDBの不整合を防ぐパターン。設計次第で不整合ウィンドウを最小化できる。

ダブルデリート（二重削除）パターン

Cache-Aside において更新処理の前後でキャッシュを削除し、不整合ウィンドウを最小化するパターン。

async function updateUserWithDoubleDelete(userId, newData) {
  const cacheKey = `user:${userId}`;

  // 1st delete: 古い値を持つ他リクエストのキャッシュ格納を防ぐ
  await redis.del(cacheKey);

  // DB更新（この間に別リクエストが古い値をキャッシュするリスクがある）
  await db.updateUser(userId, newData);

  // 短い待機（レプリカラグを考慮）
  await sleep(100);

  // 2nd delete: DB更新後に古い値をキャッシュした可能性を排除
  await redis.del(cacheKey);
}
// 不整合ウィンドウ: 約100ms（sleep時間）のみ
注意: 二重削除は不整合を最小化するが完全には防げない。厳密な一貫性が必要な場合はWrite-Through + 分散ロックを検討する。

Versioned Cache（バージョン付きキャッシュキー）

キャッシュキーにバージョン番号を含めることで、古いキャッシュが新しいキャッシュに上書きされることを防ぐ。

# DBにバージョン番号を持つスキーマ
# users: {id, name, email, version}

async function getUser(userId) {
  // バージョン付きキャッシュキー
  const version = await redis.get(`user:${userId}:version`);
  if (version) {
    const cached = await redis.get(`user:${userId}:v${version}`);
    if (cached) return JSON.parse(cached);
  }
  const user = await db.findUser(userId);
  // バージョン番号を含めてキャッシュ
  await redis.setex(`user:${userId}:v${user.version}`, 3600, JSON.stringify(user));
  await redis.set(`user:${userId}:version`, user.version);
  return user;
}

async function updateUser(userId, newData) {
  // DB更新でバージョンをインクリメント
  const updated = await db.updateUserWithVersionBump(userId, newData);
  // バージョンポインタだけ更新 → 古いバージョンのキャッシュは自然消滅
  await redis.set(`user:${userId}:version`, updated.version);
}

Pub/Sub を使ったキャッシュ無効化

データ更新時にRedisのPub/Subチャンネルに変更イベントを発行し、全アプリサーバーのローカルキャッシュを同時に無効化する。

// Publisher（更新サービス）
async function updateProduct(productId, newData) {
  await db.updateProduct(productId, newData);
  // 全ノードに無効化イベントを配信
  await redis.publish('cache:invalidate', JSON.stringify({
    type: 'product',
    id: productId,
    timestamp: Date.now()
  }));
}

// Subscriber（各アプリサーバーで実行）
const subscriber = redis.duplicate(); // 専用コネクション
await subscriber.subscribe('cache:invalidate');
subscriber.on('message', (channel, message) => {
  const event = JSON.parse(message);
  // ローカルキャッシュから削除
  localCache.delete(`${event.type}:${event.id}`);
  // 分散キャッシュからも削除
  redis.del(`product:${event.id}`);
});

Read-Your-Writes 保証

ユーザーが書き込んだ直後に、その内容を確実に読み取れるようにする設計パターン。DB読み書きのレプリカラグが問題になる。

// 解決策1: 書き込んだユーザーの次のリクエストはマスターDBへ
async function readAfterWrite(userId, key, userJustWrote) {
  if (userJustWrote) {
    // 直後はマスターから読む（レプリカラグを回避）
    return masterDb.get(key);
  }
  // 通常はレプリカから
  return replicaDb.get(key);
}

// 解決策2: 書き込み後にキャッシュを直接更新（キャッシュ優先）
async function writeAndCache(userId, data) {
  await db.write(userId, data);
  // 書き込んだ本人向けに短いTTLでキャッシュを即時更新
  await redis.setex(`user:${userId}`, 30, JSON.stringify(data));
  // 30秒後にレプリカが追いついた後はDBから再取得される
}

演習1: Eコマースの商品ページキャッシュ設計

Eコマースサイトの商品詳細ページには3種類のデータが混在する。それぞれ適切なキャッシュ戦略を設計せよ。

商品情報（低更新頻度）

商品名・説明・画像URL・カテゴリ。月数回の更新。

推奨設計:
  • TTL: 24時間（86400秒）
  • パターン: Cache-Aside + Write-Through
  • キー: product:{id}:info:v{version}
  • Invalidation: 更新時に明示的削除 + バージョンバンプ
  • CDNキャッシュ: 1時間（s-maxage=3600）

在庫数（高更新頻度）

残り在庫数。注文のたびに変動。0になると購入不可。

推奨設計:
  • TTL: 30秒（在庫表示は多少古くてOK）
  • パターン: Redis INCRBY（アトミック減算）
  • キー: product:{id}:stock
  • 注意: 購入確定時はDBとRedisをトランザクションで整合
  • 表示: 「残りわずか（10以下）」のみ表示し実数は隠す

価格（随時変動）

タイムセール・クーポン・ユーザー別価格など複雑な計算が必要。

推奨設計:
  • TTL: 5分（セール価格は最悪5分古くてもOK）
  • パターン: Cache-Aside。購入確定時はDBで再計算
  • キー: product:{id}:price:{userId}:{timestamp_5min}
  • 注意: タイムセール開始時刻にキャッシュを一括削除
  • CDN禁止: ユーザー別価格はCDNキャッシュNG
// 商品ページデータ取得の実装例
async function getProductPageData(productId, userId) {
  const [info, stock, price] = await Promise.all([
    // 1. 商品情報（24時間TTL）
    redis.get(`product:${productId}:info`) ||
      db.getProduct(productId).then(p => {
        redis.setex(`product:${productId}:info`, 86400, JSON.stringify(p));
        return p;
      }),

    // 2. 在庫数（30秒TTL、Redisの整数値として管理）
    redis.get(`product:${productId}:stock`) ||
      db.getStock(productId).then(s => {
        redis.setex(`product:${productId}:stock`, 30, s);
        return s;
      }),

    // 3. 価格（ユーザー別・5分TTL）
    getUserPrice(productId, userId),
  ]);

  return { ...info, stock: parseInt(stock), price };
}

演習2: リーダーボードをRedisで実装する

ゲームの1億ユーザーリーダーボードをRedisのSorted Setで構築する完全実装。

# Python実装（redis-py）
import redis
import math

r = redis.Redis(host='localhost', port=6379)
LEADERBOARD_KEY = "game:season2:leaderboard"

def add_score(player_id: str, score: int) -> None:
    """スコアを追加（既存スコアは置き換え）"""
    r.zadd(LEADERBOARD_KEY, {player_id: score})

def increment_score(player_id: str, delta: int) -> float:
    """スコアをインクリメント（アトミック）"""
    return r.zincrby(LEADERBOARD_KEY, delta, player_id)

def get_top_n(n: int = 10) -> list:
    """トップN取得（降順・スコア付き）"""
    results = r.zrevrange(LEADERBOARD_KEY, 0, n-1, withscores=True)
    return [
        {"rank": i+1, "player": pid.decode(), "score": int(score)}
        for i, (pid, score) in enumerate(results)
    ]

def get_player_rank(player_id: str) -> dict:
    """プレイヤーの順位とスコアを取得"""
    rank = r.zrevrank(LEADERBOARD_KEY, player_id)
    score = r.zscore(LEADERBOARD_KEY, player_id)
    if rank is None:
        return {"error": "player not found"}
    return {"rank": rank + 1, "score": int(score)}

def get_around_player(player_id: str, window: int = 5) -> list:
    """プレイヤーの前後window人を取得"""
    rank = r.zrevrank(LEADERBOARD_KEY, player_id)
    if rank is None:
        return []
    start = max(0, rank - window)
    end = rank + window
    results = r.zrevrange(LEADERBOARD_KEY, start, end, withscores=True)
    return [
        {"rank": start+i+1, "player": pid.decode(), "score": int(score)}
        for i, (pid, score) in enumerate(results)
    ]

# 使用例
add_score("player:tanaka", 9850)
increment_score("player:tanaka", 150)  # → 10000
print(get_top_n(3))
print(get_player_rank("player:tanaka"))

Sharded Leaderboard設計

# 1億ユーザーを10シャードに分割
# 各シャードが1000万ユーザーを担当

def get_shard(player_id: str) -> int:
    return hash(player_id) % 10

def add_score_sharded(player_id, score):
    shard = get_shard(player_id)
    key = f"lb:shard:{shard}"
    r.zadd(key, {player_id: score})

# グローバルTOP10取得: 全シャードのTOP10を取り出してマージ
def get_global_top10() -> list:
    candidates = []
    for shard in range(10):
        top = r.zrevrange(f"lb:shard:{shard}", 0, 9, withscores=True)
        candidates.extend(top)
    candidates.sort(key=lambda x: -x[1])
    return candidates[:10]

演習3: キャッシュ設計アンチパターン診断（5問）

以下の各設計の問題点を特定し、修正案を考えよ。答えは下に示す。

問題1: 同一TTLで全商品をキャッシュ

// 毎朝5時のバッチ処理
const products = await db.getAllProducts();
for (const p of products) {
  await redis.setex(`product:${p.id}`, 3600, JSON.stringify(p));
}
問題: 毎朝6時に全キャッシュが同時に期限切れ → Cache Avalanche発生。
修正: 3600 + Math.floor(Math.random() * 600) のようにジッターを追加する。

問題2: キャッシュミス時の処理なし

async function getUser(userId) {
  return JSON.parse(await redis.get(`user:${userId}`));
}
// キャッシュミス時はnullが返る
問題: キャッシュミス時にnullが返り、呼び出し元がDBアクセスしないためデータが取得できない。また、存在しないユーザーへのリクエストでもDBへの保護がない（Cache Penetration）。
修正: キャッシュミス時にDBから取得し結果をキャッシュするCache-Asideパターンを実装する。存在しない場合もnull値をキャッシュする。

問題3: セッション情報をアプリメモリに保存

// アプリサーバーのグローバル変数
const sessions = new Map();
sessions.set(sessionId, { userId: 123, role: 'admin' });
// 別サーバーへのリクエストはこのMapにアクセスできない
問題: スケールアウト時に別サーバーへリクエストが届くとセッション情報が見つからない（スティッキーセッション問題）。サーバー再起動でセッション消失。
修正: Redis等の外部分散ストアにセッションを保存し、全サーバーが共有できるようにする。

問題4: ホットなユーザーIDを単一キーで管理

// 有名人のプロフィール（毎秒10万アクセス）
await redis.get('user:celebrity:1');
// このキーを持つノードのCPUが100%に
問題: Hot Key Problem。単一Redisノードに負荷集中。
修正: ①アプリサーバー内のローカルキャッシュ（5秒TTL）を追加する。②キーをシャーディングして複数ノードに分散させる（user:celebrity:1:0〜:9）。

問題5: キャッシュキーのバージョン管理なし

// v1: ユーザー情報 {name, email}
redis.setex(`user:${id}`, 86400, JSON.stringify({name, email}));

// アプリアップデートでv2に: {name, email, phoneNumber}
// 古いキャッシュが残っていると parseError の可能性
問題: スキーマ変更時に古いフォーマットのキャッシュが残り、デシリアライズエラーが発生する可能性がある。
修正: キャッシュキーにバージョンを含める（user:v2:{id}）か、キャッシュデプロイ時に全キーを削除（FLUSHDBは本番で危険なため、Scanを使った選択的削除を推奨）。
