SECTION 01ロードバランシングとは

単一サーバーには物理的な限界がある。ロードバランサーはその限界を超えるための最初の一手だ。

単一サーバーの限界

どれだけ高性能なサーバーでも、扱えるリクエスト数には上限がある。具体的には以下の3つのボトルネックが発生する。

🔴 CPU の限界

コア数・クロック周波数の上限。マルチスレッドでも並列処理可能数に限りがある。大量リクエストでCPU使用率100%に張り付き、レスポンスタイムが急増する。

🟡 メモリの限界

物理メモリが枯渇するとスワップが発生し、レイテンシが100倍以上悪化。セッションデータやキャッシュで特に顕著。

🔵 ネットワーク帯域の限界

NICの最大スループット（例：10Gbps）を超えることはできない。動画配信・大量データ転送では特にボトルネックになりやすい。

垂直スケーリング vs 水平スケーリング

垂直スケーリング（スケールアップ）

  • 既存サーバーのCPU・RAM・ディスクを強化
  • シンプルだがハードウェアの物理的上限がある
  • コストが指数的に上昇（ハイエンドサーバーは高価）
  • SPOFが残る — サーバーが1台なら停止でサービス全断

水平スケーリング（スケールアウト）

  • 同スペックのサーバーを複数台追加
  • コストが線形に増加（汎用サーバーを利用可能）
  • 理論上無限にスケール可能
  • ロードバランサーが必須になる

SPOF の排除とフォールトトレランス

SPOF (Single Point of Failure) とは、障害が発生するとシステム全体が停止してしまう単一コンポーネントのこと。ロードバランサーは複数のバックエンドサーバーに分散することで、1台が落ちても残りが処理を継続できるようにする。

可用性の計算: サーバー1台の可用性が 99% の場合、2台を並列にすると
1 - (1-0.99)² = 1 - 0.0001 = 99.99% に向上する。
左：単一サーバーはSPOF。右：ロードバランサーが複数サーバーへ分散し冗長性を確保。

SECTION 02L4 vs L7 ロードバランサー

OSIモデルのどのレイヤーで動作するかによって、速度・機能・適用シナリオが大きく異なる。

L4 ロードバランサー（トランスポート層）

TCP/UDPのヘッダー情報（IPアドレス・ポート番号）のみを見てルーティングする。HTTPの中身は見ない。

  • 処理速度が非常に高速（パケット単位で転送）
  • プロトコル非依存（HTTP/FTP/SSH/SMTP すべて扱える）
  • コンテキストなし — リクエストの内容は不透明
  • SSL終端は不可（パケットを解釈しないため）
  • 代表例：AWS NLB、HAProxy（TCPモード）、LVS

L7 ロードバランサー（アプリケーション層）

HTTPリクエストを完全にデコードし、URL・ヘッダー・Cookieなどの内容を見てルーティングする。

  • コンテンツベースルーティング（/api/*→APIサーバー等）
  • SSL/TLS終端が可能
  • ヘッダーの追加・変更・削除
  • A/Bテスト・カナリアデプロイに対応
  • 代表例：AWS ALB、Nginx、Envoy、Traefik
L4はTCP/UDPヘッダのみを処理し高速。L7はHTTPを完全解析し細かなルーティングが可能。

SECTION 03ロードバランシングアルゴリズム（全6種）

どのサーバーにリクエストを転送するかを決めるロジック。ワークロードの特性に応じて選択が変わる。

1. Round Robin
均等分散

サーバーリストを順番に循環してリクエストを割り当てる。最もシンプルなアルゴリズム。

適用シナリオ: サーバーのスペックが均一で、リクエストの処理時間が同程度の場合。

欠点: 処理時間が長いリクエストが集中すると、一部サーバーがホットスポットになる。

2. Weighted Round Robin
重み付き均等分散

各サーバーに重みを設定し、重みに比例した頻度でリクエストを割り当てる。

例: Server A (weight=3) : Server B (weight=1) → A に 75%、B に 25% のリクエスト

適用シナリオ: サーバーのスペックに差がある場合（古いサーバーと新しいサーバーの混在）。

3. Least Connections
最小コネクション

現在のアクティブ接続数が最も少ないサーバーにリクエストを送る。

適用シナリオ: 長時間持続するコネクション（WebSocket・DB・ファイル転送）で Round Robin より均等に分散できる。
4. Weighted Least Connections
重み付き最小コネクション

最小コネクション数を重みで割った値が最小のサーバーを選ぶ。

スコア = active_connections / weight
スコアが最小のサーバーを選択する。

スペック差があり、かつ接続時間も不均一な場合に最適。

5. IP Hash
スティッキーセッション

クライアントのIPアドレスをハッシュ化し、常に同一サーバーへルーティングする。

server_index = hash(client_ip) % server_count

欠点: サーバー追加・削除でハッシュ値が変化し、セッションが切れる可能性がある。一貫性ハッシュ（Consistent Hashing）で改善可能。

6. Power of Two Choices
確率的・高スケール

ランダムに2台のサーバーを選び、接続数が少ない方を選択する。

なぜ優れているか: 完全ランダムより大幅に均等な分散を実現し、最小コネクションよりも計算コストが低い。大規模クラスター（数百台以上）で特に有効。

アルゴリズム比較表

アルゴリズム計算コスト均等性スティッキー最適ユースケース
Round RobinO(1)良好なし均一スペック・短時間リクエスト
Weighted Round RobinO(1)重み依存なしスペック混在
Least ConnectionsO(n)優秀なし長時間コネクション

Round Robin は機械的に順番に振るが、Least Connections は現在の負荷を見て最適なサーバーを選ぶ。

SECTION 04スティッキーセッション (Sticky Sessions)

同一クライアントを常に同一サーバーへ振り向ける仕組み。なぜ必要で、なぜ問題なのかを理解する。

なぜスティッキーセッションが必要か

従来のWebアプリは、セッション情報（ログイン状態・カートの中身など）をサーバーのメモリに保持することが多い（ステートフルアーキテクチャ）。この場合、別のサーバーにリクエストが届くとセッション情報が見つからず、ログアウト状態として扱われてしまう。

Cookie-based Stickiness（推奨）

ロードバランサーが特定のCookieを発行・読み取り、サーバーを識別する。

  • Nginxでは sticky cookie ディレクティブ
  • クライアントのIPが変わっても継続（モバイル回線切替等）
  • LBが発行するCookieは暗号化可能

IP Hash Stickiness

クライアントIPからサーバーを決定する。Cookieが使えない環境向け。

  • NAT背後のクライアントは同一IPに見える問題
  • モバイルクライアントのIP変動で接続が切れる
  • サーバー台数変化でルーティングが変わる

スティッキーセッションの問題点

⚠️ 特定サーバーへの偏り: 一部のユーザーが長時間セッションを維持すると、特定サーバーへの負荷が集中する。Round Robin の均等分散の恩恵を失う。

⚠️ スケールアウトの困難: サーバーを追加しても、既存セッションが旧サーバーに縛られるため、新サーバーへのトラフィック移行が遅い。

⚠️ 障害時のセッション消失: スティッキー先のサーバーが落ちると、そのサーバーのセッション情報はすべて失われる。

解決策：分散セッションストア（ステートレス化）

セッション情報をサーバーのメモリではなく、外部の分散ストレージ（Redisなど）に保存することで、どのサーバーへリクエストが届いてもセッションを取得できるようになる。

Redisのような分散セッションストアを使うことでスティッキーセッション不要のステートレスアーキテクチャになる。

セッション管理コード例（Node.js + Redis）

// express-session + connect-redis でセッションをRedisに保存
const session = require('express-session');
const RedisStore = require('connect-redis')(session);
const redis = require('redis');

const client = redis.createClient({
  host: 'redis-cluster.internal',
  port: 6379
});

app.use(session({
  store: new RedisStore({ client }),
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: {
    secure: true,     // HTTPS のみ
    httpOnly: true,   // XSS対策
    maxAge: 1000 * 60 * 60 * 24  // 24時間
  }
}));
// これでどのサーバーへルーティングされても同一セッションを参照できる

SECTION 05ヘルスチェック

死んだサーバーへのリクエスト転送を防ぐ仕組み。Active と Passive の2方式がある。

Active ヘルスチェック（積極的確認）

ロードバランサーが定期的にバックエンドサーバーへプローブリクエストを送信し、応答を確認する。

  • 設定間隔（例：5秒ごと）でエンドポイントを叩く
  • 失敗が N 回連続した場合にサーバーをdown状態とする
  • 回復後は slow start で徐々にトラフィックを戻す
  • Nginx Plus、AWS ALB、Envoyが対応

Passive ヘルスチェック（受動的検知）

実際のクライアントリクエストへの応答を監視し、エラーが続いた場合にサーバーを外す。

  • 追加のプローブトラフィックが不要
  • エラーが実際に出てから検知（少しラグがある）
  • Nginxの max_fails / fail_timeout が該当
  • 無料版Nginxでも使用可能

ヘルスチェックの種類

種類チェック方法適用プロトコル特徴
HTTPGET /health → 200 OKHTTP/HTTPSアプリレベルの正常性確認
TCPTCPコネクション確立確認任意TCPアプリ非依存・軽量
gRPCgRPC Health Checking ProtocolgRPCマイクロサービス向け
ICMP (Ping)ping応答全般OSレベルのみ確認

Nginx upstream ヘルスチェック設定例

# /etc/nginx/conf.d/upstream.conf
upstream app_backend {
    # Passive health check
    server 10.0.1.10:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8080 max_fails=3 fail_timeout=30s backup;

    # keep-alive でバックエンドとのコネクションを再利用
    keepalive 32;
}

server {
    listen 80;
    location / {
        proxy_pass       http://app_backend;
        proxy_next_upstream error timeout http_500 http_502 http_503;
        proxy_next_upstream_tries 2;  # 失敗時に別サーバーへリトライ
        proxy_connect_timeout 3s;
        proxy_read_timeout    30s;
    }
}
# Nginx Plus の Active Health Check 例
upstream app_backend {
    zone backend 64k;
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
}

server {
    location / {
        proxy_pass http://app_backend;
        # アクティブヘルスチェック（Nginx Plus機能）
        health_check interval=5s fails=3 passes=2 uri=/health;
    }
}
# interval: チェック間隔, fails: down判定の連続失敗数
# passes: up復帰判定の連続成功数（slow start効果）
Slow Start（段階的復帰）の仕組み: 復帰したサーバーにいきなり全量のトラフィックを送ると、キャッシュが冷たい状態でサーバーが再び過負荷になる可能性がある。Slow Start では一定時間をかけて少しずつトラフィック割合を増やしていく。Nginxでは slow_start=30s で設定可能。

SECTION 06リバースプロキシ

クライアントの代理ではなく、サーバーの代理として動作するプロキシ。

フォワードプロキシはクライアント側に置かれクライアントを代理する。リバースプロキシはサーバー側に置かれサーバーを代理する。

リバースプロキシの機能一覧

SSL/TLS 終端

クライアントとの暗号化通信をリバースプロキシで終端し、バックエンドへはHTTPで転送。証明書管理を一元化できる。

圧縮（gzip/Brotli）

レスポンスを圧縮してクライアントへ送信。バックエンドの負荷を増やさずに転送量を削減できる。

静的コンテンツキャッシュ

画像・CSS・JSなどの静的ファイルをプロキシがキャッシュし、バックエンドへのリクエストを削減。

レート制限

特定IPやエンドポイントへのリクエスト数を制限し、DoS攻撃やAPIの過剰利用を防ぐ。

ヘッダー操作

X-Real-IPなどのカスタムヘッダーを追加し、バックエンドが本来のクライアントIPを取得できるようにする。

WAF統合

Web Application Firewallと連携し、SQLインジェクション・XSSなどの攻撃パターンを検知・ブロック。

Nginx 設定例（upstream + SSL終端）

# /etc/nginx/sites-available/api.conf
upstream api_servers {
    least_conn;  # Least Connections アルゴリズム
    server 10.0.1.10:8080 weight=3 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8080 weight=3 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8080 weight=1 max_fails=3 fail_timeout=30s;
    keepalive 64;
}

server {
    listen 443 ssl http2;
    server_name api.example.com;

    # SSL 終端（TLS Termination）
    ssl_certificate     /etc/ssl/certs/api.crt;
    ssl_certificate_key /etc/ssl/private/api.key;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    ssl_session_cache   shared:SSL:10m;  # セッション再利用
    ssl_session_timeout 10m;

    # gzip 圧縮
    gzip on;
    gzip_types application/json text/plain;

    location /api/ {
        proxy_pass         http://api_servers;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection "upgrade";
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;

        # レート制限
        limit_req zone=api_limit burst=20 nodelay;
    }

    # 静的ファイルはキャッシュ
    location ~* \.(js|css|png|jpg|gif|ico)$ {
        root /var/www/static;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
}

HAProxy 設定例

# /etc/haproxy/haproxy.cfg
global
    log /dev/log local0
    maxconn 50000
    nbproc 2

defaults
    mode http
    timeout connect 5s
    timeout client  30s
    timeout server  30s
    option httplog
    option forwardfor  # X-Forwarded-For 追加
    option http-server-close

frontend https_in
    bind *:443 ssl crt /etc/ssl/api.pem  # SSL終端
    default_backend app_servers
    acl is_api path_beg /api/
    use_backend api_servers if is_api  # コンテンツルーティング

backend app_servers
    balance leastconn
    option httpchk GET /health HTTP/1.1\r\nHost:\ localhost
    http-check expect status 200
    server app1 10.0.1.10:8080 check inter 5s rise 2 fall 3
    server app2 10.0.1.11:8080 check inter 5s rise 2 fall 3
    server app3 10.0.1.12:8080 check inter 5s rise 2 fall 3 backup

API Gateway の主要機能

認証・認可

JWTトークン検証、OAuth 2.0フロー、APIキー管理。バックエンドサービスから認証ロジックを分離できる。

レート制限

APIキー別・ユーザー別・エンドポイント別に細かくスロットリング設定。DDoS対策と課金プランの実装に利用。

リクエスト変換

外部APIのレスポンスを内部フォーマットに変換。プロトコル変換（REST→gRPC等）も担える。

サーキットブレーカー

下流サービスの障害時に自動でフォールバックレスポンスを返し、障害の連鎖（カスケード障害）を防ぐ。

BFF (Backend for Frontend) パターン

モバイルアプリ・Webアプリ・サードパーティ向けに、それぞれ専用のAPI Gatewayを用意するパターン。各クライアントの要求に最適化したAPIを提供できる。

BFFパターン：クライアントの種類ごとに最適化されたAPIゲートウェイを用意する。

SECTION 08グローバルロードバランシング

データセンターやリージョンをまたいで世界規模のトラフィック分散を行う技術群。

GeoDNS

クライアントのDNSリゾルバーのIPアドレスを基に、地理的に近いデータセンターのIPアドレスを返す。

  • DNSレベルで振り分けるため実装がシンプル
  • TTLによってフェイルオーバー速度に限界（通常30〜300秒）
  • 例：AWS Route 53の地理的ルーティング

Anycast ルーティング

複数のデータセンターに同一のIPアドレスをアナウンス。BGPルーティングにより最も近いノードへ自動転送。

  • クライアントからは単一IPに見える
  • BGPのコンバージェンス（数秒〜数分）でフェイルオーバー
  • 例：Cloudflare・AWS Global Accelerator

GSLB (Global Server Load Balancing)

ヘルスチェック・レイテンシ・重みなどを組み合わせてグローバルにトラフィックを制御する高度なDNSベースLB。

  • リアルタイムのヘルスチェックでフェイルオーバー
  • サーバー負荷に応じた動的な重み調整
  • 例：F5 BIG-IP DNS、NS1

レイテンシベース vs 地理ベースルーティング

方式ルーティング根拠メリットデメリット
地理ベースクライアントの地理的位置シンプル・予測可能地理的に近くても遅い場合がある
レイテンシベース実際の往復レイテンシ測定値最も低レイテンシなリージョンへ測定コスト・複雑
重みベース設定された重み比率カナリアリリース・段階移行に最適手動管理が必要
フェイルオーバーヘルスチェック結果プライマリ障害時に自動切替DNSのTLLキャッシュによる遅延

リージョン間フェイルオーバーのフロー

1
プライマリリージョン障害検知（ヘルスチェック失敗）
2
連続失敗N回でdownと判定
3
DNSの応答をセカンダリリージョンのIPへ更新
4
TTL経過後、クライアントが新IPを取得
5
セカンダリへのトラフィック移行完了
⚠️ DNS TTL の落とし穴: フェイルオーバー時間はDNSのTTL（Time To Live）に依存する。TTLが300秒（5分）なら、フェイルオーバー完了まで最大5分かかる。緊急時に備えてTTLを短く設定しておく（例：30秒）のが推奨だが、DNSサーバーへの問い合わせ頻度が上がるトレードオフがある。

SECTION 09コネクション最適化

コネクション確立のオーバーヘッドを削減し、スループットとレイテンシを改善する。

Keep-Alive と Connection Pooling

HTTP Keep-Alive

一度確立したTCPコネクションを複数のHTTPリクエストで再利用する。

  • TCPハンドシェイク（3-way handshake）のオーバーヘッドを削減
  • HTTP/1.1ではデフォルトで有効
  • HTTP/2は多重化でさらに効率化
# Nginx: バックエンドへのkeep-alive
upstream backend {
  keepalive 32;  # 維持するコネクション数
}
location / {
  proxy_http_version 1.1;
  proxy_set_header Connection "";
}

Connection Pooling

DBやマイクロサービスへのコネクションを事前に確立しておき、リクエストごとに再利用する。

  • コネクション確立コスト（TCP+TLS）を分散
  • 最大コネクション数でバックエンドを保護
  • プール枯渇時はキュー待機 or エラーを返す
# Node.js: pg connection pool
const pool = new Pool({
  max: 20,     // 最大コネクション数
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000
});

TLS セッション再利用（Session Resumption）

TLSハンドシェイクは通常1〜2回のRTTを要する。セッション再利用によりこのオーバーヘッドを大幅に削減できる。

Session ID（TLS 1.2以前）

サーバーがセッションIDを発行し、クライアントが次回のハンドシェイク時に提示する。サーバーがセッション情報をキャッシュする必要がある。複数サーバー環境では共有キャッシュが必要。

Session Tickets（TLS 1.2/1.3）

セッション情報をサーバーではなく暗号化されたticketとしてクライアントに返す。次回接続時にclientがticketを送信し、サーバーが復号して再利用。サーバー側でのセッションストレージが不要。

TLS 1.3 の 0-RTT

TLS 1.3では0-RTT（Zero Round Trip Time Resumption）が導入。初回接続後の再接続で、最初のデータをハンドシェイクと同時に送信できる。注意: リプレイ攻撃のリスクがあるため、べき等なGETリクエストのみに使用するのが推奨。

# Nginx TLS セッション再利用設定
http {
    ssl_session_cache   shared:SSL:50m;  # 全workerで共有するセッションキャッシュ
    ssl_session_timeout 1d;              # セッション有効期限
    ssl_session_tickets on;               # Session Tickets 有効化

    # TLS 1.3 0-RTT（注意して使用）
    ssl_protocols TLSv1.3;
    ssl_early_data on;

    # 0-RTTリプレイ攻撃対策ヘッダー
    add_header Early-Data $ssl_early_data;
}

バックエンドへの最大接続数調整

計算例: バックエンドサーバー1台が安定処理できる最大コネクション数を算出する。

最大コネクション数 = (CPU コア数) × (平均リクエスト処理時間[ms] / 1000ms) × 安全係数(0.7~0.8)

例：8コア、平均処理時間50ms → 8 × (50/1000) × 0.75 ≈ 300接続 
# Nginx: バックエンドへの接続数制限
upstream backend {
    server 10.0.1.10:8080;

    # コネクション数上限（Nginx Plus）
    max_conns=300;      # 最大アクティブ接続数
    max_fails=3;        # この回数失敗でdown
    fail_timeout=30s;   # down状態維持時間

    keepalive 64;        # keep-aliveコネクション数
    keepalive_requests 10000;  # 1コネクションで処理するリクエスト上限
    keepalive_timeout 75s;     # keep-aliveタイムアウト
}

INTERVIEW TIPS面接でよく出る設計問題

システムデザイン面接で負荷分散に関して問われるパターンと、回答のポイント。

Q1

「Twitterのようなシステムを設計せよ」

最初に「まず単一サーバーでは何億ユーザーを捌けないため、水平スケーリングとロードバランサーが必要」と言及する。L7 LBでのルーティング（タイムライン /、メディア /cdn）を説明する。

Q2

「LBアルゴリズムを比較せよ」

Round Robinの限界（長時間コネクション問題）→Least Connectionsの改善点→IP Hashのスティッキーセッション→Power of Twoの大規模向け利点、という流れで説明すると構造的でよい。

Q3

「99.99%の可用性を実現するには」

LB自体のHAを言及する（Active-Active or Active-Standby）。ヘルスチェックの設定、自動フェイルオーバー、複数リージョンへのGSLB展開まで説明できるとベスト。

Q4

「セッション管理の問題点を挙げよ」

サーバーサイドセッション→スティッキーセッションの必要性→その問題点（偏り・スケールアウト困難・障害時消失）→Redis分散セッションストアによる解決、という論理の流れを明確に。

Q5

「L4とL7はどう選ぶか」

HTTPコンテンツルーティングが必要→L7。純粋なTCP/UDPや超高スループット→L4。ゲームサーバー・DNS・VoIPはL4。Webアプリ・APIはL7。両方使うケース（L4でL7をLB配下に置く）も言及できるとよい。

Q6

「API GatewayとLBの違いは」

LBはネットワークレベルのトラフィック分散が主。API GatewayはBizロジック（認証・レート制限・変換・サーキットブレーカー）を担う。両者は補完関係にあり、実際は共存することが多い。

📝 まとめ：負荷分散の設計原則

1. ステートレス化が最優先 — セッションをRedisに逃がし、どのサーバーへ振っても正しく動く設計にする。
2. ヘルスチェックは必須 — 死んだサーバーへの転送を自動回避。Active + Passiveの組み合わせが堅牢。
3. アルゴリズムは目的に合わせる — 短時間リクエスト多数→Round Robin、長時間コネクション→Least Connections。
4. 単一障害点を排除 — LB自体も冗長化。プライマリ/セカンダリ構成かActive-Active構成で。
5. 観測可能性（Observability） — LBのアクセスログ・接続数メトリクス・エラーレートを常時監視。

10深掘り: Nginxの内部アーキテクチャ

NginxがなぜC10K問題を解決できたか、内部構造から理解する。設定の意味を原理から把握することで最適なチューニングができる。

イベント駆動型 vs スレッド型 — C10K問題の解決

Apache Worker MPM（スレッド型）

1リクエスト = 1スレッド（またはプロセス）で処理。スレッド生成コストと、1万コネクションでの1万スレッドによるコンテキストスイッチがボトルネック。

# Apache httpd.conf
StartServers     4
MinSpareThreads  25
MaxSpareThreads  75
ThreadsPerChild  25
MaxRequestWorkers 150  # 最大150スレッド
# 1万コネクション → 1万スレッド → OOM

Nginx（イベント駆動型）

シングルスレッドのイベントループ（epoll/kqueue）で全コネクションを非同期に管理。ノンブロッキングI/Oにより1ワーカーが数万コネクションを同時処理できる。

# nginx.conf
# CPU数に合わせてワーカーを設定
worker_processes auto;  # CPUコア数
events {
  use epoll;            # Linuxの高効率I/Oイベント
  worker_connections 65535;
  multi_accept on;      # 一度に複数コネクション受け入れ
}
# 4コア × 65535 = 約26万コネクション同時処理可能
C10K問題: 2000年代初頭、1台のサーバーで同時1万コネクションを扱うことが技術的に困難だった問題。Nginxはイベント駆動により解決した。現在は「C10M問題（1000万コネクション）」が次の課題になっている。

Upstream Keepalive — バックエンドコネクション再利用

バックエンドサーバーとのTCPコネクションをプールして再利用する。毎リクエストでのTCPハンドシェイクコストを削減。

# upstream keepalive の設定
upstream app_backend {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;

    # アイドルコネクションプールの最大数
    keepalive 32;
    keepalive_requests 1000;   # コネクションあたりの最大リクエスト数
    keepalive_timeout  60s;    # アイドル接続のタイムアウト
}

server {
    location / {
        proxy_pass http://app_backend;
        proxy_http_version 1.1;           # HTTP/1.1 必須
        proxy_set_header Connection "";   # Connection: close を上書き
        proxy_set_header Host $host;
    }
}
効果の目安: keepaliveなしでは1万QPS時に毎秒1万回のTCPハンドシェイク（3-wayハンドシェイクで約0.1ms × 1万 = 約1秒分の遅延）。keepalive 32設定でコネクション確立コストを大幅に削減。

レート制限 — limit_req / limit_conn

# レート制限ゾーン定義（httpコンテキスト）
# key: クライアントIP, ゾーンメモリ: 10MB, レート: 10req/秒
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

# 同時コネクション制限
limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

server {
    location /api/ {
        # バーストを20まで許可（burst=20 nodelay: 待機させずに即処理）
        limit_req zone=api_limit burst=20 nodelay;

        # 1IPからの同時コネクション数を10に制限
        limit_conn conn_limit 10;

        # 制限超過時のレスポンスコード
        limit_req_status 429;
        limit_conn_status 503;

        proxy_pass http://app_backend;
    }

    # ログイン試行の厳格なレート制限
    location /auth/login {
        limit_req zone=api_limit burst=5;  # バースト少なめ
        proxy_pass http://auth_backend;
    }
}
burst の動作: rate=10r/sでburst=20の場合、突発的な20リクエストは即処理し（nodelay指定時）、超過分は429を返す。nodedelayなしの場合は超過分をキューで待機させる（最大burst数まで）。

OpenResty（lua-nginx-module）による動的ルーティング

OpenRestyはNginxにLuaランタイムを組み込んだプラットフォーム。リクエスト処理中にビジネスロジックを実行できる。

# Nginxの各フェーズでLuaを実行できる
# access_by_lua: 認証チェック
# rewrite_by_lua: URLリライト
# content_by_lua: レスポンス生成

server {
    location /api/ {
        # A/Bテスト: User-Agentでルーティング
        access_by_lua_block {
            local user_agent = ngx.var.http_user_agent
            if string.find(user_agent, "iPhone") then
                ngx.var.upstream = "mobile_backend"
            else
                ngx.var.upstream = "web_backend"
            end
        }
        proxy_pass http://$upstream;
    }

    location /auth/ {
        # JWT検証をNginx側で実施（バックエンドに届く前に）
        access_by_lua_block {
            local jwt_token = ngx.req.get_headers()["Authorization"]
            if not jwt_token then
                ngx.status = 401
                ngx.say('{"error":"unauthorized"}')
                ngx.exit(401)
            end
            -- JWT検証ロジック（redis-lua等で実装）
        }
        proxy_pass http://app_backend;
    }
}

11深掘り: 高可用性ロードバランサーの設計

ロードバランサー自体が単一障害点になることを防ぐ。VRRP・BGP・Anycastによる冗長化戦略を理解する。

VRRP / Keepalived — Active-Standby構成

2台のNginxサーバーでVirtual IP (VIP)を共有し、プライマリが落ちた場合にセカンダリが自動でVIPを引き継ぐ。

# keepalived.conf (プライマリNginx側)
vrrp_script check_nginx {
    script "/bin/check_nginx.sh"   # Nginxが動いているか確認
    interval 2                       # 2秒ごとにチェック
    weight -20                       # 失敗時にpriorityを20下げる
}

vrrp_instance VI_1 {
    state MASTER                     # プライマリ
    interface eth0
    virtual_router_id 51
    priority 100                     # セカンダリは50に設定
    advert_int 1                     # VRRPアドバタイズ間隔
    authentication {
        auth_type PASS
        auth_pass secret123
    }
    virtual_ipaddress {
        192.168.1.100/24             # VIP（クライアントはここに接続）
    }
    track_script {
        check_nginx
    }
}

---
# フェイルオーバーの流れ:
# 1. プライマリのNginx障害
# 2. check_nginx.sh が失敗 → priority 100-20=80
# 3. セカンダリ(priority=50)より低くなったため
#    セカンダリがMASTERに昇格
# 4. VIP 192.168.1.100 がセカンダリに移動
# 5. クライアントは約1〜2秒のダウン後に自動回復

Active-Standby の特徴

  • 実装が単純でデバッグしやすい
  • スタンバイ機はほぼアイドル（コスト無駄）
  • フェイルオーバー: 通常1〜2秒
  • 小〜中規模サービスに適切

Active-Active の特徴

  • 両LBが同時にトラフィックを処理
  • DNSラウンドロビンやBGP ECMPで分散
  • 一方が落ちても残りが継続
  • 大規模サービスでリソースを有効活用

BGP ECMP と Anycast

BGP ECMP (Equal-Cost Multi-Path): 複数の等コストパスが存在する場合、トラフィックを複数のロードバランサーに分散させるルーティング手法。ネットワーク層でのロードバランシング。

Anycast: 同じIPアドレスを複数のデータセンターでアドバタイズし、ユーザーがネットワーク的に最も近いデータセンターに自動的にルーティングされる手法。CloudflareやFastlyのCDNで使われる。

# Anycastの概念（BGP設定の例）
# 東京DC・大阪DC・シンガポールDCが同じIPを持つ
# Anycast IP: 203.0.113.1

# 東京DCのBGP設定（FRRouting）
router bgp 65001
  bgp router-id 10.0.1.1
  neighbor 203.0.113.254 remote-as 65000  # 上位ISP
  address-family ipv4 unicast
    network 203.0.113.0/24               # Anycast prefixをアドバタイズ

# 大阪DCも同じ 203.0.113.0/24 をアドバタイズ
# → 日本のユーザーは近い方のDCにルーティングされる
# → DCが落ちるとBGPが引っ込み、別DCへ自動切替

DNS TTLとフェイルオーバーのトレードオフ

DNS TTLフェイルオーバー速度DNSサーバー負荷キャッシュの恩恵推奨場面
5秒極速（〜5秒）極高ほぼなし事前フェイルオーバー切替直前
60秒速（〜1分）高少ない高可用性が必要なサービス
300秒中（〜5分）普通中一般的なWebサービス
3600秒遅（〜1時間）低大きい変更の少ない静的サイト
GTM（Global Traffic Manager）の活用: Route53・Cloudflare・NS1などのGTMサービスは、ヘルスチェックと連動してDNSレスポンスを動的に変更する。フェイルオーバー時にTTLを無視して即座に新しいIPを返せる（Negative TTLキャッシュへの対処は別途必要）。

12実践演習

100万QPS設計・Blue-Greenデプロイ・面接形式の設計問題で実力を鍛える。

演習1: 100万QPS対応のNginx構成を設計する

要件と計算

# 目標: 1,000,000 QPS
# 1サーバー（32コア, 64GB RAM）での最大スループット想定

# worker設定の計算
worker_processes 32;          # CPUコア数と同じ
worker_connections 65535;     # 1ワーカーあたりの同時接続数
# 理論最大コネクション: 32 × 65535 ≈ 200万

# 1サーバーのスループット上限
# 各リクエストの処理時間: 1ms と仮定
# 1ワーカー×1スレッドで: 1000 req/sec
# 32ワーカー: 32,000 req/sec per server
# 100万QPSには: 1,000,000 / 32,000 ≈ 32台のNginxが必要
# BGP ECMPまたはDNSラウンドロビンで32台に分散

完全なNginx設定

# /etc/nginx/nginx.conf (本番用 100万QPS対応)

user nginx;
worker_processes auto;          # CPUコア数に自動設定
worker_rlimit_nofile 200000;    # ファイルディスクリプタ上限
error_log /var/log/nginx/error.log warn;

events {
    use epoll;
    worker_connections 65535;
    multi_accept on;            # 一度に複数接続受け入れ
}

http {
    # バッファサイズ最適化
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    keepalive_requests 10000;   # KAコネクション再利用回数

    # バッファサイズ（大きくするとメモリ消費増加に注意）
    proxy_buffer_size 128k;
    proxy_buffers 4 256k;
    proxy_busy_buffers_size 256k;
    client_body_buffer_size 128k;
    client_max_body_size 100m;

    # レート制限ゾーン
    limit_req_zone $binary_remote_addr zone=api:100m rate=100r/s;

    upstream app_backend {
        least_conn;             # Least Connections アルゴリズム
        server 10.0.1.10:8080 weight=1 max_fails=3 fail_timeout=30s;
        server 10.0.1.11:8080 weight=1 max_fails=3 fail_timeout=30s;
        server 10.0.1.12:8080 weight=1 max_fails=3 fail_timeout=30s;
        # バックエンドとのKeepalive（重要）
        keepalive 32;
        keepalive_requests 1000;
        keepalive_timeout 60s;
    }

    server {
        listen 443 ssl http2 reuseport;  # reuseport: SO_REUSEPORT
        server_name api.example.com;

        # TLS 最適化
        ssl_certificate /etc/ssl/api.crt;
        ssl_certificate_key /etc/ssl/api.key;
        ssl_protocols TLSv1.2 TLSv1.3;
        ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
        ssl_session_cache shared:SSL:100m;  # セッション再利用（TLS高速化）
        ssl_session_timeout 1d;

        location /api/ {
            proxy_pass http://app_backend;
            proxy_http_version 1.1;
            proxy_set_header Connection "";
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_connect_timeout 3s;
            proxy_read_timeout 30s;
            proxy_next_upstream error timeout http_500;
            limit_req zone=api burst=200 nodelay;
        }
    }
}

演習2: Blue-Greenデプロイをロードバランサーで実現する

ゼロダウンタイムデプロイの実装。Blue（現行）とGreen（新バージョン）の2環境を用意し、LBでトラフィックを切り替える。

# 段階1: Blue-Green upstream定義
upstream blue {
    server 10.0.1.10:8080 weight=1;
    server 10.0.1.11:8080 weight=1;
    keepalive 32;
}

upstream green {
    server 10.0.2.10:8080 weight=1;  # 新バージョン
    server 10.0.2.11:8080 weight=1;
    keepalive 32;
}

# nginx.conf の変数でターゲットを切り替え
server {
    location / {
        # map: Cookieやヘッダーでroutingを制御
        set $target blue;
        if ($http_x_canary = "true") {
            set $target green;
        }
        proxy_pass http://$target;
    }
}

---
# 段階的トラフィック移行スクリプト
#!/bin/bash
# split_traffic.sh: Blue-Greenの段階的切り替え
NGINX_UPSTREAM=/etc/nginx/conf.d/upstream.conf

# Phase 1: 5%をGreenに
cat > $NGINX_UPSTREAM << EOF
upstream app {
    server blue_backend weight=19;
    server green_backend weight=1;   # 5%
}
EOF
nginx -s reload

sleep 300  # 5分間モニタリング

# Phase 2: 25%をGreenに
cat > $NGINX_UPSTREAM << EOF
upstream app {
    server blue_backend weight=3;
    server green_backend weight=1;   # 25%
}
EOF
nginx -s reload

sleep 600  # 10分間モニタリング

# Phase 3: 100%Greenに切り替え
cat > $NGINX_UPSTREAM << EOF
upstream app {
    server green_backend weight=1;   # 100%
}
EOF
nginx -s reload
ヘルスチェックと組み合わせた安全な切り替え:
  • 各フェーズでエラーレート・レイテンシ・CPU使用率を監視
  • 異常を検知したら即座にBlueへロールバック（nginx -s reloadは設定変更を無停止で反映）
  • Nginx PlusのDynamic Upstream APIを使うと設定ファイル変更なしでリロードなしの切り替えが可能

演習3: スティッキーセッションなしのスケールアウト設計（面接問題）

問題文: スティッキーセッションを使わずに、ユーザーセッションを維持しながら複数サーバーにスケールアウトするにはどうするか？

解答の構成（面接での模範回答）

方法1: 分散セッションストア（推奨）

セッション情報をRedisに保存。全アプリサーバーが同じRedisを参照。

# express-session + Redis
const session = require('express-session');
const RedisStore = require('connect-redis')(session);

app.use(session({
  store: new RedisStore({
    client: redisClient,
    ttl: 1800   // 30分
  }),
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false
}));
// どのサーバーへ届いてもRedisからセッション取得

方法2: JWTによるステートレス認証

セッション情報をJWTトークンに含めてクライアント側に保存。サーバー側は検証のみ。

// JWTの発行
const token = jwt.sign({
  userId: user.id,
  role: user.role,
  exp: Math.floor(Date.now() / 1000) + 3600
}, process.env.JWT_SECRET);

// 検証（どのサーバーでも同じ秘密鍵で検証）
const decoded = jwt.verify(token, process.env.JWT_SECRET);

// 利点: サーバー側状態なし・水平スケール容易
// 欠点: トークン失効が困難（ブロックリストが必要）

方法別トレードオフ比較

方法実装難度スケール性セキュリティログアウト即時性
スティッキーセッション低低（偏り）中即時
Redis分散セッション中高高即時
JWT（ステートレス）中最高中困難（Redisブロックリスト要）
面接での加点ポイント:
  • 「セッションハイジャック対策として、セッションIDローテーション（ログイン後にIDを再生成）とSecure/HttpOnly/SameSite Cookieフラグを必ず設定する」
  • 「大規模サービスでは、Redisのセッションストアも冗長化（Redis Cluster）が必要。Redisが単一障害点にならないよう注意」
  • 「JWTのaccess tokenは短命（15分）にし、refresh tokenをRedisで管理することでログアウト即時性とスケーラビリティを両立できる」
