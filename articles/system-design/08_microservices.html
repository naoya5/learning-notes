1

モノリス vs マイクロサービス vs SOA

アーキテクチャスタイルを選択するとき、最初に理解すべきは「なぜその選択をするか」というコンテキストだ。モノリス・SOA・マイクロサービスはそれぞれ異なるトレードオフを持つ。

モノリシックアーキテクチャ

すべての機能が単一のデプロイ単位にまとまったアーキテクチャ。小チームの初期フェーズでは最も生産性が高い。

モノリスの利点

  • 開発・デバッグが簡単
  • ネットワーク遅延なし（関数呼び出し）
  • トランザクション管理が容易
  • IDE のサポートが充実
  • チームの認知負荷が低い
  • インフラが単純（1プロセス）

モノリスの限界

  • コードベースが大きくなると把握困難
  • 1機能の変更でも全体を再デプロイ
  • 特定モジュールだけスケールできない
  • 技術スタックの刷新が困難
  • チームが大きくなると変更衝突が増加
  • 障害が全体に波及しやすい

SOA（Service-Oriented Architecture）との違い

SOA は 2000 年代に流行した「サービスを ESB（Enterprise Service Bus）で統合する」スタイル。マイクロサービスとよく混同されるが本質的に異なる。

観点SOAマイクロサービス
通信基盤ESB（集中型・重厚）軽量プロトコル（REST/gRPC/MQ）
サービス粒度大きい（エンタープライズ機能単位）小さい（ビジネス能力単位）
データ管理共有データベースが一般的サービスごとに独立したDB
デプロイ一括またはスロット切替独立した CI/CD パイプライン
チーム中央組織が統制自律した小チームが所有
標準化WSDL・SOAP・WS-*実用的な慣習ベース

マイクロサービスの基本原則

🔀 独立デプロイ

各サービスは他のサービスに影響を与えずデプロイできる。これが最大の価値。API の後方互換性を維持することで実現する。

📦 ビジネスドメイン境界

サービス境界はビジネスの能力（capability）に沿って引く。技術層（DB層・UI層）ではなくドメイン（注文・在庫・決済）で分割する。

💾 分散データ管理

各サービスが自分のデータを所有する。他サービスのDBに直接アクセスしない。データの整合性は最終的整合性（Eventual Consistency）で担保する。

🔧 障害に備えた設計

ネットワーク越しの呼び出しは必ず失敗しうる前提で設計する。サーキットブレーカー・タイムアウト・リトライを標準実装する。

🤖 自動化インフラ

多数のサービスを手動管理するのは不可能。CI/CD・コンテナ・オーケストレーション（K8s）・監視の自動化が前提条件。

👥 チームの自律性

各サービスチームが技術選択・デプロイ判断・運用責任を持つ。Conway の法則を意識した組織設計が重要。

三種アーキテクチャの比較表

観点モノリスSOAマイクロサービス
開発速度（初期）速い中中（基盤整備必要）
開発速度（成熟後）低下中高い
運用複雑度低い高い（ESB）高い（多数サービス）
障害分離低い中高い
スケーラビリティ全体のみサービス単位サービス単位
データ管理ACID容易共有DB多い分散トランザクション複雑
技術多様性制限あり標準規格縛り自由（Polyglot）
チームスケール〜20人中規模大規模対応

いつモノリスを選び、いつマイクロサービスを選ぶか

マイクロサービスは「常に正解」ではない。適切なアーキテクチャはチームサイズ・ドメインの複雑さ・組織の成熟度によって変わる。

Amazon の「Two-Pizza Team」ルール

Two-Pizza Team： ピザ2枚で全員お腹いっぱいになれる人数（6〜8人）が1チームの上限。それを超えたらチームを分割し、各チームが独立したサービスを担当する。Amazon はこの哲学を組織とシステム設計の両方に適用した。

モノリスを選ぶべきケース

  • チームが10人未満
  • ドメインが未成熟（境界が不明確）
  • プロダクト初期・PMFを探している段階
  • インフラ運用のリソースが少ない
  • 開発スピードが最優先
  • バックオフィス・社内ツール系

マイクロサービスを選ぶべきケース

  • 複数チーム（各6〜8人）が並行開発
  • ドメインが明確に分離可能
  • スケール要件がサービスごとに大きく異なる
  • 高い開発頻度（1日複数回デプロイ）
  • 技術スタックの多様性が必要
  • 障害分離・高可用性が必須

「Majestic Monolith」という選択肢

DHH（Rails 作者）が提唱した概念。きちんと設計されたモジュール化モノリスは、多くのスタートアップ・中規模サービスに最適解である。

条件： 内部モジュールが明確なAPI境界を持ち、循環依存がなく、将来的にサービス分割が可能な設計になっていること。Shopify・Stack Overflow はこのアプローチで大規模成長を遂げた。

アンチパターン：Distributed Monolith（分散モノリス）

最悪の組み合わせ： マイクロサービスに分割したのに、サービス間が密結合のまま。独立デプロイができず（A をデプロイすると B が壊れる）、分散システムの複雑さだけを抱えた状態。サービス間で DB を共有したり、同期呼び出しチェーンが長い場合に発生する。

判断フローチャート

ドメイン駆動設計 (DDD) の基礎

マイクロサービスの境界設計に最も有効なアプローチが DDD（Domain-Driven Design）。Eric Evans が 2003 年に提唱した設計哲学で、複雑なドメインをソフトウェアに正確に反映させる手法を体系化している。

Bounded Context（境界づけられたコンテキスト）

DDDの中核概念。同じ言葉でも文脈（コンテキスト）によって意味が異なる。たとえば「顧客（Customer）」はマーケティング部門では「リード」、配送部門では「受取人住所」、請求部門では「支払い情報」を意味する。

Bounded Context はその「言葉の意味が一意に決まる範囲」。マイクロサービスの境界は原則として Bounded Context と一致させる。

主要な DDD 戦術パターン

Entity（エンティティ）

一意のIDで識別されるオブジェクト。属性が変化しても同じエンティティとして扱われる。例：注文（Order ID で識別）、ユーザー（User ID で識別）。

Value Object（値オブジェクト）

IDを持たず、属性の値で等価性を判断するオブジェクト。不変（Immutable）。例：金額（¥1,000）、住所、メールアドレス。

Aggregate（集約）

整合性を保つ必要があるエンティティと値オブジェクトのクラスター。Aggregate Root を通じてのみ外部からアクセス可能。例：Order Aggregate（Order + OrderItems + ShippingAddress）。

Domain Event（ドメインイベント）

ドメイン内で起きた過去の出来事。「OrderPlaced」「PaymentProcessed」など。イベント駆動アーキテクチャとの親和性が高く、Bounded Context 間の通信に使う。

Context Map（コンテキストマップ）

複数の Bounded Context の関係を可視化したもの。上流（Upstream）が下流（Downstream）に影響を与える方向を矢印で示す。

関係パターン説明例
Upstream / Downstream変更の影響が流れる方向。上流の変更が下流に影響する商品サービス→注文サービス
Shared Kernel2チームが共有するドメインモデルの一部共通のユーザーID型定義
Customer / Supplier下流（顧客）が上流（供給者）に要件を交渉できるSLAの合意
Anti-Corruption Layer外部システムの概念が自ドメインに侵入しないよう変換レイヤーを設ける外部決済APIの吸収
Conformist下流が上流のモデルをそのまま採用する外部SaaSに合わせる

Ubiquitous Language（ユビキタス言語）

エンジニアとビジネス側が同じ言語を使う： コード・会議・ドキュメントで同じ用語を使う。「User」でなく「Customer」「Seller」「Rider」と区別する。用語の曖昧さがバグの原因になる。

実践例：ECサイトのドメイン分割

サービス間通信

マイクロサービス間の通信は「同期」と「非同期」の2種類に大別される。それぞれの特性を理解して適切に使い分けることが重要。

同期通信：REST vs gRPC

観点REST（HTTP/JSON）gRPC（HTTP/2 + Protobuf）
プロトコルHTTP/1.1 または HTTP/2HTTP/2 必須
シリアライズJSON（人間可読・大きい）Protocol Buffers（バイナリ・小さい）
パフォーマンス普通高速（5〜10倍）
ストリーミング制限あり（WebSocket別途）双方向ストリーミング対応
型安全性弱い（JSONスキーマ別途）強い（.proto定義）
ブラウザ対応ネイティブ対応grpc-web が必要
デバッグ容易性curl で確認できる専用ツール必要
採用場面外部API・ブラウザ対応・読みやすさ重視内部サービス間・低レイテンシ・高スループット

gRPC の .proto 定義例

// order_service.proto
syntax = "proto3";
package order;

service OrderService {
  rpc CreateOrder(CreateOrderRequest) returns (OrderResponse);
  rpc GetOrder(GetOrderRequest) returns (OrderResponse);
  // サーバーストリーミング：注文ステータスをリアルタイム通知
  rpc WatchOrderStatus(GetOrderRequest) returns (stream OrderStatusUpdate);
}

message CreateOrderRequest {
  string customer_id = 1;
  repeated OrderItem items = 2;
  string shipping_address = 3;
}

message OrderItem {
  string product_id = 1;
  int32 quantity = 2;
  int64 unit_price_cents = 3;
}

message OrderResponse {
  string order_id = 1;
  OrderStatus status = 2;
  int64 total_amount_cents = 3;
  google.protobuf.Timestamp created_at = 4;
}

enum OrderStatus {
  PENDING = 0;
  CONFIRMED = 1;
  SHIPPED = 2;
  DELIVERED = 3;
  CANCELLED = 4;
}

同期 vs 非同期 通信の使い分け

判断基準： 即時レスポンスが必要（認証・在庫確認）→ 同期。処理に時間がかかる・複数サービスへのファンアウト（注文後の通知・在庫引当）→ 非同期。サービス間の依存を切りたい場合も非同期を選ぶ。

データ管理パターン

マイクロサービスで最も難しいのがデータ管理。「各サービスが自分のDBを持つ」原則を守りながら、複数サービスにまたがる整合性をどう担保するかが設計の核心。

Database per Service パターン

各サービスが専用のデータストアを持つ。他サービスのDBへの直接アクセス（JOIN・外部キー）は禁止。

メリット

  • サービスが独立してデプロイ・スケール可能
  • 最適な DB 技術を選択できる（Polyglot）
  • 他サービスの障害が波及しない
  • スキーマ変更が他サービスに影響しない

デメリット

  • 複数サービスにまたがるクエリが困難
  • 分散トランザクションが必要になる
  • データの冗長性が発生する
  • 最終的整合性の受け入れが必要

Saga パターン（分散トランザクション）

複数サービスにまたがるトランザクションを、各サービスのローカルトランザクションの連鎖で実現。失敗時は補償トランザクション（Compensating Transaction）でロールバックする。

Choreography（コレオグラフィー）

中央のオーケストレーターを持たず、各サービスがイベントを受け取って次のアクションを実行する。疎結合だが全体のフローが把握しにくい。

Orchestration（オーケストレーション）

中央のオーケストレーター（Saga Coordinator）が各サービスを呼び出して制御する。フローが明確だが中央サービスへの依存が生まれる。

Saga Choreography の流れ（注文処理）

CQRS（Command Query Responsibility Segregation）

コマンド（書き込み）とクエリ（読み込み）の責務を分離するパターン。

  • Command Side： 書き込み最適化。正規化されたデータモデル。イベントを発行。
  • Query Side： 読み込み最適化。非正規化されたリードモデル。高速クエリ用に別途ストア（Elasticsearch・Redis）を持つ。

Event Sourcing（イベントを正とし、状態を都度再構築）と組み合わせることが多い。

API Composition（複数サービスのデータ集約）

// API Gateway での Composition 例（Node.js）
async function getOrderDetail(orderId) {
  // 並行リクエストで複数サービスからデータ取得
  const [order, payment, shipment] = await Promise.all([
    orderService.getOrder(orderId),
    paymentService.getPaymentByOrder(orderId),
    shippingService.getShipmentByOrder(orderId),
  ]);

  // 商品詳細を取得（order.itemsの数だけ）
  const productIds = order.items.map(i => i.productId);
  const products = await productService.getProductsByIds(productIds);
  const productMap = new Map(products.map(p => [p.id, p]));

  return {
    order,
    items: order.items.map(item => ({
      ...item,
      product: productMap.get(item.productId),
    })),
    payment,
    shipment,
  };
}

ストラングラーフィグパターン（Strangler Fig）

既存のモノリスを段階的にマイクロサービスへ移行するための戦略。マーティン・ファウラーが命名。モノリスをいきなり全部作り替える「Big Bang 移行」は失敗リスクが極めて高い。

名前の由来： 熱帯雨林のストラングラーフィグ（絞め殺しの木）は、宿主の木に巻き付きながら成長し、最終的に宿主の木に取って代わる。ソフトウェア移行も同様に、新システムが旧システムを徐々に置き換えていく。

移行フェーズ

Phase 1：ファサード設置
  • APIゲートウェイ／リバースプロキシを前段に置く
  • 全リクエストを既存モノリスへルーティング
  • モノリスには一切手を加えない
Phase 2：機能の抽出
  • 移行しやすい機能から新サービスを作成
  • プロキシが新サービスへルーティング
  • モノリスの該当機能は無効化
  • データ同期を一時的に行う
Phase 3：繰り返し
  • 機能ごとに Phase 2 を繰り返す
  • モノリスがどんどん縮小する
  • 各サービスが独立DBを持つ
  • リスクを小さく保ちながら進める
Phase 4：完了
  • モノリスが廃止または最小化
  • 全機能がマイクロサービスへ移行
  • APIゲートウェイが唯一のエントリポイント

SVG図：移行フェーズ

アンチパターン — Big Bang 移行： モノリスをすべて停止して新マイクロサービスに一気に切り替える方法。開発期間中に本番が止まる、バグが大量発生、ロールバックが困難、という三重苦を招く。Amazon・Netflix はいずれも段階的移行でマイクロサービスへ移行した。

サービスディスカバリ

コンテナ・オートスケール環境ではサービスのIPアドレスが動的に変わる。ハードコードされたエンドポイントは使えない。サービスディスカバリはこの問題を解決する仕組み。

Client-side Discovery

クライアントがService Registryに問い合わせ、利用可能なインスタンスを取得して自分でロードバランシングする。

例： Netflix Eureka + Ribbon

✓ シンプル・クライアントが柔軟なLBポリシーを持てる

✗ 各クライアントが Discovery ロジックを実装する必要あり

Server-side Discovery

ロードバランサーやAPIゲートウェイがService Registryに問い合わせ、適切なインスタンスへルーティングする。クライアントは意識しない。

例： AWS ALB + ECS Service Discovery / Kubernetes Service

✓ クライアントが単純・言語非依存

✗ ロードバランサーが単一障害点になりうる

ツール特徴採用場面

サービスメッシュ（Service Mesh）

多数のサービス間通信の横断的関心事（mTLS・リトライ・サーキットブレーカー・観測性）をアプリケーションコードから切り離し、インフラレイヤーで統一管理する仕組み。

サイドカーパターン

各サービスのPodにサイドカーコンテナ（代表的なのは Envoy Proxy）を注入する。すべてのネットワークトラフィックはサイドカーを経由するため、アプリケーションコードを変更せずに高度な機能を追加できる。

Istio が提供する主要機能

mTLS（相互TLS）

すべてのサービス間通信を自動的に暗号化・認証。サービスIDを証明書で管理。Zero-Trust ネットワークの実現。

トラフィック管理

Canary デプロイ（5%→10%→100%）、A/Bテスト、ヘッダーベースルーティング、フォールト注入（Fault Injection）。

観測性（Observability）

分散トレーシング（Jaeger/Zipkin）・メトリクス（Prometheus/Grafana）・アクセスログを自動収集。アプリコード変更なし。

サーキットブレーカー

Envoy の outlier_detection で失敗率の高いインスタンスを自動的にプールから除外。Hystrix のような実装を各サービスに書く必要がない。

導入判断基準

サービスメッシュは銀の弾丸ではない。 サービスが 5〜10 個以下なら導入コストの方が大きい。サービス数が多く、mTLS の一括適用・分散トレーシングの必要性・精緻なトラフィック制御が求められる場合に導入を検討する。Istio は強力だが学習コストも高い。Linkerd はシンプルで軽量な代替案。

コンテナと Kubernetes

マイクロサービスの運用基盤として Kubernetes（K8s）は事実上の標準になっている。コンテナ技術と K8s の主要概念を理解することはマイクロサービス設計に不可欠。

Docker コンテナのメリット

環境統一（Build Once Run Anywhere）

「自分のマシンでは動く」問題を解消。開発・ステージング・本番で同じイメージを使う。OS依存ライブラリの差異が消える。

軽量・高速起動

VMより軽量（MBオーダー）。秒単位で起動・停止。オートスケールに最適。

イミュータブルインフラ

コンテナイメージは不変。デプロイ = 古いコンテナを新しいイメージで置き換える。ロールバックも一瞬で完了。

Kubernetes の主要概念

リソース役割ポイント
PodK8sのデプロイ最小単位。1つ以上のコンテナをまとめたもの通常1Podに1メインコンテナ＋サイドカー
DeploymentPodのReplicaSetを管理。ローリングアップデートを実行replicas: 3 でPodを3つ維持
ServicePodへの安定したエンドポイント（ClusterIP/NodePort/LoadBalancer）Podが入れ替わってもDNS名が変わらない
Ingress外部HTTPトラフィックをServiceへルーティング。L7のルール定義パスベース・ホストベースのルーティング
ConfigMap設定値をPodへ注入（環境変数やファイルとして）機密情報はSecret（Base64エンコード）
HPACPU/メモリ使用率に基づいてPod数を自動増減HorizontalPodAutoscaler
Namespaceクラスタ内の論理的な分離区画dev/staging/prod を分ける

          periodSeconds: 10

3種のデプロイ戦略

Rolling Update（ローリング）

古いPodを少しずつ新しいPodに置き換える。ダウンタイムなし。Kubernetes のデフォルト。切り戻し（rollout undo）が容易。

✓ シンプル・リソース効率が良い

Blue-Green デプロイ

Blue（現行）と Green（新版）の2環境を用意し、LBのルーティングを切り替える。問題があれば即座にBlueに戻せる。

✗ リソースが2倍必要

Canary デプロイ

新バージョンを全体の5%などの少量トラフィックで試験し、問題がなければ段階的に100%へ拡大。サービスメッシュ（Istio）との組み合わせが強力。

✓ リスクを最小化した本番テスト

ヘルスチェック3種

Probe役割失敗時の挙動
Liveness Probeコンテナが「生きているか」（デッドロック・クラッシュ検出）コンテナを再起動する
Readiness Probeコンテナが「リクエストを受け付けられるか」（DB接続完了後など）ServiceのEndpointから除外（トラフィックを送らない）
Startup Probe起動が遅いアプリの初回起動を待つ（起動後はLivenessに委譲）起動タイムアウト → コンテナ再起動
10

マイクロサービスの落とし穴

マイクロサービスは強力だが、適切に設計・運用しないと深刻な問題を引き起こす。よくある失敗パターンを知ることが成功の前提条件。

1. 分散システムの8つの誤謬（Fallacies of Distributed Computing）

Peter Deutsch（Sun Microsystems）が提唱した、分散システム設計者がしがちな誤った前提：

① ネットワークは信頼できる

実際はパケットロス・タイムアウト・ネットワーク分断が常に起きる。タイムアウト・リトライ・サーキットブレーカーは必須。

② レイテンシはゼロ

同一データセンター内でも数ms。別リージョンなら100ms以上。ネットワーク呼び出しは関数呼び出しと桁が違う。

③ 帯域幅は無限

大量の細かいサービス呼び出しは帯域を消費する。バッチリクエスト・GraphQL の利用を検討する。

④ ネットワークはセキュア

内部通信でも暗号化（mTLS）・認証が必要。サービスメッシュや Zero-Trust 設計を採用する。

2. 「分散モノリス」を避ける

症状根本原因解決策
サービスAをデプロイするとBが壊れるAPIの後方互換性がないConsumer-Driven Contract Testing（Pact）
複数サービスの同時デプロイが必要共有ライブラリの密結合ライブラリをAPIに変換する
サービスBのDBに直接アクセスDatabase per Service 原則違反サービスAのAPIを呼ぶ or イベント購読
同期呼び出しチェーンが10段以上過度な機能分割サービスを統合する・非同期化する
全サービスが同じDB技術を使う共有DBから逃げられない段階的なデータ分離（ Strangler Fig）

3. Conway の法則とチームトポロジー

Conway の法則（1968年）：「システムを設計する組織は、その組織のコミュニケーション構造を模倣した設計を産み出す。」マイクロサービスの境界は、チームの組織構造と一致させる必要がある。逆コンウェイ戦略として、「作りたいアーキテクチャに合わせてチームを組織する」アプローチもある。

Stream-aligned Team

特定のビジネスドメイン（注文・商品）に専念するチーム。E2Eの責任を持つ。マイクロサービスチームの理想型。

Platform Team

他チームが使う共通インフラ（K8s基盤・CI/CD・監視）を提供するチーム。開発者体験（DevEx）の向上が責務。

Enabling Team

特定技術の専門知識（セキュリティ・パフォーマンス）を他チームへ移転する一時的なコーチングチーム。

4. 適切な粒度のサービス分割

分割が細かすぎる（Nanoservices）分割が大きすぎる
ネットワーク呼び出しのオーバーヘッドが支配的になるチームが再び大きくなりCognitive Load増加
単純な操作に多数のサービス協調が必要になる独立デプロイの効果が薄れる
分散トレーシングが複雑極まりないスケール粒度が荒くなる
対策：関連する機能を一つのサービスに統合する対策：Bounded Context に沿って再分割する

5. 分散トレーシングの必要性

モノリスではスタックトレースで問題を追えたが、マイクロサービスでは1リクエストが10以上のサービスをまたぐ。どこで遅延が起きているか・どこでエラーが発生しているかを追跡するには分散トレーシングが必須。

標準： OpenTelemetry（ベンダー非依存の計装API）

実装： Jaeger・Zipkin・AWS X-Ray・Datadog APM

各サービスが Trace ID を伝播することで、マイクロサービス群全体でのリクエストフローを可視化できる。

面接頻出：マイクロサービスQ&A

  • Q: モノリスとマイクロサービスをどう使い分けますか？
    → チームサイズ・ドメインの明確さ・スケール要件・インフラ成熟度の4軸で判断。初期は Majestic Monolith から始め、Two-Pizza Team を超えたら分割を検討する。
  • Q: マイクロサービス間の分散トランザクションをどう扱いますか？
    → 2PC（Two-Phase Commit）は分散環境では使わない。Saga パターン（Choreography または Orchestration）で補償トランザクションを実装する。最終的整合性を受け入れる設計にする。
  • Q: サービス間の循環依存をどう防ぎますか？
    → DDD の Bounded Context と Context Map で上流・下流の関係を明確にする。循環が生まれる場合はサービスの境界引き直しのサインと捉える。
  • Q: ECサイトの注文処理をマイクロサービスで設計してください。
    → 注文・商品・ユーザー・決済・配送・通知の6サービスを設計。OrderPlaced イベントを Kafka に発行 → 在庫サービス・決済サービスが購読。Saga で整合性を担保。API Gateway でクライアントに公開。
  • Q: マイクロサービスの監視で重要なのは何ですか？
    → Metrics（Prometheus/Grafana）・Distributed Tracing（Jaeger）・Logs（ELK Stack）の「観測性の3本柱」。さらに Availability・Latency・Error Rate・Saturation の「ALES指標」を各サービスに定義する。
  • Q: サービスメッシュは何のために必要ですか？
    → サービス間の mTLS・リトライ・サーキットブレーカー・分散トレーシングをアプリケーションコードから分離してインフラレイヤーで統一管理するため。ただし10サービス以下なら導入コスト過多。

深掘り: Kubernetes本番運用

Kubernetesは強力だが、正しく設定しないと本番環境で重大な問題が発生する。各設定の意味と落とし穴を理解する。

Resource Requests vs Limits — CPU throttlingとOOMKiller

Requests（保証リソース）

Podをスケジュールする際に保証するリソース量。Nodeにこれだけの空きがなければスケジュールされない。

Limits（上限）

Podが使用できる最大リソース量。CPU: Limitを超えるとthrottle（速度制限）。メモリ: Limitを超えるとOOMKillerに強制終了される。

# deployment.yaml — Resource設定の例
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: order-service
        image: order-service:v1.2.3
        resources:
          requests:
            memory: "256Mi"   # スケジュール保証
            cpu: "100m"       # 0.1 CPU core
          limits:
            memory: "512Mi"   # これを超えるとOOMKill
            cpu: "500m"       # 0.5 CPU core (超えるとthrottle)

---
# CPU Throttling の問題
# CPUはcgroups v1のCFS (Completely Fair Scheduler)で管理
# 100ms周期で使用量をチェック。Limitを超えると残り時間はthrottleされる
# 短期的スパイク(P99レイテンシ増加)の原因になりやすい
# 対策: Requests/Limitsの比率を小さくする(1:2以内推奨)
# または Burstable ではなく Guaranteed QoS (req=limits)にする

演習3: マイクロサービス設計問題診断（5問）

問題1: 全サービスで共有DBを使用している

問題: OrderServiceとInventoryServiceが同じDBに直接アクセスしている。スキーマ変更が全サービスに影響し、独立デプロイができない（分散モノリス）。
修正: 「Database per Service」パターン。各サービスが自分のDBを所有し、他サービスのDBには直接アクセスしない。データが必要な場合はAPIを呼ぶか、イベントで同期する。

問題2: サービス間の長い同期呼び出しチェーン

OrderService → InventoryService → ProductService → CategoryService
# 全サービスの合計レイテンシ: 各10ms × 4 = 40ms
# CategoryServiceがダウンするとOrderServiceも503

問題: 深い同期呼び出しチェーンにより、末端サービスの障害が全体に伝播（カスケード障害）。レイテンシも蓄積。
修正: ①チェーンを短くする（必要なデータをOrderService自身がキャッシュ）。②CategoryServiceへの呼び出しを非同期化。③Circuit Breakerで障害伝播を防ぐ。

問題3: サービスに複数のビジネスドメインが混在

問題: 「UserService」が認証・プロフィール管理・注文履歴・決済履歴を全て担当している。変更頻度・スケール要件・所有チームが異なる機能が混在。
修正: DDDのBounded Contextに基づいて分割する。AuthService・ProfileService・OrderHistoryServiceは別サービスへ。

問題4: サービスが他サービスの内部実装に依存

// OrderServiceがInventoryServiceのテーブルを直接JOINしている
const order = await inventoryDB.query(
    "SELECT o.*, i.stock_count FROM orders o JOIN inventory i ON ...");

問題: InventoryServiceのDB実装に直接依存。DBスキーマ変更やDB技術変更（PostgreSQL→DynamoDB）で即座に壊れる。
修正: InventoryServiceのAPIを呼ぶ（GET /inventory/{productId}）。データが必要なら自分のDBにコピーをキャッシュするRead Modelを作る（CQRS）。

問題5: サービスのバージョン管理なしにAPIを変更

問題: OrderServiceのAPIを変更したところ、そのAPIを呼んでいた全クライアントが壊れた。マイクロサービスの「独立デプロイ」の原則を破っている。
修正: ①APIのバージョニング（/v1/orders → /v2/orders）。②後方互換性を維持（古いフィールドはdeprecatedにし一定期間は維持）。③Consumer Driven Contract Testing（PactなどのCDCテスト）でクライアントとの互換性を自動検証。
