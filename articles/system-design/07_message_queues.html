1

なぜメッセージキューが必要か

モダンなシステムでは、複数のサービスが連携して動作する。同期呼び出し（HTTP REST等）だけに頼ると深刻な問題が生じる。

同期呼び出しの問題点

🔗 カスケード障害

サービスAがBを呼び、BがCを呼ぶ連鎖構造では、Cが落ちるとB→Aと障害が伝播する。一部の障害が全体を巻き込む。

⚡ スパイク対応不可

注文が急増しても下流サービスの処理能力を超えてリクエストが殺到し、タイムアウトや502エラーが多発する。

🐢 レイテンシの蓄積

A→B→C→Dと4サービスを経由すると、各サービスのレイテンシが加算される。合計レイテンシ = Σ(各サービスのレイテンシ)。

🔒 密結合

呼び出し元が呼び先のAPIを直接知っている。呼び先のインターフェース変更が即座に呼び出し元に影響する。

非同期化のメリット

疎結合 (Decoupling)

プロデューサーはコンシューマーの存在を知らなくてよい。システムの変更・デプロイを独立して行える。

スパイク吸収 (Load Leveling)

リクエスト急増時もキューがバッファとして機能。コンシューマーは処理可能なペースで安定して消化できる。

再試行と信頼性

処理失敗したメッセージはキューに残る（またはDLQへ）。消費者が復旧後に再処理できる。データロストを防ぐ。

スケーラビリティ

コンシューマーを増やすだけで処理能力を線形にスケールアウト。プロデューサーに変更不要。

2

メッセージキューの基本概念

主要コンポーネント

Producer（プロデューサー）

メッセージを生成してキューまたはブローカーに送信するサービス。送信後の処理は関知しない。

Consumer（コンシューマー）

キューからメッセージを取り出して処理するサービス。複数のコンシューマーが並列処理可能。

Queue / Topic

メッセージを保持する中間ストレージ。キュー（FIFO）またはトピック（Pub/Sub）の形態がある。

Broker（ブローカー）

メッセージの受信・ルーティング・配信を仲介するミドルウェア。Kafka・RabbitMQ等が該当。

メッセージ配信保証（Delivery Guarantees）

保証レベル説明リスクユースケース
At-most-once 最大1回配信。受信確認なしで即削除 メッセージ損失あり ログ収集、メトリクス（損失許容）
At-least-once 最低1回配信。ACK受信まで再送 重複処理あり（冪等性が必要） 多くのメッセージングシステムのデフォルト
Exactly-once 厳密に1回のみ配信・処理 実装コスト大、パフォーマンス低下 金融取引、在庫更新（重複不可）

Exactly-onceの現実： 分散システムで真のExactly-onceを保証するには、トランザクション的な仕組みが必要。Kafkaはプロデューサー+コンシューマー間でExactly-once Semantics (EOS)をサポートするが、外部システムとの統合には冪等性設計が不可欠。

Apache Kafka 詳解

Kafkaは分散型ログストリーミングプラットフォーム。LinkedIn社が開発し、2011年にOSSとして公開。毎秒数百万メッセージを処理できる高スループットが特徴。

Kafkaの本質：分散コミットログ

Kafkaはメッセージを「キュー（取り出したら消える）」ではなく追記専用の不変ログとして保存する。これがKafkaの高性能と再処理能力の根源。

なぜKafkaのディスクI/Oは速いか

順次書き込み (Sequential Write)

Kafkaはログを追記（Append-only）するだけ。ランダムI/Oではなく順次I/Oのため、HDDでも500MB/s以上のスループットが出る。

ゼロコピー (Zero-Copy)

OSのsendfile()システムコールを使い、カーネル空間からネットワークへ直接転送。ユーザー空間へのコピーを省いてCPU使用を削減。

バッチ処理 (Batching)

プロデューサーはメッセージをバッチにまとめて送信。ネットワークラウンドトリップとI/O回数を大幅削減。

ページキャッシュ活用

OS管理のページキャッシュを最大限活用。最近のデータはほぼメモリから読める。JVMのGCを回避できる利点もある。

Consumer GroupとPartitionの関係

重要ルール： 1つのパーティションは、同一Consumer Group内の1つのコンシューマーにしか割り当てられない。これにより順序保証と並列処理を両立。Consumer数がPartition数を超えると、超過分のConsumerはアイドル状態になる。

// Consumer Group設定の例 (Java)
Properties props = new Properties();
props.put("bootstrap.servers", "kafka1:9092,kafka2:9092");
props.put("group.id", "order-processing-group"); // Consumer Group ID
props.put("enable.auto.commit", "false"); // 手動コミット推奨
props.put("max.poll.records", "500");
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("orders"));

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
        for (ConsumerRecord<String, String> record : records) {
            // Partition, Offset, Key, Value にアクセス可能
            System.out.printf("partition=%d offset=%d key=%s value=%s%n",
                record.partition(), record.offset(), record.key(), record.value());
            processOrder(record.value());
        }
        consumer.commitSync(); // 処理完了後にオフセットをコミット
    }
} finally {
    consumer.close();
}

RabbitMQ vs Kafka

RabbitMQのExchange Types

Direct Exchange

Routing Keyと完全一致するキューにルーティング。1対1の明確な宛先指定に使用。

Topic Exchange

orders.*や#.errorのようなワイルドカードパターンでルーティング。柔軟なフィルタリングに最適。

Fanout Exchange

バインドされた全キューにブロードキャスト。Routing Keyを無視。ログやキャッシュ無効化に使用。

Headers Exchange

メッセージのヘッダー属性でルーティング。Routing Keyより柔軟だが複雑。x-matchで AND/OR 条件指定。

総合比較表

特徴RabbitMQApache Kafka
モデル スマートブローカー、シンプルコンシューマー シンプルブローカー、スマートコンシューマー
メッセージ保持 ACK後は削除（デフォルト） 設定期間保持（デフォルト7日）。再読取り可能
スループット 〜数万msg/sec 〜数百万msg/sec（順次I/O）
ルーティング Exchange/Binding/RoutingKeyで柔軟 Topic/Partitionのみ（シンプル）
順序保証 キュー内FIFO（競合コンシューマーで崩れる） パーティション内で厳密に順序保証
スケーラビリティ 水平スケールは複雑 Partitionを増やすだけで線形スケール
ストリーム処理 不向き Kafka Streams / KSQL で強力にサポート
向いているケース タスクキュー、RPC、複雑ルーティング ログ収集、イベントソーシング、ストリーム分析
学習コスト 低〜中 中〜高

選択の指針： タスクを「処理して消す」ならRabbitMQ。「イベントログを永続化して複数サービスが独立して読む」ならKafka。大量のストリームデータやイベントソーシングにはKafkaが圧倒的に向いている。

メッセージングパターン

Command vs Event の違い

Command（命令）

  • 特定のサービスへの指示（「注文を処理せよ」）
  • 宛先が明確（1対1）
  • 処理されることを期待する
  • 例：ProcessOrder, SendEmail

Event（イベント）

  • → 起きた事実の宣言（「注文が完了した」）
  • → 誰が聞くかを意識しない（1対多）
  • → 過去形で命名される
  • → 例：OrderPlaced, PaymentProcessed

Dead Letter Queue (DLQ)

処理に繰り返し失敗したメッセージを隔離するための特別なキュー。本番システムで必須のパターン。

// RabbitMQでのDLQ設定 (Python pika)
import pika

channel.queue_declare(
    queue='orders',
    arguments={
        'x-dead-letter-exchange': 'dlx',       # DLQ用Exchange
        'x-dead-letter-routing-key': 'orders.dead',
        'x-message-ttl': 30000,                 # 30秒でDLQへ
        'x-max-retries': 3                      # 3回失敗でDLQへ
    }
)
# DLQキューを宣言
channel.queue_declare(queue='orders.dead', durable=True)
channel.queue_bind(
    exchange='dlx',
    queue='orders.dead',
    routing_key='orders.dead'
)

Retry with Exponential Backoff

失敗時に即座に再試行するのではなく、待機時間を指数的に増やすことで、障害中のシステムへの負荷を軽減する。

import asyncio, random

async def process_with_retry(message, max_retries=5):
    for attempt in range(max_retries):
        try:
            await process_message(message)
            return # 成功
        except RetryableError as e:
            if attempt == max_retries - 1:
                await send_to_dlq(message, str(e))
                return
            # 指数バックオフ + ジッター: 2^attempt + random(0,1)
            wait = (2 ** attempt) + random.random()
            # attempt: 0→1s, 1→2s, 2→4s, 3→8s, 4→16s (最大)
            print(f"Attempt {attempt+1} failed. Retry in {wait:.1f}s")
            await asyncio.sleep(wait)
        except NonRetryableError as e:
            await send_to_dlq(message, str(e)) # 即DLQ
            return

Poison Pill Problem（毒メッセージ問題）

問題： 永遠に処理に失敗するメッセージ（バグのあるデータ等）がキューに残り続け、コンシューマーがクラッシュし続ける現象。

対処法1: 最大再試行数 + DLQ

N回失敗したらDLQへ移動。人間が後でDLQを確認・修正して再処理できる。

対処法2: メッセージバリデーション

処理前にスキーマ検証。不正フォーマットのメッセージを即座にDLQへ。処理ループに入らない。

対処法3: 試行回数をメッセージに付与

メッセージヘッダーにx-retry-countを持たせ、閾値超えで自動的にDLQへルーティング。

Competing Consumers パターン

同一キューに複数のコンシューマーを接続し、メッセージを並列処理することでスループットをスケールする。

スケール計算： コンシューマー1台が100msg/secの場合、10台並列で1000msg/sec。キューが受信スパイクを吸収するため、一時的な急増も問題なく処理できる。

イベントソーシング (Event Sourcing)

従来のCRUDは「現在の状態」を保存するが、イベントソーシングは「状態を変えた出来事（イベント）の履歴」を保存する。現在の状態はイベントを最初から順番に適用して再現する。

従来のCRUD vs イベントソーシング

従来のCRUD

  • → 現在の状態をDB行として保存
  • → UPDATEで上書き。過去は消える
  • → 「今どうなっているか」は分かる
  • → 「なぜそうなったか」は分からない

イベントソーシング

  • すべての変更をイベントとして追記
  • 過去の状態を任意の時点で再現可能
  • 完全な監査ログが自動で得られる
  • イベントを再生して別の射影を作れる
# イベントストアのイメージ（口座残高の例）
events = [
    { "type": "AccountOpened",   "amount": 0,    "ts": "2024-01-01" },
    { "type": "MoneyDeposited",  "amount": 10000, "ts": "2024-01-02" },
    { "type": "MoneyWithdrawn",  "amount": 3000,  "ts": "2024-01-05" },
    { "type": "MoneyDeposited",  "amount": 5000,  "ts": "2024-01-10" },
]

def rebuild_state(events):
    balance = 0
    for event in events:
        if event["type"] == "MoneyDeposited":
            balance += event["amount"]
        elif event["type"] == "MoneyWithdrawn":
            balance -= event["amount"]
    return balance  # → 12000円

# 任意の日時の残高を再現できる！
state_on_jan5 = rebuild_state([e for e in events if e["ts"] <= "2024-01-05"])
# → 7000円（2024/01/05時点の残高）

スナップショット戦略

イベントが大量に蓄積すると再生に時間がかかる。定期的にその時点の状態をスナップショットとして保存し、最後のスナップショット以降のイベントのみを再生することで高速化する。

メリット

  • 完全な監査ログ・変更履歴
  • 任意時点へのタイムトラベル
  • バグ修正後に全データを再計算可能
  • 新機能のためのプロジェクションを後から追加

デメリット

  • 実装の複雑さが増す
  • クエリが直感的でない（都度再生が必要）
  • イベントスキーマの変更が困難
  • 大量イベントの再生コスト（スナップショットで対処）

CQRS（コマンドクエリ責任分離）

CQRSは読み取り（Query）と書き込み（Command）を別々のモデル・データストアで扱うアーキテクチャパターン。イベントソーシングと組み合わせることが多い。

トレードオフ：最終整合性 (Eventual Consistency)
CQRSでは書き込み後、Read DBへの反映に遅延が生じる（通常ミリ秒〜秒単位）。書いた直後に読んでも古いデータが返る可能性がある。UIでの「楽観的更新」や「書き込み直後は書き込みDBから読む」戦略で対処。

8

バックプレッシャー (Backpressure)

プロデューサーがコンシューマーより速くメッセージを生成し続けると、キューが無限に膨張してメモリを枯渇させる。これをバックプレッシャー問題という。

バックプレッシャー戦略

1. バッファリング

一定量までバッファに蓄積。キュー上限（max-length）を設定し、溢れたらDLQへ、または最古を削除（drop-head）。RabbitMQのx-max-length設定。

2. ドロップ (Shedding)

閾値超えたリクエストを意図的に破棄。ログ収集などロスが許容できるケースで使用。シンプルだが情報ロストのリスクあり。

3. プロデューサー速度制御

コンシューマーが処理可能な速度をフィードバックし、プロデューサーの送信速度を下げる。Reactive Streamsのrequest(n)がこのモデル。

4. コンシューマースケールアウト

オートスケーリングでコンシューマー数を増やす。KubernetesのHPA（Horizontal Pod Autoscaler）＋KEDAでKafkaラグに応じて自動スケール。

Circuit Breaker との組み合わせ

class CircuitBreakerConsumer:
    def __init__(self):
        self.failure_count = 0
        self.state = "CLOSED"  # CLOSED/OPEN/HALF_OPEN
        self.threshold = 5
        self.reset_timeout = 60  # 秒

    async def process(self, message):
        if self.state == "OPEN":
            # サーキット開放中 → 処理をスキップ、キューに蓄積させる
            await asyncio.sleep(0.1)  # 少し待って再確認
            return

        try:
            await call_downstream_service(message)
            self.failure_count = 0
            self.state = "CLOSED"
        except Exception:
            self.failure_count += 1
            if self.failure_count >= self.threshold:
                self.state = "OPEN"
                # 60秒後に HALF_OPEN へ移行
                asyncio.create_task(self.reset_after(self.reset_timeout))
                # バックプレッシャー: キューにメッセージが溜まる
                # コンシューマーが処理を止めることで自然に圧力をかける

Reactive Streams の考え方： コンシューマーが「今N件処理できる」とプロデューサーに通知することで、プッシュ型ではなくプル型の流量制御を実現する。RxJava・Project Reactor・Akka Streamsがこの仕様に準拠。

Kafkaの実践設定

Partition数の決め方

目安：目標スループット ÷ 1Partition速度

例：目標1GB/s、1パーティションが100MB/sなら10パーティション必要。Consumer数を見越して多めに設定（後から増やすと順序が崩れる）。

注意：多すぎも問題

パーティション数はZooKeeperのメタデータ負荷、Broker間のレプリケーション接続数に直結。1Broker当たり数千パーティションが目安の上限。

Producer acks 設定（信頼性 vs スループット）

acks設定動作信頼性スループット
acks=0 Brokerからの確認なし。送りっぱなし 最低 最高
acks=1 Leaderが書き込んだら確認（デフォルト） 中 中
acks=all (or -1) 全In-Sync Replica(ISR)が書き込んだら確認 最高 低い

重要な設定パラメータ

# ==========================================
# Producer 設定（Python kafka-python）
# ==========================================
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers=['kafka1:9092', 'kafka2:9092'],
    acks='all',            # 全ISRからのACKを待つ（高信頼性）
    retries=5,             # 失敗時の再試行回数
    retry_backoff_ms=300,  # 再試行間隔（ms）
    linger_ms=10,          # バッチ待機時間（スループット向上）
    batch_size=16384,      # バッチサイズ（bytes）
    compression_type='gzip', # 圧縮（ネットワーク帯域削減）
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    key_serializer=lambda k: k.encode('utf-8') if k else None
)

# キーを指定することで同じキーのメッセージが同じPartitionへ
# → 同一ユーザーの注文を順番通りに処理できる！
future = producer.send(
    'orders',
    key='user-12345',       # 同じユーザーは常に同じPartition
    value={'order_id': 789, 'item': 'laptop', 'price': 120000}
)
result = future.get(timeout=10)  # ブロッキングで確認
print(f"Sent to partition {result.partition} offset {result.offset}")

# ==========================================
# Consumer 設定
# ==========================================
from kafka import KafkaConsumer

consumer = KafkaConsumer(
    'orders',
    bootstrap_servers=['kafka1:9092'],
    group_id='inventory-service-group',
    auto_offset_reset='earliest',   # 初回: 最初から読む
    enable_auto_commit=False,        # 手動コミット（重要！）
    max_poll_records=100,             # 1回のpollで取得する最大数
    fetch_min_bytes=1024,             # 最低1KB集まってから取得
    fetch_max_wait_ms=500,            # 最大500ms待つ
    session_timeout_ms=30000,         # Rebalanceのタイムアウト
)

for message in consumer:
    try:
        order = json.loads(message.value)
        update_inventory(order)
        consumer.commit()  # 処理成功後にのみコミット
    except Exception as e:
        # エラー時はコミットしない → 再起動後に再処理される
        log_error(e, message)

Replication Factor の指針

環境推奨Replication Factor理由
開発・検証1シンプル、リソース節約
本番 (3 Broker)31台障害時も稼働継続
重要データ (5+ Broker)3〜5より高い耐障害性

min.insync.replicas との組み合わせ：
acks=all と min.insync.replicas=2 を組み合わせることで、「最低2台のレプリカに書き込まれるまで成功とみなさない」を強制できる。Replication Factor 3 の場合、1台故障しても書き込み継続可能。

Kafkaトピックの設計チェックリスト

1. トピック命名規則を統一する

<domain>.<entity>.<event> 形式を推奨。例：commerce.order.placed, payment.invoice.created

2. Partition数は多めに設定（後から増やしにくい）

少なすぎると後でスケールアウトが困難。最初から想定最大Consumer数 × 2〜3倍を目安に。

3. 保持期間（retention）をユースケースに合わせる

デフォルト7日。イベントソーシング用途は無制限も検討（ストレージコストとのトレードオフ）。

4. キー選択でローカル順序を保証する

同一エンティティ（ユーザーID、注文ID等）をキーにすることで、そのエンティティのイベントが必ず同じPartitionへ → 順序保証。

5. Consumer GroupのLag監視を設定する

Kafka Consumer Lag（処理遅延）をDatadog/Prometheusで監視。Lag増加はスケールアウトのシグナル。

まとめ：重要ポイント

メッセージキューの本質

非同期化により疎結合・スパイク吸収・再試行を実現。同期HTTP連鎖でのカスケード障害を防ぐ最も重要な設計判断の一つ。

KafkaはログDB

Kafkaはキューではなく分散コミットログ。メッセージを削除せず保持し、Consumer Groupが独立してOffsetを管理。ストリーム処理の基盤。

ツール選択基準

タスクキュー・複雑ルーティングはRabbitMQ。大量イベント・ストリーム・複数Consumer・イベントソーシングはKafka。

イベントソーシング + CQRS

書き込みを不変イベントの追記に変え、読み取り用に最適化されたビューを非同期で構築。完全な監査ログと再処理能力を得る。

信頼性パターン

DLQ + Exponential Backoff は本番必須。Poison Pillには最大再試行数を設定。Exactly-onceには冪等性設計が必要。

バックプレッシャー

コンシューマーが追いつかないときはスケールアウト or Circuit Breaker。キュー長とConsumer Lagを監視して早期検出。

Zero-Copy — sendfile()によるカーネルコピーの省略

通常のread()→write()では4回のコピーが発生する。Kafkaはsendfile()syscallを使い2回に削減する。

通常のコピー（4回）

  1. ディスク → カーネルバッファ（DMA）
  2. カーネルバッファ → ユーザー空間
  3. ユーザー空間 → ソケットバッファ
  4. ソケットバッファ → NIC（DMA）

コンテキストスイッチ: 4回（user ↔ kernel）

sendfile()（2回）

  1. ディスク → カーネルバッファ（DMA）
  2. カーネルバッファ → NIC（DMA）

ユーザー空間コピーなし。コンテキストスイッチ: 2回のみ。CPU使用率を大幅削減。

// Javaでのsendfile()（Kafkaの内部実装）
// FileChannel.transferTo() がsendfile()syscallにマップされる
long transferred = fileChannel.transferTo(
    position, count, socketChannel);
// これだけでカーネルが最適なzero-copyを実行

# linger.ms=100: 高レイテンシ・最高スループット（バッチ処理向け）

# スループット計算例
# バッチなし: 1msg/リクエスト × 1KBmsg = 1KB/req
# バッチあり(linger=5ms,batch=64KB): ~64msg/リクエスト
# スループット: 64倍向上（ネットワークRTTの削減効果）

Outbox Pattern — アトミックなDB書き込みとメッセージ送信

DBへの書き込みとKafkaへのメッセージ送信を同一トランザクションで実行する。2フェーズコミットなしに分散トランザクションを実現する重要なパターン。

-- アウトボックステーブルの定義（PostgreSQL）
CREATE TABLE outbox (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type VARCHAR(50) NOT NULL,  -- 'order', 'payment'等
    aggregate_id   VARCHAR(50) NOT NULL,
    event_type     VARCHAR(100) NOT NULL, -- 'OrderPlaced'等
    payload        JSONB NOT NULL,
    created_at     TIMESTAMP DEFAULT NOW(),
    published_at   TIMESTAMP             -- NULL = 未送信
);

-- 注文処理: DBとOutboxを同一トランザクションで
BEGIN;
  INSERT INTO orders (id, customer_id, total) VALUES (1, 101, 5000);
  INSERT INTO outbox (aggregate_type, aggregate_id, event_type, payload)
    VALUES ('order', '1', 'OrderPlaced',
            '{"orderId":1,"customerId":101,"total":5000}');
COMMIT;
-- 片方が失敗したら両方ロールバック（アトミック）

Polling Publisher

// アウトボックスをポーリングして送信
async function publishOutboxEvents() {
  while (true) {
    const events = await db.query(`
      SELECT * FROM outbox
      WHERE published_at IS NULL
      ORDER BY created_at
      LIMIT 100
      FOR UPDATE SKIP LOCKED
    `);
    for (const event of events) {
      await kafka.produce(event.event_type, event.payload);
      await db.query(`
        UPDATE outbox SET published_at=NOW()
        WHERE id=$1`, [event.id]);
    }
    await sleep(1000); // 1秒毎にポーリング
  }
}

# リクエスト側（非同期で送り、応答を待つ）
async def call_async_service(request_data: dict) -> dict:
    correlation_id = str(uuid.uuid4())
    reply_queue = f"reply:{correlation_id}"

    # 返信先キューを作成（30秒TTL）
    r.expire(reply_queue, 30)

    # リクエスト送信
    channel.basic_publish(
        exchange='',
        routing_key='requests',
        body=json.dumps(request_data),
        properties=pika.BasicProperties(
            reply_to=reply_queue,
            correlation_id=correlation_id
        )
    )

    # 応答を待つ（Redisでブロッキング待機）
    _, response = r.blpop(reply_queue, timeout=25)
    return json.loads(response)

# サービス側（リクエストを受け取って処理し返信）
def handle_request(ch, method, props, body):
    request = json.loads(body)

演習1: 注文システムのSagaをKafkaで実装する

注文処理の完全なSaga実装。OrderService→InventoryService→PaymentService→ShippingServiceの流れと補償トランザクションを実装する。

# kafka-python を使ったSaga実装
from kafka import KafkaProducer, KafkaConsumer
import json, uuid

# === Topic設計 ===
# orders          → OrderPlaced イベント
# inventory-cmd   → ReserveInventory コマンド
# inventory-events → InventoryReserved | InventoryReserveFailed
# payment-cmd     → ProcessPayment コマンド
# payment-events  → PaymentProcessed | PaymentFailed
# shipping-cmd    → CreateShipment コマンド
# compensation    → 補償イベント（ロールバック用）

producer = KafkaProducer(
    bootstrap_servers=['kafka:9092'],
    value_serializer=lambda v: json.dumps(v).encode()
)

# --- OrderService ---
def place_order(customer_id: str, items: list) -> str:
    order_id = str(uuid.uuid4())
    # 1. DBにPENDING状態で保存
    db.save_order(order_id, customer_id, items, status='PENDING')
    # 2. OrderPlacedイベントを送信
    producer.send('orders', {
        'type': 'OrderPlaced',
        'orderId': order_id,
        'customerId': customer_id,
        'items': items
    })
    return order_id

# --- InventoryService ---
def handle_order_placed(event: dict):
    try:
        # 在庫確保を試みる
        for item in event['items']:
            db.reserve_stock(item['productId'], item['quantity'])

        producer.send('inventory-events', {
            'type': 'InventoryReserved',
            'orderId': event['orderId']
        })
    except InsufficientStockError:
        # 在庫不足 → 補償イベントを送信
        producer.send('inventory-events', {
            'type': 'InventoryReserveFailed',
            'orderId': event['orderId'],
            'reason': 'INSUFFICIENT_STOCK'
        })

# --- PaymentService ---
def handle_inventory_reserved(event: dict):
    try:
        order = db.get_order(event['orderId'])
        charge_result = payment_gateway.charge(order.total, order.customer_id)
        producer.send('payment-events', {
            'type': 'PaymentProcessed',
            'orderId': event['orderId'],
            'transactionId': charge_result.transaction_id
        })
    except PaymentFailedError:
        # 決済失敗 → 在庫を解放する補償トランザクション
        producer.send('compensation', {
            'type': 'ReleaseInventory',
            'orderId': event['orderId']
        })
        producer.send('payment-events', {
            'type': 'PaymentFailed',
            'orderId': event['orderId']
        })

# --- Consumer Group設計 ---
# 各サービスは独自のConsumer Groupを持つ
inventory_consumer = KafkaConsumer(
    'orders',               # OrderPlacedを購読
    group_id='inventory-service',
    bootstrap_servers=['kafka:9092'],
    value_deserializer=lambda m: json.loads(m.decode()),
    enable_auto_commit=False  # 手動コミット

演習2: Kafkaのパーティション数とスループットを計算する

目標: 毎秒10万メッセージ（平均1KB）

"""
スループット計算:
  目標: 100,000 msg/sec × 1 KB = 100 MB/sec

Consumerスループットから逆算:
  1 Consumer の処理能力:
    - ネットワーク: 1Gbps NIC → 125 MB/sec
    - CPU処理: 仮に50 MB/sec (JSONデシリアライズ含む)
    - ボトルネック: CPU → 50 MB/sec

必要なConsumer数:
  100 MB/sec ÷ 50 MB/sec = 2 Consumer (最低)
  安全マージン2倍 → 4 Consumer

パーティション数:
  1 Consumer = 1 Partition (同一Consumer Group内)
  → 4 Partition が最小
  実際は将来のスケールアップ考慮し 12 Partition 推奨

Replication Factor の計算:
  Replication Factor = 3 (標準)
  Write Amplification = メッセージ数 × Replication Factor
  100 MB/sec × 3 = 300 MB/sec の書き込み
  各Brokerへの書き込み分散を考慮した容量計画が必要

ディスク容量見積もり:
  保持期間: 7日 (604800秒)
  100 MB/sec × 604800 sec × 3 (replication) = 約179 TB
  圧縮率(lz4で約60%削減): 179 × 0.4 ≈ 72 TB
"""

# 計算コード
msg_per_sec = 100_000
msg_size_kb = 1
throughput_mb = (msg_per_sec * msg_size_kb) / 1024
consumer_capacity_mb = 50
min_partitions = math.ceil(throughput_mb / consumer_capacity_mb)
recommended_partitions = min_partitions * 3  # 将来の拡張用
retention_days = 7
replication_factor = 3

disk_tb = (throughput_mb * 86400 * retention_days * replication_factor * 0.4) / 1024**2
print(f"スループット: {throughput_mb:.0f} MB/sec")
print(f"推奨パーティション数: {recommended_partitions}")
print(f"必要ディスク容量: {disk_tb:.0f} TB")

演習3: Kafkaアンチパターン診断（5問）

問題1: 1つのTopicにすべてのイベントを送信

producer.send("all-events", {"type": "OrderPlaced", ...})
producer.send("all-events", {"type": "UserRegistered", ...})
producer.send("all-events", {"type": "PaymentProcessed", ...})

問題: 全イベントが1Topicに混在するため、特定イベントにしか関心のないConsumerも全メッセージを受信してフィルタリングする必要がある。パーティション内の順序保証が崩れる。
修正: イベント種別ごとにTopicを分ける（orders、users、payments等）。

問題2: Consumer Group IDを設定しない

consumer = KafkaConsumer("orders")  # group_id未設定

問題: Consumer Groupなしでは、複数インスタンスを起動しても同じメッセージを全インスタンスが受信する（Pub/Sub動作になる）。水平スケールによる負荷分散ができない。
修正: group_id="order-processor"を必ず設定する。

問題3: 処理前にオフセットをコミット

for msg in consumer:
    consumer.commit()      # 先にコミット
    result = process(msg)  # 処理中にクラッシュ!

問題: コミット後に処理がクラッシュすると、そのメッセージは永久に失われる（At-most-onceになる）。
修正: 処理完了後にコミットする（At-least-once）。冪等性を担保した上でExactly-onceを目指す。

問題4: パーティション数 < Consumer数

問題: パーティション=6、Consumer=10の場合、4つのConsumerがアイドル状態になる。パーティション≥Consumer数でなければスケールアウトの恩恵を受けられない。
修正: パーティション数を多めに設定する（後から増やせるが、Consumerのリバランスが発生するため慎重に）。

問題5: enable.auto.commit=trueで重い処理

問題: auto.commit.interval.ms（デフォルト5000ms）ごとに自動コミットされる。メッセージのpollingから処理完了まで5秒以上かかると、処理未完了のオフセットがコミットされてデータロストが発生する可能性がある。
修正: enable.auto.commit=Falseにして手動コミット。処理完了後に明示的にconsumer.commit()を呼ぶ。
