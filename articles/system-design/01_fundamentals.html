システムデザインの進め方

Senior Engineerが面接・実務で使うフレームワーク：要件定義から詳細設計・トレードオフ分析まで体系的に習得する

🎯 なぜシステムデザインが重要か

システムデザインは「正解のない問題に対して最善解を導く力」です。 コーディング試験が「与えられた問題を解く力」を見るのに対し、システムデザインは 「どんな問題を解くべきかを定義し、設計し、トレードオフを語れるか」を評価します。

🗺️ 5ステップ設計フレームワーク

どんなシステムデザイン問題にも適用できる普遍的なフレームワークです。面接では45〜60分でこの流れをたどります。

1

要件定義（5〜7分）

曖昧な問題文を具体的な仕様に変換します。機能要件と非機能要件を必ず分けて整理します。

✅ 機能要件 (What)

  • システムが何をするか
  • 主要なユースケース
  • 対象ユーザー
  • データの入出力

⚡ 非機能要件 (How Well)

  • スケール（DAU、QPS）
  • 可用性（99.9% vs 99.99%）
  • レイテンシ（p50, p99）
  • 整合性（強一貫 vs 結果整合）
💡 実践Tips
「Twitterを設計してください」→ 「書き込みより読み込みが多い？フォロワー数の上限は？リアルタイム性は必要？」と質問を重ねる

スケール & 最終確認（5〜10分）

  • データが10倍になったらどうなるか？
  • 特定機能だけ急増した場合は？（例：特定時間帯のアクセス集中）
  • コストと可用性のトレードオフをどう判断するか？

⚖️ トレードオフ分析の思考法

Senior Engineerとしての核心は「トレードオフを語れること」です。「AよりBがいい」ではなく「AとBにはこういうトレードオフがある、今回の要件ではBが適切」と語れるかが重要です。

観点一方の選択もう一方の選択判断の基準
整合性 vs 可用性 強一貫性（データが常に正確） 結果整合性（高可用性・低レイテンシ） 金融→強一貫性、SNSフィード→結果整合
正規化 vs 非正規化 正規化（データ整合性↑、書き込み速度↑） 非正規化（読み込み速度↑、ストレージ↑） 読み書き比率・クエリパターンで決定
SQL vs NoSQL SQL（ACID、柔軟クエリ、JOINが得意） NoSQL（水平スケール、スキーマ柔軟） データ構造の複雑さとスケール要件で決定
同期 vs 非同期 同期（シンプル、即時確認可能） 非同期（スケーラブル、疎結合） ユーザーが即時レスポンスを必要とするか
モノリス vs マイクロサービス モノリス（シンプル、デプロイ容易） マイクロサービス（独立スケール、技術多様性） チームサイズとシステムの複雑さ

⚠️ よくある設計の罠とアンチパターン

  • ⚠️
    SPOFを見落とす
    DBが1台、LBが1台などの単一障害点は必ずレビューする。Active-Standbyかレプリカを検討する。
  • ⚠️
    キャッシュを無思慮に追加する
    キャッシュは magic bullet ではない。invalidation戦略とcache stampede（雪崩）への対策が必要。
  • ⚠️
    DBに全てのロジックを押し込む
    DBのストアドプロシージャやトリガーへの依存はスケールを妨げる。ロジックはアプリ層へ。
  • ⚠️
    非機能要件を後回しにする
    可用性・一貫性・レイテンシは設計の最初から考慮する。後付けでは大幅なリアーキテクチャになる。
  • ⚠️
    「とりあえずマイクロサービス」にする
    チームが小さいうちはモノリスの方がシンプルで保守しやすい。Premature decompositionは避ける。
  • ⚠️
    セキュリティを設計から除外する
    認証・認可・暗号化は後付けではなく設計時から組み込む（Security by Design）。

💼 面接攻略Tips

Q: 面接でまず何をすべきか？
まず要件を確認する質問をする。「どのくらいのスケール？」「読み書き比率は？」「可用性の目標は？」—— これだけで面接官への印象が大きく変わる。
Q: 詳細を聞かれたら何を話すべきか？
「なぜその選択をしたか」「代替案と比較してどうか」「何がボトルネックか」を語る。知識の羅列ではなく設計思考のプロセスを見せる。
Q: 知らない技術が出てきたらどうする？
正直に「詳しくないが、こういう用途の技術だと理解している。代わりに〜という選択肢を考えている」と言う。Bluffingは逆効果。

🔬 深掘り: 見積もり演習（詳細版）

演習1: Instagramのような写真共有サービスの完全見積もり

与えられた条件をもとに、書き込みQPS・必要ストレージ・帯域幅を算出する。

前提条件
  • DAU = 500M（5億人）
  • 写真投稿ユーザー = 2%/日 → 500M × 0.02 = 10M 投稿/日
  • 1写真 = 500KB（サムネイル含む平均）
  • 動画投稿ユーザー = 5%/日のうち 10% が動画 → 500M × 0.05 × 0.1 = 2.5M 動画/日
  • 1動画 = 50MB（圧縮後平均）
// ===== 書き込みQPS =====
写真投稿QPS = 10M ÷ 86,400秒
           = 115.7 QPS  // ≈ 116 QPS

動画投稿QPS = 2.5M ÷ 86,400秒
           = 28.9 QPS  // ≈ 29 QPS

合計書き込みQPS ≈ 146 QPS
ピーク（×3）  ≈ 438 QPS

// ===== ストレージ（5年分） =====
写真ストレージ/日 = 10M × 500KB = 5 TB/日
動画ストレージ/日 = 2.5M × 50MB = 125 TB/日
合計/日           = 130 TB/日

5年分           = 130 TB × 365 × 5
               = 237,250 TB
               ≈ 237 PB

// ===== 読み込み帯域幅 =====
// 1ユーザーが1日20枚の写真を閲覧すると仮定
読み込みQPS   = 500M × 20枚 ÷ 86,400秒
             ≈ 115,740 QPS

帯域幅        = 115,740 × 500KB
             ≈ 57.9 GB/s  // ≈ 58 GB/s の読み込み帯域
解答のポイント
写真より動画の方がストレージへの影響が圧倒的に大きい（125TB vs 5TB）。動画の圧縮・解像度別エンコード戦略が設計の核心になる。読み込み帯域はCDNで分散させることが必須。

演習2: WhatsAppのメッセージング見積もり

前提条件
  • DAU = 2B（20億人）
  • 1人1日 = 50通送受信
  • テキストメッセージ = 100 bytes
  • メディア（画像・動画）= 全メッセージの5%、平均200KB
// ===== メッセージQPS =====
メッセージ数/日 = 2B × 50 = 100B（1000億通/日）

メッセージQPS  = 100B ÷ 86,400
              ≈ 1,157,407 QPS  // ≈ 1.16M QPS

// ===== テキストストレージ =====
テキスト/日   = 100B × 95% × 100 bytes
             = 9.5T bytes
             ≈ 9.5 TB/日

// ===== メディアストレージ =====
メディア数/日 = 100B × 5% = 5B 件
メディア容量  = 5B × 200KB
             = 1,000T bytes
             = 1 PB/日

// ===== 帯域幅（送受信合計） =====
テキスト帯域  = 9.5TB ÷ 86,400 ≈ 110 MB/s
メディア帯域  = 1PB ÷ 86,400  ≈ 11.6 GB/s
合計          ≈ 11.7 GB/s
設計上の注意点
1.16M QPS は単一サーバーでは到底処理できない。メッセージキュー（Kafka）+ シャーディング（ユーザーIDベース）が必須。メディアはオブジェクトストレージ（S3互換）に分離する。

🔬 深掘り: トレードオフの実例

Amazonカートの整合性問題

強一貫性を選んだ場合：

// 分散トランザクション（2PC）
BEGIN DISTRIBUTED TRANSACTION
  lock(cart_item)
  lock(inventory)
  update cart
  update inventory
COMMIT

// 問題: 在庫DBがダウン→カートも使えない
// 可用性 ↓、レイテンシ ↑（ロック待ち）

結果整合性を選んだ場合（Amazonの実際）：

// カートはDynamoDBで独立管理
// 在庫チェックはチェックアウト時のみ
cart.add(item_id, qty)  // 即時成功
// → 稀に「在庫切れ」エラーが発生するが
// → 99.99%の可用性を優先
教訓
ユーザー体験（カートが使える）を可用性で優先。在庫不整合のリスクはビジネスロジックで吸収。

Facebookタイムライン: モノリス→マイクロサービス

Facebookは2004年のPHP monolithから段階的に分解してきた。

2004〜2008: PHP + MySQL monolith。全機能が1コードベース。
2008〜2011: HipHop（PHP→C++トランスパイラ）で性能問題を解決しながらも単一コードベース維持。
2011〜2014: GraphQL/Thrift RPC で内部APIを分離。タイムライン生成を独立サービスに。
2014〜: React + Flux でフロントエンド分離。バックエンドは数百のマイクロサービスへ。
教訓
最初からマイクロサービスは不要。ユーザー規模・チーム規模が拡大してから分解する。

Netflixのカオスエンジニアリング: なぜ本番を意図的に壊すか

背景（2010年）：

Netflixが AWS に移行した際、依存関係の複雑化で「どこかが壊れた時に何が起きるか」が予測不能になった。本番障害で初めて問題を発見する事態が続いた。


Chaos Monkeyの誕生：

本番環境でランダムにEC2インスタンスを停止するツール。最初はエンジニアに嫌われたが、障害耐性が劇的に向上。

// Chaos Monkey の動作イメージ
class ChaosMonkey {
  run() {
    const instances = getRunningInstances()
    const target = randomPick(instances)

    // 営業時間内に意図的に停止
    // → エンジニアが対応できる時間に壊す
    target.terminate()

    alertTeam("Chaos Monkey struck: " + target.id)
  }
}
具体的な学び
Chaos Engineeringで発見した問題: タイムアウト設定漏れ、Retry storm、フォールバックなしの依存関係。これらを先手で修正できた。

🛠️ 実践演習: ホワイトボード設計チャレンジ

問題1: オンライン投票システムの設計

問題文
1000万人が24時間以内に投票するシステムを設計せよ。不正投票の防止、リアルタイム集計、99.99%の可用性が必要。

Step1: 要件定義

機能要件：

  • 1ユーザー = 1票のみ
  • 投票結果のリアルタイム表示
  • 投票受付の開始・終了時刻管理

非機能要件：

  • ピーク: 投票開始直後に集中 → 500K QPS
  • 不正防止: 同一ユーザーの2重投票を防ぐ
  • 可用性: 24時間以内に99.99% = 最大2.4分のダウンのみ許容

Step2: 見積もり

総投票数 = 10M
平均投票QPS = 10M ÷ 86,400 ≈ 116 QPS
ピークQPS（開始直後）= 500K QPS

// 投票データサイズ
1投票 = user_id(8B) + choice(1B)
      + timestamp(8B) = 17 bytes
総容量 = 10M × 17B = 170 MB
// → ストレージはほぼ問題なし

Step3: 設計解答

// 不正防止: Redis で冪等性チェック
SETNX voted:{user_id} 1 EX 86400
// → 既にキーが存在する場合 0 を返す → 2重投票拒否
// → アトミック操作なので競合なし

// アーキテクチャ
Client → API Gateway（レート制限: 1 req/user/日）
       → Vote Service（水平スケール × 50台）
       → Redis Cluster（冪等性チェック）
       → Kafka（投票イベントキュー）
       → Vote Counter（集計ワーカー）
       → PostgreSQL（永続化）
       → Redis（リアルタイム集計キャッシュ）
       → WebSocket Server → Browser

// リアルタイム集計
Kafkaで投票を受け取るたびに:
  HINCRBY vote_count choice_id 1
  // → 1ms以下でカウント更新 → WebSocketで配信
重要なトレードオフ
Redisの冪等性チェックと PostgreSQL への永続化を非同期にすることで、高いスループットを実現。Kafkaがバッファとなり、DBへの書き込みが遅れても投票は受け付けられる。ただし障害時には Kafka のオフセット管理で重複を防ぐ（exactly-once semantics）。

問題2: ファイルストレージサービス（Dropbox風）の設計

問題文
100M DAU のファイル同期・共有・バージョニングサービスを設計せよ。

核心技術1: チャンク分割

// 4MB チャンクに分割してアップロード
function uploadFile(file) {
  chunks = split(file, 4MB)
  for chunk in chunks:
    hash = SHA256(chunk)
    // 既存チャンクは再アップロード不要
    if not exists(hash):
      upload(S3, hash, chunk)
    db.record(file_id, hash, order)
}

核心技術2: デデュプリケーション

// 同じチャンクは1つだけ保存
// 1MBの変更 → 残り3チャンクは再送不要

// デルタ同期
old_chunks = [A, B, C, D]
new_chunks = [A, B, C', D]
// → C' のみアップロード
// → 転送量: 4MB → 4MB（1チャンク）

// バージョン管理
versions[file_id] = [
  {v:1, chunks:[A,B,C,D]},
  {v:2, chunks:[A,B,C',D]},
]
アーキテクチャ全体像
Client → Block Service（チャンク管理） → S3（オブジェクト保存）
Client → Metadata Service → MySQL（ファイルツリー・バージョン情報）
変更通知: Long polling / WebSocket → 他クライアントに同期トリガー送信

問題3: ニュースフィード（Facebook風）— Push vs Pull vs Hybrid

Push モデル（Fan-out on write）

投稿時に全フォロワーのフィードに書き込む

// 投稿時
followers = getFollowers(user_id)
for f in followers:
  feed[f].prepend(post_id)

// 読み込み時 → O(1)
return feed[user_id]

問題: フォロワー100万人のセレブが投稿したら？→ 書き込みストーム

Pull モデル（Fan-out on read）

読み込み時にフォロイーの投稿をマージ

// 読み込み時
followees = getFollowing(user_id)
posts = []
for f in followees:
  posts += getLatestPosts(f)
return sort(posts)

// 問題: 1000人フォロー→1000クエリ
// 読み込みが遅く、キャッシュ効率も悪い

問題: 1000人フォローしているユーザーのフィード生成が重い

Hybrid モデル（Facebookの実際の解決策）

// セレブ判定（フォロワー閾値: 例 10万人）
if follower_count(post.user) < 100_000:
  // 通常ユーザー: Push（Fan-out on write）
  fanOutToFollowers(post)
else:
  // セレブ: Pull（読み込み時にマージ）
  storeInCelebrityFeed(post)
  // 読み込み時: ユーザーフィード + フォロー中セレブの最新投稿をマージ

// 読み込み時
feed = redis.get("feed:" + user_id)  // 通常ユーザー分（Push済み）
for celeb in followingCelebrities(user_id):
  feed += getLatestPosts(celeb, limit=10)
return sort(feed)[:N]
判断基準のまとめ
フォロワーが少ない（一般ユーザー）→ Push が効率的。フォロワーが多い（セレブ・著名人）→ Pull に切り替え。閾値はシステム負荷に応じて動的に調整可能。
