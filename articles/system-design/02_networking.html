OSI / TCP IP モデル

なぜ層に分けるのか

ネットワーク通信の複雑さを管理するため、機能を独立した「層（レイヤー）」に分割します。各層は隣接する層とのみやり取りし、内部実装を隠蔽します。たとえばイーサネットをWi-Fiに変えても、上位層のHTTPは変更不要です。

カプセル化（Encapsulation）の流れ

送信時はデータが上位層から下位層へ渡され、各層がヘッダを付加します。受信時は逆に下位層から上位層へデヘッダを剥がしながら渡されます。

→
L3: IPヘッダ → パケット
→
L2: MACヘッダ → フレーム
→
L1: 電気/光信号
層PDU名称主なプロトコル識別子
L7 アプリケーションデータ / メッセージHTTP, HTTPS, DNS, SMTPURL, ドメイン名

  • 「L4ロードバランサーとL7ロードバランサーの違いは？」→ L4はIP/ポートで振り分け、L7はHTTPヘッダ・Cookie・URLパスで振り分け可能
  • ファイアウォールはL3/L4で動作、WAF（Web Application Firewall）はL7で動作
  • 実際の現場ではOSI 7層よりTCP/IP 4層モデルが多用される

HTTP の進化

HTTP/1.1 1997

Keep-Alive: 1つのTCP接続を複数リクエストで再利用。ハンドシェイクのオーバーヘッドを削減。

Head-of-Line Blocking: パイプライニングを使っても、先のレスポンスが来るまで次を処理できない致命的な問題。

ヘッダ冗長: 同一ドメインへの毎リクエストでCookieや大きなヘッダを繰り返し送信。

並列化の回避策: ブラウザは6接続/ホストに制限。スプライト・ドメインシャーディングで対処していた。

HTTP/2 2015

Multiplexing: 1つのTCP接続上で複数ストリームを並列処理。アプリケーション層のHOLブロッキングを解消。

HPACK圧縮: ヘッダをHuffman符号化＋動的テーブルで圧縮。

Server Push: クライアントのリクエスト前にサーバがリソースをプッシュ。

残課題: TCP層でのHOLブロッキングは残存。パケットロスで全ストリームが影響を受ける。

HTTP/3 RFC 9114 — 2022

HTTP/2の根本課題（TCP HOLブロッキング）を解決するため、QUIC（Quick UDP Internet Connections）プロトコル上に構築されました。

0-RTT / 1-RTT 接続確立: TLSハンドシェイクをQUICに統合。以前接続したサーバーには0-RTTで再接続可能。

ストリーム独立性: UDPベースのため、パケットロスが特定ストリームのみに影響。他ストリームは継続する。

接続マイグレーション: モバイルでWi-Fi → 4G切り替え時も接続IDで維持できる。

注意点: UDP 443番ポートを使用。企業ファイアウォールでブロックされる場合は自動的にHTTP/2へフォールバックする。

機能HTTP/1.1HTTP/2HTTP/3
転送プロトコル TCPTCP QUIC (UDP)
多重化 (Multiplexing) なし あり あり（改善）
ヘッダ圧縮 なし HPACKQPACK
TCP HOLブロッキング あり あり なし
接続確立RTT 1-RTT + TLS 1-RTT + TLS 0-RTT / 1-RTT
Server Push なし あり 限定的

💡 面接頻出ポイント

  • HTTP/2のmultiplexingとHTTP/1.1のパイプライニングの違いを説明できるか
  • HTTP/3がUDPを使う理由：TCPはカーネルに実装されておりHOL問題を修正できないため、QUICをユーザー空間に実装
  • 「なぜまだHTTP/1.1を使うサービスがあるか」→ 後方互換性、プロキシの問題、HTTP/2の実装コスト

TCP vs UDP

特性TCPUDP
接続確立 3ウェイハンドシェイク（1.5 RTT） コネクションレス（0 RTT）
信頼性 順序保証 + 再送制御 保証なし（ベストエフォート）
フロー制御 あり（ウィンドウサイズ） なし
輻輳制御 あり（Slow Start, AIMD） なし
ヘッダサイズ 20〜60バイト 8バイト（固定）
速度 比較的低速（オーバーヘッドあり） 高速
典型的ユースケース HTTP, HTTPS, FTP, SMTP, SSH DNS, ビデオ会議, ゲーム, QUIC

ユースケース別選択ガイド

ユースケース選択理由
Web API / RESTTCPデータの完全性が必須
ファイル転送TCP欠落したら再送が必要
ライブ映像ストリーミングUDP多少の欠落より低レイテンシ優先
オンラインゲーム（FPS等）UDP古いフレームより最新状態が重要
DNS クエリUDP小さいパケット、速度優先
VoIP / 音声通話UDPリアルタイム性優先（RTP over UDP）
金融トランザクションTCP整合性・順序が絶対必須

💡 面接頻出ポイント

  • 「TCPの3ウェイハンドシェイクを説明してください」→ SYN / SYN-ACK / ACK と各シーケンス番号の役割
  • 「ビデオ会議でTCPではなくUDPを使う理由」→ 再送で遅延が累積すると会話が成立しなくなるため
  • QUICはUDP上に信頼性・暗号化を独自実装。「UDP = 信頼性なし」は現代では必ずしも正確ではない

DNS と名前解決

DNS とは

DNS（Domain Name System）はインターネットの「電話帳」です。人間が読める www.example.com をコンピュータが通信に使う 93.184.216.34（IPアドレス）に変換します。分散階層型のデータベースとして設計されており、単一障害点を排除しています。

主な DNS レコードタイプ

レコード用途
Aドメイン → IPv4アドレス
AAAAドメイン → IPv6アドレス
CNAME別名 → 別のドメイン
MXメールサーバ指定
TXTテキスト情報（SPF, DKIM等）
NS権威DNSサーバ指定
PTR逆引き（IP → ドメイン）

TTL とキャッシュ戦略

TTL（Time To Live）はDNSレコードがキャッシュされる秒数です。

  • TTL短め（60〜300秒）: 変更を素早く反映できる。クエリ数が増加しコスト増。
  • TTL長め（3600〜86400秒）: キャッシュ効率が高い。変更反映に時間がかかる。
  • CDN切替・ブルーグリーンデプロイ前はTTLを事前に短くするのが定石。
ℹ️
障害復旧時は低TTL（60秒）にしておくと、フェイルオーバーが迅速になります。

💡 面接頻出ポイント

  • 「DNSの名前解決の流れを説明してください」→ ブラウザキャッシュ → OS → リゾルバ → ルート → TLD → 権威DNSの順
  • 再帰クエリ（クライアントが1回問い合わせる）と反復クエリ（リゾルバが各サーバに逐次問い合わせる）の違い
  • Anycastでルートサーバを世界中に分散させることで、単一障害点と低レイテンシを両立している

CDN（Content Delivery Network）

CDN とは

世界各地のエッジサーバーにコンテンツをキャッシュし、ユーザーに最も近いサーバーから配信するネットワークです。レイテンシを削減しつつ、オリジンサーバーの負荷を大幅に軽減します。

Push CDN vs Pull CDN

Push CDNPull CDN
仕組み事前にコンテンツをアップロード初回リクエスト時にオリジンから取得
更新手動/自動でCDNへ配信TTL切れ or Purge後に自動更新
向いている静的ファイル、動画、大容量ほとんどのWebサイト
ストレージ事前に全て格納（コスト大）必要分のみキャッシュ

キャッシュ制御ヘッダ

# 1日キャッシュ（静的アセット）
Cache-Control: public, max-age=86400

# キャッシュ禁止（APIレスポンス）
Cache-Control: no-store, no-cache

# バックグラウンドで再検証
Cache-Control: stale-while-revalidate=60

# ブラウザ1時間、CDN1日
Cache-Control: max-age=3600, s-maxage=86400

動的コンテンツの CDN 活用

  • エッジキャッシュ（短TTL）: ニュース記事・商品情報など頻繁に変わるが全ユーザー共通のレスポンス
  • エッジコンピューティング: Cloudflare Workers / Lambda@Edge でオリジンの代わりにエッジで処理
  • Vary ヘッダ: Accept-Language や Accept-Encoding 別にキャッシュを分岐させる
  • バイパス条件: Cookie やクエリパラメータを含む個人化された動的 API は CDN をバイパスし直接オリジンへ
  • Cache-Tag / Surrogate-Key: コンテンツのグループ単位（商品IDなど）でのキャッシュ Purge が可能

💡 面接頻出ポイント

  • 「CDNキャッシュの更新方法」→ ① TTL満了 ② Purge API ③ ファイル名にhashを付与するキャッシュバスティング
  • 「動的コンテンツをCDNでキャッシュできるか？」→ 短TTL・Cache-Tagによる選択的Purge・エッジコンピューティングの組み合わせで対応可能
  • PoPとエッジサーバーの違い：PoPは物理的なデータセンター拠点、エッジサーバーはその中の実サーバー

リアルタイム通信

方式プロトコル方向レイテンシサーバ負荷主なユースケース
Short PollingHTTPクライアント主導 高（間隔依存） 高（無駄リクエスト） シンプルな状態確認
Long PollingHTTPクライアント主導 中中 通知、古いブラウザ対応
WebSocketws:// / wss:// 双方向 低 中（接続維持コスト） チャット, ゲーム, コラボ編集
SSEHTTP (text/event-stream) サーバ → クライアント 低 低 通知, ライブフィード, AI応答

WebSocket 実装例

// クライアント (JavaScript)
const ws = new WebSocket('wss://api.example.com/ws');

ws.onopen = () => {
  ws.send(JSON.stringify({
    type: 'join', room: 'lobby'
  }));
};

ws.onmessage = (e) => {
  const data = JSON.parse(e.data);
  updateUI(data);
};

// 自動再接続
ws.onclose = () =>
  setTimeout(reconnect, 3000);

SSE 実装例 (Node.js)

app.get('/stream', (req, res) => {
  res.setHeader('Content-Type',
    'text/event-stream');
  res.setHeader('Cache-Control','no-cache');
  res.setHeader('Connection','keep-alive');

  const send = (d) =>
    res.write(`data: ${JSON.stringify(d)}\n\n`);

  const iv = setInterval(() =>
    send({ token: nextToken() }), 50);

  req.on('close', () =>
    clearInterval(iv));
});
💡
ChatGPT / Claude のストリーミング応答はSSEを使用しています。トークンが生成されるたびにSSEイベントとして送信し、クライアントが順次表示します。WebSocketではなくSSEが選ばれる理由は、一方向通信で十分かつHTTPベースでCDN・プロキシ互換性が高いためです。

💡 面接頻出ポイント

  • 「WebSocketとSSEをいつ使い分けるか？」→ 双方向が必要（チャット・ゲーム）はWS、サーバーからの一方向通知（ライブフィード・AI応答）はSSE
  • 「Long PollingとWebSocketの違い」→ LPはHTTP接続を都度張り直す、WSは1回の接続を維持し続ける
  • WebSocketはHTTP Upgradeヘッダで接続を切り替え（TCPは同じ接続を継続使用）

TLS / HTTPS の仕組み

TLS とは

TLS（Transport Layer Security）はHTTP通信を暗号化し、盗聴・改ざん・なりすましを防ぐプロトコルです。HTTPS = HTTP over TLS です。現在の標準はTLS 1.3（RFC 8446、2018年）です。

暗号化の種類

種類鍵用途例
対称暗号同一鍵データ暗号化（高速）AES-256-GCM
非対称暗号公開鍵/秘密鍵鍵交換・デジタル署名RSA, ECDSA
鍵交換—安全な共通鍵合意ECDHE（楕円曲線DH）
ハッシュ—整合性確認SHA-256, SHA-384

証明書チェーン

🔒 ルートCA証明書（ブラウザ/OSに組み込み済み）
↓ 署名で信頼を伝搬
🔒 中間CA証明書（Let's Encrypt 等）
↓ 署名で信頼を伝搬
🔒 サーバー証明書（example.com）

ブラウザはルートCAの署名を検証し、チェーンを辿ってサーバー証明書の正当性を確認します。

# Nginx TLS 1.2/1.3 設定例
server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # TLS 1.3 推奨 (1.2も後方互換で残す)
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers   ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers off;

    # HSTS: 1年間 HTTPS を強制
    add_header Strict-Transport-Security
      "max-age=31536000; includeSubDomains; preload" always;

    # OCSP Stapling でレスポンス高速化
    ssl_stapling on;
    ssl_stapling_verify on;
}

💡 面接頻出ポイント

  • 「TLSハンドシェイクで何を決めるか」→ ① TLSバージョン ② 暗号スイート ③ セッション鍵（ECDHE経由）
  • 「なぜ対称・非対称暗号を組み合わせるか」→ 非対称暗号は遅いため鍵交換のみに使い、実データは高速な対称暗号で処理する
  • Perfect Forward Secrecy（PFS）: ECDHE使用時、秘密鍵が漏洩しても過去の通信を復号できない
  • TLS 1.3はTLS 1.2比で1-RTT削減（ハンドシェイクのラウンドトリップが1回少ない）

レイテンシとスループット

レイテンシ（Latency）

リクエスト送信から最初のバイト受信までの時間。ユーザー体験に直結します。

  • RTT（Round Trip Time）: 往復時間
  • TTFB（Time To First Byte）: 最初のバイトを受け取るまでの時間
  • P50 / P95 / P99: パーセンタイルレイテンシ。P99はテール（外れ値）ユーザー体験を示す

スループット（Throughput）

単位時間当たりに処理できるデータ量またはリクエスト数。

  • QPS / RPS: Queries / Requests Per Second
  • 帯域幅: bps（bits per second）
  • TPS: Transactions Per Second
  • Little's Law: L = λ × W（同時接続数 = 到着率 × 平均応答時間）
操作目安レイテンシ規模感
L1キャッシュ参照~0.5 ns光が0.15m進む時間
L2キャッシュ参照~7 nsL1の14倍
メインメモリ (RAM) アクセス~100 nsL1の200倍
SSD ランダム読み取り~100 µs (0.1ms)メモリの1000倍
HDD シーク時間~10 msSSDの100倍
同一DC内ネットワーク RTT~0.5 msローカルネットワーク

ネットワーク最適化テクニック

Keep-Alive: TCP接続を再利用しハンドシェイクのオーバーヘッドを削減。HTTP/1.1以降デフォルト有効。

# Nginx upstream Keep-Alive
upstream backend {
  server 127.0.0.1:3000;
  keepalive 32;
  keepalive_requests 1000;
  keepalive_timeout 60s;
}

Connection Pooling: 接続を事前に確保し都度接続のコストを排除。DBやHTTPクライアントで必須。

// Node.js HTTPS エージェント
const agent = new https.Agent({
  keepAlive: true,
  maxSockets: 50,
  maxFreeSockets: 10,
  timeout: 60000,
});

手法効果実装例

⚠️
レイテンシとスループットはトレードオフ — バッファリング（大量データをまとめて送る）はスループットを上げますがレイテンシが増します。ストリーミング（小さいチャンクを逐次送る）は低レイテンシですがスループットが下がることがあります。ユースケースに応じてバランスを取ることが重要です。

💡 面接頻出ポイント

  • 「P99レイテンシとは？」→ 99%のリクエストがこの時間以内に完了する。テールレイテンシはユーザー体験の最悪ケースを評価するのに重要
  • 「東京から米国西海岸のRTTは？」→ 約100–150ms。光速のみで計算すると最低約65ms（9000km ÷ 光速 × 2）
  • 「Connection Poolingが重要な理由」→ DBへの接続確立は100–300msかかることがあり、毎回接続するとレイテンシが爆発する
  • 「CDNの主なパフォーマンス改善効果3つ」→ ① 地理的近接化 ② キャッシュでオリジン負荷削減 ③ エッジでのTCP接続事前確立

📋 Chapter 02 まとめ

  • OSI / TCP IP: L4がTCP/UDP、L3がIP、L7がアプリケーション
  • HTTP/3: QUICでTCP HOLブロッキングを完全解消、0-RTT接続
  • TCP: 信頼性重視。UDP: 速度重視（ゲーム・動画）
  • DNS: TTLでキャッシュ。権威DNSが最終的な答えを持つ

  • レイテンシ: 東京↔米国は約100–150ms（光速限界は約65ms）

深掘り: HTTP/2 Multiplexingの実際の動作

ストリームIDとフレームの構造（バイナリプロトコル詳細）

HTTP/2はHTTP/1.1のテキストベースプロトコルをバイナリフレームに置き換えた。フレームはすべて9バイトのヘッダで始まる。

// HTTP/2 フレーム構造（9バイトヘッダ）
+-----------------------------------------------+
|                 Length (24 bits)               |
+---------------+---------------+---------------+
|   Type (8)    |   Flags (8)   |
+-+-------------+---------------+
|R|          Stream ID (31)     |
+=+=============+===============+
|         Frame Payload         |
+-----------------------------------------------+

// 主要フレームタイプ
DATA         = 0x0  // 実際のデータ
HEADERS      = 0x1  // HTTPヘッダ
PRIORITY     = 0x2  // ストリーム優先度
RST_STREAM   = 0x3  // ストリームのリセット
SETTINGS     = 0x4  // 接続設定
PUSH_PROMISE = 0x5  // Server Push予告
PING         = 0x6  // 生存確認
GOAWAY       = 0x7  // 接続終了
WINDOW_UPDATE= 0x8  // フロー制御

// Multiplexing: 複数リクエストを1TCP接続で並行送信
Stream 1: [HEADERS] → [DATA chunk1] → [DATA chunk2 END_STREAM]
Stream 3: [HEADERS] → [DATA chunk1] → [DATA chunk2 END_STREAM]
Stream 5: [HEADERS] → [DATA chunk1 END_STREAM]
// 奇数ストリームID = クライアント起点
// 偶数ストリームID = サーバー起点（Server Push）

HPACK: Header Compression（静的テーブル + 動的テーブル）

HTTP/2ではヘッダを圧縮してオーバーヘッドを削減する。HPACKは2種類のテーブルを使う。

// 静的テーブル（61エントリ、仕様に固定）
Index  Name                Value
  1    :authority
  2    :method             GET
  3    :method             POST
  4    :path               /
  7    :scheme             https
 13    :status             200
 ...

// 動的テーブル（接続ごとに構築）
// 初回リクエスト: フルヘッダを送信しテーブルに追加
":method: GET"        → Index 2  (静的)
"user-agent: Chrome" → Index 62 (動的テーブルに追加)

// 2回目以降: インデックス番号だけ送信
// 例: user-agent は 1バイト（0x3E = 62）だけ送れば済む
// → 典型的なAPIリクエストで 85〜88% のヘッダ圧縮率

HTTP/2でも残るHoL Blocking → HTTP/3がUDPに移行した理由

HTTP/2のストリーム多重化はアプリケーション層のHoLブロッキングを解決したが、TCP層の問題が残った。

// TCP上のHTTP/2: パケットロス発生時
Stream 1: [frame1] [frame2] [LOST] [frame4] ...
Stream 3: [frame1] [frame2] [frame3] ...
Stream 5: [frame1] [frame2] [frame3] ...

// TCPは順序保証のためLOSTパケットの再送まで
// Stream 3,5も含めて全ストリームが停止!
// → TCP Head-of-Line Blocking

// HTTP/3 (QUIC over UDP) の解決策
// QUICは各ストリームを独立して管理
QUIC Stream 1: [frame1] [frame2] [LOST] [frame4]
              // → Stream 1 だけが再送待ち
QUIC Stream 3: [正常に進む]  // 影響なし!
QUIC Stream 5: [正常に進む]  // 影響なし!

// さらに QUIC は TLS 1.3 を組み込み
// → ハンドシェイクが 1-RTT（新規）/ 0-RTT（再接続）
Server Pushが廃止方向になった理由
Server PushはクライアントがキャッシュしているリソースもPushしてしまう問題があった。103 Early Hints（Link preloadヘッダ）が代替手段として普及中。

深掘り: TLSハンドシェイク詳細（TLS 1.2 vs 1.3）

TLS 1.2: 2-RTTハンドシェイク

// RTT1: 暗号スイート交渉
C→S: ClientHello
     (supported ciphers, TLS version,
      client random, SNI)
S→C: ServerHello
     (chosen cipher, server random)
S→C: Certificate (公開鍵証明書)
S→C: ServerHelloDone

// RTT2: 鍵交換
C→S: ClientKeyExchange
     (pre-master secret, RSA暗号化 or
      DH公開鍵)
C→S: ChangeCipherSpec
C→S: Finished (MAC)

S→C: ChangeCipherSpec
S→C: Finished (MAC)

// 計: 2-RTT + TCP 1-RTT = 3-RTT
// RSA鍵交換は PFS なし（秘密鍵漏洩で過去の通信も復号可能）

TLS 1.3: 1-RTT ＆ 0-RTT再接続

// RTT1: 鍵交換と暗号化を同時に
C→S: ClientHello
     (key_share: DH公開鍵,
      supported groups, ciphers,
      PSK identity [再接続時])
S→C: ServerHello
     (key_share: DH公開鍵応答)
S→C: {EncryptedExtensions}
S→C: {Certificate}
S→C: {CertificateVerify}
S→C: {Finished}

// 接続確立!（1-RTT）
C→S: {Finished}
C→S: {Application Data}

// 0-RTT（再接続時）
C→S: ClientHello + EarlyData
// リプレイ攻撃リスクあり → 冪等GETのみ推奨

// TLS 1.3 の廃止された機能
// × RSA鍵交換（PFSがない）
// × RC4, DES, 3DES
// × SHA-1証明書
// × 圧縮（CRIME攻撃対策）

実際の確認コマンド（openssl s_client）

# TLSバージョンと証明書を確認
openssl s_client -connect example.com:443 -tls1_3 </dev/null 2>&1 | grep -E "Protocol|Cipher|issuer"

# 出力例
Protocol  : TLSv1.3
Cipher    : TLS_AES_128_GCM_SHA256
issuer=/C=US/O=Let's Encrypt/CN=R3

# サポートされる CipherSuite を列挙
openssl ciphers -v 'TLSv1.3'

# Server Name Indication (SNI) を指定して接続
openssl s_client -connect 93.184.216.34:443 -servername example.com

# 証明書チェーンの検証
openssl s_client -connect example.com:443 -verify_return_error 2>&1 | tail -5

# TLS 1.3 の 0-RTT 対応確認（curl）
curl --http3 -I https://example.com -v 2>&1 | grep -E "TLS|QUIC|Early"
PFS（Perfect Forward Secrecy）とは
サーバーの長期秘密鍵が漏洩しても、過去の通信を復号できない性質。TLS 1.3ではすべての鍵交換がDH（Diffie-Hellman）ベースとなり、セッションごとに一時鍵を生成するためPFSが強制される。

深掘り: DNS 完全解説

DNSレコードの全種類と用途

A / AAAA

ドメイン → IPv4(A) / IPv6(AAAA) アドレスへのマッピング。最も基本的なレコード。

example.com. A 93.184.216.34
example.com. AAAA 2606:2800:220:1:248:1893:25c8:1946

CNAME

別名（エイリアス）。CDNで使用される。CNAME先を変えるだけでIPを切り替えられる。

www.example.com. CNAME d1234.cloudfront.net.
# ルートドメインにはCNAME不可 → ALIAS/ANAMEレコード

MX

メールサーバー。優先度（priority）が低いほど優先される。

example.com. MX 10 mail1.example.com.
example.com. MX 20 mail2.example.com.

TXT

テキスト情報。SPF/DKIM/DMARECメール認証、ドメイン所有権確認に使用。

example.com. TXT "v=spf1 include:_spf.google.com ~all"
_dmarc.example.com. TXT "v=DMARC1; p=reject"

NS / SOA

NS: ゾーンのネームサーバー。SOA: ゾーン管理情報（シリアル番号・TTLデフォルト）。

SRV / PTR

SRV: サービス発見（Kubernetes内部DNS等）。PTR: 逆引き（IP→ドメイン）。

_https._tcp.example.com. SRV 0 5 443 server.example.com.

DNSキャッシュポイゾニング と DNSSEC

// 攻撃の仕組み（Kaminsky攻撃 2008年）
1. 攻撃者が example.com の偽回答を大量送信
2. リゾルバが正規のクエリを送った直後に
   偽の UDP応答（送信元:53）を大量に試みる
3. トランザクションID(16bit)が一致すると
   偽のAレコードがキャッシュに書き込まれる
4. ユーザーが evil-server.com のIPに誘導される

// DNSSEC による対策
// 各レコードに電子署名(RRSIG)を付与
example.com. A 93.184.216.34
example.com. RRSIG A 8 2 3600 (
  20231231000000 20231201000000 12345 example.com.
  base64encodedSignature... )

// 検証チェーン: ルートゾーン → TLD → ドメイン
// 偽の回答は署名が合わないため拒否される

// DNS over HTTPS (DoH)
GET https://cloudflare-dns.com/dns-query?name=example.com&type=A
// → DNS クエリをHTTPS暗号化 → ISPによる盗聴・改ざん防止
// → Firefox, Chrome が標準搭載
GeoDNS + エニーキャストの組み合わせ
CDNはGeoDNSで地域ごとに異なるIPを返し、さらにそのIPがエニーキャストで最寄りのPoPサーバーにルーティングされる。例: Cloudflareの 1.1.1.1 は世界中の270+のデータセンターで同じIPアドレスを持つ。

設計問題2: WebSocketライブスコア配信システム

問題文
スポーツスコアのリアルタイム配信。100万人が同時接続、スコア更新は1分に数回、接続断時の自動再接続が必要。 
// 接続管理のアーキテクチャ
Client
  ↓ WebSocket (ws://scores.example.com)
Load Balancer (Sticky Session: IP Hash or Cookie)
  ↓                    ↓
WebSocket Server A    WebSocket Server B ...
  ↓ Subscribe         ↓ Subscribe
            Redis Pub/Sub
              ↑ Publish
         Score Updater Service
              ↑
         スポーツデータAPI

// スケールアウト: WebSocketサーバーの水平展開
// 問題: サーバーをまたいだブロードキャストが難しい
// 解決: Redis Pub/Subで全サーバーに配信

// WebSocket サーバー実装（Node.js / pseudocode）
const redis = createClient()
const subscriber = redis.duplicate()

subscriber.subscribe('score:updates', (message) => {
  // このサーバーに接続中の全クライアントにブロードキャスト
  connectedClients.forEach(ws => ws.send(message))
})

// クライアント側: 自動再接続
function connectWithRetry() {
  const ws = new WebSocket('wss://scores.example.com')
  ws.onclose = () => {
    // Exponential backoff: 1s, 2s, 4s, 8s, 最大30s
    setTimeout(connectWithRetry, Math.min(delay * 2, 30000))
  }
  ws.onmessage = (e) => updateScore(JSON.parse(e.data))
}

// 見積もり
接続数: 1M 同時接続
メモリ/接続: ~10KB (バッファ込み)
必要メモリ: 1M × 10KB = 10GB → 5台 × 2GB = 10台で冗長化
更新頻度: 10回/分 × メッセージ200B = 2KB/分 → 帯域は軽微
WebSocket vs SSE vs Long Polling の比較
WebSocket: 双方向・最低レイテンシ。SSE (Server-Sent Events): サーバー→クライアント一方向だが HTTP/2対応・自動再接続付き、スコア配信には最適解の一つ。Long Polling: レガシー環境向け、効率は低い。
