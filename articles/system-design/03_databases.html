Section 01

SQL vs NoSQL

どちらを選ぶか ― ユースケース別の判断基準

観点 SQL (リレーショナル) NoSQL
データ構造スキーマ固定、テーブル・行・列スキーマレス or 柔軟（JSON、KV、グラフ等）
クエリ言語標準 SQL — 強力な JOIN・集計DB 固有 API / 限定的なクエリ能力
トランザクション完全 ACID 保証多くは BASE（一部は ACID 対応）
スケーリング主に垂直スケール（水平は難しい）水平スケール前提で設計
一貫性強一貫性結果整合性が多い
代表製品PostgreSQL, MySQL, Oracle, SQLiteMongoDB, Redis, Cassandra, DynamoDB
得意なユースケース金融・決済・ERP・OLTPSNS・IoT・リアルタイム分析・大量書き込み
選択の黄金律

「NoSQL の方が速い」は誤解。データ構造・アクセスパターン・一貫性要件で選ぶ。多くのシステムは SQL + NoSQL を組み合わせる Polyglot Persistence を採用する。

NoSQL の 4 タイプ

Document

ドキュメント型

JSON / BSON 形式でデータを保存。ネストした構造をそのまま格納できる。スキーマ変更が容易。

代表: MongoDB, CouchDB, Firestore
✓ カタログ, CMS, ユーザープロファイル
Key-Value

キー・バリュー型

最もシンプルな構造。キーで値を O(1) 取得。超高速だが複雑なクエリは不可。

代表: Redis, DynamoDB, Memcached
✓ セッション管理, キャッシュ, リーダーボード
Wide Column

ワイドカラム型

行キー + 列ファミリーでデータを整理。大量の書き込みと時系列データが得意。

代表: Cassandra, HBase, Bigtable
✓ IoT センサー, ログ, 時系列分析
Graph

グラフ型

ノードとエッジでデータを表現。多段リレーション探索が極めて高速。

代表: Neo4j, Amazon Neptune, JanusGraph
✓ SNS の友人関係, レコメンド, 不正検知
Section 02

ACID 特性と BASE

トランザクション保証の原則と結果整合性の世界

A — Atomicity（原子性）

すべて成功 or すべて失敗

トランザクション内の操作は不可分。銀行振込: 送金元の引き落としと送金先への入金は両方成功するか両方ロールバックする。途中状態は存在しない。

SQL 例
BEGIN;
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
COMMIT; -- 両方成功時のみ確定
C — Consistency（一貫性）

制約を常に満たす

トランザクション前後でデータは必ず整合状態を維持する。口座残高は 0 以上という制約があれば、残高不足の引き落としは拒否される。

  • 外部キー制約
  • UNIQUE / NOT NULL 制約
  • CHECK 制約（残高 >= 0 など）
  • アプリレベルのビジネスルール
I — Isolation（分離性）

並行トランザクションの相互影響を防ぐ

複数トランザクションが並行実行されても、それぞれが独立して実行されたように見える。分離レベル（READ COMMITTED, REPEATABLE READ, SERIALIZABLE）で調整。

分離レベルDirty ReadNon-RepeatablePhantom
Read Uncommitted×××
Read Committed✓××
Repeatable Read✓✓×
Serializable✓✓✓
D — Durability（永続性）

コミット後のデータは失われない

コミットされたトランザクションは、電源断やクラッシュが発生してもデータが保持される。WAL（Write-Ahead Log）により実現。

  • WAL（先行書き込みログ）
  • fsync によるディスク同期
  • redo ログによるリカバリ
  • クラッシュリカバリプロセス

BASE ― NoSQL の哲学

BA

Basically Available

システムは常に応答を返す。障害時も部分的に利用可能。SLA よりも可用性を優先する設計。

S

Soft State

入力がなくても時間経過で状態が変化しうる。ノード間の同期が進行中のため、瞬間的な不整合が存在しうる。

E

Eventual Consistency

更新後、一定時間経過すれば全レプリカが最終的に同じ値になる。DNS の伝播やSNSのいいね数が典型例。

CAP 定理

分散システムは Consistency（一貫性）・Availability（可用性）・Partition Tolerance（分断耐性） の 3 つを同時に満たせない。ネットワーク分断は避けられないため、CP（一貫性重視: HBase, Zookeeper）か AP（可用性重視: Cassandra, DynamoDB）かを選択することになる。

Section 03

インデックス

検索を劇的に高速化する仕組みと落とし穴

B-Tree Index

B-Tree インデックス

最も一般的。範囲検索・等値検索・ORDER BY に対応。木の高さが O(log n) なので大規模でも安定したパフォーマンス。

  • = 、<、>、BETWEEN、LIKE 'abc%'
  • ORDER BY / GROUP BY の最適化
  • PostgreSQL / MySQL のデフォルト
Hash Index

Hash インデックス

等値検索 (=) のみ対応。O(1) で非常に高速だが、範囲検索は不可。メモリ上のハッシュテーブル構造。

  • = のみ（<, > は使えない）
  • ORDER BY に使えない
  • Redis の内部実装に活用
Composite

複合インデックス

複数カラムを組み合わせたインデックス。左端のカラムから順に有効（最左プレフィックス原則）。

-- (a, b, c) の複合インデックス
-- 有効: WHERE a=1
-- 有効: WHERE a=1 AND b=2
-- 無効: WHERE b=2 AND c=3

クラスタードインデックス vs ノンクラスタードインデックス

クラスタードインデックス

インデックスの順序 = データの物理的な格納順序。テーブルに 1 つだけ作れる。PRIMARY KEY が通常これになる。範囲スキャンが高速。

注意

PRIMARY KEY を UUID にすると挿入時に物理ページの再配置が頻発し書き込みが遅くなる（ページスプリット）。連番 (SERIAL / AUTO_INCREMENT) が推奨。

ノンクラスタードインデックス

インデックスとデータは別領域。テーブルに複数作れる。等値検索に強いが、実データへのポインタ参照（ブックマーク参照）が追加コスト。

カバリングインデックス

クエリで必要な全カラムをインデックスに含めると、テーブル本体へのアクセスが不要になる（Index Only Scan）。非常に高速。

-- クエリ
SELECT email, name FROM users WHERE email = 'a@b.com';
-- カバリングインデックス
CREATE INDEX idx_cover ON users (email, name);
インデックスの弊害 ― 書き込みオーバーヘッド

インデックスは SELECT を高速化する一方、INSERT / UPDATE / DELETE のたびにインデックスツリーも更新が必要。インデックスが多いほど書き込みが遅くなる。書き込みヘビーなテーブルでは最低限のインデックスに留めること。通常は 5〜6 個が上限の目安。

インデックス作成例 (PostgreSQL)
-- 通常のインデックス
CREATE INDEX idx_users_email ON users (email);

-- 複合インデックス（order_date が先 → 範囲スキャンに有利）
CREATE INDEX idx_orders_date_user ON orders (order_date, user_id);

-- 部分インデックス（条件付き — アクティブユーザーのみ）
CREATE INDEX idx_active_users ON users (email) WHERE is_active = true;

-- 関数インデックス（大文字小文字を無視した検索）
CREATE INDEX idx_lower_email ON users (LOWER(email));
Section 04

データベースレプリケーション

高可用性と読み取りスケールアウトを実現するアーキテクチャ

同期 vs 非同期レプリケーション

同期レプリケーション

Primary がコミットを確定する前に、Replica が受信・書き込みを確認するまで待機する。

  • データロスなし（RPO = 0）
  • 強一貫性を保証
  • Replica の遅延が書き込みレイテンシに直結
  • Replica 障害で Primary も停止リスク
PostgreSQL: synchronous_standby_names で設定

非同期レプリケーション

Primary はコミット後すぐに応答を返す。WAL は非同期でバックグラウンド送信される。

  • 書き込みレイテンシが低い
  • Replica 障害の影響を受けない
  • フェイルオーバー時にデータロスの可能性
  • Replica から古いデータを読む可能性（Replication Lag）
MySQL: デフォルトが非同期。Semi-sync もオプションで可能
Replication Lag の問題

非同期レプリケーションでは、書き込み直後に Replica から読み取ると古いデータが返ることがある（Read-Your-Writes 問題）。対策: ① 書き込み直後の読み取りは Primary へ、② 読み取り時にタイムスタンプチェック、③ Sessions Consistency ミドルウェア。

Section 05

シャーディング (Sharding)

データを水平分割して書き込みをスケールアウトする

シャーディングとは

単一 DB では捌けない大量データ/書き込みに対応するため、データを複数の DB インスタンス（シャード）に水平分割する手法。各シャードは独立したデータを保持する。レプリケーション（同じデータのコピー）とは異なる。

シャーディング戦略の比較

方式仕組みメリットデメリット適用例
Hash hash(key) % N でシャード決定 均一な分散、ホットスポットが発生しにくい シャード追加時に大規模なデータ移動が必要 ユーザーデータ、メッセージ
Range キーの値域でシャード分割
(例: A-M, N-Z)		 範囲クエリが効率的、追加が容易 アクセス集中（ホットスポット）が発生しやすい 時系列データ、アーカイブ
Directory ルックアップテーブルでシャードを管理 柔軟な割り当て変更が可能 ルックアップが単一障害点、追加レイテンシ マルチテナント、地域分割
Consistent Hash ハッシュリングで仮想ノード管理 シャード追加時のデータ移動が最小限 実装が複雑、不均一になりやすい Cassandra, Amazon DynamoDB
ホットスポット問題

シャードキーの選択が悪いと特定のシャードに負荷が集中する。例: 「作成日時」をシャードキーにすると最新シャードに書き込みが集中。「ユーザーID の余り」など カーディナリティが高く均一に分散するキーを選ぶ。有名人の投稿のような例外は追加のシャードを割り当てる（Celebrity Problem）。

シャーディングの制約

シャードをまたぐ JOIN は非常にコスト高（アプリ側で結合）。分散トランザクションは複雑。シャードキーを後から変更するのは困難。シャーディングは最後の手段。まずはレプリケーション、キャッシュ、クエリ最適化で対処する。

Section 06

接続プール (Connection Pooling)

DB 接続コストを劇的に削減する仕組み

なぜ必要か

DB への TCP 接続確立には通常 20〜100ms かかる（TCP ハンドシェイク + SSL + 認証）。リクエストごとに接続していては高負荷時にボトルネックとなる。接続プールは確立済みの接続を再利用してこのコストをゼロにする。

pgBouncer

PostgreSQL 専用の軽量プーキサ。3 種のモード:

  • Transaction: トランザクション単位で接続を使い回す（推奨）
  • Session: 接続を専有
  • Statement: SQL 文単位
10,000 クライアント → 100 DB 接続に集約可能

HikariCP

Java / JVM 向けの超高速接続プール。Spring Boot のデフォルト。ベンチマークで他を圧倒する性能。

# application.yml
spring.datasource.hikari:
  maximum-pool-size: 10
  minimum-idle: 5
  connection-timeout: 30000
  idle-timeout: 600000

適切なプールサイズ

プールが大きすぎても DB 側が詰まる。計算式:

pool_size = Tdb / Tquery
Tdb=処理時間, Tquery=クエリ時間

目安: cores × 2 + effective_spindle_count（PostgreSQL 推奨公式）

pgBouncer 設定例 (pgbouncer.ini)
[databases]
mydb = host=127.0.0.1 port=5432 dbname=mydb

[pgbouncer]
listen_port = 6432
listen_addr = *
auth_type = md5
pool_mode = transaction    # Transaction mode 推奨
max_client_conn = 10000    # アプリからの最大接続数
default_pool_size = 100    # DB への実接続数
min_pool_size = 10
reserve_pool_size = 5
server_idle_timeout = 600
Section 07

クエリ最適化

EXPLAIN の読み方、N+1 問題、JOIN 最適化

EXPLAIN / EXPLAIN ANALYZE の読み方

PostgreSQL EXPLAIN ANALYZE
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 123 AND status = 'pending';

-- 出力例
Seq Scan on orders  (cost=0.00..15420.00 rows=12 width=128)
                    (actual time=0.045..234.123 rows=8 loops=1)
  Filter: ((user_id = 123) AND (status = 'pending'))
  Rows Removed by Filter: 892451
Planning Time: 0.312 ms
Execution Time: 234.456 ms

注目すべきキーワード

  • Seq Scan — テーブル全件スキャン（インデックス未使用）
  • Index Scan — インデックス使用（良い）
  • Index Only Scan — カバリングインデックス（最速）
  • Bitmap Heap Scan — 複数インデックスを合成
  • Hash Join — 大テーブル同士の JOIN
  • Nested Loop — 片方が小さい場合に有効

改善後のクエリプラン

-- インデックス追加
CREATE INDEX idx_orders_uid_status
  ON orders (user_id, status);

-- 再 EXPLAIN の結果
Index Scan using idx_orders_uid_status
  on orders
  (cost=0.43..8.47 rows=8 width=128)
  (actual time=0.031..0.048
   rows=8 loops=1)
Execution Time: 0.089 ms  -- 2600x 改善!

N+1 問題とその解決

悪い例: N+1 クエリ

# Python ORM の例（悪い）
users = User.objects.all()     # クエリ 1 回
for user in users:
    # ユーザーごとに 1 クエリ → N 回!
    orders = user.orders.all()

# 1000 ユーザー → 1001 クエリ発行!

良い例: Eager Loading / JOIN

# Django: select_related / prefetch_related
users = User.objects\
  .prefetch_related('orders').all()

-- または SQL JOIN
SELECT u.*, o.*
FROM users u
LEFT JOIN orders o
  ON u.id = o.user_id;
-- クエリ 1 回で完結!

JOIN 最適化のポイント

JOIN 順序

小さいテーブルを外側（駆動表）に配置する。オプティマイザが自動判断するが、ヒントで制御できる場合もある。

JOIN キーのインデックス

ON 句のカラム両方にインデックスがあると Nested Loop Join が使える。外部キー制約を設定するとインデックスが自動作成される DB もある。

不要な JOIN を避ける

COUNT(*) だけ取得したいなら実際のデータを JOIN しない。EXISTS / IN サブクエリが適切な場面も多い。

その他の最適化テクニック
-- SELECT * は避ける（不要なカラムの転送コスト）
SELECT id, name, email FROM users;  -- 必要なカラムだけ

-- LIMIT で早期終了
SELECT * FROM logs ORDER BY created_at DESC LIMIT 100;

-- WHERE 句に計算式を入れるとインデックスが使えない
WHERE YEAR(created_at) = 2024       -- × インデックス不使用
WHERE created_at >= '2024-01-01'   -- ○ インデックス使用

-- OR はインデックスを使いにくい → UNION ALL に変換
SELECT * FROM users WHERE email = 'a@b.com'
UNION ALL
SELECT * FROM users WHERE phone = '090-xxx';

データベースの種類と使い分け

システムデザインで即答できる DB 選択の判断軸

PostgreSQL

OLTP / 汎用

最も機能豊富な OSS RDBMS。ACID 完全対応、強力な JSON/JSONB サポート、全文検索、地理情報（PostGIS）まで対応。

  • 複雑なトランザクション、金融・EC
  • JSON カラムで半構造化データ
  • MVCC による高い並行性
  • 豊富な拡張機能エコシステム
機能の豊富さ: 95%

MySQL / MariaDB

OLTP / Web

Web アプリケーションで最もシェアが高い RDBMS。LAMP スタックの定番。InnoDB エンジンで ACID 対応。

  • WordPress, Drupal など CMS
  • シンプルな OLTP ワークロード
  • 大規模なコミュニティとエコシステム
  • リードレプリカによる読み取りスケール
シンプルさ: 80%

Redis

Cache / Session

インメモリ Key-Value Store。μs オーダーの超低レイテンシ。永続化オプションあり。Pub/Sub、Stream も対応。

  • セッション管理（ユーザーセッション）
  • キャッシュレイヤー（DB 負荷軽減）
  • レートリミット（Sliding Window Counter）
  • リーダーボード（Sorted Set）
  • ジョブキュー（List / Stream）

Cassandra

Write Heavy / IoT

分散ワイドカラム型。書き込みが非常に高速。マスターレス（P2P）アーキテクチャで単一障害点なし。

  • IoT センサーデータ（大量書き込み）
  • タイムライン・アクティビティフィード
  • メッセージ履歴（Messenger, Discord）
  • グローバルに分散したシステム

MongoDB

Document / Flexible

ドキュメント型 NoSQL の代表格。JSON 形式でそのまま保存。スキーマ変更が容易でアジャイル開発に向く。

  • スキーマが頻繁に変わる開発初期
  • カタログ、商品情報（属性が可変）
  • コンテンツ管理システム
  • リアルタイム分析（Aggregation Pipeline）

Elasticsearch

Search / Analytics

分散全文検索エンジン。Lucene ベース。転置インデックスにより高速なテキスト検索と集計分析を実現。

  • 全文検索（ECサイト商品検索）
  • ログ分析（ELK スタック）
  • オートコンプリート機能
  • リアルタイムダッシュボード（Kibana）

Interview

システムデザイン面接 頻出 Q&A

データベース設計で押さえておくべき質問と回答の骨格

Q: SQL と NoSQL、どちらを選びますか？
まず要件を確認します。複雑なトランザクションや厳密な一貫性が必要なら SQL（PostgreSQL）。大量書き込みでスキーマが可変なら NoSQL（MongoDB/Cassandra）。多くのシステムは両者を組み合わせる Polyglot Persistence を採用します。
Q: DB のスケーリング方法を教えてください
段階的に対応します: ①クエリ最適化・インデックス追加 → ②キャッシュレイヤー（Redis）追加 → ③Read Replica でスケールアウト → ④垂直スケールアップ → ⑤最終手段としてシャーディング。シャーディングは複雑度が跳ね上がるので慎重に。
Q: インデックスの弊害は？
INSERT/UPDATE/DELETE のたびにインデックスも更新されるため書き込みが遅くなります。また、インデックスはディスク容量を消費します。書き込みが多いテーブルでは必要最小限に留め、実行計画（EXPLAIN）で実際に使用されているか確認します。
Q: シャードキーはどう選びますか？
高カーディナリティ（値のバリエーションが多い）でアクセスが均等に分散するキーを選びます。user_id などが典型的な良例です。時系列（created_at）や人気度（like_count）はホットスポットを生みやすいので避けます。クエリパターンに合わせてシャードキーで絞り込めるよう設計することも重要です。
Q: ACID のどれが最も重要ですか？
ユースケース次第ですが、金融系では Atomicity（原子性）が最重要です。送金の途中状態は絶対に許されません。Isolation（分離性）も重要で、分離レベルの選択がパフォーマンスとデータ整合性のトレードオフになります。Serializable は最も安全ですがスループットが下がります。
Q: レプリケーションラグの対処方法は？
① 書き込み直後に読む場合は Primary へルーティング、② 「Read Your Writes」一貫性保証をアプリレベルで実装（書き込みタイムスタンプ管理）、③ 一部の処理は最終整合性を許容しビジネス要件を緩和、④ 同期レプリケーションに変更（レイテンシとのトレードオフ）。
Q: N+1 問題を説明してください
親エンティティを 1 回クエリした後、子エンティティを N 回クエリする非効率なパターンです。例: ユーザー一覧を取得し、ループ内で各ユーザーの注文を取得すると 1+N クエリが発行されます。解決策は JOIN または ORM の Eager Loading（Django の prefetch_related、Rails の includes）です。
Q: 接続プールのサイズをどう決めますか？
PostgreSQL の公式推奨は「CPUコア数 × 2 + ストレージデバイス数」です。プールが多すぎると DB サーバーのメモリ/コンテキストスイッチが増加して逆効果です。実際には負荷テストで測定し、Prometheus + pg_stat_activity でアクティブな接続数をモニタリングしながら調整します。

B-Treeインデックスの内部構造

PostgreSQL/MySQLが実際にインデックスを管理する仕組みを理解する

B-Treeノード構造（内部ノード vs 葉ノード）

// B-Tree の構造（次数3の例）
//
//          [30 | 60]          ← 内部ノード（検索キーのみ）
//         /    |    \
//    [10|20] [40|50] [70|80]  ← 葉ノード（キー + 行ポインタ）
//
// 内部ノード: 子ノードへのポインタ + 分岐キー
// 葉ノード: 実際のキー値 + ヒープ上の行アドレス(ctid)
// 葉ノードは双方向リンクリスト → 範囲クエリに効率的

// SELECT id FROM orders WHERE id = 40
// 1. ルートノード[30|60] → 40 ∈ (30,60) → 中央の子
// 2. 葉ノード[40|50] → id=40 発見 → ctid取得
// 3. ヒープファイルの ctid 位置から行を取得
// O(log N) で完了

// SELECT * FROM orders WHERE id BETWEEN 30 AND 60
// 1. id=30 の葉ノードを探す
// 2. 葉ノードのリンクリストを60まで辿る
// → B+Tree の範囲スキャンが高速な理由

Partial Index（部分インデックス）

特定の条件に一致する行だけにインデックスを作成。サイズが小さく、更新も速い。

-- 未処理注文だけのインデックス
CREATE INDEX idx_pending_orders
ON orders(created_at)
WHERE status = 'pending';

-- このクエリで使用される
SELECT * FROM orders
WHERE status = 'pending'
  AND created_at > NOW() - '1 day';

-- 全行: 1000万件
-- pending行: 1万件
-- インデックスサイズ: 1/1000 に削減

Index-Only Scan（カバリングインデックス）

クエリで必要な列がすべてインデックスに含まれる場合、ヒープアクセスが不要になる。

-- (user_id, amount) の複合インデックス
CREATE INDEX idx_user_amount
ON orders(user_id, amount);

-- ヒープアクセス不要（Index-Only Scan）
SELECT SUM(amount)
FROM orders
WHERE user_id = 123;

-- EXPLAIN ANALYZE 出力例
Index Only Scan using idx_user_amount
  Index Cond: (user_id = 123)
  Heap Fetches: 0  ← VMビットが最新の場合
  Rows Removed: 0

EXPLAIN ANALYZE の読み方

-- クエリプランを取得
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE user_id = 123
ORDER BY created_at DESC
LIMIT 10;

-- 出力例と読み方

       (actual time=0.089..0.112 ms rows=10 loops=1)
  --          ↑推定コスト         ↑実際の実行時間
  -> Index Scan Backward using idx_user_created
       on orders
       (cost=0.43..2345.6 rows=500 width=128)
       (actual time=0.082..0.098 ms rows=10 loops=1)

Planning Time: 0.234 ms
Execution Time: 0.156 ms

-- Seq Scan → テーブル全件スキャン（インデックス未使用）
-- Index Scan → インデックス使用 + ヒープアクセスあり
-- Index Only Scan → ヒープアクセスなし（最速）
-- Hash Join / Merge Join / Nested Loop → JOIN方法
-- rows=500 だが actual rows=10 → 統計情報が古い可能性
-- ANALYZE orders; で統計更新

MVCC（多版型同時実行制御）

PostgreSQLがロックなしで高い並行性を実現する仕組み

PostgreSQL MVCCの仕組み（xmin, xmax）

-- 各行には隠しシステム列がある
SELECT xmin, xmax, id, amount FROM orders WHERE id = 1;

--  xmin | xmax | id | amount
-- ------+------+----+--------
--  100  |  0   |  1 |  5000    ← xmin=作成TX, xmax=0(有効)

-- UPDATEは「旧行を無効化 + 新行を作成」
BEGIN; -- TX ID = 200
UPDATE orders SET amount = 6000 WHERE id = 1;

-- 内部状態
-- xmin=100, xmax=200, id=1, amount=5000  ← 旧行（xmax=200で無効化）
-- xmin=200, xmax=0,   id=1, amount=6000  ← 新行（TX200から有効）

-- READ COMMITTED スナップショット
-- TX 300 が SELECT した時点で TX 200 がコミット済み → 6000が見える
-- TX 300 が SELECT した時点で TX 200 が実行中   → 5000が見える
-- → Dirty Read なし（コミット済みデータのみ可視）

-- SELECT FOR UPDATE（排他ロック）
SELECT * FROM orders WHERE id = 1 FOR UPDATE;
-- → この行を UPDATE しようとする他のTXはロック待ちに
-- → NOWAIT: ロック取得失敗時にエラー
-- → SKIP LOCKED: ロック中の行をスキップ（キュー処理に便利）

Vacuum処理と肥大化問題

-- 旧バージョン行の蓄積（デッドタプル）
-- 高頻度UPDATEのテーブルは旧行が残り続ける
SELECT schemaname, tablename,
       n_dead_tup,    -- デッドタプル数
       n_live_tup,    -- 有効タプル数
       last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;

-- VACUUM の動作
-- VACUUM: デッドタプルをマーク → 再利用可能に（ファイルサイズは減らない）
-- VACUUM FULL: テーブルを書き直し → ファイルサイズ削減（テーブルロック必要）
VACUUM ANALYZE orders;       -- 通常メンテナンス
VACUUM FULL orders;          -- ディスク回収（オフピーク時）

-- autovacuum チューニング
-- autovacuum_vacuum_scale_factor = 0.2 (デッドタプル20%超でトリガー)
-- 高更新テーブルは 0.01〜0.05 に下げると良い
ALTER TABLE orders
  SET (autovacuum_vacuum_scale_factor = 0.01);
Transaction ID Wraparound に注意
TX IDは32bitで約21億。使い切るとデータが見えなくなる（Wraparound）。`pg_stat_user_tables.age`で監視し、定期的なVACUUMFREEZEが必要。本番でこれを起こすと重大障害になる。

シャーディングキー選定の深掘り

シャーディング戦略の選択がシステムの性能と運用コストを決定づける

ユーザーIDシャーディング

// shard = user_id % shard_count
shard_id = user_id % 8

// メリット
// ✓ 1ユーザーのデータが同じシャードに集中
// ✓ JOIN が1シャード内で完結
// ✓ 均等分散しやすい

// Hotspot問題: セレブユーザー
// Elon Musk: 1.5億フォロワー
// → 彼のシャードに読み込みが集中
// 対策: ホットユーザーをメモリキャッシュ
// または別の「セレブシャード」に移動

地理的シャーディング

// リージョンで分割
user.region → shard
"US"  → us-shard-cluster
"EU"  → eu-shard-cluster
"APAC" → ap-shard-cluster

// メリット
// ✓ データレジデンシー要件（GDPR等）に対応
// ✓ 地理的近接でレイテンシ低減

// 問題
// 人口分布の偏り（米国+欧州に集中）
// 国際移住ユーザーのシャード移動
// クロスリージョンJOINが高コスト

時間ベースシャーディング

// ログ・タイムシリーズに最適
shard = YEAR_MONTH(timestamp)
2024_01 → shard_202401
2024_02 → shard_202402

// メリット
// ✓ 古いシャードをコールドストレージに移動
// ✓ 最新データへのアクセスが高速
// ✓ バックアップ・アーカイブが容易

// 問題
// 直近シャードに書き込みが集中（Hot Write）
// 解決: 書き込み先を複数シャードに分散
// (user_id % N) + year_month の複合キー

テナントベースシャーディング（SaaS）

// SaaSマルチテナント: 顧客企業単位で分割
tenant_id → shard

// 3つのモデル
// 1. Silo: テナントごとに独立DB
//    → 完全な分離、コスト高
// 2. Bridge: 共有DB、テナントIDで論理分離
//    → コスト効率良、データ漏洩リスク
// 3. Pool: 規模に応じてどちらか選択
//    → Tier1顧客はSilo, Tier2はBridge

// Shopify の実装
// 各ショップは1シャードに固定
// ブラックフライデーのHotspot: 人気ショップを
// 独立シャードに事前移動

シャードリバランシング: Blue-Green方式

// 問題: シャード数を8→16に増やしたい
// user_id % 8 → user_id % 16 への移行

// Blue-Green シャードマイグレーション手順

Step 1: 新シャード群（Green）を並行構築
  Old shards (Blue):  shard0〜7  [読み書き可能]
  New shards (Green): shard0〜15 [書き込み不可]

Step 2: バックグラウンドで全データをGreenにコピー
  for user_id in all_users:
    new_shard = user_id % 16
    copy(blue[user_id % 8], green[new_shard])

Step 3: 差分同期（コピー完了後、変更分を追いかける）
  binlog / WAL から変更をGreenに適用

Step 4: 短時間メンテナンス（数秒〜数分）
  書き込みを一時停止 → 最終差分同期 → Green切り替え

Step 5: トラフィック切り替え
  Router: user_id % 8 → user_id % 16 に更新
  問題があれば即座に Blue に戻す（ロールバック）
Consistent Hashingで回避
Consistent Hashingを使えばシャード追加時に移動が必要なデータを最小限に抑えられる（n/(n+1)のデータのみ移動）。仮想ノードで均等分散も実現。

SQLクエリ最適化ラボ

問題のあるクエリを見て、最適化後のクエリと理由を考えよう

問題1: N+1問題（ORM生成クエリ）

-- 悪い例: ORMがN+1クエリを生成
-- orders テーブルを取得後、各 order に対して user を別クエリ
orders = Order.all()          -- クエリ1回: 100件取得
for order in orders:
  print(order.user.name)      -- 各orderに1クエリ → 合計101クエリ!

-- 良い例: JOIN で1クエリに集約
SELECT o.id, o.amount, u.name
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.created_at > NOW() - INTERVAL '30 days';

-- ORM での解決 (eager loading)
orders = Order.includes('user').recent()
-- → SELECT * FROM orders WHERE ...  (1クエリ)
-- → SELECT * FROM users WHERE id IN (1,2,3,...) (1クエリ)
-- 合計 2クエリで完了

問題2〜3: 集計クエリとJSONインデックス

-- 問題2: GROUP BY の誤った使い方
-- 悪い例: HAVING で絞り込む前に全件集計
SELECT user_id, COUNT(*) cnt
FROM orders
GROUP BY user_id
HAVING cnt > 10;

-- 良い例: サブクエリ/CTE で先に絞り込み
WITH active_users AS (
  SELECT user_id
  FROM orders
  GROUP BY user_id
  HAVING COUNT(*) > 10
)
SELECT u.*, a.order_count
FROM users u
JOIN active_users a ON a.user_id = u.id;

-- 問題3: PostgreSQL JSONB へのインデックス
-- 悪い例: JSON 抽出に関数使用 → インデックス未使用
SELECT * FROM products
WHERE metadata->>'category' = 'electronics';
-- → Seq Scan（全件スキャン）

-- 良い例: GINインデックスで高速化
CREATE INDEX idx_products_metadata
  ON products USING GIN(metadata);

-- または特定キーへの関数インデックス
CREATE INDEX idx_products_category
  ON products ((metadata->>'category'));
-- → Index Scan が使われるようになる

問題4〜5: タイムシリーズとパーティショニング

-- 問題4: 巨大テーブルの時系列クエリ
-- 1日1億行のログテーブル → 3年で約1000億行
-- 通常のインデックスでも限界に近づく

-- 解決: パーティショニング（PostgreSQL 10+）
CREATE TABLE logs (
  id        BIGSERIAL,
  event_at  TIMESTAMPTZ NOT NULL,
  user_id   BIGINT,
  action    TEXT
) PARTITION BY RANGE (event_at);

-- 月次パーティション
CREATE TABLE logs_2024_01
  PARTITION OF logs
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

-- クエリは自動的に該当パーティションのみスキャン
SELECT * FROM logs
WHERE event_at BETWEEN '2024-01-15' AND '2024-01-16';
-- → logs_2024_01 のみスキャン（Partition Pruning）

-- 問題5: 古いデータの削除
-- 悪い例: DELETE は遅く、VACUUMも必要
DELETE FROM logs WHERE event_at < '2022-01-01'; -- 数時間かかる

-- 良い例: パーティションごと DROP → 瞬時
DROP TABLE logs_2022_01;  -- ほぼ瞬時
-- → VACUUM不要、ロックなし（DDL操作だが軽量）

データベース選定問題

5つのユースケースに最適なDBを選び、その理由を答えよう

シナリオ1: SNSのユーザープロフィールとフォロー/フォロワー関係

推奨: Neo4j（グラフDB）+ PostgreSQL

フォロー関係は本質的にグラフ構造。「友達の友達」「6次の隔たり」などの多段階経路探索はRDBよりグラフDBが圧倒的に高速。

// Cypher: 2ホップ以内の共通フォロー
MATCH (me:User {id:123})
      -[:FOLLOWS*1..2]->
      (suggested:User)
WHERE NOT (me)-[:FOLLOWS]->(suggested)
RETURN suggested, count(*) AS common
ORDER BY common DESC LIMIT 10

シナリオ2: ECサイトの商品カタログ（多様な属性、全文検索）

推奨: Elasticsearch + PostgreSQL

全文検索・ファセット検索はElasticsearchが最適。商品の多様な属性（電子機器: CPU/RAM、服: サイズ/色）はドキュメントDBやPostgreSQLのJSONBが柔軟に対応。

// ES: ファセット検索
{
  "query": {"match": {"title": "laptop"}},
  "aggs": {
    "by_brand": {
      "terms": {"field": "brand"}
    }
  },
  "filter": {"range": {
    "price": {"gte": 50000}
  }}
}

シナリオ3: IoTデバイスのセンサーデータ（1秒1データ, 数百万デバイス）

推奨: TimescaleDB（PostgreSQL拡張）または InfluxDB

タイムシリーズ特化DB。自動パーティショニング、時系列圧縮、ウィンドウ関数が充実。

-- TimescaleDB: 1時間平均を計算
SELECT
  time_bucket('1 hour', ts) AS bucket,
  device_id,
  AVG(temperature) AS avg_temp
FROM sensor_data
WHERE ts > NOW() - '7 days'
GROUP BY bucket, device_id
ORDER BY bucket;

シナリオ4: ゲームのリーダーボード（リアルタイムランキング）

推奨: Redis Sorted Set

O(log N)でのスコア更新・順位取得が可能。100万ユーザーのランキングでもミリ秒で応答。

// スコア更新
ZADD leaderboard 95000 "user:123"

// 上位10位を取得
ZREVRANGE leaderboard 0 9 WITHSCORES

// 自分の順位を取得
ZREVRANK leaderboard "user:123"
// → 0から始まる順位 (0 = 1位)

シナリオ5: 金融取引の台帳（不変な取引履歴）

推奨: Amazon QLDB または PostgreSQL with Append-Only設計

取引履歴は絶対に変更・削除できない（監査要件）。QQLBは暗号学的に改ざん検知可能な台帳DB。PostgreSQLでも行削除禁止のポリシーとUPDATEトリガー禁止で実現可能。

-- Append-Only設計（更新・削除禁止）
CREATE RULE no_update_ledger
AS ON UPDATE TO transactions
DO INSTEAD NOTHING;

CREATE RULE no_delete_ledger
AS ON DELETE TO transactions
DO INSTEAD NOTHING;

-- 残高は取引の積み上げで計算
SELECT SUM(amount) FROM transactions
WHERE account_id = 456;
