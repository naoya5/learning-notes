Section 01
REST API設計
Representational State Transfer — Webの設計思想に基づいたアーキテクチャスタイル
RESTとは

REST（Representational State Transfer）は、Roy Fieldingが2000年の博士論文で提唱したアーキテクチャスタイルです。HTTPプロトコルの機能を最大限に活用し、スケーラブルでシンプルなAPIを設計するための指針を提供します。

REST 6原則

原則 1

Uniform Interface

統一されたインターフェース。リソースはURIで識別し、HTTPメソッドで操作する。レスポンスはリソースの表現（JSON/XML）で返す。HATEOASによりクライアントはリンクを辿れる。

原則 2

Stateless（ステートレス）

サーバーはクライアントのセッション状態を保持しない。各リクエストは完全な情報を含み独立して処理される。スケールアウトが容易になる。

原則 3

Cacheable（キャッシュ可能）

レスポンスはキャッシュ可能かどうかを明示する（Cache-Control ヘッダー）。適切なキャッシュでパフォーマンスを向上し、サーバー負荷を削減。

原則 4

Client-Server

クライアントとサーバーの関心を分離する。UIはクライアント側、データストレージはサーバー側。独立した進化が可能になる。

原則 5

Layered System（階層化）

クライアントは直接エンドサーバーに接続しているか中間層（ロードバランサ・キャッシュ・ゲートウェイ）を経由しているかを知らなくてよい。

原則 6

Code on Demand（任意）

サーバーからクライアントに実行可能なコード（JavaScript等）を送信できる。唯一のオプション原則。現代のWebアプリでは一般的に使用。

Section 02
APIバージョニング
後方互換性を保ちながらAPIを進化させるための戦略
なぜバージョニングが必要か

公開APIは一度リリースすると既存クライアントを壊さずに変更する必要があります。フィールドの削除・型変更・エンドポイント廃止などの破壊的変更（Breaking Change）を安全に行うためにバージョニング戦略が必要です。

方式 1

URI バージョニング

GET /api/v1/users
GET /api/v2/users
  • ◎ メリット: 最も直感的・URLで一目瞭然・キャッシュしやすい・ブラウザのアドレスバーでテスト可能
  • ✗ デメリット: URLが汚くなる・「URIはリソースを表すべき」というRESTの理念に反する・v1とv2のコード二重管理
  • 採用: GitHub API・Twitter API・Stripe API（業界標準）
方式 2

カスタムヘッダー

GET /api/users
X-API-Version: 2
  • ◎ メリット: URLをクリーンに保てる・RESTの理念に沿う・同じURIで複数バージョン対応
  • ✗ デメリット: カスタムヘッダーは非標準・キャッシュが効きにくい（Vary ヘッダー設定が必要）・テストが面倒
  • 採用: 一部のエンタープライズAPI
方式 3

Accept ヘッダー（メディアタイプ）

GET /api/users
Accept: application/vnd.
  myapp.v2+json
  • ◎ メリット: HTTP仕様に最も忠実・コンテントネゴシエーション活用・URLは完全にクリーン
  • ✗ デメリット: 複雑で学習コスト高・キャッシュ難しい・ブラウザからの直接アクセス不可
  • 採用: GitHub API（サブオプション）

バージョニング方式の比較・採用基準

評価軸URI バージョニングカスタムヘッダーAccept ヘッダー
発見しやすさ◎ 高△ 低✗ 極低
キャッシュ互換性◎ 高△ 要設定△ 要設定
REST純粋性✗ 低◎ 高◎ 最高
実装の簡単さ◎ 易○ 中✗ 難
クライアント実装◎ 易○ 中✗ 難

推奨場面公開API・外部パートナー向け内部API・社内ツール超厳格なREST設計
実践的な推奨

公開APIを設計する場合、業界標準のURIバージョニング（/api/v1/...）を採用することを推奨します。開発者体験（DX）が最も高く、ドキュメント化・テスト・デバッグが容易です。セマンティックバージョニングのメジャーバージョンのみをURLに含めるのが一般的です（v1, v2 — パッチ・マイナーはURLに含めない）。

Section 03
ページネーション
大量データを効率的に分割取得する3つの戦略
方式 1

Offset-based（オフセット）

SQLの LIMIT/OFFSET に直接対応。最も直感的。

GET /users?page=3&limit=20
# または
GET /users?offset=40&limit=20

# SQL変換
SELECT * FROM users
LIMIT 20 OFFSET 40;
メリット
  • 実装が簡単
  • 任意ページへジャンプ可能
  • ページ番号をUIに表示しやすい
デメリット（重要）
  • レコード移動問題：取得中に新データが挿入されると同じデータが2回表示される
  • 大きなOFFSETはDBに負荷（全件スキャン後にスキップ）
  • 10万件以降は著しく遅くなる
方式 2

Cursor-based（カーソル）

最後に取得したレコードの位置をカーソルとして次のリクエストに渡す。大規模データに最適。

# 最初のリクエスト
GET /users?limit=20

# レスポンス
{
  "data": [...],
  "cursor": {
    "next": "eyJpZCI6MjB9",
    "hasMore": true
  }
}

# 次ページ
GET /users?cursor=eyJpZCI6MjB9&limit=20
メリット
  • レコード移動問題が発生しない
  • 大量データでも一定の高速性
  • インフィニットスクロールに最適
デメリット
  • 任意ページへのジャンプ不可
  • 実装が複雑（カーソルのBase64エンコードなど）
方式 3

Keyset-based（キーセット）

主キーや一意インデックス列を使って次ページを特定。Cursor-basedの一形態。

# IDベース
GET /users?after_id=100&limit=20

# タイムスタンプベース
GET /posts?after=2026-06-01T00:00:00Z&limit=20

# SQL変換
SELECT * FROM users
WHERE id > 100
ORDER BY id
LIMIT 20;
メリット
  • インデックスを最大限活用
  • 何百万件でも一定速度
  • 実装がシンプル
デメリット
  • ソート順が固定（主キー順など）
  • 複合ソートが複雑

ページネーション比較表

評価軸Offset-basedCursor-basedKeyset-based
実装難易度易中易〜中
大規模データのパフォーマンス低下安定高速
レコード移動問題発生する発生しない発生しない
任意ページジャンプ可能不可不可
ソート柔軟性高中低
推奨用途小〜中規模・管理画面SNSフィード・大規模API時系列データ・イベントログ
レコード移動問題（Offset-basedの罠）

ページ1（offset=0, limit=10）を取得中に新しいレコードが先頭に挿入されると、ページ2（offset=10, limit=10）でページ1の最後のレコードが再度返ってくる問題。SNSやリアルタイム更新のある画面ではCursor-basedを使うべき。

Section 04
冪等性 (Idempotency)
同じ操作を何度実行しても結果が変わらない性質 — 分散システムの信頼性の基礎
冪等性とは

数学的定義：f(f(x)) = f(x)。APIにおいては「同一リクエストを複数回送信しても、1回送信した場合と同じ結果になること」。ネットワーク障害・タイムアウト・リトライによる重複実行を安全にするために不可欠な性質です。

冪等 + 安全

GET / HEAD / OPTIONS

読み取り専用。何度呼んでも状態が変化しない。キャッシュ可能。

# 何度呼んでも同じ結果
GET /users/123  → 200 OK
GET /users/123  → 200 OK
GET /users/123  → 200 OK
冪等（状態変化あり）

PUT / DELETE

同じ操作を複数回しても最終状態は同じ。ただし状態は変化する。

# PUTは何度しても同じ状態
PUT /users/123 {name:"田中"}
→ 200: name = "田中"
PUT /users/123 {name:"田中"}
→ 200: name = "田中" (同じ)

# DELETEも冪等（2回目は404）
DELETE /users/123 → 204
DELETE /users/123 → 404 (既に削除済)
非冪等

POST

毎回新しいリソースを作成する。リトライすると重複が発生する。

# POSTはリトライで重複!
POST /orders {item:"本"}
→ 201: order_id=1 (注文作成)
POST /orders {item:"本"}
→ 201: order_id=2 (別の注文!)

冪等性キー（Idempotency Key）による重複防止設計

冪等性キーパターン

クライアントが一意のキー（UUID等）をリクエストヘッダーに含める。サーバーはキーをキャッシュし、同じキーの2回目以降のリクエストには最初のレスポンスをそのまま返す。Stripeが普及させた業界標準パターン。

クライアント側の実装
// クライアント: 冪等性キーを生成して送信
const idempotencyKey = crypto.randomUUID();

async function createPaymentWithRetry() {
  for (let i = 0; i < 3; i++) {
    try {
      const res = await fetch('/payments', {
        method: 'POST',
        headers: {
          'Idempotency-Key': idempotencyKey,
          'Content-Type': 'application/json'
        },
        body: JSON.stringify({
          amount: 10000,
          currency: 'jpy'
        })
      });
      if (res.ok) return res.json();
    } catch (e) {
      // リトライ — 同じキーなので安全
      await sleep(1000 * 2**i);
    }
  }
}
サーバー側の実装
// サーバー: 冪等性キーでキャッシュ管理
async function handlePayment(req, res) {
  const key = req.headers['idempotency-key'];

  if (key) {
    // Redisで既存レスポンスを確認
    const cached = await redis.get(
      `idem:${key}`
    );
    if (cached) {
      // 同じレスポンスをそのまま返す
      return res.json(JSON.parse(cached));
    }
  }

  // 初回: 実際に決済処理
  const payment = await processPayment(req.body);
  const response = { id: payment.id, status: 'succeeded' };

  if (key) {
    // 結果をRedisにキャッシュ（24時間）
    await redis.setex(
      `idem:${key}`, 86400, JSON.stringify(response)
    );
  }

  res.status(201).json(response);
}

Section 05
GraphQL
Facebookが開発したクエリ言語 — クライアントが必要なデータを正確に指定できる
GraphQLとは

GraphQLは2015年にFacebook（Meta）がオープンソース化したAPIクエリ言語・ランタイムです。「クライアントが必要なデータを正確に要求できる」という思想で、RESTのOver-fetching（取りすぎ）とUnder-fetching（取り不足）問題を解決します。

REST vs GraphQL — Over-fetching / Under-fetching問題

REST の問題

Over-fetching（不要なデータまで取得）とUnder-fetching（必要なデータが足りず複数回リクエスト）が発生しやすい。

# Over-fetching: ユーザー名だけ欲しいが
# 不要なフィールドも返ってくる
GET /users/123
{
  "id": 123,
  "name": "山田太郎",    ← 必要
  "email": "...",          ← 不要
  "address": "...",        ← 不要
  "preferences": {...},    ← 不要
  "createdAt": "..."       ← 不要
}

# Under-fetching: 投稿一覧を取るのに
# 各投稿の著者情報が別リクエスト必要
GET /posts           ← リクエスト1
GET /users/1         ← リクエスト2
GET /users/2         ← リクエスト3
GET /users/3         ← リクエスト4 (N+1問題)
GraphQL の解決

クライアントが必要なフィールドのみを指定。1リクエストで複数リソースを取得可能。

# 必要なフィールドだけ指定
query {
  user(id: 123) {
    name      # 必要なものだけ
  }
}
# → { "user": { "name": "山田太郎" } }

# 1リクエストでネストしたデータを取得
query {
  posts {
    title
    author {
      name      # ネストして著者も取得
      avatar
    }
    comments {
      body
      createdAt
    }
  }
}
# → 1リクエストで全部返ってくる!

GraphQL Schema定義

Schema Definition Language (SDL)
# 型定義
type User {
  id:       ID!           # ! = 非null必須
  name:     String!
  email:    String!
  posts:    [Post!]!      # ユーザーの投稿一覧
  createdAt: String!
}

type Post {
  id:      ID!
  title:   String!
  body:    String!
  author:  User!          # 投稿者
  tags:    [String!]!
}

# クエリ（読み取り）
type Query {
  user(id: ID!): User
  users(page: Int, limit: Int): [User!]!
  posts(authorId: ID): [Post!]!
}

# ミューテーション（書き込み）
type Mutation {
  createUser(input: CreateUserInput!): User!
  updateUser(id: ID!, input: UpdateUserInput!): User!
  deleteUser(id: ID!): Boolean!
}

# サブスクリプション（リアルタイム）
type Subscription {
  userCreated: User!              # 新規ユーザー作成を購読
  postPublished(userId: ID): Post!  # 投稿公開を購読
}

# 入力型
input CreateUserInput {
  name:  String!
  email: String!
}

N+1問題とDataLoaderによる解決

N+1問題

投稿10件の著者を取得するとき、10回別々のDBクエリが走る問題。

// N+1問題の発生するResolver
const resolvers = {
  Post: {
    author: (post) => {
      // 各投稿ごとにDBクエリ!
      // 10投稿 = 10回のDB呼び出し
      return db.users.findById(post.authorId);
    }
  }
};

// 実行されるSQLの数
// SELECT * FROM posts LIMIT 10; (1回)
// SELECT * FROM users WHERE id=1; (N回)
// SELECT * FROM users WHERE id=2;
// ... 合計N+1回!
DataLoaderで解決

複数のIDをバッチにまとめて1回のSQLで取得する。

import DataLoader from 'dataloader';

// バッチ関数: IDをまとめて1回取得
const userLoader = new DataLoader(
  async (userIds) => {
    // IN句で一括取得!
    const users = await db.users.findMany({
      where: { id: { in: userIds } }
    });
    // IDの順序を保持して返す
    return userIds.map(id =>
      users.find(u => u.id === id)
    );
  }
);

// Resolverでloaderを使用
const resolvers = {
  Post: {
    author: (post) => {
      return userLoader.load(post.authorId);
      // → 自動バッチ化! 1回のSQLに
    }
  }
};
// SELECT * FROM users WHERE id IN (1,2,...10);
Section 06
gRPC
Googleが開発した高性能RPC フレームワーク — マイクロサービス間通信の標準
gRPCとは

gRPC（gRPC Remote Procedure Call）はGoogleが開発したオープンソースのRPCフレームワークです。HTTP/2を基盤とし、Protocol Buffersによるスキーマ定義・バイナリシリアライズを使用。JSONベースのREST APIと比較して大幅に高速・軽量で、特にマイクロサービス間の内部通信に適しています。

Protocol Buffers（.protoファイル）

// user.proto
syntax = "proto3";

package user;

// サービス定義
service UserService {
  rpc GetUser (GetUserRequest)
      returns (User);

  rpc ListUsers (ListUsersRequest)
      returns (ListUsersResponse);

  rpc CreateUser (CreateUserRequest)
      returns (User);

  // サーバーストリーミング
  rpc WatchUsers (WatchRequest)
      returns (stream User);
}

// メッセージ定義（型付きスキーマ）
message User {
  string id = 1;
  string name = 2;
  string email = 3;
  int64  created_at = 4;
}

message GetUserRequest {
  string id = 1;
}

message ListUsersResponse {
  repeated User users = 1;
  int32 total = 2;
}

gRPCの4つの通信パターン

パターン 1

Unary RPC（単方向）

通常のリクエスト/レスポンス。REST APIと同じ。1リクエスト→1レスポンス。最も一般的。

Client: GetUser(id) → Server: User{}
パターン 2

Server Streaming RPC

クライアントが1リクエスト、サーバーが複数レスポンスをストリームで返す。ログ配信・リアルタイム更新に使用。

Client: Watch() → Server: User{}, User{}, ...
パターン 3

Client Streaming RPC

クライアントが複数リクエストをストリームで送り、サーバーが1レスポンスを返す。ファイルアップロード・センサーデータ収集に使用。

Client: data1, data2, ... → Server: Result{}
パターン 4

Bidirectional Streaming RPC

双方向ストリーミング。チャット・ゲーム・リアルタイム協調作業などに使用。WebSocketに相当。

Client: msg1 ⇄ Server: reply1
Client: msg2 ⇄ Server: reply2

APIゲートウェイ
クライアントとバックエンドサービスの間に立つ「玄関口」— 横断的関心事を一元管理
APIゲートウェイとは

APIゲートウェイは、すべてのクライアントリクエストの単一入口点（Single Entry Point）となるサーバーです。認証・認可、レートリミット、ロギング、ルーティング、プロトコル変換などの横断的関心事（Cross-cutting Concerns）を各マイクロサービスから切り離して一元管理できます。

BFF（Backend For Frontend）パターン

BFFとは

BFF（Backend For Frontend）は、フロントエンドの種類（Web・モバイル・TV等）ごとに専用のAPIゲートウェイ/集約層を用意するパターンです。各クライアントの要件に合わせた最適なデータ形式・フィールドを返せます。Sam Newmanが提唱。

Web BFF

PCブラウザ向け。大画面・大帯域前提。詳細なデータ・多くのフィールドを返す。Reactアプリに最適化。

  • フル解像度画像
  • 詳細なユーザー情報
  • SSE/WebSocket対応

Mobile BFF

iOS/Android向け。帯域・バッテリー考慮。必要最低限のデータ・圧縮済みレスポンス。プッシュ通知連携。

  • サムネイル画像
  • 最小限フィールド
  • オフライン対応設計

Partner API BFF

外部パートナー企業向け。厳格な認証・レートリミット。公開APIドキュメント完備。バージョニング必須。

  • OAuth2認証
  • 厳格なレートリミット
  • Webhookサポート

APIキー vs JWT vs OAuth2

方式 1

APIキー

GET /api/data
X-API-Key: sk_live_abc123xyz
  • ◎ シンプル: 発行・管理が容易
  • ◎ サーバー間通信に最適
  • ✗ ユーザー単位の認証不可
  • ✗ 有効期限なし（漏洩リスク）
  • ✗ キーが盗まれたら即使用可能
用途: 外部パートナー・CI/CD・機械間通信
方式 2

JWT（JSON Web Token）

GET /api/data
Authorization: Bearer eyJhbGci...
# header.payload.signature
  • ◎ ステートレス: DB照合不要
  • ◎ ユーザー情報を含める（ペイロード）
  • ◎ 有効期限設定可能
  • ✗ 失効（Revoke）が困難
  • ✗ ペイロードは暗号化されない（Base64のみ）
用途: Webアプリ・モバイルアプリ認証
方式 3

OAuth2 + OIDC

# 認可コードフロー
1. ユーザー → 認可サーバーへリダイレクト
2. 認可コードを受け取る
3. コードをアクセストークンに交換
4. トークンでAPI呼び出し
  • ◎ 業界標準の委任認証
  • ◎ スコープで細かな権限制御
  • ◎ リフレッシュトークンで長期利用
  • ✗ 実装が複雑
用途: 「Googleでログイン」・サードパーティアプリ認可

JWTの構造

JWTの3部構成（Base64URLエンコード）
// Header（アルゴリズム情報）
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload（クレーム — 公開情報!）
{
  "sub": "user_123",     // Subject
  "name": "山田太郎",
  "role": "admin",
  "iat": 1750000000,    // Issued At
  "exp": 1750003600     // Expiration (1h後)
}

// Signature（秘密鍵で署名）
HMACSHA256(
  base64url(header) + "." + base64url(payload),
  SECRET_KEY
)
JWTの注意点

ペイロードはBase64URLエンコードされているだけで暗号化されていません。誰でもデコードして中身を見られます。パスワード・個人情報をペイロードに含めないこと。

ベストプラクティス

1) アクセストークンは短命（15分〜1時間）に設定する。2) リフレッシュトークンでサイレント更新する。3) HTTPOnly Cookieでトークンを保存（XSS対策）。4) HTTPS必須。5) トークンの失効にはブラックリスト（Redis）を使用。

CORS（Cross-Origin Resource Sharing）

CORSとは

ブラウザのセキュリティ機能。異なるオリジン（ドメイン/ポート/スキーム）へのリクエストをデフォルトでブロックし、サーバーが明示的に許可したオリジンのみアクセスを認める仕組み。

CORSヘッダー設定例
# サーバーのレスポンスヘッダー
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400  # プリフライトキャッシュ
Access-Control-Allow-Credentials: true  # Cookie送信許可

# 注意: * (ワイルドカード) はCredentialsと併用不可
# NG: Access-Control-Allow-Origin: *
#     + Access-Control-Allow-Credentials: true

レートリミット設計 — 4つのアルゴリズム

アルゴリズム 1

固定窓（Fixed Window）

一定時間窓（例: 1分間）でリクエスト数をカウント。実装最も簡単。

// 1分間に100リクエストまで
window = floor(now / 60)
key = `rate:${userId}:${window}`
count = redis.incr(key)
if (count == 1) redis.expire(key, 60)
if (count > 100) return 429
問題点

窓の境界でバースト可能。0:59に100回 + 1:00に100回 = 2秒間に200リクエスト通過。

アルゴリズム 2

スライディング窓（Sliding Window）

現在時刻から遡った直近N時間のリクエストをカウント。固定窓の境界問題を解消。

// Redisのsorted setで実装
now = Date.now()
key = `rate:${userId}`

// 古いエントリを削除（1分以前）
redis.zremrangebyscore(key, 0, now - 60000)

// 現在のカウントを確認
count = redis.zcard(key)
if (count >= 100) return 429

// 新リクエストを追加
redis.zadd(key, now, now + random())
アルゴリズム 3

トークンバケット（Token Bucket）

バケツにトークンが一定レートで補充される。トークンがあればリクエスト処理。バーストを許容。

// 概念的な実装
capacity = 100    // バケツの容量
refillRate = 10   // 1秒あたり10トークン補充

tokens = min(capacity,
  lastTokens + (now - lastRefill) * refillRate
)
if (tokens < 1) return 429
tokens -= 1      // トークン消費
特長

Stripe・Twilioが採用。一時的なバーストを許容しながら平均レートを制御。

アルゴリズム 4

リーキーバケット（Leaky Bucket）

キューにリクエストを溜め、一定レートで処理する。出力レートが一定になりトラフィックを平滑化。

// FIFOキューで実装
queue = []
processRate = 10  // 1秒あたり10件処理

// リクエスト到着時
if (queue.length >= capacity) {
  return 429  // キューが満杯
}
queue.push(request)

// 一定間隔で処理
setInterval(() => {
  if (queue.length > 0) process(queue.shift())
}, 1000 / processRate)
特長

トラフィックの平滑化に最適。Nginx のリミット実装に使用されている。

システムデザイン面接でAPI設計について問われたときの回答フレームワーク
Q: RESTとGraphQLをどう使い分けますか？
公開APIや単純なCRUDにはREST（業界標準・キャッシュ容易）。モバイルアプリや複雑なデータ要件にはGraphQL（Over-fetching回避・1リクエスト完結）。マイクロサービス内部通信にはgRPC（高速・型安全・ストリーミング）。クライアントの多様性とデータ複雑度で判断。
Q: APIのバージョニング戦略を設計してください
公開APIはURIバージョニング（/api/v1/）を推奨。破壊的変更（フィールド削除・型変更・エンドポイント廃止）はメジャーバージョンを上げる。マイナー追加は後方互換を保つ。廃止予定エンドポイントにはDeprecationヘッダーとSunsetヘッダーで事前通知し、最低6〜12ヶ月の移行期間を設ける。
Q: 大量データのAPIページネーションをどう設計しますか？
データ量・ユースケースで選択。管理画面や小規模（〜10万件）はOffset-based（ページ番号UIとの相性が良い）。SNSフィードやリアルタイム更新はCursor-based（レコード移動問題回避）。時系列データやイベントログはKeyset-based（主キーでINDEX活用、高速）。10万件以上ではOffset避けてCursor/Keysetを選択。
Q: 決済APIで二重課金を防ぐにはどうしますか？
冪等性キー（Idempotency Key）パターンを使用。クライアントがUUIDをリクエストヘッダー（Idempotency-Key）に付与し、サーバーはRedisにキー→レスポンスをTTL付きでキャッシュ。同じキーの2回目以降は実際の処理をスキップして初回のレスポンスをそのまま返す。Stripeが業界標準として普及させた手法。
Q: APIゲートウェイは何のために使いますか？
横断的関心事（Cross-cutting Concerns）の一元管理。具体的には：認証・認可の共通化、レートリミット、ロギング・メトリクス収集、SSL終端、リクエストルーティング、プロトコル変換（外部REST→内部gRPC）、レスポンス集約。マイクロサービスアーキテクチャでは実質必須。単一障害点リスクはHA構成で対策。
Q: 401 Unauthorized と 403 Forbidden の違いは？
401は「誰だかわからない（認証が必要）」。ログインしていない・トークンが無効・期限切れ。クライアントは認証情報を付けて再試行すべき。403は「誰だかはわかるが、権限がない（認可失敗）」。ログイン済みだが他ユーザーのリソースや管理者専用機能へのアクセス。再試行しても無駄で権限付与が必要。
Q: GraphQLのN+1問題とは何ですか？解決法は？
投稿10件の著者を取得する際、各投稿ごとにDB呼び出しが走り合計N+1回のクエリになる問題。解決策はDataLoader（Facebookが開発）を使ったバッチ処理：複数IDを集めて1回のIN句クエリ（SELECT * FROM users WHERE id IN (1,2,...10)）に変換。リクエストスコープでインスタンスを生成しキャッシュも兼ねる。
Q: APIのレートリミットはどう設計しますか？
ユーザーID・APIキー・IPアドレスなどでバケット分割。アルゴリズムはStripe採用のトークンバケットが汎用的（バーストを許容しつつ平均レートを制御）。実装はRedis + Lua スクリプトでアトミックに。レスポンスヘッダーにX-RateLimit-Limit/Remaining/Resetを含める。制限超過は429 Too Many Requestsを返す。

REST成熟度モデル（Richardson Maturity Model）

APIがどの程度「RESTful」かを4段階で評価するモデル

Level 0

RPCスタイル（1エンドポイント）

全操作を1つのURLに対してPOSTで送る。SOAPなどの古いアーキテクチャがこれに当たる。

// すべての操作が同じURL
POST /api
{"action": "getUser", "id": 123}

POST /api
{"action": "createOrder",
 "userId": 123, "items": [...]}

// 問題: HTTPの機能（キャッシュ・ステータス）
// を全く活用できない
Level 1

リソース分離

URLをリソースごとに分けるが、HTTPメソッドは統一されていない。

// URLはリソースを表すが…
POST /users/getUser
POST /users/createUser
POST /orders/getOrder
POST /orders/createOrder

// 改善: URLがリソースを示す
// 問題: POSTしか使わず、動詞をURLに入れている
// HTTP のセマンティクスを活用できていない
Level 2

HTTP Verbsの正しい使用 ★推奨

REST APIの現実的な目標。GET/POST/PUT/PATCH/DELETEを正しいセマンティクスで使う。

GET    /users/123      ← 取得（冪等・キャッシュ可）
POST   /users          ← 作成（非冪等）
PUT    /users/123      ← 全更新（冪等）
PATCH  /users/123      ← 部分更新
DELETE /users/123      ← 削除（冪等）

// 正しいステータスコード
200 OK, 201 Created, 204 No Content
400 Bad Request, 401 Unauthorized
404 Not Found, 409 Conflict
429 Too Many Requests, 500 Internal Error
Level 3

HATEOAS（ハイパーメディア）

レスポンスに「次に取れるアクション」のリンクを含める。APIが自己記述的になる。

// GET /orders/456 のレスポンス
{
  "id": 456,
  "status": "pending",
  "_links": {
    "self": {"href": "/orders/456"},
    "cancel": {"href": "/orders/456/cancel",
               "method": "POST"},
    "payment": {"href": "/orders/456/pay"}
  }
}
// クライアントはURLを知らなくてもナビゲート可能
// 現実: 実装コスト高 → Level 2が現実的な目標
深掘り +

GraphQL N+1問題の完全解決策

DataLoaderのバッチ処理とキャッシュで効率的なデータ取得を実現する

DataLoaderの仕組み（JavaScript実装）

// 問題: N+1クエリ
// Query: users { posts { author { name } } }
// → 100ユーザー × 各ユーザーの投稿数 分のSQLが発行される

// DataLoader: バッチ + キャッシュで解決
const DataLoader = require('dataloader')

// バッチ関数: IDの配列を受け取り、一括取得
const userLoader = new DataLoader(async (userIds) => {
  // 1回のSQLで全ユーザーを取得
  const users = await db.query(
    `SELECT * FROM users WHERE id IN (?)`,
    [userIds]
  )
  // DataLoaderはIDの順序通りに結果を返す必要がある
  return userIds.map(id => users.find(u => u.id === id))
})

// GraphQL Resolver
const resolvers = {
  Post: {
    author: (post, _, { loaders }) => {
      // load() はキューに追加 → イベントループの次ティックで一括処理
      return loaders.user.load(post.author_id)
    }
  }
}

// 結果: 100投稿でも userIds=[1,2,3,...,100] で1クエリ
// キャッシュ: 同じリクエスト内で同じIDは2回取得しない

@defer / @stream による段階的レスポンス

# @defer: 重い部分を後で送信
query GetUser {
  user(id: "123") {
    name
    email
    ... on User @defer {
      # 計算コストの高いフィールド
      recommendedPosts { title }
      friendSuggestions { name }
    }
  }
}
# レスポンス1: name, email を即時返却
# レスポンス2: recommendedPosts を後で送信
# → 初期表示を高速化

# @stream: リストを逐次送信
query GetFeed {
  feed @stream(initialCount: 3) {
    id title
  }
}
# → 最初の3件を即時表示、残りは続々送信

Persisted Queries によるセキュリティ強化

// 通常のGraphQL: 任意のクエリを送れる
POST /graphql
{"query": "{ users { id email password } }"}
// → 攻撃者が任意のクエリを試せる
// → Introspectionで全スキーマを調査可能

// Persisted Queries: 事前登録クエリのみ許可
// 登録時（デプロイ時）
hash = SHA256(queryString)
registry.set(hash, queryString)

// クライアントはハッシュのみ送信
POST /graphql
{"extensions": {
  "persistedQuery": {
    "sha256Hash": "abc123..."
  }
}}
// → 未登録クエリはサーバーが拒否
// → 帯域削減 + セキュリティ向上

Circuit Breaker をAPIゲートウェイで実装

// サーキットブレーカーの3状態
CLOSED  → 正常: リクエストを通過させる
OPEN    → 異常: 全リクエストをフォールバック
HALF_OPEN → 回復確認: 一部リクエストを試験的に通過

// 実装例（Resilience4j）
CircuitBreakerConfig config = CircuitBreakerConfig
  .custom()
  .failureRateThreshold(50)      // 50%失敗でOPEN
  .waitDurationInOpenState(
    Duration.ofSeconds(30))       // 30秒後にHALF_OPENへ
  .slidingWindowSize(10)         // 直近10リクエストで判断
  .build();

// フォールバック
return circuitBreaker
  .executeSupplier(() -> userService.getUser(id))
  .recover(ex -> User.fromCache(id));  // キャッシュから返す

REST API設計レビュー

以下のAPI設計のどこが問題か？正しい設計を答えよう

問題1

HTTPメソッドの誤り

悪い設計

POST /getUser
POST /getUsersByAge
POST /searchProducts?q=laptop

問題: GETすべき操作をPOSTで実装。キャッシュが効かず、ブラウザの戻るボタンで再送確認ダイアログが表示される。

正しい設計

GET /users/123
GET /users?age=25
GET /products?q=laptop

GETはべき等かつキャッシュ可能。CDNやブラウザキャッシュが効き、パフォーマンスが大幅に向上する。

問題2

危険な一括削除エンドポイント

悪い設計

DELETE /users/all
DELETE /products

誤ってアクセスされると全データが削除される。バグやCSRFで壊滅的な被害が出る。

正しい設計

DELETE /users/123     ← 1件ずつ
# 一括削除が必要な場合
POST /users/bulk-delete
{"ids": [1, 2, 3], "confirm": true}
# → 明示的なIDリスト + 確認フラグ
問題3

エラー時も常に200を返す

悪い設計

HTTP/1.1 200 OK
{
  "success": false,
  "error": "User not found",
  "data": null
}

モニタリングツール・ロードバランサー・CDNがエラーを検知できない。ログ解析も困難。

正しい設計

HTTP/1.1 404 Not Found
{
  "error": "USER_NOT_FOUND",
  "message": "ユーザーID 123 は存在しません",
  "request_id": "req-abc123",
  "docs_url": "/docs/errors#USER_NOT_FOUND"
}
問題4

ページネーションなしの大量データ返却

悪い設計

GET /orders
→ 100万件のJSONを一気に返す
→ メモリ不足、タイムアウト
→ クライアント側もクラッシュ

正しい設計（カーソルページネーション）

GET /orders?limit=20&cursor=xxx
{
  "data": [...],  // 20件
  "pagination": {
    "next_cursor": "eyJpZCI6MTAwfQ",
    "has_more": true
  }
}
// カーソルはオフセットより高速
// （大量データでもページ移動が O(1)）
問題5

パスワードをGETパラメータで渡す

危険な設計

GET /login?username=alice
         &password=secret123

// URLはアクセスログに記録される
// ブラウザ履歴に残る
// リファラーヘッダで外部に漏洩
// → パスワードが平文で流出!

正しい設計

POST /auth/login
Content-Type: application/json
{
  "username": "alice",
  "password": "secret123"
}
// POSTボディはHTTPS暗号化で保護
// ログにはURLのみ記録（ボディは記録しない）
// ブラウザ履歴・リファラーに残らない
問題6

レスポンスに不要な機密情報を含む

危険な設計

GET /users/123
{
  "id": 123,
  "name": "Alice",
  "email": "alice@example.com",
  "password_hash": "$2b$10$...",
  "api_key": "sk-live-xxxxx",
  "internal_user_id": "USR-123",
  "credit_card_number": "4111..."
}

正しい設計（必要最小限のみ）

GET /users/123
{
  "id": 123,
  "name": "Alice",
  "email": "alice@example.com",
  "created_at": "2024-01-15"
}
// 機密情報は専用エンドポイントで
// 認証後に最小権限で取得
// Allowlist（返してよいフィールド）方式推奨
// Denylist（返してはいけないフィールド）は漏れやすい

OpenAPI 仕様書のベストプラクティス
components で再利用: スキーマ・レスポンス・パラメータを $ref で参照し、重複を避ける。
examples を充実させる: 複数のexampleで正常系・異常系を示す。
operationId: SDK自動生成時のメソッド名になるため、動詞+名詞で一意に命名する（listTodos, createTodo等）。
エラーレスポンスを統一: 全エンドポイントで同じ ErrorResponse スキーマを使うことでクライアント実装が簡単になる。
