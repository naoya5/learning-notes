Chapter 12

🔐 セキュリティ設計

認証・認可・暗号化・DDoS防御・ゼロトラストまで、システムを守る設計パターンを体系的に学ぶ

1 セキュリティ設計の原則

セキュリティは後付けではなく、設計段階から組み込む必要がある。Security by Design の考え方と主要な原則を理解する。

🛡️ Security by Design

セキュリティを後から追加するのではなく、要件定義・設計段階から考慮する。脆弱性は発見が遅れるほどコストが増大する（SDLC早期発見の重要性）。

🔑 最小権限の原則

各コンポーネント・ユーザーは、必要最低限の権限のみを持つ。DB接続もREAD ONLYで十分なら書き込み権限を与えない。侵害時の被害範囲を最小化する。

🧅 多層防御（Defense in Depth）

単一のセキュリティ層に依存せず、ネットワーク・アプリ・データ層に複数の防御を重ねる。1層が突破されても次の層で防御できる設計。

🔒 Fail Secure

エラー発生時はアクセス拒否側に倒す（Fail Secure）。認証サービスがダウンしたとき「全員通過」ではなく「全員拒否」。Fail Safeは物理安全で使う概念（火災時にドアを開放など）。

📉 攻撃面の最小化

不要なAPIエンドポイント・ポート・サービスを無効化する。外部に公開するサービスが少ないほど攻撃の入口が減る。使われていない機能は削除する。

🎯 OWASP Top 10

Open Web Application Security Project が発表するWebアプリ上位10脆弱性。Broken Access Control、Cryptographic Failures、Injection などが含まれる。設計・コードレビューの基準として活用。

多層防御（Defense in Depth）の概念図

💬 面接でよく聞かれること

  • 「セキュリティ設計で最初に考えることは？」→ 脅威モデリング（STRIDE）、攻撃面の特定
  • 「最小権限をDBに適用するには？」→ 読み取り専用ユーザー、スキーマ分離、Row-Level Security
  • 「Fail SecureとFail Safeの違いは？」→ デジタルセキュリティ vs 物理安全の文脈で説明
2 認証（Authentication）

「あなたは誰か」を検証する仕組み。セッションベース、JWT、APIキー、MFAの仕組みと使い分けを理解する。

📦 セッションベース認証（Cookie + Session Store）

🎫 JWTトークン認証

JWT（JSON Web Token）は Header.Payload.Signature の3部構成。サーバー側でセッション状態を持たないステートレス認証。

// JWT 構造 Header (Base64URL): { "alg": "HS256", "typ": "JWT" } Payload (Base64URL): { "sub": "user123", "name": "Naoya", "role": "admin", "iat": 1719225600, "exp": 1719229200 // 1時間 } Signature: HMACSHA256( base64(header) + "." + base64(payload), secretKey )
JWS vs JWE

JWS (JSON Web Signature): Payloadに署名。内容は誰でも読めるが改ざん不可。通常の JWTはJWS。

JWE (JSON Web Encryption): Payloadを暗号化。内容自体を隠したい場合（例: PII含む）。

⚠️ JWTの注意点: Payloadはデコード可能なため機密情報を入れない。トークン失効管理が難しい（JTI blacklistやshort expiry推奨）。

🔐 多要素認証（MFA / 2FA）

TOTP（Time-based OTP）

共有シークレット鍵 + 現在時刻（30秒単位）をHMACでハッシュし6桁を生成。Google Authenticatorはこの仕組み。オフラインで動作可能。

// TOTP計算 counter = floor(epoch / 30) OTP = HMAC-SHA1(secret, counter) .slice(-6) // 下6桁

SMS OTP

ランダムな6桁をSMSで送信。実装は簡単だがSIMスワップ攻撃のリスクあり。TOTPより安全性は低い。高セキュリティ要件ではTOTP/FIDO2を推奨。

FIDO2 / WebAuthn

パスキー（Passkeys）の基盤。生体認証+デバイス内公開鍵ペア。フィッシング耐性が最も高い。

セッション vs JWT 比較表

3 OAuth 2.0 と OpenID Connect

OAuth 2.0 は「認可」フレームワーク、OpenID Connect は OAuth 2.0 上に構築された「認証」プロトコル。

🔄 Authorization Code Flow（最も安全・推奨）

🔒 PKCE（Proof Key for Code Exchange）

SPAやモバイルアプリでは client_secret をハードコードできない。Authorization Code を傍受されてもトークン取得を防ぐ仕組み。

// PKCE フロー（クライアント側） const code_verifier = randomBase64URL(43-128文字); const code_challenge = base64url(SHA256(code_verifier)); // 1. 認可リクエスト時に code_challenge を含める GET /authorize? code_challenge=<hash>& code_challenge_method=S256& ... // 2. トークン交換時に code_verifier を送る（code_secret不要） POST /token code=AUTHCODE& code_verifier=<originalRandom> // サーバーがSHA256(verifier)==challengeを検証 → 傍受者は verifier を知らない

OAuth 2.0 Grant Type 比較

Grant Type用途推奨度特徴
Authorization Code + PKCE SPA・モバイル・Webアプリ 推奨 最も安全、ユーザー認証あり
Client Credentials M2M（サービス間通信） 推奨（M2M用） ユーザー不在、client_id+secretで取得
Device Code スマートTV・CLI 限定用途 ブラウザなし環境でユーザーが別デバイスで認可
Implicit （旧SPA） 非推奨 トークンがURLに露出、PKCE付きCodeに移行
Resource Owner Password （旧ファーストパーティ） 非推奨 パスワードをアプリが直接扱う、リスク大

Access Token

APIにアクセスするための短命トークン（15分〜1時間）。Bearer Tokenとして Authorization ヘッダで送信。漏洩時の影響を最小化するため短命に。

Refresh Token

Access Tokenを再取得するための長命トークン（数日〜数ヶ月）。Auth ServerのAPIにのみ使用。Rotation（使用のたびに新しいものと交換）を実装する。

ID Token（OIDC）

OpenID ConnectのJWT。ユーザーの identity 情報（sub, name, email）を含む。APIアクセスではなく、ユーザーがログインしていることを示すために使用。

4 認可（Authorization）

「あなたは何をできるか」を制御する仕組み。RBAC・ABAC・ACL・OPAの特性と使い分けを理解する。

👥 RBAC（Role-Based Access Control）

ユーザーにロールを割り当て、ロールにパーミッションを付与。シンプルで管理しやすい。エンタープライズシステムで広く採用。

User → Role → Permissions alice → admin → [read, write, delete] bob → editor → [read, write] carol → viewer → [read] // DB設計例 users ←→ user_roles ←→ roles ↓ role_permissions ←→ permissions

🧮 ABAC（Attribute-Based Access Control）

ユーザー・リソース・環境の属性に基づいてアクセス制御。細粒度の制御が可能だが複雑。GDPR対応やマルチテナントに適する。

// ABAC ポリシー例 Allow IF: user.department == resource.owner_dept AND user.clearance >= resource.sensitivity AND time.hour BETWEEN 9 AND 18 AND request.ip IN corporate_ranges

🤖 OPA（Open Policy Agent）によるポリシー管理

ポリシーをコードとして管理（Policy as Code）。Regoという宣言的言語でポリシーを記述し、マイクロサービス・Kubernetes・APIゲートウェイで横断的に適用。

# Rego ポリシー例（OPA） package authz default allow = false allow { input.method == "GET" input.user.roles[_] == "viewer" } allow { input.method == "POST" input.user.roles[_] == "editor" input.resource.owner == input.user.id }
方式管理単位柔軟性複雑度適用場面
ACLリソースごとのリスト低低ファイルシステム、S3バケット
RBACロール中中ほとんどのWebアプリ
ABAC属性ポリシー高高複雑な組織・マルチテナント
OPAコードとしてのポリシー最高高マイクロサービス、K8s Admission
5 TLS / HTTPS

転送中のデータを暗号化するTLS。1.3ではラウンドトリップが削減され、より高速・安全になった。

🤝 TLS 1.3 ハンドシェイクフロー

📜 証明書チェーン

Root CA（信頼のアンカー） └─ Intermediate CA（中間CA） └─ End Entity 証明書 （example.com のサーバー証明書） 検証: End Entity → Intermediate → Root CA ブラウザはRoot CAを事前にインストール済み Intermediate CAをバンドルして送信することが重要

🔄 mTLS（相互TLS認証）

サーバーだけでなくクライアントも証明書を提示。サービスメッシュ（Istio）やゼロトラストでサービス間認証に使用。

# Nginx: クライアント証明書要求 ssl_verify_client on; ssl_client_certificate /etc/ssl/ca.crt;

🔒 HSTS と Certificate Pinning

HTTP Strict Transport Security (HSTS)

ブラウザに「このドメインは常にHTTPS」を記憶させる。初回HTTPアクセスのSSLストリッピング攻撃を防ぐ。preloadでブラウザの事前インストールリストに登録。

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

Certificate Pinning

モバイルアプリなどで特定の証明書/公開鍵のみを信頼する。中間者攻撃に対して強力だが、証明書更新時にアプリ更新が必要というトレードオフがある。

6 暗号化（Encryption）

転送中・保管時・エンドツーエンドの3つの暗号化。パスワードの安全な保存と鍵管理も重要。

🚀 転送中の暗号化

TLS 1.3 による HTTPS。内部サービス間通信にも mTLS を使用（ゼロトラスト）。データベース接続・メッセージキューも TLS 必須。

💾 保管時の暗号化

AES-256-GCM が標準。DBの透過的暗号化（TDE）、ディスク暗号化（BitLocker/LUKS）。クラウドではKMSで鍵管理し、BYOK（持ち込み鍵）も可能。

🔐 E2E暗号化

サーバーでも復号不可。クライアント側で公開鍵暗号化（例：Signal Protocol）。医療・法律など高秘匿要件で使用。サーバーは暗号文のみを保存。

🔑 パスワードの安全な保存

⚠️ MD5/SHA-1/SHA-256 でのパスワードハッシュは危険 — 高速ハッシュはGPUで総当たり攻撃可能。パスワードには必ず遅い（コスト設定可能な）ハッシュ関数を使う。

// bcrypt（推奨、cost factor=12） const hash = await bcrypt.hash(password, 12); // $2b$12$... コスト12 = 約300ms/ハッシュ（意図的に遅い） // Argon2id（最新推奨 - メモリハード） argon2id( password, salt, memory=64MB, // メモリコスト（GPU攻撃を困難に） iterations=3, parallelism=4 ) // scrypt（メモリハード） scrypt(password, salt, N=32768, r=8, p=1)
アルゴリズム特性推奨コスト評価
bcrypt時間コスト（CPU）cost=12〜14推奨
argon2id時間+メモリコストmem=64MB, iter=3最推奨
scrypt時間+メモリコストN=32768良好
PBKDF2時間コストのみiter=600,000許容（FIPS環境）
MD5/SHA-1高速ハッシュ-使用禁止

🗝️ 鍵管理（Key Management Service）

AWS KMS

  • CMK（Customer Managed Key）でデータ暗号化鍵(DEK)を暗号化
  • エンベロープ暗号化: DEK→データ暗号化、CMK→DEK暗号化
  • KMSにマスター鍵は保存されない（HSMで保護）
  • 自動ローテーション（1年ごと）

HashiCorp Vault

  • シークレット（DBパスワード・APIキー）の一元管理
  • Dynamic Secrets: アクセスごとに短命な認証情報を発行
  • Transit Engine: 暗号化をAPIとして提供（「暗号化サービス」）
  • Seal/Unseal: Shamir秘密分散で起動時の鍵シェアを分散
7 レートリミットと DDoS 防御

APIの乱用防止・DoS対策としてのレートリミット実装と、DDoS攻撃の種類・防御方法を理解する。

レートリミット 4アルゴリズム比較

// Token Bucket（Redisで実装） async function rateLimit(userId: string) { const key = `rate:${userId}`; const pipe = redis.multi(); // Tokenを追加（最大100まで） pipe.set(key, MAX_TOKENS, 'NX', 'EX', 60); pipe.decrby(key, 1); pipe.expire(key, 60); const [, remaining] = await pipe.exec(); if (remaining < 0) { throw new Error('Rate limit exceeded'); } return remaining; }
// Sliding Window（Redis ZSETで実装） async function slidingWindowLimit(userId) { const now = Date.now(); const windowMs = 60_000; const key = `sw:${userId}`; await redis.multi() .zremrangebyscore(key, 0, now - windowMs) .zadd(key, now, `${now}`) .expire(key, 60) .exec(); const count = await redis.zcard(key); if (count > MAX_REQUESTS) throw new Error('429 Too Many Requests'); }

🌊 DDoS 攻撃の種類と防御

種類攻撃方法防御手段
Volumetric（帯域埋め）大量のパケットで回線を消費（UDP Flood, ICMP Flood）Cloudflare/AWS Shield のアップストリーム吸収、Anycast拡散
Protocol（プロトコル）SYN Flood（半開きコネクションを大量生成）SYN Cookie、Connection Rate Limiting
Application Layer（L7）HTTP GET/POST Flood、SlowLoris（低速接続）WAF、CAPTCHA、行動分析、レートリミット

💬 面接でよく聞かれること

  • 「APIレートリミットをどう実装する？」→ Token Bucket + Redis、ユーザーIDとIPの両方でキー設定
  • 「Fixed Windowの問題点は？」→ 窓境界のバースト（59秒に100件+1秒後に100件=実質2倍）
  • 「DDoS対策を設計に組み込むには？」→ CDN/WAFをエッジに置き、オリジンのIP非公開化
8 OWASP Top 10 対策

Webアプリケーションの代表的な脆弱性と具体的な対策コード。

💉 A03: SQLインジェクション → Prepared Statement

❌ 脆弱なコード

// 文字列結合でSQL構築 → 危険！ const query = `SELECT * FROM users WHERE name='${username}'`; // username = "'; DROP TABLE users;--" // → テーブル削除が実行される！

✅ 安全なコード

// Prepared Statement（パラメータ化クエリ） const query = 'SELECT * FROM users WHERE name=?'; db.query(query, [username]); // または ORM（Prisma/TypeORM）を使用 // 入力は常にエスケープされる

⚡ A03: XSS（Cross-Site Scripting）→ エスケープ + CSP

// 出力エスケープ（Stored XSS対策） function escapeHtml(str: string) { return str .replace(/&/g, '&') .replace(/'<') .replace(/>/g, '>') .replace(/"/g, '"'); } // ReactはデフォルトでJSXをエスケープ // dangerouslySetInnerHTMLは使わない
// CSP ヘッダ（inline script禁止） Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-RANDOM'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; object-src 'none'; base-uri 'self'; // nonce: リクエストごとにランダム生成 // inline scriptにも nonce を付与

9 セキュリティヘッダー

HTTP レスポンスヘッダーでブラウザに指示を出し、XSS・クリックジャッキング・情報漏洩を防ぐ。

Content-Security-Policy

リソースの読み込み元を制限。XSSの実害を防ぐ最強の武器。script-src でインラインスクリプトを禁止。

Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{random}'; object-src 'none'

X-Content-Type-Options

ブラウザのMIMEスニッフィングを禁止。Content-Typeを信頼させることで、偽ファイルの実行を防ぐ。

X-Content-Type-Options: nosniff

X-Frame-Options

クリックジャッキング防止。iframeでサイトを埋め込むことを禁止。CSPの frame-ancestors で代替可能。

X-Frame-Options: DENY # または CSP で Content-Security-Policy: frame-ancestors 'none'

Strict-Transport-Security

HTTPSを強制。max-ageの間ブラウザがHTTPSを記憶。プリロードリストへの登録で初回アクセスも保護。

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

Referrer-Policy

外部サイトへ遷移時にReferrerヘッダをどこまで送るか制御。URLにトークンが含まれる場合に重要。

Referrer-Policy: strict-origin-when-cross-origin

Permissions-Policy

カメラ・マイク・位置情報など特定のブラウザ機能を制限。不要な機能へのアクセスを禁止する。

Permissions-Policy: camera=(), microphone=(), geolocation=(self)

⚙️ Nginx セキュリティヘッダ設定例

# nginx.conf - セキュリティヘッダ一括設定 server { listen 443 ssl http2; server_name example.com; # TLS設定 ssl_protocols TLSv1.2 TLSv1.3; ssl_prefer_server_ciphers on; ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384'; ssl_session_timeout 1d; ssl_session_cache shared:SSL:50m; # セキュリティヘッダ add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always; add_header X-Frame-Options DENY always; add_header X-Content-Type-Options nosniff always; add_header Referrer-Policy "strict-origin-when-cross-origin" always; add_header Permissions-Policy "camera=(), microphone=(), payment=(self)" always; add_header Content-Security-Policy "default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self'" always; # サーバー情報の隠蔽 server_tokens off; more_clear_headers Server; }
10 ゼロトラストアーキテクチャ

「信頼しない、常に検証する」という原則に基づく現代のセキュリティアーキテクチャ。VPN依存の境界型セキュリティからの移行。

❌ 境界型セキュリティ（従来型）の限界

  • 「社内ネットワークにいれば信頼できる」という前提
  • VPN接続後は横移動（Lateral Movement）が自由
  • クラウド・リモートワーク・SaaSで境界が無意味に
  • 内部犯行やVPNクレデンシャル漏洩に無力
  • すべての通信をVPNに集中させることでボトルネック

✅ ゼロトラストの原則

  • Never Trust, Always Verify: ネットワーク位置で信頼しない
  • Verify Explicitly: すべてのアクセスをID・デバイス・コンテキストで検証
  • Least Privilege Access: Just-in-Time/Just-Enough-Accessを実装
  • Assume Breach: 侵害を前提に設計、影響範囲を最小化
  • アイデンティティが新たな「境界」となる

ゼロトラストアーキテクチャ概念図

🚀 ゼロトラスト実装 5ステップ

  1. アイデンティティの確立（Identity as Perimeter） MFAを全ユーザーに強制。SSO/IdP（Okta, Azure AD）で一元管理。条件付きアクセスポリシーを設定。
  2. デバイスの信頼確認（Device Trust） MDM（Mobile Device Management）でデバイス状態を管理。コンプライアンス未達デバイスはアクセス拒否（OSパッチ・ディスク暗号化の確認）。
  3. マイクロセグメンテーション（Micro-Segmentation） ネットワークを細分化し、サービス間通信を最小権限に。East-West トラフィックを mTLS で保護。不要な横移動を防ぐ。
  4. 最小権限アクセス（Just-in-Time Access） JIT（Just-in-Time）アクセス：必要な時だけ必要なリソースへの権限を一時的に付与。特権アクセス管理（PAM）ツールで実装。
  5. 継続的な監視と検証（Continuous Monitoring） 全アクセスログを収集・分析。UEBA（User and Entity Behavior Analytics）で異常を検知。セッション中も継続的にリスクを再評価。

🏢 BeyondCorp（Googleのゼロトラスト実装）

Googleが2014年に発表した世界最大規模のゼロトラスト実装。VPNを廃止し、すべてのアクセスをデバイスとIDで検証。

主要コンポーネント

  • Device Inventory DB: 全デバイスの信頼レベルを管理
  • Access Proxy: すべてのリクエストを仲介・検証
  • Trust Inferer: デバイス証明書+IdPでトラストスコア算出
  • Resource Tier: アプリの機密度に応じたアクセス要件

現代の商用実装

  • Google BeyondCorp Enterprise: GCPネイティブ

A 深掘り: JWT の脆弱性と対策

JWTは正しく実装しないと重大な脆弱性につながる。代表的な攻撃パターンと対策を詳しく解説する。

alg:none 攻撃

JWTヘッダの alg を "none" に書き換えると、一部のライブラリは署名を検証せずにpayloadを信頼してしまう。 攻撃者は任意のユーザーIDや権限を設定したJWTを作成できる。

# 攻撃者が細工するJWTヘッダ # 元のヘッダ: {"alg":"HS256","typ":"JWT"} # 攻撃用: {"alg":"none","typ":"JWT"} # base64url({"alg":"none","typ":"JWT"}).base64url({"sub":"admin","role":"superuser"}). # 末尾の署名部分を省略 → 脆弱なライブラリは検証をスキップ import jwt # 対策: allowedアルゴリズムを明示的に制限 def verify_token(token: str, secret: str) -> dict: try: payload = jwt.decode( token, secret, algorithms=["HS256"], # "none" を許可しない options={"verify_exp": True} ) return payload except jwt.InvalidAlgorithmError: raise ValueError("不正なアルゴリズム")

対策: 検証時にアルゴリズムをハードコードし、トークンヘッダのalgを信用しない。algorithms=["HS256"]のように明示的に許可リストを指定する。

RS256 vs HS256 の使い分け

項目HS256RS256
鍵の種類共有秘密鍵（対称）RSA公開鍵/秘密鍵（非対称）
署名同じ鍵で署名・検証秘密鍵で署名、公開鍵で検証
鍵の配布すべてのサービスに秘密鍵を共有公開鍵だけ配布すれば検証可能
マイクロサービス向き低（秘密鍵漏洩リスク）高（認証サーバーだけが署名）
推奨用途シングルサービス、テストマイクロサービス、本番環境

推奨: マイクロサービスではRS256を採用。各サービスはJWKS（JSON Web Key Set）エンドポイントから公開鍵を取得して検証する。

JKU / X5U Header Injection

JWTヘッダに jku（JSON Web Key URL）を含め、攻撃者のサーバーが提供する公開鍵で検証させる攻撃。

# 攻撃用JWTヘッダ { "alg": "RS256", "typ": "JWT", "jku": "https://attacker.com/evil.json" } # → 攻撃者のサーバーから公開鍵を取得して # → 攻撃者の秘密鍵で署名されたJWTが有効になる

対策: jku/x5uヘッダは使用しない。公開鍵はコード内にハードコードするか、信頼済みJWKSエンドポイントURLをホワイトリストで管理する。

Short-lived AT + Long-lived RT のリフレッシュフロー実装

アクセストークン（AT）は短命（15分〜1時間）、リフレッシュトークン（RT）は長命（7〜30日）にし、RTでATを再発行する。

import jwt, secrets, redis from datetime import datetime, timedelta REDIS = redis.Redis(host='localhost') SECRET = "your-256-bit-secret" def issue_tokens(user_id: str) -> dict: # アクセストークン: 15分 at_payload = { "sub": user_id, "exp": datetime.utcnow() + timedelta(minutes=15), "type": "access" } access_token = jwt.encode(at_payload, SECRET, algorithm="HS256") # リフレッシュトークン: 30日（Redisで管理） refresh_token = secrets.token_urlsafe(32) REDIS.setex( f"rt:{refresh_token}", timedelta(days=30), user_id ) return {"access_token": access_token, "refresh_token": refresh_token} def refresh_access_token(refresh_token: str) -> str: user_id = REDIS.get(f"rt:{refresh_token}") if not user_id: raise ValueError("無効なリフレッシュトークン") # Rotation: 旧RTを削除 → 新RTを発行（リプレイ攻撃防止） REDIS.delete(f"rt:{refresh_token}") return issue_tokens(user_id.decode()) def revoke_token(refresh_token: str): # ログアウト時: RTをRedisから削除 REDIS.delete(f"rt:{refresh_token}") # アクセストークンの即時失効はBlacklistが必要 # REDIS.setex(f"blacklist:{jti}", timedelta(minutes=15), "revoked")

Blacklist vs Stateless失効: JWTは完全ステートレスのため、署名が有効なら検証が通る。即時失効が必要な場合はRedisのBlacklist（blacklist:{jti}）を使い、ATの有効期限まで失効済みjtiを保持する。ATを15分以内に抑えることでBlacklistのサイズを最小化できる。

B 深掘り: OAuth 2.0 の脆弱性と対策

State Parameter による CSRF 防止

# OAuth認可リクエスト生成 import secrets, hashlib def start_oauth(session): state = secrets.token_urlsafe(32) session["oauth_state"] = state # セッションに保存 auth_url = ( "https://auth.example.com/authorize" f"?response_type=code&client_id=CLIENT_ID" f"&redirect_uri=https://app.example.com/callback" f"&state={state}" ) return auth_url def handle_callback(session, request_params): # コールバックでstate検証 if request_params["state"] != session.get("oauth_state"): raise ValueError("CSRF攻撃の可能性") # stateは一度だけ使用 session.pop("oauth_state")

Redirect URI Validation（オープンリダイレクト脆弱性）

攻撃者が redirect_uri=https://attacker.com を指定すると、認可コードが攻撃者に送られる。

# 不十分な検証（前方一致は危険） # NG: redirect_uri.startswith("https://app.example.com") # → https://app.example.com.evil.com も通過してしまう # 正しい検証: 完全一致ホワイトリスト ALLOWED_REDIRECT_URIS = { "https://app.example.com/callback", "https://app.example.com/mobile/callback", } def validate_redirect_uri(uri: str) -> bool: return uri in ALLOWED_REDIRECT_URIS

対策: redirect_uriは事前登録し、完全一致で検証する。ワイルドカードやパターンマッチは使わない。

PKCE の実装（Proof Key for Code Exchange）

SPAやモバイルアプリはclient_secretを安全に保管できないため、PKCEで認可コード横取り攻撃を防ぐ。

import secrets, hashlib, base64 def generate_pkce_pair() -> tuple[str, str]: # code_verifier: 43〜128文字のランダム文字列 code_verifier = secrets.token_urlsafe(96) # 128バイト # code_challenge = BASE64URL(SHA256(code_verifier)) digest = hashlib.sha256(code_verifier.encode()).digest() code_challenge = base64.urlsafe_b64encode(digest).rstrip(b'=').decode() return code_verifier, code_challenge # 認可リクエスト code_verifier, code_challenge = generate_pkce_pair() auth_url = ( "https://auth.example.com/authorize" f"?code_challenge={code_challenge}" f"&code_challenge_method=S256" # ... 他のパラメータ ) # トークンリクエスト（code_verifierを送信） token_request = { "grant_type": "authorization_code", "code": "...認可コード...", "code_verifier": code_verifier, # サーバーはSHA256して検証 "redirect_uri": "https://app.example.com/callback" }

C 深掘り: 暗号化の実践

AES-GCM vs AES-CBC: 認証暗号の重要性

AES-CBCは暗号化のみで完全性を保証しない。Padding Oracle攻撃で平文を復号できる。

# AES-GCMの実装（認証付き暗号） from cryptography.hazmat.primitives.ciphers.aead import AESGCM import os def encrypt_aes_gcm(plaintext: bytes, key: bytes) -> bytes: aesgcm = AESGCM(key) nonce = os.urandom(12) # 96ビット推奨 ciphertext = aesgcm.encrypt(nonce, plaintext, None) return nonce + ciphertext # nonce + ciphertext + 16バイトタグ def decrypt_aes_gcm(data: bytes, key: bytes) -> bytes: aesgcm = AESGCM(key) nonce, ciphertext = data[:12], data[12:] return aesgcm.decrypt(nonce, ciphertext, None) # タグ検証失敗時: InvalidTag例外 → 改ざん検知

GCMの利点: 暗号化と同時に認証タグ（HMAC相当）を生成。復号時に改ざんを自動検知。CBCのように別途HMACが不要。

Argon2id のパラメーター選定

OWASP推奨: argon2id（メモリハード関数）でブルートフォースを計算コスト的に困難にする。

from argon2 import PasswordHasher from argon2.exceptions import VerifyMismatchError # OWASP推奨パラメーター (2023) ph = PasswordHasher( time_cost=3, # 反復回数（最低2） memory_cost=65536, # 64 MiB（最低19 MiB） parallelism=4, # 並列度（CPUコア数に合わせる） hash_len=32, # 出力ハッシュ長 salt_len=16 # ソルト長 ) def hash_password(password: str) -> str: return ph.hash(password) # ソルト自動生成 def verify_password(hashed: str, password: str) -> bool: try: return ph.verify(hashed, password) except VerifyMismatchError: return False

Envelope Encryption: AWS KMS による2層暗号化

データ暗号化鍵（DEK）をデータ鍵暗号化鍵（KEK）で暗号化する2層構造。大量データを効率的に暗号化しつつ、鍵管理を一元化できる。

import boto3, os from cryptography.hazmat.primitives.ciphers.aead import AESGCM kms = boto3.client('kms', region_name='ap-northeast-1') CMK_KEY_ID = "arn:aws:kms:ap-northeast-1:123456789012:key/xxx" def encrypt_data(plaintext: bytes) -> dict: # 1. KMSからDEK（データ暗号化鍵）を生成 # PlaintextDEK: 平文DEK（メモリ内のみ使用） # CiphertextBlob: CMKで暗号化されたDEK（DBに保存） response = kms.generate_data_key( KeyId=CMK_KEY_ID, KeySpec='AES_256' ) dek_plaintext = response['Plaintext'] # メモリ内のみ dek_encrypted = response['CiphertextBlob'] # DBに保存 # 2. 平文DEKでデータを暗号化 aesgcm = AESGCM(dek_plaintext) nonce = os.urandom(12) ciphertext = aesgcm.encrypt(nonce, plaintext, None) # 3. 平文DEKはメモリから破棄 return { "encrypted_dek": dek_encrypted, # DBに保存 "nonce": nonce, "ciphertext": ciphertext } def decrypt_data(encrypted_dek: bytes, nonce: bytes, ciphertext: bytes) -> bytes: # 1. KMSでDEKを復号（KMSへのAPI呼び出し、CloudTrailに記録） response = kms.decrypt(CiphertextBlob=encrypted_dek) dek_plaintext = response['Plaintext'] # 2. DEKでデータを復号 aesgcm = AESGCM(dek_plaintext) return aesgcm.decrypt(nonce, ciphertext, None)

なぜEnvelope Encryptionが重要か

KMSに直接大量データを送ると遅い・コスト高い。DEKをローカルで生成してデータを暗号化し、DEK自体だけKMSで暗号化することで、KMS API呼び出しをデータ単位に1回に抑えられる。鍵のローテーションはDEKの再暗号化のみでデータ本体を再暗号化する必要がない。

演習2: OWASPに基づいたコードレビュー（4問）

問1: SQLインジェクション（脆弱なコード）

修正後（プリペアドステートメント）

# 安全: パラメータを分離 def get_user(username): query = "SELECT * FROM users WHERE username = %s" cursor.execute(query, (username,)) return cursor.fetchone()

// 危険: ユーザー入力をそのままHTML挿入 function showComment(comment) { document.getElementById('output') .innerHTML = comment; // XSS! }

// 安全: テキストとして扱う function showComment(comment) { document.getElementById('output') .textContent = comment; // エスケープ // HTMLが必要な場合: DOMPurify.sanitize(comment) }

問3: IDOR（脆弱なコード）

# 安全: 自分のリソースのみ返す def get_order(request, order_id): order = Order.objects.get( id=order_id, user_id=request.user.id # 所有者確認 ) return JsonResponse(order.to_dict())
